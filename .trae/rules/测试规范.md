# Bevy SRPG 测试宪法 v4.0
> **最高约束力**：优先级高于任何通用测试实践，与《SRPG 项目总宪法 v5.0》第十二编配套执行。违反即视为不合格输出。

## 核心原则（宪法 12.2 🟥）

> **测试跟领域走（Feature First），但不写在源码文件内部。**

- 🟥 禁止 `#[cfg(test)] mod tests` 内联测试（对AI上下文污染严重）
- 🟥 禁止将所有测试平铺到根 `tests/unit/`（后期变成大杂烩）
- 🟩 测试与被测领域同目录放置，形成 Feature Folder 结构
- 🟩 根 `tests/` 仅保留跨领域测试（战斗流程、存档、回归、E2E）

## 核心规则优先级（绝对不可违反）
`docs/02-domain/` > `docs/01-architecture/README.md` > test_spec.md

## 测试哲学
测试验证**行为**，不验证实现；关注**What**，不关注How。

## 测试分类与优先级（领域内聚四层）

```
<domain>/
├── tests/
│   ├── unit/          # 单元测试：验证领域纯函数、核心规则
│   ├── integration/   # 集成测试：验证领域内多组件协作
│   ├── invariant/     # 不变量测试：验证领域不变量（**最高价值**）
│   └── fixtures/      # 测试数据（Builder模式 / RON文件）
```

| 层 | 名称 | 职责 | 示例 |
|------|------|------|------|
| **unit** | 单元测试 | 验证单个函数/纯规则的正确性 | HP计算、Tag包含检查、Modifier优先级 |
| **integration** | 集成测试 | 验证领域内多组件协作 | 装备穿戴→Modifier→Attribute联动 |
| **invariant** | 不变量测试 | 验证领域不变量（**最高价值**） | Tag bit唯一、Buff不重复叠加、HP>=0 |
| **fixtures** | 测试数据 | Builder模式构造的测试数据 | RON格式角色模板、技能配置 |

### 不变量测试（最重要）

SRPG核心架构有大量领域不变量：

| 不变量 | 说明 |
|--------|------|
| Tag bit唯一 | 同一Tag不能在位掩码中重复设置 |
| Buff不重复叠加 | 同源同类型Buff不会无限堆叠 |
| Effect不修改不存在属性 | Effect引用的AttributeId必须已注册 |
| HP永远>=0 | HP计算结果不能为负 |
| Modifier不改变基础值 | Modifier只影响聚合后的当前值 |
| 回合先攻排序稳定 | 同先攻值的单位顺序确定 |
| 技能消耗原子性 | 消耗失败时不产生部分效果 |

> 不变量测试的价值远大于普通单元测试，是架构稳定性的最后防线。

### 跨领域测试（根 tests/）

```
tests/
├── battle_flow/     # 完整战斗流程
├── save_load/       # 存档/读档完整性
├── regression/      # 回归测试（历史Bug复现）
├── replay/          # 回放确定性
├── golden/          # 金文件对比
├── simulation/      # 战斗模拟与数值平衡
├── performance/     # 性能回归
└── e2e/             # 端到端测试
```

## 强制要求
- 所有测试必须确定性，随机数固定Seed=42
- 测试用例必须遵循：Test ID → Title → Given → When → Then → Assertions
- 必须使用标准测试单位：Unit_001(战士100/30/10/10)、Unit_002(法师80/40/5/12)、Unit_003(坦克150/20/20/5)
- 覆盖目标：100%核心领域规则，追求规则覆盖而非代码覆盖
- 发现Bug必须先写失败测试，再修复代码
- 每个核心领域必须有invariant层（不变量测试）

### ID 体系测试（详见 `docs/04-data/foundation/id-taxonomy.md`）

| 不变量 | 说明 |
|--------|------|
| ID 唯一性 | 同一 Runtime ID 不会分配给两个不同对象 |
| Generation 保护 | 回收后的 ID 再次分配时 generation 递增 |
| Entity 映射一致 | Entity ↔ 业务 ID 双向映射始终同步 |
| 序列化往返 | ID 序列化后反序列化结果一致 |
| 跨层隔离 | Domain 层代码中无裸 Entity/u64 |

## AI决策规则（测试失败时严格执行）
1. 先查 `docs/02-domain/` 领域规则
2. 再查 `docs/01-architecture/README.md`
3. 最后检查测试本身
- 测试违反领域规则→改测试；代码违反领域规则→改代码；都符合→更新领域规则
- 禁止为过测试改业务逻辑，禁止为过代码改测试

## AI约束
禁止测试私有实现、内部缓存、System顺序、组件布局。

## 最终原则
Domain Rules First. Tests Second. Implementation Third.

## 架构适应度测试
- Fitness Function 是不变量测试的架构级泛化
- 所有 PR 必须通过 `tools/check-architecture-budget.sh --ci`
- 复杂计算的 Explain 输出应作为测试断言的一部分
