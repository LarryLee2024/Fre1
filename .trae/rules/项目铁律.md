# 一、架构铁律（最高优先级）
- **Feature First**：按业务拆模块，不按技术拆模块；例如 `character/skill/battle/map/ui` 优于 `components/systems/events`。
- **Definition / Instance 分离**：配置（职业、技能、装备）永远独立于运行时状态（HP、Buff、CD）。
- **Rule / Content 分离**：代码负责规则，配置负责内容；新增职业、技能、装备优先修改配置而不是代码。
- **Logic / Presentation 分离**：逻辑产生结果，表现负责展示；伤害计算不能依赖UI和特效。
- **数据驱动优于硬编码**：新增内容尽量不修改逻辑代码。
- **组合优于继承**：角色由 Trait、Modifier、能力组合构成，而不是继承树。
- **架构优先解决当前复杂度**：不要为了未来可能出现的问题增加今天的维护成本。
- **优先使用官方能力，谨慎自研基础设施**：能用 Bevy 原生能力解决的问题不要重复造轮子。

# 二、ECS铁律
- **Entity只是ID**：不要把 Entity 当对象使用。
- **Component存状态，System存行为**：数据与逻辑保持分离。
- **Hook负责组件固有行为**：例如 `Dead` 添加时自动移除移动组件。
- **Observer负责局部响应**：例如角色死亡后播放动画、刷新UI、更新任务。
- **Message负责跨Feature广播**：例如回合结束、战斗结束、任务完成。
- **模块内部优先函数调用**：不要把所有逻辑都事件化。
- **ECS是数据流，不是调用链**：不要模拟 `player.attack(enemy)`。
- **Tag Component优于bool**：`Dead` 优于 `is_dead`。
- **Required Components优先表达依赖关系**：不要手动补组件。
- **Added / Changed / Removed 优先于手写状态标记**。
- **Resource不是全局变量仓库**：只保存真正的全局状态。
- **高频逻辑避免Observer风暴**：频繁变化优先直接System处理。

# 三、模块化铁律
- **模块只暴露公共接口**：内部实现默认私有。
- **模块边界比目录结构更重要**：目录能调整，耦合难清理。
- **跨模块不要访问内部细节**：通过 Message、Observer、Command 通信。
- **Plugin职责过大就拆分**：按业务拆而不是按数量拆。
- **路径表达业务含义**：例如 `battle/damage/`、`inventory/equipment/`。

# 四、代码组织铁律
- **一个文件一个主题**。
- **优先按业务拆文件，不按代码类型拆文件**。
- **500~1000行开始警觉**。
- **函数超过100行开始警觉**。
- **不要存在 systems.rs 巨文件**。
- **不要存在 components.rs 巨文件**。
- **不要存在 utils.rs 垃圾桶文件**。
- **内部辅助函数使用 `_` 前缀仅在模块内有明确价值时使用**；Rust更推荐通过私有可见性表达边界，而不是依赖命名约定。

# 五、函数设计铁律
- **函数像目录，不像小说**：例如 `validate → calculate → apply`。
- **一个函数一个主要职责**。
- **函数名描述意图，不描述过程**。
- **优先 Early Return**。
- **超过3层嵌套需要重构**。
- **重复三次以上再抽象**。
- **先保证可读性，再考虑复用性**。

# 六、Trait铁律
- **Trait表示能力，不表示分类**。
- **Trait用于扩展点，不用于模拟继承树**。
- **Trait只解决变化点**。
- **重复出现三次以上再抽象Trait**。
- **不要为了优雅创造Trait**。

# 七、SRPG角色铁律
- **种族 = Trait + Modifier集合**。
- **职业 = 成长率 + 技能池 + Trait集合**。
- **天赋 = 特殊Trait**。
- **装备 = Modifier + Trait**。
- **Buff/Debuff = 临时Trait**。
- **状态效果统一走同一套扩展机制**。
- **所有能力来源统一进入Modifier管线**。
- **EntityEvent适合复杂战斗链**：例如装备→护盾→角色→被动技能。

# 八、属性系统铁律
- **Primary Stat 与 Derived Stat 分离**。
- **Derived Stat优先实时计算**。
- **统一Modifier管线**。
- **不要到处直接修改最终属性**。
- **最终属性来源统一**。
- **属性公式集中管理**。
- **配置型数据尽量保持不可变**。

# 九、地图系统铁律
- **地图优先看成Grid数据**。
- **Tile是否Entity由需求决定**。
- **Chunk按需引入**。
- **OccupancyGrid独立存在**。
- **地图数据与渲染分离**。
- **寻路数据运行时生成**。
- **地图逻辑层独立于渲染层**。

# 十、UI铁律
- **产品UI优先 bevy_ui**。
- **开发工具优先 bevy_egui**。
- **UI只展示状态，不保存业务真相**。
- **业务逻辑不要直接操作UI**。
- **UI监听状态变化刷新自己**。
- **UI也是独立Feature模块**。
- **优先复用官方 Widget 与 UI 能力**。
- **输入焦点统一管理**。
- **拖拽系统统一实现**。
- **复杂UI必须支持调试与可视化检查**。

# 十一、数据驱动铁律
- **配置定义内容，代码解释配置**。
- **新增内容尽量不修改逻辑代码**。
- **配置结构优先稳定**。
- **配置兼容性优先于配置优雅**。
- **统一Asset Pipeline管理配置资源**。
- **配置优先支持热重载**。
- **配置引用关系自动校验**。
- **内容生产效率优先于配置格式争论**。

# 十二、Reflect铁律
- **Reflect服务工具链，不服务核心运行时逻辑**。
- **需要编辑器支持的数据优先Reflect**。
- **Reflect注册统一管理**。
- **文档注释优先作为编辑器说明来源**。
- **反射用于展示、检查、编辑，而不是计算。**

# 十三、测试铁律
- **测试优先于手工验证**。
- **发现Bug先写测试再修复**。
- **单元测试验证规则**：伤害、Buff、属性、寻路。
- **集成测试验证Feature**：装备、背包、战斗、升级。
- **场景测试验证玩家流程**：战斗回合、技能释放、胜负结算。
- **Battle Replay优先于BDD**：复杂SRPG更适合验证状态流。
- **测试覆盖核心规则而非覆盖率数字**。
- **所有Bug最终都应沉淀为测试资产**。

# 十四、可观测性铁律（日志 + 调试）
- **统一使用 tracing**。
- **日志必须结构化**。
- **日志记录状态变化，不记录函数进入退出**。
- **重要状态流必须可追踪**。
- **Error必须包含完整上下文**。
- **不要记录每帧日志**。
- **Inspector、Replay、Debug Panel 优先于日志堆砌**。
- **关键系统必须支持单步执行与状态回溯**。
- **系统执行顺序必须可观察**。
- **复杂系统必须拥有可视化观察窗口**。

# 十五、资源与内容生产铁律
- **统一Settings体系**。
- **资源加载必须可追踪**。
- **资源生命周期显式管理**。
- **字体、音频、配置统一管理**。
- **高频修改资源优先支持热重载**。
- **编辑器属于正式产品的一部分**。
- **内容生产能力决定项目上限**。
- **工具链是长期项目核心资产**。

# 十六、性能铁律
- **先正确，再优化**。
- **先Profile，再优化**。
- **性能问题必须测量**。
- **优先优化热点，而不是全局重构**。
- **Changed过滤优于全量扫描**。
- **Reflect不要参与高频计算**。
- **缓存必须定义失效条件**。
- **Feature裁剪优于无脑开启全部功能**。
- **大多数独立游戏死于复杂度，而不是性能。**

# 十七、生命周期铁律
- **OnEnter / OnExit保持轻量**。
- **重型初始化拆成加载阶段**。
- **状态切换不要隐藏副作用**。
- **状态机负责流程，不负责业务细节**。
- **初始化过程可追踪、可恢复、可中断**。

# 十八、长期维护铁律
- **代码首先写给人看，其次写给机器执行**。
- **明确优于聪明**。
- **简单优于优雅**。
- **稳定优于炫技**。
- **删代码通常比写代码更有价值**。
- **社区维护成本通常低于自维护成本**。
- **每引入一个自研系统，都要考虑未来五年的维护成本**。
- **架构必须定期复盘**。
- **工具链与内容生产能力最终决定项目成败。**

# SRPG项目最终10条核心原则（AI最高优先级）
1. **Feature First**
2. **Definition / Instance 分离**
3. **Rule / Content 分离**
4. **Logic / Presentation 分离**
5. **Hook = 固有行为，Observer = 局部响应，Message = 跨Feature广播**
6. **Trait + Modifier 统一扩展体系**
7. **数据驱动优于硬编码**
8. **小函数、小模块、小依赖**
9. **Battle Replay + 测试优先于手工验证**
10. **组合优于继承**

这一版基本已经接近我认为的 **Bevy 0.18+ SRPG AI开发宪法版**：规则数量控制在 100 条以内，覆盖架构、ECS、测试、工具链、内容生产和长期维护，同时不会因为 Bevy 0.19、0.20 的 API 变化而大面积过时。
