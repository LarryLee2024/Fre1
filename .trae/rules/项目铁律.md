如果结合 **Bevy 0.18+（Message / Observer / Hook / Required Components）**、**Rust工程实践**、**SRPG长期项目维护** 三方面，我会进一步升级成下面这个版本。

原则：

* 保留长期有效的架构思想；
* 吸收 Bevy 0.18 新特性；
* 去掉容易过时的 API 细节；
* 每条都保持「一句话原则 + 同行例子」。

---

# 架构铁律

* **业务优先于技术分类**：优先拆 `character/skill/battle/map/ui`，不要先拆 `components.rs`、`systems.rs`、`events.rs`。
* **Feature First**：一个业务功能一个模块，一个模块一个 Plugin，例如 `battle/damage/`、`battle/turn/`、`battle/targeting/`。
* **Definition 与 Instance 分离**：Definition 保存职业、技能、装备配置，Instance 保存 HP、Buff、CD 等运行时状态，不要修改职业配置实现升级。
* **Rule 与 Content 分离**：代码负责规则，配置负责内容，例如新增职业优先改配置而不是改伤害公式。
* **Logic 与 Presentation 分离**：伤害计算属于 Logic，飘字、动画属于 Presentation，不要在 `apply_damage()` 中播放特效。
* **数据驱动优先于硬编码**：新增技能优先增加配置，而不是增加 `match SkillType` 分支。
* **组合优于继承**：不要思考“骑士继承战士”，而要思考“骑士拥有防御Trait、骑乘Trait、长枪Trait”。
* **避免过早设计未来需求**：当前只有 10 个职业时，不要提前设计支持 1000 个职业的架构。
* **架构优先解决当前复杂度**：不要为了未来可能出现的问题增加今天的维护成本。

# ECS铁律

* **Entity只是ID**：Entity 本质是索引，不是对象实例，不要写 Java 风格的 EntityManager。
* **Component主要存数据，System主要存行为**：例如 `Health { current, max }` 放 Component，伤害逻辑放 System。
* **Component固有行为优先 Hook**：例如 `Dead` 添加时自动移除 `MoveTarget`，适合使用 Hook。
* **外部响应优先 Observer**：例如 `Dead` 添加后播放动画、更新任务、刷新UI，适合使用 Observer。
* **Tag Component 优于 bool**：例如 `Dead`、`Poisoned`、`Stunned` 通常优于 `is_dead`、`is_poisoned`、`is_stunned`。
* **跨模块优先 Message/Event，模块内部优先函数调用**：Battle → UI 用 Message 或 Event，DamageCalculator → CritCalculator 用普通函数即可。
* **Message负责通知，Observer负责响应**：例如“回合结束”适合 Message，“角色死亡”适合 Observer。
* **ECS是数据流，不是调用链**：不要试图模拟 OOP 的 `player.attack(enemy)`。
* **Query职责清晰优先于Query数量**：一个职责明确的 8 组件 Query 通常优于拆成 5 个相互依赖的 System。
* **System参数超过7~8个需要警觉**：通常意味着职责开始扩散。
* **能用 Added/Changed/Removed 或 Observer 就不要自己维护状态标记**：例如不要手写 `is_dirty`。
* **Resource不是全局变量仓库**：`SelectedUnit`、`HoveredTile` 合理，`EverythingManager` 不合理。
* **Required Components优先表达依赖关系**：例如 `Player` 自动要求拥有 `Health`、`Inventory`、`Faction`。

---

# 模块化铁律

* **模块只暴露公共接口**：外部只知道 `CharacterPlugin`，不知道内部实现。
* **内部实现默认私有**：能不 `pub` 就不要 `pub`。
* **不要跨模块访问内部细节**：Battle 不应该直接修改 Inventory 内部状态。
* **模块通信优先 Message、Event、Observer 或 Command**：降低耦合。
* **Plugin职责过大就拆分**：例如拆成 `DamagePlugin`、`TurnPlugin`、`SkillPlugin`，不要纠结具体数字。
* **路径表达业务含义**：例如 `character/progression/level_up.rs` 比 `systems.rs` 更容易定位问题。
* **模块边界比目录结构更重要**：目录可以调整，耦合关系很难清理。

---

# 文件拆分铁律

* **一个文件一个主题**：不要把装备、Buff、升级逻辑塞进同一个文件。
* **500~1000行开始警觉**：不是必须拆分，但必须思考为什么这么大。
* **函数超过100行开始警觉**：通常说明职责过多。
* **不要有 systems.rs 巨文件**：最终一定会变成垃圾场。
* **不要有 components.rs 巨文件**：100多个组件塞一起后无法维护。
* **不要有 utils.rs 垃圾桶文件**：看到 `utils.rs` 超过 500 行通常说明缺少领域划分。
* **优先按业务拆文件，不按代码类型拆文件**：例如 `inventory/equipment.rs` 优于 `systems/equipment_system.rs`。

---

# 函数设计铁律

* **函数像目录，不像小说**：例如 `validate_attack()` → `calculate_damage()` → `apply_damage()`。
* **一个函数一个主要职责**：升级函数不要同时负责升级、回血、学技能、播动画。
* **函数名描述意图，不描述过程**：好例子 `apply_damage()`；坏例子 `calculate_and_apply_damage_and_notify_everything()`。
* **优先 Early Return**：减少嵌套层级。
* **超过3层嵌套需要警觉**：通常可以提取函数或提前返回。
* **重复三次以上再抽象**：避免过早抽象。
* **先写清晰代码，再考虑复用代码**：可读性优先于抽象层数。

---

# Trait系统铁律

* **Trait表示能力，不表示分类**：好例子 `DamageSource`、`Healable`；坏例子 `Character`、`Monster`、`Boss`。
* **Trait用于扩展点，不用于模拟继承树**：Rust Trait 不是 Java Class。
* **重复三次再抽象Trait**：不要项目第一天就造 30 个 Trait。
* **不要为了优雅创造Trait**：很多 Trait 最终只会有一个实现。
* **Trait解决变化点，不解决所有问题**：优先解决扩展需求，而不是追求设计模式。

---

# SRPG角色系统铁律

* **种族本质是 Trait + Modifier 集合**：例如精灵是“夜视+高感知”，而不是特殊代码分支。
* **职业本质是成长率 + 技能池 + Trait 集合**：职业不应该散落在几十个 if/else 中。
* **天赋本质是特殊Trait**：不要单独搞一套完全独立的天赋系统。
* **装备本质是 Modifier + Trait**：装备提供能力，而不是写死逻辑。
* **Buff/Debuff本质是临时Trait**：统一进入同一个扩展体系。
* **状态效果统一走同一套机制**：不要中毒一套、燃烧一套、冻结一套。
* **不要让每个系统维护自己的属性加成逻辑**：最终一定失控。
* **EntityEvent适合复杂战斗链**：例如装备→护盾→角色→被动技能的伤害传播链。

---

# 属性系统铁律

* **Primary Stat 与 Derived Stat 分离**：力量、敏捷属于 Primary，攻击、防御属于 Derived。
* **Derived Stat优先实时计算**：不要同时维护 `attack` 和 `base_attack + buffs` 两份数据。
* **统一 Modifier 管线**：装备、Buff、职业加成都走同一套流程。
* **不要到处 `attack += 10`**：统一通过 Modifier 系统修改。
* **最终属性来源统一**：不要 UI 一套算法、战斗一套算法。
* **属性公式集中管理**：不要散落在十几个文件。
* **不可变数据尽量声明不可变**：例如配置ID、职业ID、种族ID 不应频繁修改。

---

# 地图系统铁律

* **地图优先看成 Grid 数据而不是 Entity 集合**：SRPG 通常更适合 Grid 思维。
* **Tile是否做Entity取决于需求**：静态SRPG地图通常用 Grid，大量动态交互地图可以考虑 Tile Entity。
* **Chunk按需引入**：64×64 以下地图通常没必要提前做 Chunk。
* **OccupancyGrid独立存在**：寻路、攻击范围、AI 都会依赖它。
* **地图数据与地图渲染分离**：不要通过 Sprite 状态反推地图状态。
* **寻路数据运行时生成**：不要存储大量冗余导航信息。
* **地图逻辑层优先独立于渲染层**：未来切换 TileMap 插件成本更低。

---

# UI铁律

* **最终产品UI优先 bevy_ui**：官方生态整合度最高。
* **开发工具优先 bevy_egui**：编辑器、调试器、GM工具非常适合。
* **根据场景选择UI方案**：推荐不等于强制。
* **业务逻辑不要直接操作UI**：逻辑发消息，UI自行响应。
* **UI监听状态变化刷新自己**：不要每个系统都知道UI细节。
* **UI也是 Feature 模块**：例如 `CharacterPanel`、`InventoryPanel`、`SkillPanel`。
* **封装自己的 Widget 库**：例如 `VBox`、`HBox`、`Panel`、`Window`、`ScrollView`。
* **不要满项目手写 Node 树**：后期维护极其痛苦。

---

# 数据驱动铁律

* **配置定义内容，代码解释配置**：代码负责规则引擎。
* **职业数值不要写在代码里**：职业配置化。
* **技能倍率不要写在代码里**：技能配置化。
* **装备属性不要写在代码里**：装备配置化。
* **新增内容尽量不修改逻辑代码**：新增火球术不应该修改 DamageSystem。
* **数据结构优先稳定**：配置格式不要频繁推翻重做。
* **配置兼容性比配置优雅更重要**：长期项目迟早面临存档兼容问题。

---

# 调试铁律

* **日志优于猜测**：先看日志再猜问题。
* **消息链路必须可追踪**：关键 Message、Event、Observer 最好能打印链路。
* **关键状态必须可观察**：不要出现只能靠断点看的状态。
* **开发期保留 Inspector**：极大提高调试效率。
* **不要调试黑盒系统**：核心流程必须可视化。
* **复杂系统必须具备调试入口**：例如战斗回放、属性面板、事件追踪器。

---

# 性能铁律

* **先正确，再优化**：不要为了未来性能把代码写复杂。
* **先Profile，再优化**：不要凭感觉优化。
* **不要为不存在的问题设计架构**：例如小地图先做 Chunk。
* **性能问题必须测量**：不要“感觉这里慢”。
* **大部分独立游戏死于复杂度而不是性能**：维护成本往往比 CPU 更可怕。
* **优先优化热点而不是全局重构**：先找到瓶颈再动刀。

---

# 生命周期铁律

* **OnEnter/OnExit保持轻量**：不要在里面加载地图、生成全部角色、构建所有UI。
* **重型初始化拆成加载阶段**：例如 `Loading → Preparing → Ready`。
* **状态切换不要隐藏大量副作用**：否则后期很难排查问题。
* **状态机负责流程，不负责业务细节**。
* **初始化过程可中断、可追踪、可恢复**：方便调试与热重载。

---

# 长期维护铁律

* **半年后的自己也是陌生人**：代码首先写给未来的自己看。
* **代码首先是给人看的，其次才是给机器执行的**。
* **明确优于聪明**：少写炫技代码。
* **简单优于优雅**：简单方案能解决问题就不要追求架构艺术。
* **稳定优于炫技**：成熟方案往往胜过复杂方案。
* **可扩展优于一次性完成**：给未来留扩展点。
* **删代码通常比写代码更有价值**：复杂度是长期敌人。
* **架构要定期复盘**：每个大版本都应审视哪些规则已经不再适合项目。

---

# SRPG项目最终10条核心原则

* **Feature First**：按业务拆模块，而不是按技术拆模块。
* **Definition / Instance 分离**：配置与运行时状态彻底隔离。
* **Rule / Content 分离**：规则在代码，内容在配置。
* **Logic / Presentation 分离**：逻辑与表现解耦。
* **跨模块 Message/Event，模块内函数调用**：避免过度事件化。
* **Hook负责组件固有行为，Observer负责外部响应**：职责边界清晰。
* **Trait + Modifier 统一扩展机制**：统一职业、装备、Buff、天赋体系。
* **数据驱动优于硬编码**：新增内容优先改配置。
* **小函数、小模块、小依赖**：降低复杂度。
* **组合优于继承**：能力组合永远比继承树更适合SRPG。
