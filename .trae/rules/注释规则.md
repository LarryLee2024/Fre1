# Bevy 0.19+ SRPG AI注释宪法 v1.0

## 核心原则

### 1. 注释解释为什么（Why），而不是解释做什么（What）

AI和开发者可以直接阅读代码。

禁止写：

```rust
// 遍历所有单位
for unit in units {}
```

```rust
// 增加生命值
hp += 10;
```

```rust
// 判断是否死亡
if hp <= 0 {}
```

允许写：

```rust
// 死亡判定必须发生在本回合结束前，
// 否则会导致Battle Replay与正式战斗结果不一致。
if hp <= 0 {}
```

---

### 2. 注释优先描述领域规则

优先解释：

* 战斗规则
* 技能规则
* Buff规则
* 回合规则
* 存档规则

而不是实现细节。

推荐：

```rust
/// SRPG规则：反击只能触发一次。
/// 连击产生的额外攻击不会再次触发反击。
```

不推荐：

```rust
/// 遍历反击列表并计算结果。
```

---

### 3. 注释必须能够指导未来重构

推荐：

```rust
/// 当前采用实时计算而非缓存。
///
/// 原因：
/// 1. 当前单位数量极少
/// 2. Replay一致性优先
/// 3. 尚未成为性能热点
///
/// 当Profile确认属性计算成为瓶颈时，
/// 可引入缓存层。
```

---

### 4. 注释优先记录架构决策

推荐：

```rust
/// 使用Observer而非Message。
///
/// 原因：
/// 本事件仅在Battle Feature内部传播，
/// 无跨Feature需求。
```

---

## 强制注释场景

以下场景必须存在注释。

---

### 5. 所有公开API必须拥有文档注释

必须：

```rust
/// 计算最终伤害。
///
/// 不产生任何副作用。
///
/// 保证：
/// - 不修改Entity状态
/// - 不修改Buff
/// - Replay安全
pub fn calculate_damage(...)
```

禁止：

```rust
pub fn calculate_damage(...)
```

---

### 6. 所有Trait必须解释存在理由

必须：

```rust
/// 表示可响应伤害事件的能力。
///
/// 存在原因：
/// Shield、Thorns、Barrier共享相同行为扩展点。
pub trait OnDamageTaken {}
```

禁止：

```rust
pub trait Damageable {}
```

没有解释为什么存在。

---

### 7. 所有领域事件必须解释业务含义

必须：

```rust
/// 单位受到最终伤害后触发。
///
/// 已完成：
/// - 护盾结算
/// - 减伤结算
///
/// 尚未完成：
/// - 死亡检查
/// - Buff响应
pub struct UnitDamaged;
```

---

### 8. 所有状态机必须描述状态流转

必须：

```rust
/// 回合状态机
///
/// Idle
///   ↓
/// SelectUnit
///   ↓
/// SelectAction
///   ↓
/// SelectTarget
///   ↓
/// Execute
///   ↓
/// EndTurn
```

---

### 9. 所有复杂公式必须解释来源

必须：

```rust
/// 来源：设计文档 Combat_v3.2
///
/// 最终伤害公式：
/// Damage = (Atk - Def) * Multiplier
///
/// 保证最小伤害为1。
```

禁止：

```rust
let damage = ...
```

无说明。

---

### 9a. Explain vs Comments 边界

推荐：

```rust
/// 来源：设计文档 Combat_v3.2
/// 最终伤害公式：Damage = (Atk - Def) * Multiplier
/// 保证最小伤害为1。
pub fn calculate_damage(...) -> CalcBreakdown {
    // CalcBreakdown::explain() 负责记录运行时计算输入和中间步骤
    // 注释（§9）记录的是静态公式的事实源（设计文档引用 + 表达式）
    // 二者缺一不可，服务于不同阶段：设计时文档 vs 运行时诊断
}
```

规则说明：

- **Comments (§9)** document the static formula SOURCE (doc reference + expression string).
- `CalcBreakdown::explain()` documents the RUNTIME calculation inputs and intermediate steps.
- Both are required — they serve different purposes (design-time docs vs runtime diagnostics).
- 参考实现：`src/shared/diagnostics/explain.rs` — `CalcBreakdown` struct, `Explain` trait

---

必须：

```rust
// 注意：
///
/// 这里允许HP变为负数。
///
/// 原因：
/// Replay需要记录真实超伤害值。
```

## Invariant（不变量）注释

---

### 11. 核心数据结构必须声明不变量

推荐：

```rust
/// 不变量：
///
/// current_hp <= max_hp
/// current_hp >= 0
/// max_hp > 0
pub struct Health {}
```

---

### 12. Resource必须说明唯一性

推荐：

```rust
/// 全局唯一。
///
/// 同时只能存在一个BattleState。
pub struct BattleState;
```

---

### 13. 缓存必须声明事实源

推荐：

```rust
/// 缓存。
///
/// 非事实源。
///
/// 可随时删除并重新计算。
pub struct CachedFinalStats;
```

## AI协作专属规则

---

### 14. TODO / FIXME / HACK 规范（详见宪法 §16.5）

🟥 禁止无上下文 TODO/FIXME。结构化注释是 AI 协作项目的核心工程资产。

格式：`// TODO[P0-P3][领域][日期]:` + 原因 + 完成条件

- **TODO** = 缺功能（未来要做）
- **FIXME** = 有 Bug（已知问题）
- **HACK** = 已知绕过（临时方案）

优先级：P0 阻塞发布 / P1 一个迭代 / P2 技术债 / P3 可长期存在

P0/P1 的 FIXME 不允许进入主分支。

禁止：
```rust
// TODO: 优化
// TODO: 重构
// FIXME
```

必须：
```rust
// TODO[P2][ATTRIBUTE][2026-06-16]:
// 原因: Aggregator 当前每次全量重算。
// 完成条件: Battle Benchmark 提升 20%+

// FIXME[P1][REPLAY][2026-06-16]:
// 问题: Buff dispel 时触发两次移除事件。
// 复现步骤: 施加 Buff → dispel → 观察事件日志

// HACK[ADR-021]:
// 临时绕过 Trigger 递归问题，待 Runtime v2 重构后删除。
```

---

### 15. 临时方案必须写退出条件

推荐：

```rust
// 临时方案。
//
// 当前直接遍历全部单位。
// 当单位数超过500时重构为空间索引。
```

---

### 16. 宪法豁免必须记录

推荐：

```rust
// [宪法豁免]
//
// 条款：8.0.2
//
// 原因：
// Profile证明缓存收益显著。
//
// 评估日期：2026-09-01
```

---

## 注释禁令

### 17. 禁止废话注释

禁止：

```rust
// 创建角色
let character = ...
```

---

### 18. 禁止代码翻译式注释

禁止：

```rust
// 如果HP小于等于0则死亡
if hp <= 0 {}
```

---

### 19. 禁止保留过期注释

注释与代码不一致时：

优先修改注释。

---

### 20. 禁止注释解释Rust语法

禁止：

```rust
// mut表示可变
let mut hp = 10;
```

---

## 最终原则

好的代码告诉你：

它在做什么。

好的注释告诉你：

为什么这样做。

优秀的注释告诉你：

未来什么时候应该改掉它。

这套规范和你的《AI开发宪法》是互补的。

宪法负责：

```text
架构边界
ECS原则
模块通信
测试规范
```

而这套注释规范负责：

```text
架构决策记录（ADR）
领域规则记录
Replay约束记录
未来重构条件记录
AI上下文保存
```

对于你这种长期 SRPG 项目，我甚至会把 **“公开 API、Trait、领域事件、状态机、不变量、公式、临时方案”** 这 7 类注释设为强制，其余注释全部从严控制。这样代码库几年后仍然能保持很高的 AI 可读性和可维护性。
