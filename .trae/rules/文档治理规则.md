# Document Governance Rules v3.0

## 效力说明

本规范适用于项目内所有文档：

* Architecture
* Domain
* Technical
* Data
* Testing
* AI
* ADR
* Migration
* Roadmap

本规范目标：

* 单一事实源（SSOT）
* AI快速检索
* 长期可维护
* 支撑大型项目演化

---

# 1. 核心原则

## 1.1 单一事实源（SSOT）

同一主题只能存在一个权威文档。

允许：

```text
domain/skill/skill.md
```

禁止：

```text
skill.md
skill_v2.md
skill_final.md
new_skill.md
```

---

## 1.2 Feature First

文档必须按照领域边界组织。

禁止：

```text
按时间分类
按作者分类
按版本分类
```

优先：

```text
按Feature分类
按Domain分类
```

---

## 1.3 文档优先于聊天记录

长期规则必须落盘。

禁止仅存在于：

* AI聊天记录
* Issue评论
* 临时讨论

---

## 1.4 永不创建 Final 文件

禁止：

```text
xxx_final.md
xxx_v2.md
xxx_latest.md
```

文档状态通过：

```yaml
status:
```

管理。

---

## 1.5 优先聚合，必要拆分

禁止为了拆分而拆分。

原则：

```text
一个概念 = 一个文档
```

当文档明显承担多个概念时再拆分。

---

# 2. 文档优先级

冲突时按以下顺序执行：

```text

↓

```

低优先级不得覆盖高优先级。

# 5. 领域目录规则

示例：

```text
domain/

character/
skill/
buff/
equipment/
turn/
victory/
```

每个领域必须存在：

```text
<domain>.md
```

作为：

```text
领域主文档
```

例如：

```text
skill/

skill.md
targeting.md
cost.md
cooldown.md
```

其中：

```text
skill.md
```

是唯一主文档。

# 6. README导航机制

每个目录必须存在：

```text
README.md
```

作用：

* 导航
* 文档索引
* 阅读顺序
* 事实源入口

示例：

```text
domain/README.md
```

列出：

```text
Character
Skill
Buff
Turn
Victory
```

# 7. 文件命名规则

统一：

```text
英文
小写
kebab-case
```

允许：

```text
skill-execution-pipeline.md
```

禁止：

```text
SkillSystem.md
skill_system.md
技能系统.md
```

---

# 8. Frontmatter

所有文档必须包含：

```yaml
---
id: domain.skill

title: Skill

status: stable

owner: game-domain-designer

created: 2026-06-14
updated: 2026-06-14

related:
  - ../buff/buff.md

tags:
  - combat
---
```

---

# 9. 文档状态

允许：

```yaml
status: draft
status: stable
status: deprecated
status: archived
```

含义：

draft:
开发中

stable:
当前事实源

deprecated:
已废弃

archived:
历史归档

---

# 10. ADR规则

ADR必须使用：

```text
ADR-XXX-topic.md
```

模板：

```text
Status
Context
Decision
Rationale
Alternatives
Consequences
```

ADR：

```text
永不删除
永不复用编号
允许被后续ADR推翻
```

---

# 11. 生命周期规则

禁止删除稳定文档。

废弃流程：

```text
stable
↓
deprecated
↓
archived
```

必须保留历史。

# 12. AI执行流程

新增文档前必须：

## Step 1

查重。

确认不存在同主题事实源。

---

## Step 2

确定归属目录。

优先：

```text
Domain
→ Architecture
→ Technical
```

---

## Step 3

判断是否属于独立概念。

不是独立概念：

```text
更新已有文档
```

不要新建。

---

## Step 4

填写完整 Frontmatter。

---

## Step 5

使用对应模板生成。

---

## Step 6

更新 README 导航。

---

## Step 7

执行自检。

# 13. 文档复杂度预算

建议：

```text
≤1000行
```

超过：

```text
1000行：警觉
1500行：评估拆分
```

禁止按行数机械拆分。

必须按：

```text
概念边界
```

拆分。

---

# 14. AI自检清单

提交前必须全部满足：

* [ ] 无重复事实源
* [ ] 文件归属正确
* [ ] 文件命名正确
* [ ] Frontmatter完整
* [ ] 存在唯一主文档
* [ ] README已更新
* [ ] Related链接有效
* [ ] 状态合法
* [ ] 未创建Final/V2文件
* [ ] 未破坏SSOT

---

# 15. 术语一致性

所有新增术语必须与 `ubiquitous_language.md`（待创建）中的术语表保持一致。

要求：

```text
代码类型名 / 函数名 / 配置键名 必须与术语表对齐
```

禁止：

```text
同一概念在不同模块中使用不同命名
```

---

# 16. 废弃机制对齐

代码级、配置级、文档级三者的废弃状态必须同步：

```text
Code: #[deprecated]
Config: status: deprecated
Docs:   status: deprecated (frontmatter)
```

三者始终保持一致，任一处标记废弃则其余两处必须同步标记。

# 最终原则

目录决定认知结构。

领域决定目录结构。

ADR记录决策历史。

代码描述系统现状。

文档保存长期记忆。

任何超过5秒无法定位的知识，视为不存在。
