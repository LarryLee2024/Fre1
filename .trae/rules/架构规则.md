# Bevy 0.18+ SRPG 项目总宪法 v5.0 · 架构篇
> 定位：项目顶层骨架与边界规则，负责宏观架构设计、模块划分、长期治理
> 适用场景：新建模块、调整架构、制定技术方案、治理技术债

## 效力说明
1. 本篇为宪法核心分册，对所有架构决策、模块划分、边界定义具有最高约束力
2. 条款强制等级与全文一致：🟥绝对禁止 / 🟩必须遵守 / 🟨优先选择 / 🟦最佳实践 / ⚠️警觉阈值
3. 所有违反条款的设计必须标注`[宪法豁免]`并说明理由，每3个月重新评估

## 一、核心架构原则
### 1.1 模块划分第一原则
- 🟩 **Feature First**：顶层模块必须按业务领域拆分，**绝对禁止**按技术类型拆分全局目录
  - 合法顶层目录：`character/`、`skill/`、`battle/`、`map/`、`ui/`
  - 非法顶层目录：`components/`、`systems/`、`events/`、`utils/`
  - 单个Feature内部允许按技术类型拆分子文件（如`battle/components.rs`）
  - 规则本质：降低跨模块耦合，代码结构反映业务逻辑
- 🟩 **定义与实例强制分离**：静态配置（职业、技能、装备定义）与运行时状态（HP、Buff、位置）完全隔离
- 🟩 **规则与内容强制分离**：代码只实现通用规则，可变内容全部放入配置文件
  - 成熟扩展点支持纯配置扩展；新增玩法机制允许修改逻辑代码
  - 🟥 禁止为了硬凑"纯配置"强行扭曲架构
- 🟩 **逻辑与表现强制分离**：业务逻辑不依赖任何UI、特效、音效，表现层仅负责展示
- 🟩 **组合绝对优先于继承**：所有差异化通过组件、Trait、Modifier组合实现
  - 禁止通过Trait嵌套模拟继承树，禁止用继承思路实现角色差异
- 🟩 **只解决当前复杂度**：禁止为未明确的未来需求提前设计完整架构
- 🟨 **官方能力优先**：优先使用Bevy原生功能与社区成熟插件，禁止重复造轮子

### 1.2 复杂度预算原则
- 🟩 架构复杂度预算优先级高于性能优化预算
- 🟩 每新增一个抽象层、一套新机制、一个Feature，必须证明收益大于长期维护成本
- 🟥 禁止为未落地的需求预留复杂框架，仅允许轻量扩展点

### 1.3 Feature成熟度分级
- 🟩 **Core级**：核心战斗、角色、属性等基础系统，修改必须经过完整测试与兼容性验证
- 🟩 **Stable级**：装备、任务、地图等成熟系统，新增功能不得破坏现有兼容性
- 🟨 **Experimental级**：玩法实验、辅助工具，允许快速迭代

## 二、纵向三层 + 横向四层架构体系

### 2.1 架构总览

```
src/
├── main.rs
├── lib.rs
│
│                           ┌─ DDD 纵向三层 ─┐
├── shared/                   # L0：底层原子层。零业务语义的通用工具
├── core/                     # L1：领域规则层。能力机制 + 业务子系统
├── infra/                    # L2：技术实现层。渲染/持久化/网络等"脏活"
│                           └─────────────────┘
│                           ┌─ 横切四层 ─┐
├── app/                      # 横切1：启动装配层（Composition Root）
├── content/                  # 横切2：内容桥接层（数据驱动核心）
├── tools/                    # 横切3：开发工具层（feature-gated）
└── modding/                  # 横切4：Mod 扩展层（跨层聚合）
                            └─────────────┘
```

#### DDD 纵向三层

| 层级 | 名称 | 核心定位 | 依赖权限 |
|------|------|----------|----------|
| **L0** | Shared | 全局基础原子能力，零业务语义 | 允许稳定基础库依赖（serde/thiserror/smallvec/bitflags），禁止业务依赖 |
| **L1** | Core | 纯游戏规则与领域逻辑（内部双轴结构） | 仅允许依赖 Shared + bevy_ecs 核心 |
| **L2** | Infrastructure | 技术实现与引擎封装 | 允许依赖 Core + Shared |

#### 横切四层

| 层级 | 名称 | 核心定位 | 依赖权限 |
|------|------|----------|----------|
| **横切1** | App | 游戏启动与全局装配（Composition Root） | 允许依赖所有层 |
| **横切2** | Content | 数据与规则的桥接层 | 允许依赖 Core + Infrastructure |
| **横切3** | Tools | 开发期工具链 | 允许依赖 Core + Shared，永不进入发布构建 |
| **横切4** | Modding | Mod 扩展层 | 通过 Core/mod_api 访问核心功能 |

### 2.2 Core 层双轴架构

Core 层内部采用「**Capabilities 能力层 + Domains 业务域**」双轴结构：

```
src/core/
├── capabilities/               # 纵向：15个核心能力领域（通用机制，玩法无关）
│   ├── tag/                    # 标签
│   ├── attribute/              # 属性
│   ├── modifier/               # 修改器
│   ├── aggregator/             # 聚合器
│   ├── gameplay_context/       # 上下文/载荷
│   ├── spec/                   # 规格/配置
│   ├── ability/                # 技能逻辑
│   ├── trigger/                # 触发器
│   ├── condition/              # 条件/限制/免疫
│   ├── targeting/              # 目标选择
│   ├── execution/              # 执行计算
│   ├── effect/                 # 效果
│   ├── stacking/               # 堆叠规则
│   ├── event/                  # 系统通信
│   ├── cue/                    # 表现层信号
│   └── runtime/                # C3：跨领域运行时编排底座
│
├── domains/                    # 横向：15个业务子系统（承载全部玩法复杂度）
│   ├── combat/                 # 战斗
│   ├── spell/                  # 法术
│   ├── reaction/               # 反应/援护
│   ├── progression/            # 成长养成
│   ├── inventory/              # 背包物品
│   ├── quest/                  # 任务
│   ├── narrative/              # 叙事/对话
│   ├── tactical/               # 战术/网格
│   ├── terrain/                # 地形
│   ├── faction/                # 阵营关系
│   ├── party/                  # 队伍
│   ├── camp_rest/              # 营地休息
│   ├── crafting/               # 制作
│   ├── economy/                # 经济
│   └── summon/                 # 召唤
│
└── mod_api/                    # Mod 稳定 API（Facade + Gateway 模式）
```

#### 核心分工

| 维度 | 职责定位 | 核心产出 | 稳定性 |
|------|----------|----------|--------|
| Capabilities 纵向 | 通用机制与原子能力 | 可复用的系统骨架、数据结构、执行框架 | 极高，极少变更 |
| Domains 横向 | 特定玩法的规则编排 | 业务逻辑、玩法规则、流程控制 | 中等，随玩法迭代 |

> **Capabilities 管「机制」，Domains 管「规则」。**

#### 边界铁则
- 🟥 Capabilities 中出现任何具体业务玩法的硬编码逻辑
- 🟥 Domains 绕过 Capabilities 直接实现底层通用机制
- 🟥 Domain 之间直接 `use` 对方内部类型
- 🟩 跨 Domain 交互：写操作走事件（Message），读操作走 Query API
- 🟩 新增 Capability 必须通过 ADR 审批

### 2.3 依赖方向铁则
- 🟥 允许上层依赖下层：Shared ← Core ← Infrastructure
- 🟥 绝对禁止反向依赖
- 🟥 Domain 间禁止直接依赖，写操作走事件，读操作走 Query API
- 规则本质：保证核心领域逻辑稳定，不受上层表现层变更影响

### 2.4 领域纯度原则
- 🟨 核心领域逻辑（伤害计算、属性公式、技能规则）优先实现为纯函数，参数为数据结构体而非ECS Query/Entity
- 🟩 Bevy系统负责从ECS读取数据、调用领域函数、写回结果
- 🟩 领域纯函数不得直接触发事件、修改全局状态、产生IO，所有副作用由应用层统一执行

### 2.5 Domain × Capabilities 集成规范
- 🟥 Domain 绕过 `integration/` 直接 import Capabilities 组件类型（TagSet / AttributeContainer / ModifierContainer）进行字段访问
- 🟥 `integration/` 单文件膨胀超过 500 行未按能力域拆分
- 🟩 `integration/` 采用 **Facade + SystemParam** 模式：View Types（Domain 自己的类型）+ SystemParam（封装 Capabilities 查询）
- 🟩 Systems 通过 SystemParam + View Types 交互，不知道 Capabilities 内部类型
- 🟩 完整规范见 `docs/01-architecture/README.md` §6.2

## 三、模块化与Plugin边界
- 🟩 **接口最小化**：每个Feature只暴露必要公共接口，内部实现全部私有
  - 🟥 绝对禁止外部模块直接访问其他Feature的internal子模块
- 🟩 **Plugin是唯一对外入口**：每个Feature必须通过Plugin暴露能力
  - 推荐每个Feature根目录包含`plugin.rs`（入口）与`integration/`（跨域访问 ACL，ADR-046）
  - 🟥 禁止新增`api.rs`，跨域访问统一通过`integration/`
- 🟩 **跨模块交互规范**
  - 写操作：必须通过Message、Command、Trigger三种方式
  - 读操作：允许通过公开的Query、Resource、State读取
  - 🟥 绝对禁止跨模块直接修改其他Feature的内部组件与状态
- 🟩 **边界优先于目录**：模块边界清晰度比目录结构更重要
- 🟩 **路径命名规范**：文件路径必须清晰表达业务含义，禁止`base/`、`core/`等模糊命名
- 🟨 通用代码仅可放入`common/`目录，且只能存放纯工具代码，禁止包含业务逻辑

## 四、代码组织与质量原则
- 🟩 **单一文件单一主题**：每个文件只包含一个明确业务主题
- 🟥 禁止创建全局顶层的`systems.rs`、`components.rs`、`utils.rs`巨文件
- 🟩 **Domain 内部结构**：`components/`、`events/`、`error/`、`resources/` 初始可为单文件，组件/事件/错误/资源数 ≥ 5 时升级为目录；`rules/`、`systems/`、`integration/`、`tests/` 初始即为目录（详见 `ai-constitution-complete.md` §3.4）
- ⚠️ **文件大小阈值**：超过500行且内聚性下降时建议拆分，超过1000行强制评估
  - 禁止机械按行数拆分，内聚性优先于文件大小
- 🟩 **函数设计原则**：单一职责、命名表意、Early Return优先
- ⚠️ 函数超过100行且逻辑可拆解时建议拆分，超过3层嵌套必须重构
- 🟩 **抽象原则**：代码重复3次以上再抽象，可读性优先于复用性
- 🟩 **Trait原则**：Trait只表示能力、只用于扩展点，禁止模拟继承树
  - 🟥 禁止为了"代码优雅"创建无实际价值的Trait
- 🟩 **AI可读性优先**：优先直白线性逻辑，避免宏套宏、深度泛型、类型体操
  - 可读性优先级：人类开发者 > AI理解 > 编译器优化 > 代码"优雅"

## 五、数据驱动与内容生产
- 🟩 配置定义内容，代码解释配置；配置稳定性优先于格式优雅性
- 🟩 向后兼容性优先于任何其他考虑
- 🟩 **配置Schema强制要求**：所有配置结构体必须对应明确的Schema定义
  - 规则本质：AI生成配置有明确依据，避免字段漂移
- 🟩 **CI配置Lint**：所有配置必须支持CI自动校验（引用合法性、数值范围、循环依赖）
- 🟩 **数据所有权原则**：每份配置数据有唯一归属Feature，禁止多模块同时维护同一份核心数据
- 🟩 **数值平衡隔离**：所有平衡参数必须配置化，**绝对禁止**代码中硬编码魔法数字
- 🟩 **字段三步删除原则**：核心字段禁止直接删除，必须遵循「标记Deprecated → 迁移逻辑 → 大版本移除」
- 🟩 所有配置优先支持热重载，通过统一Asset Pipeline管理

## 六、可观测性规范
### 6.1 日志体系
- 🟩 统一使用`tracing`库，禁止`println!`、`dbg!`输出运行时信息
- 🟩 日志核心定位为领域事件履历，禁止记录技术执行流水账
- 🟩 分级规范：ERROR异常 / WARN可恢复异常 / INFO核心业务边界 / DEBUG调试信息 / TRACE算法细节
- 🟩 所有INFO级日志必须携带`event`字段，target与所属Feature目录名一致
- 🟥 Release版本禁止在每帧系统中输出INFO/DEBUG日志，禁止循环内输出INFO日志
- 🟩 **领域事件驱动日志**：INFO级业务日志必须通过领域事件+Observer统一输出，业务代码禁止直接调用`info!`
- 🟩 日志 Observer 统一放在基础设施层，绝对不侵入业务模块

### 6.2 错误体系
- 🟩 每个领域定义独立错误枚举，**绝对禁止**全局统一`AppError`、`anyhow::Error`作为业务层错误类型
- 🟩 强制区分四类失败：规则失败（不用Err）、领域错误（用Err）、基础设施错误、程序Bug
- 🟩 所有错误必须携带完整上下文信息，推荐带编号错误码
- 🟥 核心业务领域绝对禁止`unwrap()`、`expect()`、`panic!()`，仅测试/工具/原型可豁免

## 七、工程质量与技术债治理
### 7.1 质量预算原则
- Warning Budget = 0：主分支不允许存在未处理的必修编译警告
- Bug Budget = 0：P0/P1级缺陷不允许流入主分支
- Tech Debt Budget = 可控：技术债必须登记跟踪，设定偿还节点

### 7.2 问题分级标准
- **P0 致命级**：发现即修，禁止提交合并。包括业务panic、ECS Borrow冲突、事件无限循环、Entity泄漏、存档损坏、双轴边界突破
- **P1 高优先级**：一个迭代内修复，CI拦截。包括必修Rust警告、废弃API、必修Clippy警告、配置校验失败、Domain边界违规
- **P2 中优先级**：登记跟踪，阈值触发处理。包括非热点性能问题、Archetype抖动、非核心日志噪音
- **P3 低优先级**：技术债，架构复盘时集中清理。包括局部重复代码、命名不统一、文件偏大

### 7.3 警告分级
- **必修清理（P1）**：unused_imports、unused_variables、dead_code、deprecated、unused_must_use
- **可暂缓（P3）**：missing_docs
- **可豁免**：测试、工具、原型代码可通过`#[allow]`豁免
- Clippy同理，必修项必须修复，建议项按需处理，禁止机械修改

### 7.4 CI门禁强制标准
- 🟩 主分支准入必须同时满足：`cargo fmt` 0问题、`cargo clippy` 0必修警告、`cargo test`全绿、配置校验全绿、架构依赖检查无违规
- 🟥 任意一项不满足，禁止合并

## 八、长期维护与运营
- 🟩 代码可读性优先，明确优于聪明，简单优于优雅，稳定优于炫技
- 🟩 删除无用代码优先于写新代码
- 🟩 每引入一个自研系统，必须评估未来五年的维护成本
- 🟩 架构每3个月复盘一次，重点清理过度设计与无用代码
- 🟨 **扩展预留**：核心系统预留轻量扩展点，为未来Mod支持、国际化铺路
  - 禁止硬编码玩家可见文本，统一走本地化资源
  - 公共API、事件体系保持稳定，不提前实现完整框架
- 🟨 核心领域事件设计预留遥测采集能力，不提前实现完整埋点系统

## 核心速查（10条）
1. Feature First，按业务拆顶层模块
2. 纵向三层+横切四层，禁止反向依赖
3. Capabilities管机制，Domains管业务，边界不可突破
4. 定义/实例、规则/内容、逻辑/表现三重分离
5. Plugin是模块唯一入口，跨模块写操作严格收口
6. 数据驱动，配置有Schema，数值不硬编码
7. 领域纯函数化，不绑定引擎
8. 日志走领域事件，错误分领域定义
9. Warning预算为0，CI门禁硬拦截
10. 技术债可登记但必须有偿还节点
