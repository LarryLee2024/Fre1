# Bevy 0.19+ SRPG 项目总宪法 v5.0 · SRPG专项篇
> 定位：战棋游戏业务规则与系统设计规范，负责所有玩法系统的实现标准
> 适用场景：开发角色、属性、技能、Buff、回合、战斗、存档等核心玩法

## 效力说明
1. 本篇约束所有SRPG业务系统的设计与实现，是玩法开发的统一标准
2. 所有核心玩法系统必须遵循本篇规范，保证逻辑一致性与可维护性
3. 条款强制等级与全文一致，违反核心规则的玩法实现视为不合格

---

## 零、Core层双轴架构

Core 层内部采用「**Capabilities 能力层 + Domains 业务域**」双轴结构：

- **Capabilities（纵向）**：15个核心能力领域，提供通用机制骨架，玩法无关
  - 核心基石：Tag / Attribute / Modifier / Aggregator / GameplayContext
  - 逻辑骨架：Spec / Ability / Trigger / Condition
  - 行为表现：Targeting / Execution / Effect / Stacking / Event / Cue
  - 新增 Capability 必须通过 ADR 审批

- **Domains（横向）**：15个业务子系统，承载全部玩法复杂度
  - combat / spell / reaction / progression / inventory / quest / narrative
  - tactical / terrain / faction / party / camp_rest / crafting / economy / summon

- **边界铁则**：
  - 🟥 Capabilities 只提供机制，不实现业务规则
  - 🟥 Domains 只编排规则，不重复造机制
  - 🟥 Domain 之间禁止直接依赖，写操作走事件，读操作走 Query API

---

## 一、角色系统
### 1.1 分层扩展体系
- 🟩 **数值能力**：统一通过Modifier管线实现，所有属性增减进入统一计算流程
- 🟩 **行为扩展**：优先使用Trait定义能力接口，由系统统一调度
- 🟩 **特殊机制**：召唤、传送、复活等非数值效果，允许通过独立系统+组件实现，不强制塞入Modifier
- 规则本质：不同能力用最合适的实现方式，避免强行统一导致架构膨胀

### 1.2 核心定义
- 种族 = Trait集合 + Modifier集合
- 职业 = 成长率表 + 技能池 + Trait集合
- 天赋 = 特殊Trait + 专属Modifier
- 装备 = Modifier集合 + Trait集合
- Buff/Debuff = 临时Modifier + 临时Trait + 生命周期组件

### 1.3 战斗事件链
- 🟩 复杂战斗交互必须通过**Trigger + Observer**构建事件链
- 示例：伤害Trigger → 护盾Observer吸收 → 吸血Observer结算 → 反击Observer触发

---

## 二、属性系统
- 🟩 基础属性与派生属性必须完全分离
- 🟩 **计算策略**：默认实时计算保证一致性，Profile证明确为热点后允许引入缓存
- 🟩 **修改规范**
  - 🟥 基础/派生属性的最终值禁止直接修改，必须通过添加/移除Modifier实现
  - HP、MP等资源型数值的增减，可通过专用系统直接修改，不强制走Modifier
- 🟩 最终属性值只有一个统一计算来源，公式集中管理
- 🟨 **缓存约定**：缓存不是事实源，必须允许随时删除重建，属性变化时立即失效

---

## 三、地图系统
- 🟩 地图优先视为Grid数据结构
- 🟩 Tile是否实体化按需决定，不强制统一
- 🟩 Chunk机制按需引入，小地图不提前拆分
- 🟩 占用网格必须作为独立数据结构存在
- 🟩 地图逻辑数据与渲染层完全分离
- 🟩 寻路数据运行时动态生成

---

## 四、回合系统
- 🟩 **阶段划分标准化**：回合必须划分为「回合开始 → 阶段判定 → 单位行动 → 回合结算 → 回合结束」明确阶段
- 每个阶段对应独立的系统集与触发点
- 🟩 回合切换通过状态机驱动，禁止手动调用回合切换函数
- 规则本质：统一时序，避免Buff结算、技能效果的顺序依赖Bug

---

## 五、技能执行管线
- 🟩 **五阶段固定流程**：所有技能执行必须遵循「目标校验 → 消耗扣除 → 施法前摇 → 效果执行 → 结果结算」
- 禁止跳过阶段或乱序执行
- 🟩 **预览与执行分离**：伤害、范围预览走独立只读路径，绝对禁止产生任何副作用
- 规则本质：统一技能执行流程，便于扩展、调试与回放

---

## 六、Buff生命周期
- 🟩 **四阶段标准化**：所有Buff必须遵循「Apply（施加）→ Tick（周期触发）→ Expire（到期）→ Remove（移除）」
- 每个阶段有明确的触发时机与Hook点
- 规则本质：统一Buff行为，避免生命周期混乱导致的残留、叠加Bug

---

## 七、命令层与输入抽象
### 7.1 命令层
- 🟩 所有操作入口为标准化业务命令
- 玩家输入、AI决策、回放执行、网络同步，最终都必须转换为命令
- 典型命令：`MoveUnitCommand`、`CastSkillCommand`、`EndTurnCommand`
- 🟩 执行系统不区分命令来源，只处理命令本身
- 规则本质：所有入口统一，真人、AI、回放复用同一套执行链

### 7.2 输入抽象层
- 🟩 输入为独立Feature，鼠标/键盘/手柄/触摸统一转换为业务命令
- 🟥 绝对禁止业务模块直接读取原始输入按键
- 规则本质：屏蔽平台差异，多端适配无需修改核心业务

---

## 八、核心设计原则
### 8.1 读写分离（CQRS Lite）
- 🟩 读路径（伤害预览、范围查询、状态展示）必须无副作用
- 🟥 绝对禁止在读路径中修改状态、触发事件、消耗资源
- 🟩 所有状态修改必须通过命令与执行系统统一收口

### 8.2 仿真优先
- 🟩 核心战斗规则必须支持脱离Bevy运行时独立执行
- AI决策、伤害预览、自动测试均可直接调用同一套规则
- 规则本质：一次实现，多场景复用，保证逻辑一致性

### 8.3 随机数分层管理
- 🟩 按用途拆分独立RNG流：战斗RNG、掉落RNG、世界RNG、AI RNG
- 禁止全局共用单一RNG实例
- 🟩 业务逻辑禁止直接调用`rand::random()`，必须通过统一随机数服务获取
- 规则本质：不同系统随机结果互不干扰，保证战斗回放确定性

### 8.4 状态分离原则
- 🟩 临时状态（选中单位、悬停格子、技能预览）属于表现层，绝对禁止混入事实状态
- 临时状态不参与存档、不进入回放
- 规则本质：避免交互状态污染核心游戏数据

---

## 九、存档与兼容性
- 🟩 **强制版本字段**：所有存档必须包含显式version字段
- 🟩 **向后兼容原则**：新版本必须支持读取上一个大版本存档，通过自动迁移升级
- 🟥 禁止版本升级后旧存档直接无法读取
- 🟩 核心字段删除必须遵循三步原则：标记废弃 → 迁移 → 大版本移除

---

## 十、测试与确定性

> **测试跟领域走（Feature First），但不写在源码文件内部。**

### 10.1 领域内聚四层测试

```
<domain>/
├── tests/
│   ├── unit/          # 单元测试：验证领域纯函数、核心规则
│   ├── integration/   # 集成测试：验证领域内多组件协作
│   ├── invariant/     # 不变量测试：验证领域不变量（**最高价值**）
│   └── fixtures/      # 测试数据（Builder模式 / RON文件）
```

- 🟥 禁止 `#[cfg(test)] mod tests` 内联测试
- 🟥 禁止将所有测试平铺到根 `tests/unit/`
- 🟩 根 `tests/` 仅保留跨领域测试（battle_flow / save_load / regression / replay / golden / simulation / performance / e2e）

### 10.2 不变量测试（最高价值）

| 不变量 | 说明 |
|--------|------|
| Tag bit唯一 | 同一Tag不能在位掩码中重复设置 |
| Buff不重复叠加 | 同源同类型Buff不会无限堆叠 |
| Effect不修改不存在属性 | Effect引用的AttributeId必须已注册 |
| HP永远>=0 | HP计算结果不能为负 |
| Modifier不改变基础值 | Modifier只影响聚合后的当前值 |

### 10.3 测试基础设施
- 🟩 核心测试使用Builder模式构造测试数据（TestCharacterBuilder、TestBattleBuilder）
- 🟩 稳定输出使用Golden Test金文件对比，版本变更必须显式确认

### 10.4 确定性强制要求
- 🟩 相同初始状态 + 相同输入 + 相同种子，必须得到完全一致的战斗结果
- 🟩 核心业务逻辑禁止直接依赖系统时间，必须使用统一GameTime服务
- 🟩 所有战斗Bug必须通过Battle Replay重现并沉淀为永久测试用例

---

## 十一、UI系统专项
- 🟩 产品UI用bevy_ui，工具UI用bevy_egui
- 🟩 状态单向流动：UI只展示状态，不保存业务真相，不修改业务状态
- 🟩 业务逻辑绝对禁止直接操作UI组件
- 🟩 输入焦点、拖拽功能全局统一管理

---

## 核心速查（10条）
1. 数值走Modifier，特殊机制独立实现，不强行统一
2. 属性分基础/派生，资源值可直接修改
3. 回合五阶段、技能五阶段、Buff四阶段标准化
4. 所有操作转命令，多来源复用同一执行链
5. 读写分离，预览路径绝对无副作用
6. 随机数分层，统一RNG服务，禁止直接调用rand
7. 临时状态与事实状态严格隔离
8. 核心规则支持离线仿真，不绑定引擎
9. 存档带版本，支持自动迁移向后兼容
10. 战斗必须可重现，Replay是核心测试手段
