对于你的项目最大的风险已经不是：

> "代码写错了"

而是：

> "代码已经没用了，但没人敢删"

这是大型项目最常见的问题。

很多项目最后30%~50%的代码其实都是历史遗留。

---

# 第一原则

不要问：

> 哪些代码是冗余的？

而要问：

> 哪些代码已经没有业务价值了？

这是完全不同的思路。

---

# 第一层：找死代码（Dead Code）

最容易发现。

## Rust Clippy

直接开启：

```bash
cargo clippy --all-targets --all-features
```

重点看：

```rust
dead_code
unused_imports
unused_variables
unused_mut
```

例如：

```rust
fn old_damage_formula() {}
```

没有任何引用。

直接删除。

---

## Rust Analyzer

VSCode：

```text
Find All References
```

如果：

```rust
FireDamageCalculator
```

引用数：

```text
0
```

基本可删。

---

# 第二层：找幽灵Feature

这是最危险的。

例如：

```text
battle/
  old_damage/
```

代码很多。

测试全绿。

但是：

```text
根本没有Plugin注册
```

---

检查：

```rust
app.add_plugins(...)
```

是否真正注册。

---

很多项目都有：

```rust
OldInventoryPlugin
LegacyQuestPlugin
OldBattlePlugin
```

留了半年没人用。

---

# 我的习惯

直接搜：

```rust
pub struct xxxPlugin
```

然后检查：

```rust
add_plugins
```

是否引用。

---

# 第三层：找永远不会触发的System

特别常见。

---

例如：

```rust
fn apply_poison_damage(...)
```

看起来很重要。

---

结果：

```rust
PoisonTickMessage
```

已经不发了。

---

这个System：

```rust
Update
```

一直挂着。

永远不执行。

---

检查方法：

搜索：

```rust
MessageWriter<PoisonTickMessage>
```

或者：

```rust
commands.trigger(...)
```

是否存在。

---

没有发送方：

系统已经死了。

---

# 第四层：找重复规则

这是SRPG最大的问题。

---

例如：

装备：

```rust
attack += 10
```

---

Buff：

```rust
attack += 10
```

---

职业：

```rust
attack += 10
```

---

天赋：

```rust
attack += 10
```

---

你会发现：

项目出现：

```rust
calculate_attack()
```

```rust
compute_attack()
```

```rust
get_attack()
```

```rust
final_attack()
```

四套算法。

---

这时候不是删代码。

而是：

合并规则。

---

对于你的项目：

最应该检查：

```text
Modifier
Trait
Stat
Damage
Buff
```

是否存在多套实现。

---

# 第五层：找已经失效的抽象

这是高级问题。

---

例如：

以前：

```rust
trait DamageCalculator
```

下面：

```rust
PhysicalDamageCalculator
MagicDamageCalculator
```

---

后来重构：

```rust
ModifierPipeline
```

统一了。

---

结果：

```rust
DamageCalculator Trait
```

还留着。

---

实现：

```text
只有一个
```

---

这种就是：

```text
失效抽象
```

---

铁律：

> Trait只有一个实现，且未来没有扩展计划，要警觉。

---

# 第六层：找僵尸配置

大型SRPG特别容易出现。

---

例如：

```json
{
  "crit_rate": 0.1
}
```

---

结果：

```rust
crit_rate
```

根本没人读。

---

检查：

Rust Analyzer：

```text
Find References
```

---

引用：

```text
0
```

---

配置字段直接删除。

---

# 第七层：找历史兼容层

这个是重构后最大垃圾来源。

---

例如：

```rust
OldEquipmentDefinition
```

```rust
EquipmentDefinition
```

同时存在。

---

或者：

```rust
BattleEvent
```

```rust
BattleMessage
```

同时存在。

---

或者：

```rust
OldModifier
```

```rust
Modifier
```

同时存在。

---

这种通常来自：

```text
迁移完成了
兼容层没删
```

---

# 第八层：看测试覆盖反向找

你已经有：

```text
377+
```

单测。

---

很好用。

---

统计：

```text
哪些模块完全没有测试
```

---

两种可能：

### 情况1

重要代码漏测

补测试。

---

### 情况2

代码已经没人用了

直接删。

# 第九层：战斗链路审计（最适合你）

你的项目：

```text
装备
↓
属性
↓
技能
↓
伤害
↓
Buff
↓
死亡
↓
奖励
```

---

画出来。

例如：

```text
AttackIntent
↓
DamageCalculated
↓
DamageApplied
↓
DeathDetected
↓
RewardGranted
```

---

然后检查：

每个节点：

```text
谁产生
谁消费
```

---

会发现：

```text
OldDamageApplied
```

没人消费。

---

删。

# 第十层：最狠的方法（我最推荐）

建立：

## CODE GRAVEYARD 审查表

每次重构后。

检查：

| 项目        | 问题          |
| --------- | ----------- |
| Plugin    | 是否还被注册      |
| Message   | 是否还有发送方     |
| Observer  | 是否还有触发源     |
| Component | 是否还有Query引用 |
| Resource  | 是否还有读写      |
| Trait     | 是否还有多个实现    |
| Config字段  | 是否还有读取方     |
| 测试        | 是否覆盖真实业务    |
| Feature目录 | 是否仍有业务价值    |
| Debug工具   | 是否仍被使用      |

---

# 对你当前项目

我最怀疑产生冗余的地方

按概率排序：

| 排名 | 高危区域              |
| -- | ----------------- |
| 1  | 旧Damage系统         |
| 2  | 旧Buff系统           |
| 3  | 旧Modifier实现       |
| 4  | Message/Event迁移遗留 |
| 5  | Inventory重构遗留     |
| 6  | Trait抽象层          |
| 7  | Config字段          |
| 8  | 调试工具              |
| 9  | 测试辅助代码            |
| 10 | Plugin注册链         |

因为你已经经历过多轮重构，这些地方最容易形成：

```text
代码还在
测试还绿
编译还过
实际上已经没人用了
```

这种“幽灵代码”才是长期项目复杂度爆炸的真正来源。
