# 一、Bevy 0.18+ SRPG AI开发宪法 v1.6（工业级完整版）
## 宪法效力说明
1. 本宪法所有条款对AI生成的所有代码具有**最高约束力**，优先级高于任何通用编程规范、语言习惯或AI默认输出
2. 条款编号永久固定，任何AI生成代码违反条款即视为不合格输出
3. 条款分为五个强制等级：
   - 🟥 **绝对禁止**：任何情况下都不允许出现，**不可豁免**
   - 🟩 **必须遵守**：无例外强制执行，除非获得明确豁免
   - 🟨 **优先选择**：除非有明确且可验证的技术理由，否则必须采用
   - 🟦 **最佳实践**：推荐但非强制，无技术理由时优先采用
   - ⚠️ **警觉阈值**：达到阈值时AI必须主动提出重构建议
4. 本宪法专为Bevy 0.18及以上版本设计，API变更不影响架构原则
5. 所有违反宪法的代码必须标注`[宪法豁免]`并说明理由，豁免代码每3个月必须重新评估

## 第一部分：架构宪法（最高优先级）
### 1.1 模块划分原则
- 🟩 **1.1.1 Feature First**：必须按业务领域拆分顶层模块，**绝对禁止**按技术类型拆分全局目录
  - 规则本质：降低模块间耦合，让代码结构反映业务逻辑而非技术实现
  - 合法顶层目录：`character/`、`skill/`、`battle/`、`map/`、`ui/`
  - 非法顶层目录：`components/`、`systems/`、`events/`、`utils/`
  - 允许单个Feature内部按技术类型拆分子文件（如`battle/components.rs`）
- 🟩 **1.1.2 定义与实例强制分离**：所有静态配置（职业、技能、装备、天赋定义）必须完全独立于运行时状态（HP、Buff、CD、位置）
  - 规则本质：配置可复用、可热重载、可编辑，运行时状态仅存在于游戏进程中
- 🟩 **1.1.3 规则与内容强制分离**：代码只实现通用规则，所有可变内容必须放在配置文件中
  - 成熟扩展点支持纯配置扩展；新增玩法机制允许修改逻辑代码
  - 🟥 **绝对禁止**：为了硬塞配置而强行扭曲架构
  - 规则本质：内容生产无需改代码，同时避免为了“纯配置”过度设计
- 🟩 **1.1.4 逻辑与表现强制分离**：业务逻辑计算结果，表现层只负责展示
  - 🟥 **绝对禁止**：伤害计算、属性计算、战斗规则依赖任何UI组件或视觉特效
  - 规则本质：保证逻辑可测试、可回放，表现层可独立替换
- 🟩 **1.1.5 数据驱动优先**：所有可配置内容必须通过数据驱动实现
- 🟩 **1.1.6 组合绝对优先于继承**：
  - 所有游戏对象必须由Component、Trait、Modifier组合构成
  - Rust无类继承机制，禁止通过Trait嵌套模拟继承树
  - 🟥 **绝对禁止**：使用继承思路实现角色差异化
  - 规则本质：组合比继承更灵活、更易扩展、更符合ECS范式
- 🟩 **1.1.7 只解决当前复杂度**：
  - 🟥 **绝对禁止**：为未来可能出现但未明确的需求提前设计完整架构
  - 🟥 **绝对禁止**：过度抽象增加当前维护成本
  - 允许为明确规划的能力预留轻量扩展点
  - 规则本质：预测未来需求的准确率极低，过度设计是最大的技术债务
- 🟨 **1.1.8 官方能力优先**：
  - 优先使用Bevy原生提供的所有功能
  - 优先使用社区成熟插件而非自研
  - 🟥 **绝对禁止**：重复实现Bevy已有的基础设施

### 1.2 强类型标识约定
- 🟩 **1.2.1 核心领域必须使用强类型ID**
  - 所有业务实体必须定义专属强类型标识，禁止直接使用裸`Entity`作为业务标识跨模块传递
  - 必须覆盖核心实体：`UnitId`、`SkillId`、`BuffId`、`ItemId`、`QuestId`
  - 规则本质：编译期避免传参类型错误，大幅提升日志、错误、事件的可读性与长期可维护性
- 🟩 **1.2.2 对外输出必须使用可读标识**
  - 所有日志、错误信息、领域事件中，必须输出业务可读的字符串ID，禁止直接打印`Entity(xxx)`原生格式
  - 规则本质：保证半年后日志、报错依然可直接定位对应业务对象，无需二次排查

### 1.3 架构分层与依赖方向
- 🟩 **1.3.1 三层架构划分**
  - 领域层（Domain）：核心业务规则、数值计算、状态定义，纯逻辑无引擎依赖
  - 应用层（Application）：ECS调度、系统编排、状态流转，粘合领域逻辑与引擎
  - 表现层（Presentation）：UI、特效、音效、输入处理，仅读取状态不修改核心逻辑
  - 规则本质：明确权责边界，核心逻辑可脱离引擎独立测试、仿真
- 🟥 **1.3.2 依赖方向铁则**
  - 允许上层依赖下层：表现层 → 应用层 → 领域层
  - 绝对禁止反向依赖：领域层不得依赖应用层与表现层，应用层不得依赖表现层
  - 规则本质：保证核心领域逻辑的稳定性，不受上层表现层变更影响

### 1.4 领域纯度原则
- 🟨 **1.4.1 核心领域与引擎解耦**
  - 伤害计算、属性公式、技能规则、Buff效果等纯领域逻辑，应优先实现为纯函数，参数为数据结构体而非ECS Query/Entity
  - Bevy系统负责从ECS中读取数据、调用领域函数、写回结果
  - 规则本质：核心规则可离线测试、仿真、回放，不绑定引擎生命周期
- 🟩 **1.4.2 领域无副作用**
  - 领域纯函数不得直接触发事件、修改全局状态、产生IO
  - 所有副作用由应用层系统统一执行
  - 规则本质：保证预览、仿真、AI决策等场景调用核心规则时不产生意外副作用

### 1.5 复杂度预算原则
- 🟩 **1.5.1 复杂度优先于性能**
  - 架构复杂度预算优先级高于性能优化预算
  - 每新增一个抽象层、一个新Feature、一套新机制，必须证明其收益大于长期维护成本
  - 规则本质：大多数独立项目死于复杂度失控，而非性能不足
- 🟥 **1.5.2 禁止为未落地需求预留复杂框架**
  - 仅允许预留轻量扩展点，禁止提前实现完整的底层能力
  - 规则本质：预测需求的准确率极低，过度设计是最大的技术债务

### 1.6 Feature成熟度分级
- 🟩 **1.6.1 三级成熟度划分**
  - Core级：核心战斗、角色、属性等基础系统，稳定性要求最高，修改必须经过完整测试
  - Stable级：装备、任务、地图等成熟系统，新增功能不得破坏现有兼容性
  - Experimental级：玩法实验、辅助工具等功能，允许快速迭代
  - 规则本质：不同成熟度的模块适用不同约束强度，AI生成代码时自动匹配严格度

## 第二部分：ECS宪法
### 2.1 核心概念
- 🟩 **2.1.1 Entity只是ID**：
  - 🟥 **绝对禁止**：在Entity上调用任何方法或将其当作面向对象实例使用
  - 允许将Entity作为纯ID参数传递
  - 规则本质：Entity没有任何行为，只是组件的集合标识
- 🟩 **2.1.2 数据与行为强制分离**：
  - Component只能存储纯数据状态，**绝对禁止**包含任何逻辑
  - System只能包含纯逻辑，**绝对禁止**存储任何状态
  - 规则本质：ECS的核心优势就是数据与行为分离带来的并行性和可维护性
- 🟩 **2.1.3 Tag Component优先原则**
  - 实体持久状态必须优先使用空Tag Component实现
    - 合法：`Dead`、`Frozen`、`Stunned`
    - 非法：`is_dead: bool`、`is_frozen: bool`
  - 配置字段、临时计算值、非实体级状态允许使用bool类型
    - 合法：`is_unique: bool`（配置属性）、`is_critical: bool`（结算临时值）
  - 规则本质：Tag Component可被Bevy优化为位掩码，查询性能远超布尔字段；配置与临时场景无需过度设计
- 🟨 **2.1.4 从属关系使用官方Relationship**
  - 实体间从属关系（Buff归属单位、背包归属角色、召唤物归属主人）优先使用Bevy官方`Relationship`机制实现
  - 禁止手动维护父子实体ID字段
  - 规则本质：复用引擎原生能力，保证关系查询与生命周期管理的一致性

### 2.2 四级通信机制（Hook / Trigger / Observer / Message）
- 🟩 **2.2.1 Hook = 组件生命周期固有行为**
  - 组件添加/移除时的轻量固有副作用，必须通过`#[component(on_add=..., on_remove=...)]`声明
  - 仅用于简单状态联动，禁止承载复杂业务逻辑
  - 示例：`#[component(on_add=remove_moveable)] struct Dead;`
  - 规则本质：组件副作用与定义绑定，生命周期行为内聚
- 🟩 **2.2.2 Trigger = Feature内事件链载体**
  - 同一Feature内的多段响应逻辑、战斗事件链，必须使用`commands.trigger()`机制实现
  - 典型场景：伤害触发护盾、吸血、反击等连锁效果
  - 规则本质：比全局Message轻量，天然绑定实体，适合构建线性事件链
- 🟩 **2.2.3 Observer = 局部状态变化响应**
  - 同一Feature内的组件变化、Trigger触发的响应逻辑，必须使用Observer实现
  - 典型场景：角色死亡播放动画、血量变化刷新UI
  - 规则本质：比系统轮询更高效，关注点更聚焦
- 🟩 **2.2.4 Message = 跨Feature全局广播**
  - 跨业务模块的通知必须使用全局事件（Event）
  - 典型场景：回合结束、战斗胜利、任务完成
  - 规则本质：跨模块通信保持松耦合，不直接依赖内部实现
- 🟩 **2.2.5 模块内部优先函数调用**
  - 🟥 绝对禁止将同一模块内的普通逻辑全部事件化
  - 🟥 绝对禁止滥用事件/Trigger模拟函数调用
  - 规则本质：事件增加调试复杂度，仅在需要解耦的场景使用
- 🟩 **2.2.6 领域事件是唯一业务事实源**
  - 核心业务动作完成后必须发布对应领域事件，日志、战斗回放、UI履历、成就、任务等均为事件的下游消费者
  - 禁止业务代码直接调用日志、UI刷新等副作用逻辑
  - 规则本质：一次事件触发，全链路复用，消除重复代码，保持领域层纯净
- 🟩 **2.2.7 领域事件白名单管理**
  - 所有领域事件必须纳入统一白名单文档管理，新增事件必须先更新白名单
  - 🟥 **绝对禁止**：为临时副作用随意新增领域事件
  - 规则本质：控制事件数量爆炸，保证领域事件的权威性与一致性

### 2.3 ECS执行模型
- 🟩 **2.3.1 ECS是数据流，不是调用链**：
  - 🟥 **绝对禁止**：模拟面向对象的调用方式，如`player.attack(enemy)`
  - 正确方式：创建命令组件，由对应系统统一处理
  - 规则本质：ECS通过系统处理组件数据，而不是通过对象之间互相调用
- 🟩 **2.3.2 依赖关系表达**：组件依赖必须通过`#[require(Component)]`属性声明
  - 🟥 **绝对禁止**：手动检查并补全缺失的组件
  - 规则本质：Bevy会自动管理组件依赖，确保依赖组件同时存在
- 🟩 **2.3.3 状态变更检测原则**
  - 组件生命周期变化必须优先使用Bevy原生`Added`、`Changed`、`Removed`过滤器
  - 🟥 绝对禁止手写bool脏标记字段检测组件变化
  - 业务状态机（如TurnPhase、BattleState）允许使用显式状态组件，自行管理状态流转
  - 规则本质：引擎原生变更检测高度优化；业务状态机有自身流转逻辑，不强制套用生命周期检测
- 🟩 **2.3.4 Resource使用规范**：
  - Resource只能存储真正的全局唯一状态
  - 🟥 **绝对禁止**：将Resource当作全局变量仓库使用
  - 规则本质：Resource会引入全局状态，增加代码耦合和测试难度
- 🟩 **2.3.5 性能优化**：
  - 🟥 **绝对禁止**：在高频逻辑中使用Observer造成风暴
  - 每帧执行10次以上的逻辑必须直接使用System处理
- 🟩 **2.3.6 状态管理规范**
  - 全局流程状态优先使用`States`、`SubStates`、`ComputedStates`实现
  - 禁止手动维护全局状态枚举与切换逻辑
  - 规则本质：复用官方状态机能力，统一状态切换与调度时机
- 🟨 **2.3.7 运行条件优先**
  - 系统的执行前置判断优先使用`run_if()`条件表达
  - 避免在系统内部写大段if判断是否执行
  - 规则本质：调度层统一控制执行时机，系统内部专注业务逻辑
- 🟩 **2.3.8 Schedule权责划分**
  - `PreUpdate`：输入处理、命令执行、状态同步
  - `Update`：核心业务逻辑、规则结算
  - `PostUpdate`：事件响应、表现层更新、UI刷新
  - 禁止跨Schedule乱放系统，禁止在错误阶段执行业务逻辑
  - 规则本质：统一执行时序，避免系统顺序依赖导致的隐性Bug

## 第三部分：模块化与Plugin边界宪法
- 🟩 **3.0.1 接口最小化原则**
  - 每个Feature只暴露必要的公共接口，所有内部实现必须设为私有
  - 🟥 绝对禁止外部模块直接访问其他Feature的internal子模块
  - 规则本质：隐藏内部实现，保证模块边界可控
- 🟩 **3.0.2 Plugin是唯一对外入口**
  - 每个Feature必须通过Plugin对外暴露能力
  - 推荐每个Feature根目录包含`plugin.rs`（入口）与`api.rs`（公共类型与接口）
  - 规则本质：统一模块接入方式，外部只依赖公开API
- 🟩 **3.0.3 边界优先于目录**：模块边界的清晰度比目录结构更重要
- 🟩 **3.0.4 跨模块交互规范**
  - 跨模块写操作必须通过Message、Command、Trigger三种方式
  - 跨模块读操作允许通过公开的Query、Resource、State读取
  - 🟥 绝对禁止跨模块直接修改其他Feature的内部组件与状态
  - 规则本质：写操作严格收口，读操作保持灵活，兼顾解耦与效率
- 🟩 **3.0.5 Plugin拆分原则**：Plugin职责过大时按业务领域拆分，而非按代码数量拆分
- 🟩 **3.0.6 路径命名规范**：文件路径必须清晰表达业务含义
  - 合法：`battle/damage/`、`inventory/equipment/`
  - 非法：`base/`、`core/`
- 🟨 **3.0.7 通用代码规范**：
  - 优先不创建通用顶层目录
  - 如确需创建`common/`，只能存放与业务无关的纯工具代码
  - 🟥 **绝对禁止**：在`common/`中放入任何业务逻辑

## 第四部分：代码组织宪法
- 🟩 **4.0.1 单一文件单一主题**：每个文件只能包含一个明确的业务主题
- 🟩 **4.0.2 按业务拆文件**：必须优先按业务主题拆分文件，**绝对禁止**按技术类型拆分全局文件
  - 🟥 **绝对禁止**：创建全局顶层的`systems.rs`、`components.rs`巨文件
  - 🟥 **绝对禁止**：创建`utils.rs`垃圾桶文件
- ⚠️ **4.0.3 文件大小阈值**
  - 单个文件超过500行且内聚性下降时，AI必须主动提出拆分建议
  - 超过1000行时强制评估拆分方案
  - 禁止机械按行数拆分，内聚性优先于文件大小
  - 规则本质：文件拆分的目的是提升可维护性，而非满足行数指标
- ⚠️ **4.0.4 函数大小阈值**
  - 单个函数超过100行且逻辑可拆解时，AI必须主动提出拆分建议
  - 线性流程的简单逻辑不受行数强制限制
- 🟩 **4.0.5 AI可读性优先**
  - 优先使用直白的线性逻辑，避免宏套宏、深度泛型、类型体操、隐藏控制流
  - 代码可读性优先级：人类开发者 > AI理解 > 编译器优化 > 代码“优雅”
  - 规则本质：AI重度协作项目中，代码可被AI准确理解的价值远高于技巧性写法
- 🟦 **4.0.6 私有函数规范**：
  - 优先使用Rust私有可见性表达边界
  - 不使用`_`前缀命名私有函数
  - 仅在模块内有明确区分价值时才使用`_`前缀

## 第五部分：函数设计宪法
- 🟩 **5.0.1 单一职责**：每个函数只能有一个主要职责
- 🟩 **5.0.2 函数命名规范**：函数名必须描述意图，而非描述实现过程
  - 合法：`calculate_damage()`
  - 非法：`loop_through_modifiers_and_sum_values()`
- 🟩 **5.0.3 控制流规范**：必须优先使用Early Return减少嵌套
- ⚠️ **5.0.4 嵌套深度阈值**：超过3层嵌套时AI必须主动提出重构建议
- 🟩 **5.0.5 抽象原则**：代码重复出现三次以上时才进行抽象
- 🟩 **5.0.6 可读性优先**：必须优先保证代码可读性，其次才考虑复用性

## 第六部分：Trait宪法
- 🟩 **6.0.1 Trait表示能力**：Trait只能用于定义对象具备的能力，**绝对禁止**用于表示分类
- 🟩 **6.0.2 Trait用于扩展点**：Trait只能用于定义需要扩展的接口，**绝对禁止**用于模拟继承树
- 🟩 **6.0.3 Trait抽象原则**：
  - Trait只能解决明确的变化点
  - 重复出现三次以上时才抽象为Trait
  - 🟥 **绝对禁止**：为了"代码优雅"而创建无实际价值的Trait

## 第七部分：SRPG角色系统宪法
- 🟩 **7.0.1 分层扩展体系**
  - 角色数值能力：统一通过Modifier管线实现，所有属性增减必须进入统一计算流程
  - 角色行为扩展：优先使用Trait定义能力接口，由系统统一调度
  - 特殊机制：召唤、传送、复活等非数值效果，允许通过独立系统+组件实现，不强制塞入Modifier
  - 规则本质：不同类型的能力使用最合适的实现方式，避免强行统一导致架构膨胀
- 🟩 **7.0.2 核心定义**
  - 种族 = Trait集合 + Modifier集合
  - 职业 = 成长率表 + 技能池 + Trait集合
  - 天赋 = 特殊Trait + 专属Modifier
  - 装备 = Modifier集合 + Trait集合
  - Buff/Debuff = 临时Modifier + 临时Trait + 生命周期组件
- 🟩 **7.0.3 状态效果统一**：所有数值类状态效果必须走统一的Modifier管线
- 🟩 **7.0.4 Modifier管线统一**：所有影响基础/派生属性的数值来源必须进入统一计算管线
- 🟩 **7.0.5 战斗事件链**
  - 复杂战斗交互必须通过**Trigger + Observer**构建事件链
  - 示例：伤害Trigger → 护盾Observer吸收 → 吸血Observer结算 → 反击Observer触发
  - 规则本质：使用Bevy原生机制实现连锁效果，避免自定义事件体系

- 🟩 **8.0.6 配置不可变**：所有配置型属性数据必须保持不可变
- 🟨 **8.0.7 属性缓存约定**
  - 缓存组件不是事实源，必须允许随时删除并重新生成
  - 属性源变化时必须立即失效对应缓存
  - 规则本质：避免缓存与真实数据不一致导致的Bug

## 第十一部分：SRPG核心系统宪法
### 11.1 回合系统规范
- 🟩 **11.1.1 阶段划分标准化**
  - 回合必须划分为明确阶段：回合开始 → 阶段判定 → 单位行动 → 回合结算 → 回合结束
  - 每个阶段对应独立的系统集与触发点
  - 规则本质：统一时序，避免Buff结算、技能效果的顺序依赖Bug
- 🟩 **11.1.2 状态驱动回合流转**
  - 回合切换通过状态机驱动，禁止手动调用回合切换函数
  - 规则本质：保证回合计数、结算逻辑的一致性，便于回放与测试

### 11.2 技能执行管线
- 🟩 **11.2.1 五阶段固定管线**
  - 所有技能执行必须遵循统一流程：目标校验 → 消耗扣除 → 施法前摇 → 效果执行 → 结果结算
  - 禁止跳过阶段或乱序执行
  - 规则本质：统一技能执行流程，便于扩展、调试与回放
- 🟩 **11.2.2 预览与执行分离**
  - 技能伤害、范围预览必须走独立的只读路径，绝对禁止产生任何副作用
  - 规则本质：保证预览、AI仿真不会修改游戏状态

### 11.3 Buff生命周期规范
- 🟩 **11.3.1 四阶段标准化**
  - 所有Buff必须遵循：Apply（施加）→ Tick（周期触发）→ Expire（到期）→ Remove（移除）四个生命周期阶段
  - 每个阶段有明确的触发时机与Hook点
  - 规则本质：统一Buff行为，避免生命周期混乱导致的残留、叠加Bug

### 11.4 胜利/失败条件
- 🟨 **11.4.1 条件配置化**
  - 战斗胜负条件优先通过配置实现，支持多条件组合
  - 禁止硬编码单一胜负判定逻辑
  - 规则本质：适配多样化的关卡设计，无需修改核心代码

### 11.5 命令层（Command Layer）
- 🟩 **11.5.1 所有操作入口为命令**
  - 玩家输入、AI决策、回放执行、网络同步，最终都必须转换为标准化业务命令
  - 典型命令：`MoveUnitCommand`、`CastSkillCommand`、`EndTurnCommand`
  - 规则本质：所有执行入口统一，天然支持回放、测试、多端输入复用同一套执行链
- 🟩 **11.5.2 命令无差别执行**
  - 执行系统不区分命令来源，只处理命令本身
  - 规则本质：保证真人操作、AI操作、回放操作的结果完全一致

### 11.6 输入抽象层
- 🟩 **11.6.1 输入为独立Feature**
  - 鼠标、键盘、手柄、触摸等输入，统一在input模块转换为业务命令
  - 🟥 绝对禁止业务模块直接读取原始输入按键
  - 规则本质：屏蔽平台差异，未来适配多端时无需修改核心业务逻辑

### 11.7 读写分离原则（CQRS Lite）
- 🟩 **11.7.1 读路径无副作用**
  - 伤害预览、范围查询、状态展示等读操作，必须使用纯函数或只读查询
  - 绝对禁止在读路径中修改状态、触发事件、消耗资源
  - 规则本质：避免预览、仿真、UI刷新影响真实游戏状态
- 🟩 **11.7.2 写路径收口**
  - 所有状态修改必须通过命令与执行系统统一处理
  - 禁止零散分布的状态写入逻辑

### 11.8 仿真优先原则
- 🟩 **11.8.1 核心规则支持离线仿真**
  - 战斗核心规则必须可以脱离Bevy运行时独立执行
  - AI决策、伤害预览、自动测试，均可直接调用同一套规则
  - 规则本质：一次实现，多场景复用，保证逻辑一致性

### 11.9 随机数分层管理
- 🟩 **11.9.1 多RNG流独立**
  - 必须按用途拆分独立随机数流：战斗RNG、掉落RNG、世界RNG、AI RNG
  - 禁止全局共用单一RNG实例
  - 规则本质：避免不同系统的随机结果互相干扰，保证战斗回放的确定性
- 🟩 **11.9.2 统一RNG服务**
  - 业务逻辑禁止直接调用`rand::random()`，必须通过统一的随机数服务获取
  - 规则本质：收口随机源，是确定性与可回放的基础前提

### 11.10 系统职责分类
- 🟨 **11.10.1 五类系统划分**
  - Collector：收集输入、状态变化
  - Validator：校验规则、前置条件
  - Executor：执行业务逻辑、修改状态
  - Projector：将状态投射到UI、特效等表现层
  - Observer：监听事件触发次级响应
  - 规则本质：统一系统定位，AI生成代码时职责更清晰

## 第十二部分：数据驱动与存档宪法
### 12.1 数据驱动核心原则
- 🟩 **12.1.1 职责划分**：配置定义内容，代码解释配置
- 🟩 **12.1.2 配置稳定性**：配置结构的稳定性优先于配置格式的优雅性
- 🟩 **12.1.3 兼容性优先**：配置的向后兼容性优先于任何其他考虑
- 🟩 **12.1.4 资源管理**：所有配置资源必须通过统一的Asset Pipeline管理
- 🟩 **12.1.5 热重载优先**：所有配置必须优先支持热重载
- 🟩 **12.1.6 自动校验**：配置之间的引用关系必须实现自动校验
- 🟩 **12.1.7 效率优先**：内容生产效率优先于配置格式的争论

### 12.2 配置Schema与校验
- 🟩 **12.2.1 所有配置必须有Schema**
  - SkillConfig、BuffConfig、CharacterConfig等所有配置结构体，必须对应明确的Schema定义
  - 规则本质：AI生成配置时有明确的结构依据，避免字段漂移与格式错误
- 🟩 **12.2.2 CI配置Lint**
  - 所有配置必须支持CI自动校验：引用合法性、数值范围、循环依赖
  - 校验不通过禁止合并入主分支
  - 规则本质：提前发现配置错误，避免运行时崩溃

### 12.3 数据所有权原则
- 🟩 **12.3.1 唯一事实源**
  - 每份配置数据必须有唯一的归属Feature与维护入口
  - 禁止多个Feature同时维护同一份核心数据
  - 示例：CharacterDef归属character模块，SkillDef归属skill模块
  - 规则本质：避免多源修改导致的数据不一致

### 12.4 数值平衡隔离
- 🟩 **12.4.1 平衡参数全配置化**
  - 所有数值平衡参数（暴击倍率、地形加成、系数常量）必须放入配置文件
  - 🟥 绝对禁止在代码中硬编码魔法数字
  - 规则本质：数值调整无需修改代码，便于平衡与热更新

### 12.5 字段删除与兼容性
- 🟩 **12.5.1 三步删除原则**
  - 核心配置字段、存档字段禁止直接删除
  - 必须遵循：标记Deprecated → 新增迁移逻辑 → 下个大版本移除
  - 规则本质：保证配置与存档的向后兼容

### 12.6 存档版本与迁移
- 🟩 **12.6.1 强制版本字段**
  - 所有存档文件必须包含显式的version字段
  - 规则本质：是存档迁移与兼容的基础前提
- 🟩 **12.6.2 向后兼容原则**
  - 新版本必须支持读取上一个大版本的存档，通过自动迁移完成升级
  - 🟥 禁止版本升级后旧存档直接无法读取
  - 规则本质：长期连载游戏的核心体验保障

## 第十三部分：可观测性宪法（日志 + 错误 + 调试）
### 13.1 日志框架与核心定位
- 🟩 **13.1.1 日志框架统一**：必须统一使用`tracing`库进行日志记录
  - 🟥 **绝对禁止**：使用`println!`、`dbg!`输出运行时信息
  - 规则本质：统一日志收集、过滤、格式化能力，支持结构化输出与动态级别调整
- 🟩 **13.1.2 日志核心定位：领域事件履历**
  - 日志必须记录业务事件事实，**绝对禁止**记录技术执行流水
    - 合法：`UnitAttacked`、`BuffApplied`、`TurnEnded`
    - 非法：`AttackSystem running`、`function entered`、`query finished`
  - 一个完整业务动作最多输出一条INFO级别日志；DEBUG / TRACE级可按需细化，但发布版默认关闭
  - 规则本质：让日志可阅读、可检索、可还原业务流程，而非调试代码的流水账

### 13.2 日志分级规范
- 🟥 **ERROR**：理论上不应发生的程序异常
  - 适用场景：Entity丢失、配置缺失、非法状态、存档损坏
  - 强制要求：必须附带完整复现上下文
- 🟩 **WARN**：可继续运行的异常情况
  - 适用场景：配置不完整、数据异常、回退逻辑触发
- 🟩 **INFO**：核心业务事件边界
  - 适用场景：战斗开始/结束、回合开始/结束、单位移动/攻击/死亡、Buff施加/移除、任务完成
- 🟦 **DEBUG**：开发调试辅助信息
  - 适用场景：寻路路径长度、Modifier计算过程、资源加载详情
  - 发布版默认关闭
- 🟦 **TRACE**：极细粒度算法细节
  - 适用场景：A*节点探索、单帧循环内部逻辑
  - 仅专项调试时临时开启

### 13.3 结构化日志强制要求
- 🟩 **13.3.1 标准字段**：所有INFO级别日志必须携带`event`字段，值与事件名完全一致
  - 示例：`info!(event = "UnitDamaged", attacker=?unit_id, target=?unit_id, damage=25, "unit received damage");`
- 🟩 **13.3.2 Target规范**：日志`target`必须与所属Feature目录名完全一致
  - 示例：battle模块日志target = "battle"，buff模块日志target = "buff"
  - 规则本质：支持通过`RUST_LOG=buff=debug`精准过滤单一领域日志

### 13.4 日志禁令
- 🟥 绝对禁止记录函数进入、退出、系统执行等技术流水账
- 🟥 Release版本绝对禁止在每帧执行的系统中输出INFO / DEBUG级别日志（仅允许ERROR）
- 🟥 绝对禁止在循环、迭代器内部输出INFO级别日志
- 🟥 绝对禁止堆砌日志替代专业调试工具
- 🟥 绝对禁止业务代码直接调用`info!`输出核心业务事件（必须走领域事件链路）

### 13.5 战斗日志分离
- 🟩 游戏内战斗履历使用独立的事件体系，不依赖tracing
  - 用途：UI战斗记录、Battle Replay、录像回放、Bug复现
- 🟩 tracing仅负责程序运行时的可观测性，不承担玩家可见的日志功能
- 规则本质：领域事件资产与运维工具解耦，各自独立演进

### 13.6 错误处理与日志
- 🟩 所有ERROR级别日志必须包含完整的上下文信息
  - 禁止只输出"operation failed"，必须附带所有相关实体ID、状态与参数
- 🟩 可失败逻辑优先返回Result，错误由上层统一记录，避免层层重复打日志

### 13.7 调试工具与可观测性
- 🟩 **13.7.1 调试工具优先**：
  - 必须优先使用Inspector、Replay、Debug Panel进行问题排查
  - 🟥 **绝对禁止**：通过堆砌临时日志定位问题
  - 规则本质：可视化调试工具的效率远高于文本日志
- 🟨 **13.7.2 可调试性要求**
  - 核心战斗系统优先支持单步执行与状态回溯
  - 所有系统的执行顺序必须可观察
  - 复杂系统优先配备可视化观察窗口

### 13.8 日志架构规范（领域事件驱动）
- 🟩 **13.8.1 核心架构模式**：日志是领域事件的消费者，而非业务代码主动调用的功能
  - 链路：**领域事件触发 → 统一Log Observer监听 → 输出tracing日志**
  - 规则本质：所有业务日志统一收口，一次修改全量生效，支持未来无限扩展下游
- 🟩 **13.8.2 强制适用范围**：所有INFO级别的核心业务事件，必须通过触发领域事件的方式生成日志
  - 复用现有业务事件，**绝对禁止**为了打日志单独创建事件
  - 日志Observer统一放在基础设施层，绝对不侵入业务模块
- 🟩 **13.8.3 例外范围**：以下场景可直接调用tracing宏，不受此限
  - ERROR / WARN 级别异常日志
  - DEBUG / TRACE 级别调试日志
  - 基础设施层、工具层代码
  - 测试代码
- 🟩 **13.8.4 扩展一致性**：Battle Replay、战斗履历UI、成就系统、任务系统与日志共用同一套领域事件源

### 13.9 错误体系规范
#### 13.9.1 分领域错误原则
- 🟩 **每个领域定义独立错误枚举**
  - 技能领域使用`SkillError`、战斗领域使用`BattleError`、Buff领域使用`BuffError`，依此类推
  - 🟥 **绝对禁止**：使用全局统一的`AppError`大枚举、`anyhow::Error`、`Box<dyn Error>`作为业务层返回错误类型
  - 规则本质：错误自带领域上下文，AI可天然识别边界，避免单一错误枚举无限膨胀
- 🟨 **基础设施层薄封装**
  - 可在公共基础设施层定义`GameResult<T>`类型别名与通用错误转换Trait
  - 基础设施层仅收纳通用基础设施错误，绝对不包含任何业务错误变体

#### 13.9.2 失败分类学（强制区分）
必须严格区分四类失败场景，不得混用：
1. **规则失败（RuleFailure）**：业务规则的正常不满足，不属于程序错误
   - 示例：法力不足、目标超出范围、冷却未结束
   - 实现方式：使用专门的结果枚举表达，**禁止**用`Result::Err`返回
2. **领域错误（DomainError）**：领域内预期内的异常，理论上不应发生
   - 示例：技能配置不存在、BuffID无效、单位未找到
   - 实现方式：对应领域错误枚举的变体，用`Result::Err`返回
3. **基础设施错误（InfrastructureError）**：底层通用能力异常
   - 示例：资源加载失败、存档IO错误、序列化失败
4. **程序Bug（Bug）**：非法状态、逻辑断言失败，属于代码缺陷
   - 示例：状态机非法跳转、数据一致性破坏
- 规则本质：回归Result的本义，避免将正常业务失败当成错误处理，提升代码语义清晰度

#### 13.9.3 错误强制要求
- 🟩 所有错误必须携带完整上下文信息
  - 示例：`SkillError::InvalidTarget { caster: UnitId, target: UnitId, skill: SkillId }`
  - 🟥 **绝对禁止**：仅返回无上下文的错误变体（如`InvalidTarget`）
- 🟩 推荐使用带编号的错误码
  - 格式：领域前缀+序号，如`S001`（技能类1号错误）、`B003`（Buff类3号错误）
  - 规则本质：配合日志可快速定位问题，便于AI、测试、回放系统识别

#### 13.9.4 业务层Panic禁令
- 🟥 **绝对禁止**：在battle、skill、buff、character、quest、turn等核心业务领域代码中使用`unwrap()`、`expect()`、`panic!()`
- 仅允许使用场景：测试代码、工具代码、编辑器代码、原型验证代码
- 规则本质：保证核心战斗逻辑的稳定性，避免运行时意外崩溃

### 13.10 领域事件与审计轨迹
- 🟩 **13.10.1 事件统一事实源**
  - 所有核心业务事实必须通过领域事件表达，日志、战斗回放、UI战斗履历、成就系统、任务系统均监听同一事件源
  - 禁止为不同下游重复生成同类业务记录
- 🟩 **13.10.2 事件白名单**
  - 所有正式领域事件必须收录在白名单文档中
  - 核心事件包括但不限于：BattleStarted/BattleEnded、TurnStarted/TurnEnded、UnitMoved/UnitAttacked/UnitDamaged/UnitDied、BuffApplied/BuffRemoved、SkillCastStarted/SkillCastFinished、QuestAccepted/QuestCompleted
- 🟩 **13.10.3 审计轨迹（Audit Trail）**
  - 核心战斗流程必须生成结构化的审计轨迹，作为所有可观测性能力的底层数据
  - 用途：战斗回放、录像、Bug复现、自动化测试、存档校验、数值平衡分析
  - 规则本质：结构化事件流是比文本日志更高阶的可观测性形态，支撑长期工具链建设

## 第十八部分：测试与确定性宪法
### 18.1 测试核心原则
- 🟩 **18.1.1 测试优先**：所有功能必须优先编写测试，其次才是手工验证
- 🟩 **18.1.2 Bug修复流程**：发现Bug后必须先编写重现测试，再修复Bug

### 18.2 测试金字塔
- 🟩 **18.2.1 三层测试体系**
  - 单元测试：验证领域纯函数、核心规则（伤害公式、Buff结算、寻路算法）
  - 集成测试：验证完整Feature的端到端行为（装备系统、技能系统、战斗流程）
  - 回放测试：通过Battle Replay验证完整战斗场景的确定性
  - 规则本质：不同层级测试各司其职，用最低成本覆盖最多风险
- 🟩 **18.2.2 SRPG测试优先**：复杂SRPG逻辑必须优先使用Battle Replay测试，而非BDD

### 18.3 测试基础设施
- 🟩 **18.3.1 测试数据工厂**
  - 核心测试必须使用Builder模式构造测试数据（TestCharacterBuilder、TestBattleBuilder）
  - 禁止每个测试手动构造大量实体与组件
  - 规则本质：降低测试编写成本，保证测试数据一致性
- 🟩 **18.3.2 Golden Test（金文件测试）**
  - 战斗回放、配置导出等稳定输出，必须使用金文件对比测试
  - 版本升级后输出变化必须显式确认
  - 规则本质：快速发现非预期的逻辑变更，保护核心玩法稳定性

### 18.4 确定性要求
- 🟩 **18.4.1 战斗完全可重现**
  - 相同初始状态 + 相同输入序列 + 相同RNG种子，必须得到完全一致的战斗结果
  - 规则本质：是Battle Replay、Bug复现、自动化测试的核心基础
- 🟩 **18.4.2 禁止业务逻辑依赖系统时间**
  - 核心战斗逻辑禁止直接读取系统时间，必须使用统一的GameTime服务
  - 规则本质：保证仿真与回放的时间可控
- 🟩 **18.4.3 战斗Bug规范**：所有战斗相关Bug必须通过Battle Replay重现并转化为永久测试用例
- 🟩 **18.4.4 覆盖原则**：测试必须覆盖所有核心规则，不追求表面的覆盖率数字
- 🟩 **18.4.5 资产沉淀**：所有修复的Bug最终都必须沉淀为测试资产

## 第二十部分：工程质量与技术债治理宪法
### 20.1 核心原则
- 🟩 **20.1.1 质量预算原则**
  - Warning Budget = 0：主分支不允许存在未处理的编译警告
  - Bug Budget = 0：P0/P1级缺陷不允许流入主分支
  - Tech Debt Budget = 可控：技术债必须登记跟踪，设定偿还节点，禁止无限制积累
  - 规则本质：在项目早期以极低成本建立质量门禁，避免规模扩大后质量失控
- 🟩 **20.1.2 分级治理原则**：所有问题按优先级分级处理，既守住核心底线，又避免过度教条影响开发效率

### 20.2 问题分级标准
#### 20.2.1 P0 致命级（必须立即修复，禁止提交/合并）
- 🟥 处理规则：发现即修，任何情况下不得进入主分支
- 典型问题：
  1. 核心业务代码出现`unwrap()`、`expect()`、`panic!()`、`todo!()`、`unimplemented!()`
  2. ECS系统Query Borrow冲突，影响调度并行性与正确性
  3. 事件/Observer/Trigger触发无限循环，导致程序卡死
  4. Entity泄漏：实体生成后无对应销毁逻辑，长期运行资源持续增长
  5. 存档兼容性损坏：旧版本存档无法在新版本读取
  6. 核心数据损坏、状态机非法跳转

#### 20.2.2 P1 高优先级（一个迭代内必须修复）
- 🟩 处理规则：当前Sprint内必须修复，CI门禁直接拦截
- 典型问题：
  1. Rust编译警告（unused_imports、unused_variables、dead_code、deprecated等必修项）
  2. 已废弃API调用
  3. Clippy必修项警告
  4. 配置数据校验失败：技能/Buff/地图/任务引用不存在的资源
  5. 跨模块边界违规、架构约束被破坏

#### 20.2.3 P2 中优先级（允许短期存在，必须登记跟踪）
- 🟨 处理规则：记录入技术债台账，达到阈值或对应模块重构时统一处理
- 典型问题：
  1. 非热点代码存在可优化的性能问题（如O(n²)复杂度但当前数据量极小）
  2. ECS Archetype频繁抖动：短时间内反复insert/remove组件
  3. 非核心路径存在过度日志、日志噪音
  4. Archetype数量膨胀风险

#### 20.2.4 P3 低优先级（技术债，统一偿还）
- ⚠️ 处理规则：登记记录，在版本迭代、架构复盘时集中清理
- 典型问题：
  1. 局部重复代码，未达到抽象阈值
  2. 命名风格不统一，不影响功能与可读性
  3. 单文件体积偏大但内聚性良好，暂不影响维护
  4. 可补充的文档注释

### 20.3 Rust Warning 分级规范
- 🟩 **20.3.1 必修清理项（P1级，必须修复）**
  - `unused_imports`、`unused_mut`、`unused_variables`
  - `dead_code`、`unreachable_code`
  - `deprecated`、`unused_must_use`
  - 规则本质：此类警告大概率隐含真实问题，放任积累会快速降低代码库质量
- 🟨 **20.3.2 可暂缓项（P3级，可后补）**
  - `missing_docs`：非公开类型可暂缓，核心公共接口建议补充
- 🟦 **20.3.3 可豁免项**
  - 测试代码、工具代码、原型代码中，可通过`#[allow(dead_code)]`等属性豁免，无需修复

### 20.4 Clippy 检查分级规范
- 🟩 **20.4.1 必修项（P1级，必须修复）**
  - `clone_on_copy`、`redundant_clone`、`needless_collect`
  - `needless_return`、`manual_map`、`map_clone`
  - `large_enum_variant`、`redundant_field_names`
  - `await_holding_lock`
  - 规则本质：此类问题多为AI生成代码高频出现的冗余写法，修复成本极低，收益明确
- 🟨 **20.4.2 建议修项（P2级，按需处理）**
  - `too_many_arguments`：结合业务场景判断，战斗效果执行等固有多参数场景可豁免
  - 禁止机械修改，避免为了满足规范破坏业务语义

### 20.5 Bevy 专项检查
- 🟩 **20.5.1 必修检查项（P1级，必须监控与修复）**
  - Commands滥用：禁止每帧高频生成/销毁大量实体
  - 事件洪泛：禁止单帧触发大量无意义事件
  - Observer/Trigger递归：事件链必须设置递归深度限制，防止循环触发
  - Archetype爆炸监控：关注组件组合导致的Archetype数量膨胀
  - 规则本质：聚焦Bevy真实性能瓶颈，而非教条化指标

### 20.6 AI协作专属规则
- 🟩 **20.6.1 提交代码禁令**
  - 🟥 禁止在业务领域代码中提交`todo!()`、`unimplemented!()`
  - 🟥 禁止在业务领域代码中提交`unwrap()`、`expect()`
- 🟩 **20.6.2 交付物要求**
  - 新增核心业务系统必须至少附带1个基础测试用例
  - 新增领域事件必须同步更新领域事件白名单文档
  - 新增的编译警告、Clippy警告必须同步处理，无法处理的必须标注豁免并说明理由
- 🟩 **20.6.3 豁免规范**
  - 所有`#[allow(...)]`属性必须附带注释说明豁免理由与有效期
  - 禁止无理由批量屏蔽警告与Clippy检查

### 20.7 CI 门禁强制标准
- 🟩 **20.7.1 主分支准入标准**
  - `cargo fmt` 0格式问题
  - `cargo clippy` 0必修项警告
  - `cargo test` 全部测试通过
