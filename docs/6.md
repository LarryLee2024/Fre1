# 第一铁律：日志服务于定位问题，不服务于记录一切

坏例子：

```rust
info!("Update");
info!("Update");
info!("Update");
info!("Update");
```

---

运行一分钟：

```text
200万行日志
```

没人会看。

---

好例子：

```rust
info!(
    unit=?unit_id,
    hp_before,
    hp_after,
    "Damage Applied"
);
```

---

日志必须回答：

```text
发生了什么
为什么发生
影响了谁
```

---

# 第二铁律：统一使用 tracing

Bevy 0.18 已经全面建立在：

```rust
tracing
```

之上。

不要：

```rust
println!()
dbg!()
eprintln!()
```

---

统一：

```rust
use tracing::*;

info!();
warn!();
error!();
debug!();
trace!();
```

---

例如：

```rust
debug!(
    attacker=?attacker,
    target=?target,
    damage,
    "Attack resolved"
);
```

---

# 第三铁律：所有日志必须结构化

不要：

```rust
info!(
    "Player {} attacked {} for {} damage",
    player,
    target,
    damage
);
```

---

要：

```rust
info!(
    attacker=?player,
    target=?target,
    damage,
    "Attack resolved"
);
```

---

因为以后：

```text
ELK
Grafana
Tracy
Chrome Trace
```

都能直接分析字段。

---

# 第四铁律：业务日志必须带Entity

例如：

```rust
info!(
    unit=?entity,
    "Unit died"
);
```

---

不要：

```rust
info!("Knight died");
```

---

因为以后：

```text
Knight
Knight
Knight
Knight
```

可能有20个。

---

推荐：

```rust
info!(
    entity=?entity,
    unit_id=%unit.id,
    "Unit died"
);
```

---

# 第六铁律：Message必须可追踪

Bevy 0.18：

```text
Message
Observer
```

越来越重要。

---

所以：

```rust
message_writer.write(
    UnitDiedMessage {
        entity,
    }
);
```

最好：

```rust
trace!(
    entity=?entity,
    "Message: UnitDied"
);
```

---

否则：

```text
消息发没发？
谁消费了？
```

很难查。

---

# 第七铁律：Observer必须可追踪

例如：

```rust
commands.trigger(
    UnitKilledEvent {
        killer,
        victim,
    }
);
```

---

Observer：

```rust
fn on_unit_killed(
    trigger: Trigger<UnitKilledEvent>,
)
```

---

建议：

```rust
trace!(
    killer=?trigger.killer,
    victim=?trigger.victim,
    "Observer: UnitKilled"
);
```

---

否则：

```text
事件触发了

Observer没响应

还是Observer报错
```

根本看不出来。

---

# 第八铁律：不要记录每帧日志

禁止：

```rust
fn update() {
    info!("Running");
}
```

---

60FPS：

```text
一分钟3600条
```

---

改：

```rust

```

或者：

```rust

    debug!()
}
```

# 最终版（我会写进你的SRPG规范）

* **统一使用 tracing，不使用 println!/dbg!**：Bevy 0.18 生态已经全面围绕 tracing 构建。
* **日志必须结构化**：记录字段，不记录拼接字符串，例如 `attacker=?entity, damage=50` 优于 `"Knight dealt 50 damage"`。
* **业务日志必须带 Entity 与业务ID**：例如 `entity=?entity, unit_id=%unit_id`。
* **Message、Observer、Hook 必须可追踪**：重要状态流必须能看到从发送到响应的完整链路。
* **不要记录每帧日志**：Update 系统日志默认使用 `trace!` 或状态变化触发。
* **按 Feature 设置 target**：例如 `battle`、`character`、`inventory`、`quest`。
* **Error 必须包含上下文**：记录谁失败、为什么失败、影响什么。
* **日志记录状态变化，不记录函数执行**：关注“HP变化”“Buff添加”，不要关注“进入函数A”。
* **复杂系统必须有可视化调试工具**：Inspector、AI Viewer、Buff Viewer、Grid Viewer 比日志更重要。
* **战斗回放记录优先级高于普通日志**：BattleRecord 最终会成为回放、同步、复盘和排查Bug的核心基础设施。

对于你的项目，我甚至会再加一句：

> **能做成 BattleRecord 的信息，不要只写进 Log；Log 会消失，而 BattleRecord 会成为未来整个战斗系统的基础资产。** 这条在战棋项目里价值极高。
