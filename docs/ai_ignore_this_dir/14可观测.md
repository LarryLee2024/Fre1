这是一个非常好的问题。

很多项目做到 5~10 万行代码时还觉得：

```rust
info!()
warn!()
error!()

metrics::counter!()

tracing::instrument()
```

挺简单。

但到了 **30~50万行代码**，真正的问题已经不是「怎么打日志」，而是：

> **如何让整个项目只有一种观测方式（Single Source of Observability）**

否则后期会出现：

```rust
fn execute_ability() {
    info!(...);

    metrics::record(...);

    analytics::record(...);

    audit::record(...);

    replay::record(...);

    telemetry::record(...);
}
```

整个项目到处都是这种东西。

---

# 一、什么是 Observability Layer

本质上：

```text
业务代码
    ↓
Observability Layer
    ↓
─────────────────
Log
Metric
Trace
Audit
Replay
Profiling
─────────────────
```

业务代码：

```rust
emit_ability_activated(...)
```

而不是：

```rust
info!(...)
counter!(...)
audit!(...)
```

---

# 二、游戏项目最容易犯的错误

很多项目：

```rust
Combat Domain
    -> info!

Ability Domain
    -> metrics!

Buff Domain
    -> tracing!

AI Domain
    -> audit!
```

每个领域自己决定记录什么。

结果：

```text
风格不统一
字段不统一
Code 不统一
统计不统一
```

几年后没人敢改。

---

# 三、核心原则：业务事件优先

不要记录：

```rust
info!("进入函数");
info!("开始计算");
info!("结束计算");
```

而是记录：

```rust
AbilityActivated
DamageApplied
BuffAdded
TurnStarted
BattleEnded
```

即：

```text
Domain Event
而不是
Code Event
```

这是最重要的一条。

# 四、统一事件模型

例如：

---

每个事件都有固定结构：

```rust
pub struct EventMeta {
```

---

例如：

```rust
pub struct AbilityActivatedEvent {

}
```

---

整个项目都统一。



```rust

```

```rust
counter!("damage.total");
```

业务代码根本不知道。

# 十一、最成熟的架构

对于你的项目（Bevy + DDD + SRPG + 50万行代码），我最终会变成：

```text
src/shared/telemetry/
│
├── event.rs
├── code.rs
├── domain.rs
├── sink.rs
│
├── logger_sink.rs
├── metric_sink.rs
├── replay_sink.rs
├── audit_sink.rs
│
└── facade.rs
```

业务代码：

```rust
telemetry.emit(
    AbilityActivated {
        caster,
        spec_id,
    }
);
```

永远不要直接：

```rust
info!()
counter!()
audit!()
```

---

这是大型项目里最核心的一条经验：

> **不要让业务代码知道日志、指标、审计、回放的存在。**

业务代码只产生 **领域事件（Domain Event）**。

Observability Layer 决定这些事件如何被记录、统计、追踪、回放。

当项目从 5 万行长到 50 万行时，这种设计的价值会越来越大，而不是越来越小。

看了你的目录，我觉得你现在最大的优势是：

```text
src/
├── app/
├── content/
├── core/
├── infra/
├── modding/
├── shared/
├── tools/
├── ui/
```

已经有比较明确的：

```text
Core Domain
Infrastructure
Shared Kernel
Application
```

这意味着你根本不需要再新建一个独立的大型 `telemetry` 顶层模块。

很多项目会变成：

```text
src/
├── telemetry/
├── logging/
├── metrics/
├── tracing/
```

最后又形成新的横向耦合层。

对于你的架构，我更推荐：

```text
Observability = Infra能力
```

而不是：

```text
Observability = 新领域
```

# 最终我认为最适合你项目的结构

```text
core/
├── domains/
├── events/
│   ├── ability.rs
│   ├── combat.rs
│   ├── buff.rs
│   └── event_code.rs

shared/
└── diagnostics/
    ├── observable.rs
    ├── telemetry_level.rs
    ├── metric_name.rs

infra/
├── observability/
│   ├── facade.rs
│   ├── sink.rs
│   ├── log_sink.rs
│   ├── metric_sink.rs
│   ├── replay_sink.rs
│   └── audit_sink.rs

├── logging/
├── replay/
```

然后整个50万行项目遵守一个铁律：

```text
Core 产生 Domain Event

Observer 调用 Observability Facade

Observability Facade 分发到
    Log
    Metric
    Replay
    Audit

业务代码禁止直接调用
    tracing
    metrics
    replay
```

这是我认为最符合你当前目录、DDD边界以及未来 50 万行 SRPG 项目的 Observability Layer 落地方案。它利用了你已经存在的 `infra/replay` 和 `shared/diagnostics`，而不是再创造一套新的横切体系。
