这个问题其实触及到了：

> **Godot 的“场景树思想”如何映射到 Bevy 的 ECS 思想？**

对于地图系统来说，很多 Godot 开发者会下意识寻找：

```text
TileMapLayer
NavigationRegion2D
YSort
ParallaxBackground
Area2D
Marker2D
```

在 Bevy 里对应什么。

实际上：

```text
Godot
=
节点组织世界

Bevy
=
数据组织世界
```

所以不要试图 1:1 复制 TileMapLayer。

---

# 先看 Godot TileMapLayer 本质

假设一个战棋地图：

```text
地形层
障碍层
装饰层
阴影层
事件层
出生点层
导航层
```

Godot 会做：

```text
Map
├── GroundLayer
├── ObstacleLayer
├── DecorationLayer
├── EventLayer
├── SpawnLayer
└── NavigationLayer
```

每层都是：

```text
TileMapLayer
```

非常直观。

---

# 如果直接搬到 Bevy

很多人会写：

```rust
GroundLayer

ObstacleLayer

DecorationLayer
```

然后：

```rust
Entity
```

管理一堆 Tile。

结果：

```text
10万Tile
```

变成：

```text
10万个Entity
```

性能和管理都会很难受。

---

# Bevy最佳实践

地图不要以 Tile 为中心。

而要以：

```text
Chunk
```

为中心。

---

# 第一层：MapDefinition

类似 Godot 场景。

```yaml
map_id: village

width: 128
height: 128

layers:
  - terrain
  - obstacle
  - decoration
  - event
```

---

运行时：

```rust
MapDefinition
```

---

# 第二层：MapRuntime

```rust
MapRuntime {
    width,
    height,

    terrain,

    obstacle,

    event,
}
```

---

这里的数据结构一般是：

```rust
Vec<TileData>
```

或者：

```rust
Grid<T>
```

而不是 Entity。

---

例如：

```rust
terrain[x][y]
```

直接访问。

---

# Tile 本身最好不是 Entity

不要：

```rust
Tile Entity
Tile Entity
Tile Entity
```

---

而是：

```rust
TerrainGrid
```

```rust
Vec<TerrainTile>
```

---

例如：

```rust
Grass
Water
Mountain
Road
```

---

查询：

```rust
terrain.get(pos)
```

即可。

---

# 第三层：Chunk

大型地图必须有。

例如：

```text
256×256
```

地图。

拆：

```text
16×16
```

一个 Chunk。

---

形成：

```text
Map

├── Chunk0
├── Chunk1
├── Chunk2
```

---

类似：

```rust
HashMap<ChunkCoord, Chunk>
```

---

这样：

```text
超大地图
```

仍然可管理。

---

# Layer 如何做？

Godot：

```text
TileMapLayer
```

---

Bevy 推荐：

```rust
pub struct TerrainLayer;

pub struct ObstacleLayer;

pub struct DecorationLayer;

pub struct EventLayer;
```

---

但不要是 Entity。

而是：

```rust
MapResource
```

中的数据。

例如：

```rust
Map {
    terrain_grid,
    obstacle_grid,
    event_grid,
}
```

---

# 战棋地图最推荐结构

```rust
Map
│
├── TerrainGrid
├── HeightGrid
├── OccupancyGrid
├── EventGrid
├── RegionGrid
└── SpawnGrid
```

---

# OccupancyGrid（非常重要）

这是 Godot TileMap 没有强调的。

---

记录：

```text
哪个格子有人
哪个格子有箱子
哪个格子有机关
```

---

例如：

```rust
occupancy[(5,6)]
```

返回：

```rust
Some(Entity)
```

---

这样：

```text
寻路
攻击判定
技能范围
```

都很快。

---

# EventLayer

不要用节点。

---

而是：

```rust
HashMap<GridPos, EventId>
```

---

例如：

```rust
(10,20)
=
Treasure001
```

---

角色踩上去：

```rust
TriggerEvent
```

---

# SpawnLayer

Godot会画：

```text
PlayerSpawn
EnemySpawn
```

图块。

---

Bevy推荐：

```yaml
spawns:
  - player_knight
    pos: [10,5]

  - enemy_goblin
    pos: [20,12]
```

---

加载地图时：

```rust
spawn_character()
```

---

# 导航层

不要保存。

运行时生成。

---

例如：

```rust
MovementGrid
```

来源：

```text
Terrain
+
Obstacle
+
Occupancy
```

---

实时计算。

---

# Bevy ECS中的地图Entity

通常只需要：

```rust
MapEntity
```

一个。

---

挂：

```rust
CurrentMap
MapMetadata
```

---

真正数据：

```rust
MapResource
```

存储。

# 如果想获得 Godot Container 的体验

其实要自己封装。

---

例如：

```rust
MapBuilder
```

---

像这样：

```rust
MapBuilder::new()
    .terrain_layer(...)
    .obstacle_layer(...)
    .event_layer(...)
    .spawn_layer(...)
```

---

或者：

```rust
map! {
    terrain: ...
    obstacle: ...
    events: ...
}
```

DSL。

---

体验会非常接近：

```text
Godot TileMapLayer
```

# 对《铃兰之剑》类战棋的推荐结构

我会这样拆：

```text
map/

├── definitions/
│
│   ├── map_definition.rs
│   ├── terrain_definition.rs
│   └── event_definition.rs
│
├── runtime/
│
│   ├── terrain_grid.rs
│   ├── occupancy_grid.rs
│   ├── event_grid.rs
│   ├── region_grid.rs
│   └── height_grid.rs
│
├── rendering/
│
│   ├── chunk_renderer.rs
│   ├── tile_renderer.rs
│   └── visibility.rs
│
├── systems/
│
│   ├── load_map.rs
│   ├── unload_map.rs
│   └── update_chunks.rs
│
└── plugin.rs
```

然后把地图理解成：

```text
Godot
=
TileMapLayer组织地图

Bevy
=
Grid数据组织地图
```

这是最关键的思想转变。

对于类似《铃兰之剑》的项目，我甚至会进一步采用：

```text
MapDefinition
+
Chunk
+
Grid
+
OccupancyGrid
+
EventGrid
```

作为底层，而不是把每个 Tile 都变成 ECS Entity。这样后面加地形效果、高低差、机关、天气、区域Buff、战争迷雾时，扩展性会远远好于传统 TileMapLayer 思路。
