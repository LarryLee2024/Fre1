# 六、更高级的方案（推荐）

对于你的项目。

我甚至不建议：

```rust
Name::new("Knight_001")
```

---

而是：

```rust
#[derive(Component)]
pub struct UnitId(pub String);

#[derive(Component)]
pub struct UnitName(pub String);
```

---

然后：

```rust
commands.spawn((
    UnitId("knight_001".into()),
    UnitName("艾琳".into()),
    Name::new("Knight"),
));
```

---

这样：

Inspector

```text
Knight
```

---

日志：

```text
knight_001
```

---

UI：

```text
艾琳
```

---

三者职责完全分离。

这是大型项目最佳实践。

# 如果写入你的铁律

我会增加一条：

| 铁律                    | 说明                           |
| --------------------- | ---------------------------- |
| **Name用于调试，不用于业务逻辑**  | Name服务Inspector、日志、开发工具      |
| **重要实体必须命名，海量实体不要命名** | 角色、UI根节点、系统根实体推荐Name         |
| **业务ID与Name分离**       | UnitId、SkillId、QuestId负责业务身份 |
| **显示名称与业务ID分离**       | UnitName用于本地化与UI展示           |
| **不要通过Name查找实体**      | ECS查询依赖Component，不依赖字符串名称    |

对于你的SRPG项目，我会给 Name 的使用等级：

| 类型          | 建议        |
| ----------- | --------- |
| 主角/NPC/BOSS | ★★★★★ 必加  |
| UI根节点       | ★★★★★ 必加  |
| Debug实体     | ★★★★★ 必加  |
| 系统Root实体    | ★★★★☆ 推荐  |
| Tile        | ★☆☆☆☆ 不推荐 |
| Projectile  | ★☆☆☆☆ 不推荐 |
| 粒子特效        | ☆☆☆☆☆ 禁止  |
| 飘字          | ☆☆☆☆☆ 禁止  |

这样既保留 Inspector 的可读性，又不会把 Name 滥用成 Unity 风格的“对象名称系统”。
