对于你的项目，我建议：

```text
shared/
└── error/
```

而不是：

```text
core/error/
```

也不是：

```text
infrastructure/error/
```

---

原因是很多人会犯一个认知错误：

```text
Error = Infrastructure
```

实际上不对。

错误横跨所有层。

---

# 如果放 core

例如：

```text
core/
└── error/
```

会产生一个问题：

```text
core 依赖错误
错误又依赖各种领域
```

很容易变成：

```rust
GameError
├── SkillError
├── BuffError
├── BattleError
├── AssetError
├── SaveError
```

然后：

```text
core
↓
依赖所有东西
```

最后形成循环依赖倾向。

---

# 如果放 infrastructure

例如：

```text
infrastructure/
└── error/
```

问题更大。

因为：

```rust
SkillError
BattleError
BuffError
```

根本不是基础设施错误。

它们属于领域。

---

例如：

```rust
pub enum SkillError {
    SkillNotFound,
    InvalidTarget,
}
```

这是：

```text
Domain Error
```

不是：

```text
Infrastructure Error
```

---

# DDD里面实际上有三种错误

## 第一层

领域错误

放领域内部。

```text
skill/
└── skill_error.rs

buff/
└── buff_error.rs

battle/
└── battle_error.rs
```

---

例如：

```rust
pub enum SkillError {
    SkillNotFound,
    InvalidTarget,
}
```

---

# 第二层

基础设施错误

放 infrastructure。

```text
infrastructure/
└── persistence/
    └── save_error.rs
```

---

例如：

```rust
pub enum SaveError {
    FileNotFound,
    DeserializeFailed,
}
```

---

```text
assets/
└── asset_error.rs
```

---

例如：

```rust
pub enum AssetError {
    MissingAsset,
    InvalidAsset,
}
```

---

# 第三层

共享错误工具

放 shared。

```text
shared/
└── error/
```

---

例如：

```rust
pub type GameResult<T> = Result<T, GameError>;
```

---

或者：

```rust
pub trait ErrorExt
```

---

或者：

```rust
pub trait LogIfError
```

---

# 还有一个更符合你项目的建议

对于你现在的阶段：

```text
Character
Skill
Buff
Turn
Victory
```

主链刚打通。

我甚至建议：

```text
不要先设计 GameError
```

---

直接：

```rust
Result<T, SkillError>

Result<T, BattleError>

Result<T, BuffError>
```

即可。

---

因为：

```text
GameError
```

通常是在：

```text
存档
联网
编辑器
热更新
任务系统
回放系统
```

开始出现以后才真正需要。

---

如果让我给你的项目定最终规则：

```text
Domain Error
    放领域内部

Infrastructure Error
    放基础设施内部

共享Result/工具Trait
    放 shared/error

不要搞一个超级 AppError
```

这是我见过对 Rust + DDD + Bevy 项目最不容易后期失控的方案。

这个问题其实是在决定你项目未来 3~5 年的演化方向。

很多 Bevy 项目最后变乱，不是 ECS 的问题，而是：

```text
不知道什么东西该放哪里
```

导致：

```text
core/
    变成垃圾桶

shared/
    变成垃圾桶

common/
    变成垃圾桶
```

最后全项目互相引用。

# 如果是你的项目，我最终会采用

```text
project/

├── src/
├── assets/
├── content/
├── mods/
├── tools/
├── scripts/
├── tests/
├── docs/
└── build/
```

其中最关键的原则是：

```text
src
    游戏逻辑

content
    游戏配置

assets
    美术音频资源

mods
    用户扩展

tools
    开发工具

scripts
    自动化脚本
```

千万不要变成：

```text
assets/
    skill.json
    buff.json
    map.json
    portrait.png
    bgm.ogg
    build.py
```

全部混在一起。

对于你未来的 SRPG，这种分层是最能支撑几年持续扩展、外包资源接入、MOD 支持和 AI 协作开发的。
