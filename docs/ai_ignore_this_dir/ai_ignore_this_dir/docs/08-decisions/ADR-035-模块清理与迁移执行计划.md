# ADR-035: 模块清理与迁移执行计划

## 状态
Accepted（2026-06-15）

## 背景

根据 ADR-029 ~ ADR-034 定义的五阶段重构方案，本 ADR 制定具体的**文件级删除/新建/修改清单**、**阶段门禁检查表**和**回归验证策略**。

这是整个重构的执行总控，确保：
1. 每阶段完成后无残留废弃文件
2. 每阶段通过编译 + 测试门禁
3. 删除操作必须在阶段内完成，不可推迟
4. 新旧代码不共存

### 风险与回滚策略

| 风险 | 级别 | 缓解措施 |
|------|------|----------|
| Phase 3 删除 Buff 模块后编译失败 | 🔴 高 | Phase 3 门禁要求在删除前确认所有引用已迁移 |
| 新 Pipeline 运行时行为与旧系统不一致 | 🔴 高 | Phase 3 + Phase 6 的回归测试必须覆盖所有核心战斗场景 |
| RON 文件迁移遗漏导致运行时崩溃 | 🟡 中 | `ValidatableRegistry::validate_references()` 在启动时检查所有引用 |
| Phase 4 重命名影响大量测试代码 | 🟡 中 | Phase 4 门禁包含 `grep` 检查确保旧名已删除 |
| FTL 文件维护跟不上内容迭代 | 🟢 低 | 语言回退链保证 Key 总有展示形式（Key 本身/Debug 模式） |

**回滚策略**：每 Phase 执行前打 Git Tag（`refactor/phase-1-start`），执行完成后打 `refactor/phase-1-end`。任何 Phase 门禁不通过时回滚到上一个 -end tag。

## Forbidden（禁止事项）

- 🟥 **禁止** 在任何 Phase 中使用 `#[allow(deprecated)]` 来绕过编译错误
- 🟥 **禁止** 保留已标记删除的文件超过其所属 Phase 的完成时间
- 🟥 **禁止** 在同一分支上同时进行多个 Phase 的开发 — 必须串行执行
- 🟥 **禁止** 在未通过当前 Phase 门禁时开始下一 Phase
- 🟥 **禁止** 删除文件后通过 Git Stash 恢复旧文件作为"参考"
- 🟥 **禁止** 创建"旧 API 兼容层"或"转发函数"来减少修改量
- 🟥 **禁止** 在 Phase 门禁中使用 `cargo check` 替代 `cargo build` — 必须完整编译

## 后果

### 正面
- 所有阶段的可执行步骤清晰明确，开发团队可严格执行
- 每阶段独立门禁保证不会出现大爆炸式集成失败
- 删除清单确保所有废弃代码被物理删除，从源码管理层面消除技术债务
- RON 文件迁移清单确保不会遗漏任何配置文件

### 负面
- 全量重构涉及 100+ 文件，预计需要多人周的开发时间
- 重构期间主分支会被锁定（或需要长周期 Feature Flag 分支）
- 回归测试覆盖面必须达到 100% 核心战斗场景，否则部署风险极高

## 替代方案（已拒绝）

| 方案 | 拒绝原因 |
|------|----------|
| 不在同一分支串行执行，改为多分支并行 | 依赖链强制串行（P3 依赖 P2 的 Attribute 变化），并行会导致 Merge 冲突不可控 |
| 不删除旧文件，留存做参考 | 违背"零技术债务"核心原则，留存一定会被误引用 |
| 分两次 Release 执行重构 | 6 个月后上下文丢失一半，不如一次性完成 |
