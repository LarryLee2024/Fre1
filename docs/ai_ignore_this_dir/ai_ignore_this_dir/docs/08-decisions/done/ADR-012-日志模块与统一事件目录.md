# ADR-012: 日志模块修复与统一事件目录

## 状态

Accepted

## 背景

### 当前状态

日志模块 `src/infrastructure/logging/` 已能工作，但有三个架构违规需要修复：

```
src/infrastructure/logging/
├── mod.rs        # LogPlugin
├── events.rs     # ⚠️ 11 个事件定义，其中 9 个不应在这里
└── observer.rs   # ⚠️ 使用 bevy::log 而非 tracing
```

**🟥 问题 1：事件定义位置错误**
`events.rs` 中定义了 `BuffApplied`、`SkillActivated`、`EquipmentEquipped` 等业务事件。这些事件不属于 Infrastructure 层——它们是**业务领域事件**，被多个下游系统消费（Logging、Replay、UI BattleLog、Achievements）。

**🟥 问题 2：使用 Entity 而非 Strong ID**
事件字段使用裸 `bevy::ecs::entity::Entity`，导致日志输出 `Entity(1v0)` 而非业务可读的 `unit_soldier_01`（宪法 1.2.2）。

**🟥 问题 3：使用 bevy::log 而非 tracing**
`bevy::log::info!()` 是对 tracing 的薄封装，但宪法 13.1.1 要求直接统一使用 `tracing`。

## 引用的领域规则

- 宪法 §13.1.1 — 统一使用 tracing，禁止 println!/dbg!
- 宪法 §13.3.1 — INFO 级别日志必须携带 event 字段
- 宪法 §13.3.2 — 日志 target 与 Feature 目录名一致
- 宪法 §13.8.1 — 日志是领域事件的消费者，非业务代码主动调用
- 宪法 §13.8.2 — 禁止为了打日志单独创建事件
- 宪法 §13.8.3 — ERROR/WARN/DEBUG 可直接调用 tracing 宏
- 宪法 §1.2.2 — 日志必须输出业务可读的字符串 ID
- 宪法 §2.2.6 — 领域事件是唯一业务事实源

- 宪法 §13.10.2 — 事件白名单管理

## 决策

### 核心决策：三层事件管理模型

```
┌─────────────────────────────────────────────────────────┐
│  治理层（Governance）：shared/event/                       │
│                                                         │
│  所有可观察事件的 struct 定义在此统一管理。                   │
│  按领域分文件，不膨胀。                                    │
│  新人看一眼这个目录就知道"项目有哪些可观察事件"。              │
│  自动满足宪法 13.10.2 事件白名单需求。                      │
│                                                         │
│  src/shared/event/                                       │
│  ├── mod.rs                                              │
│  ├── battle.rs       ← 战斗事件                           │
│  ├── buff.rs         ← Buff 事件                          │
│  ├── skill.rs        ← 技能事件                           │
│  ├── equipment.rs    ← 装备事件                           │
│  ├── inventory.rs    ← 背包事件                           │
│  ├── turn.rs         ← 回合事件                           │
│  ├── character.rs    ← 角色事件（移动等）                  │
│  ├── campaign.rs     ← 战役/关卡事件                      │
│  └── infra.rs        ← 基础设施事件                       │
└──────────────────────┬──────────────────────────────────┘
                       │ 仅 struct 定义，零业务逻辑
                       │ 被 Logging、Replay、UI 等共享
┌──────────────────────▼──────────────────────────────────┐
│  触发层（Emission）：各 Core 模块                          │
│                                                         │
│  各领域 System 在业务完成后触发对应事件。                    │
│  不需要知道谁在消费。                                     │
│                                                         │
│  core/battle/systems/ → EventWriter<DamageDealt>         │
│  core/buff/systems/  → EventWriter<BuffApplied>          │
│  core/skill/systems/ → EventWriter<SkillActivated>       │
│  ...                                                     │
└──────────────────────┬──────────────────────────────────┘
                       │ 分离触发与消费
┌──────────────────────▼──────────────────────────────────┐
│  消费层（Consumption）：infrastructure/logging/            │
│                                                         │
│  LogObserver 用 MessageReader 统一监听所有领域事件。        │
│  接收来自各 Core 模块的事件，输出 tracing 日志。            │
│                                                         │
│  infra/logging/observer.rs                               │
│  → log_damage_dealt(msg: MessageReader<DamageDealt>)     │
│  → log_buff_applied(msg: MessageReader<BuffApplied>)     │
│  → log_turn_started(msg: MessageReader<TurnStarted>)     │
│  → ...                                                  │
└─────────────────────────────────────────────────────────┘
```

**为什么这个模型适合大型项目：**

| 场景 | 分散定义在各领域 | 统一在 shared/event/ |
|------|----------------|-------------------|
| 新人找所有事件 | 翻 20 个目录 | 看一个目录 |
| 加一个新事件 | 在对应领域定义，但可能漏注册 | 在对应文件加 struct，自动可见 |
| 事件白名单 | 需要额外维护白名单文档 | `ls src/shared/event/` 就是白名单 |
| 多个消费者（log/replay/ui） | 各自 import 不同位置，可能不一致 | 统一 import 路径 |
| 事件格式一致性 | 各领域风格可能不同 | 统一管理，格式一致 |

**代价**：每个事件需要额外在 shared/event/ 中定义一次 struct。但这个代价在新领域加事件时被均摊——加一次 struct，所有消费者自动获得。

### 关于"三棵树分离"与 shared/event/ 的关系

宪法 §1.1.1 的 Feature First 原则要求按业务领域拆分模块。`shared/event/` 表面上看是按技术类型（"event"）组织的。为什么可以？

```
✅ shared/event/ 是合法的
  └── 因为这里只有纯 struct 定义，零业务逻辑，零行为
  └── 宪法 §3.0.7："common/ 只能存放与业务无关的纯工具代码"
  └── 宪法 §1.1.1 禁止的是 components/ systems/ events/ 这样的"全局功能目录"
  └── shared/event/ 不包含任何"事件处理逻辑"，只是 struct 定义
```

shared/event/ 和 `shared/error/` 一样——它们定义的是跨领域共享的**数据类型**，不是**逻辑**。

### 问：为什么不直接把事件定义在 core 各领域内？

如果 team 规模小（3-5 人）、功能耦合紧密，事件定义在各领域就够用了。

但项目要做大，必须考虑：
- 多人并行开发时，"事件定义在各自领域"意味着新人需要知道去哪找
- 多个下游系统（LogObserver、ReplayRecorder、BattleLogUI、AchievementTracker）都需要 import 统一的事件集
- 如果每个下游系统 import 不同路径，event whitelist 需要额外维护一个 markdown 文件
- `src/shared/event/` 就是这个 whitelist——目录结构即白名单

## Module Design

### Phase 1：创建 shared/event/ 统一事件目录

目标：建立统一的可观察事件管理目录。

```

├── mod.rs         # 重导出所有子模块
├── battle.rs      # 战斗事件
```

- 🟩 shared/event 只有 struct 定义，零业务逻辑

### Phase 3：修复 LogObserver

**3.1 统一使用 tracing**

```rust
// 当前（违规）
bevy::log::info!(target: "battle", event = "DamageApplied", ...);

// 修复后（合规）
use tracing::info;
info!(target: "battle", event = "DamageDealt", attacker = %msg.attacker, ...);
```

**3.2 替换 Entity 为 Strong ID**

```rust
// 当前（违规）
pub fn log_damage_applied(mut reader: MessageReader<core::battle::DamageApplied>) {
    for msg in reader.read() {
        info!(
            target: "battle",
            event = "DamageApplied",
            attacker = ?msg.attacker,     // → Entity(xxx)
            target = ?msg.target,          // → Entity(xxx)
            ...
        );
    }
}

// 修复后（合规）
pub fn log_damage_dealt(mut reader: MessageReader<shared::event::battle::DamageDealt>) {
    for msg in reader.read() {
        info!(
            target: "battle",
            event = "DamageDealt",
            attacker = %msg.attacker,     // → UnitId Display ("unit_soldier_01")
            target = %msg.target,         // → UnitId Display ("unit_goblin_03")
            amount = msg.amount,
            is_critical = msg.is_critical,
            "伤害结算"
        );
    }
}
```

**3.3 用 shared/event 替换 core 事件 import**

当前 `observer.rs` 直接从 core import 事件，修复后改为从 `shared::event` import：

```rust
// 当前
use crate::core::battle::{CharacterDied, DamageApplied, HealApplied, ...};

// 修复后
use crate::shared::event::battle::{CharacterDied, DamageDealt, HealApplied};
use crate::shared::event::turn::{TurnStarted, TurnEnded};
// 不需要 import 其他领域的——LogObserver 只 import 它消费的事件
```

**3.4 不在此次范围内的优化**

| 特性 | 优先级 | 说明 |
|------|--------|------|
| Observer/Trigger 替代 MessageReader 轮询 | P2 | 回合制 SRPG 事件频率低，轮询开销可忽略 |
| 日志文件自动滚动 | P3 | 基础设施可观测性需求 |
| OpenTelemetry 对接 | P2 | 属于 telemetry 模块范畴 |
| 运行时动态日志级别 | P2 | 开发体验增强 |
| 日志脱敏 | P2 | 安全合规需求 |
| 日志配置文件 | P2 | 可通过环境变量 RUST_LOG=... 满足 |

### Phase 4：更新 core 各模块的触发点

目标：core 各模块的 System 在业务完成后，触发 `shared::event` 中定义的事件。

```rust
// 示例：core/battle/systems/damage_system.rs
use bevy::prelude::*;
use crate::shared::event::battle::DamageDealt;

pub fn apply_damage(
    mut commands: Commands,
    // ... queries
    mut damage_events: EventWriter<DamageDealt>,
) {
    // ... 计算伤害 ...
    
    damage_events.send(DamageDealt {
        attacker: attacker_id,
        attacker_name: attacker_name.clone(),
        target: target_id,
        target_name: target_name.clone(),
        amount: final_damage,
        is_skill: false,
        skill_id: None,
        is_critical: false,
    });
}
```

**迁移策略：**

按以下顺序逐个领域迁移（与 ADR-011 Phase 2 顺序一致）：

| 顺序 | 领域 | shared/event 文件 | LogObserver 变更 | core 触发点变更 |
|------|------|-------------------|-----------------|----------------|
| 1 | turn | turn.rs | 更新 import + 函数名 | 添加 `EventWriter<TurnStarted/Ended>` |
| 2 | buff | buff.rs | 更新 import + 函数名 | 添加 `EventWriter<BuffApplied/Removed>` |
| 3 | skill | skill.rs | 更新 import + 函数名 | 添加 `EventWriter<SkillActivated>` |
| 4 | battle | battle.rs | 更新 import + 函数名 | 更新已有事件定义 |
| 5 | equipment | equipment.rs | 更新 import + 函数名 | 添加 `EventWriter<Equipped/Unequipped>` |
| 6 | inventory | inventory.rs | 更新 import + 函数名 | 添加 `EventWriter<ItemUsed/Transferred>` |
| 7 | character | character.rs | 更新 import + 函数名 | 添加 `EventWriter<UnitMoved>` |
| 8 | campaign | campaign.rs | 更新 import + 函数名 | 添加 `EventWriter<LevelCompleted>` |

**每个领域迁移后：**
```
cargo build && cargo test  # 零错误
```

**不需要立即迁移的项：** 如果 core 某领域当前没有对应的事件触发逻辑，在本次只定义 shared/event 的 struct，待 core 有触发逻辑时再补全。不强制一次全部实现。

## Communication Design

```
src/shared/event/ (struct 定义层)
    │
    │   各文件中的 Event struct
    │   (DamageDealt, BuffApplied, SkillActivated, ...)
    │
    ├──→ core/*/systems/ (触发层)
    │     EventWriter<DamageDealt>.send(...)
    │
    └──→ infrastructure/logging/observer.rs (消费层)
          MessageReader<DamageDealt>.read() → tracing::info!(...)
    
    未来消费者（同一个事件源）：
    → infrastructure/replay/ → 录制事件序列
    → ui/battle_log/ → 战斗履历展示
    → core/achievement/ → 成就触发
```

## 边界定义

### 层间依赖

```
shared/event/ → 仅引用 shared/id（强类型 ID）
  └── 零业务逻辑，纯 struct
  └── 依赖 bevy::prelude (Event trait)

core/*/ → 触发 shared::event::* 事件
  └── 通过 EventWriter 发送
  └── 不直接依赖 infrastructure

infrastructure/logging/ → 消费 shared::event::* + core::* 已有事件
  └── 通过 MessageReader 读取
  └── 输出 tracing 日志
  └── 不能在这里定义事件

禁止的依赖路径：
  🟥 shared/event/ → core/* （反对，struct 不能依赖业务逻辑）
  🟥 LogObserver → UI （日志不应知道 UI 存在）
```

### 允许

- 🟩 shared/event/ 中的 struct 被所有层 import
- 🟩 infrastructure/logging/ 消费 shared/event/ 事件
- 🟩 core 各模块触发 shared/event/ 事件

### 禁止

- 🟥 禁止在 shared/event/ 中定义任何方法、逻辑、System
- 🟥 禁止在 infrastructure/logging/ 中定义业务事件
- 🟥 禁止使用裸 `Entity` 作为事件字段（必须使用 Strong ID）
- 🟥 禁止使用 `bevy::log::*` 替代 `tracing::*`
- 🟥 禁止 INFO 级别日志不携带 `event` 字段（宪法 13.3.1）
- 🟥 禁止为临时调试新增永久领域事件

## Forbidden（禁止事项）

- 🟥 禁止在 infrastructure/logging/events.rs 中定义业务事件（BusinessEvent）
- 🟥 禁止日志事件使用裸 `bevy::ecs::entity::Entity` 类型
- 🟥 禁止核心业务代码直接调用 `info!` 输出核心业务事件（宪法 13.8.2）
- 🟥 禁止在 Phase 1-4 同时引入 Observer/Trigger 迁移（P2 再考虑）
- 🟥 禁止在 shared/event/ 中包含业务逻辑、trait 实现、方法定义
- 🟥 禁止一次性迁移所有领域的触发事件（逐个领域，保持编译通过）

## Definition / Instance Design

不适用（本 ADR 不涉及 Definition/Instance 分离）。

## 后果

### 正面

- ✅ 统一查看可观察事件的位置：`ls src/shared/event/`
- ✅ 宪法 13.10.2 的事件白名单由目录结构自动满足
- ✅ 日志格式统一（tracing + 结构化字段 + Strong ID）
- ✅ 领域事件从"为日志服务"变为"为所有下游系统服务"
- ✅ 所有事件字段可读（`UnitId` Display 而非 `Entity(xxx)`）
- ✅ 逐个领域迁移，每个步骤可验证

### 负面

- ⚠️ 需要创建约 8 个 shared/event/ 子文件
- ⚠️ 部分 core 模块当前可能还没有触发事件的机制，需要补齐
- ⚠️ 从 `bevy::log` 切换到 `tracing` 改动了所有 observer 函数
- ⚠️ 事件 struct 在 shared/event/ 和 core 模块形成"两份定义"的现象——直到旧定义完全淘汰

### 风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| shared/event/ 事件与 core 已有事件重复定义 | 高 | 中 | 优先删除 core 中的重复定义，全部指向 shared/event |
| 迁移后日志不输出（遗漏 import） | 中 | 高 | 每个领域迁移后手动运行验证日志输出 |
| 领域不完整——没有事件触发点 | 中 | 低 | 先定义 struct，触发逻辑后续补齐 |
| Entity → Strong ID 转换时需要从 ECS 查询 | 中 | 中 | 触发点在 System 中时，Entity 到 UnitId 的查询成本可接受 |

## 替代方案

### 方案 A：事件定义在各自 Core 模块（已拒绝）

**思路**：符合 Feature First 原则，每个领域定义自己的事件。

**拒绝理由**（大型项目视角）：
- 新人需要知道"事件去哪找"，增加认知负担
- 每个下游系统（log/replay/ui）需要 import 多个散落路径
- 事件白名单需要额外维护一个 markdown 文件
- `shared/event/` 的目录结构就是你不需要维护的白名单

### 方案 B：全部定义在 infrastructure/logging/（已拒绝）

**思路**：保持现状，所有事件放 infra/logging/events.rs。

**拒绝理由**：
- 违反宪法 13.8.2 "禁止为了打日志单独创建事件"
- 事件被 replay、ui 等其他系统消费时，需要依赖 infrastructure 层——依赖方向错误
- `events.rs` 会随项目膨胀到几百上千行

### 方案 C：使用单独的 `domain_events/` crate（已拒绝）

**思路**：将事件定义抽出为独立 crate，所有其他 crate 依赖它。

**拒绝理由**：
- 项目目前是单 crate（workspace 未拆分），引入独立 crate 的编译时间和管理成本不值得
- `shared/event/` 不包含业务逻辑，放在单 crate 中没有任何依赖问题
- 如果未来做 workspace 拆分，将 `shared/event/` 抽成独立 crate 的迁移成本极低
