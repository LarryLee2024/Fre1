# 性能预算领域

Version: 1.1
Status: Draft
Applies To: 游戏所有模块的帧时间预算、堆内存分配、ECS 查询优化、性能门禁

> **优化来源**: docs/01-architecture/performance_budget.md

核心原则（对标宪法条款）：
- 🟥 正确性优先：必须先保证代码正确，再考虑性能优化（宪法 15.0.1）
- 🟥 测量优先：所有性能问题必须通过 Profile 测量确认，绝对禁止凭直觉进行性能优化（宪法 15.0.2）
- 🟥 优化热点：必须优先优化热点代码，绝对禁止为了性能进行全局重构（宪法 15.0.3）
- 🟥 可读性优先：禁止为了性能牺牲代码可读性，除非有明确的 Profile 数据证明该部分是性能瓶颈（宪法 15.0.4）
- 🟥 复杂度优先于性能：架构复杂度预算优先级高于性能优化预算（宪法 1.5.1 + 15.0.9）
- 🟥 禁止为未落地需求预留复杂框架（宪法 1.5.2）

## 1. 统一术语

| 术语 | 定义 | 职责边界 |
|------|------|----------|
| 帧率（FPS） | 每秒渲染帧数，目标 60FPS | 负责：衡量整体流畅度；不负责：单个 System 的微观性能 |
| 单帧预算 | 一帧可消耗的最大时间，60FPS 下为 16.67ms | 负责：设定全局上限；不负责：具体 System 的分配 |
| 模块预算 | 各业务模块允许的最大帧时间占比 | 负责：约束单个模块的性能上限；不负责：模块内部优化策略 |
| 场景分级预算 | 按战斗规模差异化的帧时间预算 | 负责：不同规模场景的预算差异；不负责：具体优化实现 |
| 设备分级预算 | 按设备性能差异化的帧时间预算 | 负责：不同设备的预算差异；不负责：设备检测逻辑 |
| 堆内存分配 | 在主更新循环中禁止的动态内存分配 | 负责：约束运行时内存行为；不负责：具体预分配实现 |
| ECS 查询优化 | 通过查询粒度控制和变更检测减少每帧计算量 | 负责：约束 System 的查询模式；不负责：具体 System 实现 |
| 性能门禁 | 新增系统必须通过的帧时间审查机制 | 负责：控制新增系统的性能影响；不负责：具体审查流程 |
| 性能告警闭环 | 从检测到修复的完整告警处理流程 | 负责：确保性能问题被追踪修复；不负责：具体修复方案 |

## 2. 领域边界

### 本领域负责

- 定义帧率目标和单帧预算分配
- 定义模块级帧时间预算上限
- 定义场景分级和设备分级的差异化预算
- 定义堆内存分配禁令和替代方案
- 定义 ECS 查询优化规则
- 定义性能门禁审查流程
- 定义性能告警处理闭环

### 本领域不负责

- 具体 System 的实现优化（由各业务 Plugin 负责）
- Profiler 工具的具体实现（由 Infrastructure 领域负责）
- 资源加载策略（由 Asset Lifecycle 领域负责）
- UI 渲染的具体实现（由 UI Architecture 领域负责）

## 3. 生命周期

### 状态列表

| 状态 | 含义 | 可转换到 |
|------|------|----------|
| BudgetSet | 预算已定义，待验证 | Measuring |
| Measuring | 正在测量实际性能 | Alert / Pass |
| Alert | 超出预算，需处理 | Fixing |
| Fixing | 正在修复性能问题 | Verifying |
| Verifying | 修复后验证 | Pass / Alert |
| Pass | 性能达标 | BudgetSet |

### 状态转换图

```
BudgetSet → Measuring → Pass → BudgetSet
                   ↓
                 Alert → Fixing → Verifying → Pass
                                    ↓
                                  Alert（未通过）
```

## 4. 不变量（Invariants）

### 不变量1：帧率底线（🟩 宪法 15.0.1 正确性优先）

任意时刻：

游戏运行帧率不得低于 30FPS。低于此值视为性能问题，必须立即处理。

违反表现：

帧时间持续超过 33.33ms。

---

### 不变量2：可用预算上限（🟩 宪法 15.0.1 正确性优先）

任意时刻：

所有游戏逻辑 System 的总帧时间不得超过 14ms（预留 2.67ms 给系统调度和渲染）。

违反表现：

游戏逻辑总耗时超过 14ms，导致帧率低于 60FPS。

---

### 不变量3：主循环零堆分配（🟩 宪法 15.0.5 ECS优化）

任意时刻：

Update Schedule 中的所有 System 不得执行堆内存分配操作（Vec::new()、HashMap::new()、String::from()、Box::new() 等）。

违反表现：

在 Update Schedule 中出现动态内存分配，导致 GC 压力和帧率波动。

---

### 不变量4：新增系统性能门禁（🟩 宪法 15.0.2 测量优先）

任意时刻：

任何新增 System 如果使帧时间增加 >1ms，必须经过架构审查批准后才能合并。

违反表现：

未经审批的 System 导致帧时间回归超过 1ms。

---

### 不变量5：内存预算不超限（🟩 宪法 15.0.1 正确性优先）

任意时刻：

- 堆内存总占用不超过 512MB
- 单场景 Entity 数量不超过 50,000
- Bevy World 中 Resource 组件总数不超过 2,000

违反表现：

内存占用超过预算，触发 GC 或导致 OOM。

---

## 5. 业务规则

### 规则1：帧时间构成约束（🟩 宪法 15.0.1 正确性优先）

允许：
- 输入处理 ≤1ms
- ECS 调度/执行 ≤9ms（游戏逻辑）
- 渲染 ≤5ms
- 系统开销 ≤1.67ms

禁止：
- 🟥 任何模块超出其分配的帧时间预算
- 🟥 未经审批就接受帧时间回归（宪法 15.0.2：所有性能问题必须通过 Profile 测量确认）

必须：
- 各模块严格遵守其预算上限
- 超出预算时立即定位并优化

---

### 规则2：模块预算上限

| 模块 | 预算 | 说明 |
|------|------|------|
| 战斗逻辑（Effect Pipeline） | <5ms | 伤害计算、Buff 结算 |
| AI 决策 | <2ms | 策略选择、目标评估 |
| 寻路计算 | <2ms | BFS + 地形消耗 |
| 回合管理 | <1ms | 状态机转换、队列操作 |
| 属性计算 | <1ms | Modifier Stack 求值 |
| UI 渲染 | <3ms | 面板更新、飘字动画 |
| 动画系统 | <2ms | 精灵动画播放 |
| 音频系统 | <1ms | 音效触发 |
| 资源加载（异步） | <5ms/帧 | 后台加载不阻塞主线程 |
| 日志记录 | <0.5ms | 结构化日志输出 |
| ViewModel 更新 | <2ms | 数据提取和格式化 |

---

### 规则3：场景分级预算

| 场景类型 | 条件 | 战斗逻辑预算 | AI 预算 | 寻路预算 | 说明 |
|---------|------|------------|---------|---------|------|
| 小规模战斗 | ≤ 10 单位 | < 5ms | < 2ms | < 2ms | 标准预算 |
| 中规模战斗 | 11-30 单位 | < 6ms | < 3ms | < 3ms | 允许轻微超支 |
| 大规模团战 | > 30 单位 | < 8ms | < 4ms | < 4ms | 分帧计算兜底 |
| UI 模式 | 无战斗实体 | < 1ms | — | — | 仅 UI 渲染和数据展示 |

分帧执行触发条件：

- 单帧战斗逻辑预算超支时，自动将非关键计算延迟到下一帧
- 使用 FrameCounter 和 Timer 控制分帧节奏

> **优化来源**: docs/其他/60.md（场景分级预算建议）

---

### 规则4：设备分级预算

| 设备级别 | 目标帧率 | 单帧预算 | 战斗逻辑 | AI 决策 | 寻路 | UI 渲染 |
|---------|---------|---------|---------|---------|------|---------|
| PC 高配 | 120 FPS | 8.33ms | < 4ms | < 1.5ms | < 1.5ms | < 2ms |
| PC 标准 | 60 FPS | 16.67ms | < 5ms | < 2ms | < 2ms | < 3ms |
| 移动端 | 60 FPS | 16.67ms | < 5ms | < 2ms | < 2ms | < 3ms |
| 低配机 | 30 FPS | 33.33ms | < 8ms | < 4ms | < 4ms | < 5ms |

设备级别分为三档：HighEnd（PC 高配）、MidRange（PC 标准/移动端）、LowEnd（低配机）。

> **优化来源**: docs/其他/60.md（设备分级预算表建议）

---

### 规则5：堆内存分配禁令（🟥 宪法 15.0.5 ECS优化）

禁止：
- 🟥 在 Update Schedule 的 System 中使用 Vec::new()、HashMap::new()、String::from() 等堆分配操作
- 🟥 在 System 中调用 Box::new()、Rc::new() 等堆分配
- 🟥 在高频 System（每帧执行）中创建临时集合

允许的替代方案：

| 场景 | 禁止 | 允许 |
|------|------|------|
| 临时收集结果 | Vec::new() + push | 预分配的 Local<Vec<T>> |
| 字符串拼接 | format!() 创建新 String | 预分配的 Local<String> |
| 错误消息 | format!() 构造错误 | 使用 &'static str 错误描述 |
| 日志消息 | format!() 构造日志 | 结构化字段 + tracing 宏 |

延迟分配规则：

- 需要分配内存的操作应使用 Bevy Commands 的延迟分配
- 实体生成、组件创建等操作通过 Commands 延迟执行

---

### 规则6：ECS 查询优化（🟩 宪法 15.0.5 ECS优化）

查询粒度规则：

- 🟩 单个 System 的查询组件数不超过 6 个
- 🟩 优先使用 Changed<T> 和 Added<T> 过滤器（宪法 2.3.3 状态变更检测原则）
- 🟩 将大型查询拆分为多个小查询
- 🟥 禁止每帧查询所有 Entity（使用过滤器缩小范围）

变更检测规则：

- 必须优先使用 Bevy 原生变更检测（Changed<T>、Added<T>）
- 禁止每帧全量更新 UI 或其他数据

查询优化检查清单：

- 是否使用了 Changed<T> / Added<T> 过滤器？
- 查询的组件数量是否 ≤ 6 个？
- 是否避免了全 Entity 扫描？
- 是否使用了 Without<T> 过滤器排除不需要的 Entity？
- 查询结果是否被缓存（如果多处使用）？

> **优化来源**: docs/其他/60.md（ECS 进阶技巧建议）

---

### 规则7：性能门禁审查（🟩 宪法 15.0.2 测量优先）

性能优化决策流程：

- 🟩 发现性能问题 → Profile 定位瓶颈
- 🟩 是热点代码 → 优化热点（宪法 15.0.3：必须优先优化热点代码）
- 🟥 不是热点 → 不优化（宪法 15.0.3：绝对禁止为了性能进行全局重构）
- 🟩 不确定 → 先 Profile 再决定（宪法 15.0.2：绝对禁止凭直觉进行性能优化）

---

### 规则8：性能告警闭环

告警处理流程：

```
检测（puffin/tracy）
    ↓ 帧时间超阈值
告警（WARN log + 指标记录）
    ↓ 自动触发
定位（performance profile 快照）
    ↓ 分析瓶颈
修复（针对性优化代码）
    ↓ 提交 PR
验证（benchmark 回归测试）
    ↓ 通过
关闭告警
```

告警级别定义：

| 级别 | 条件 | 处理方式 | SLA |
|------|------|---------|-----|
| WARN | 帧时间超预算 10%-50% | 记录日志，下次迭代修复 | 3 天内 |
| ERROR | 帧时间超预算 50%-100% | 立即定位，当前迭代修复 | 1 天内 |
| CRITICAL | 帧时间超预算 >100% | 阻止合并，立即修复 | 当天 |

> **优化来源**: docs/其他/60.md（性能告警处理闭环建议）

---

### 规则9：内存预算管理

| 指标 | 预算 | 说明 |
|------|------|------|
| 堆内存总预算 | 512MB | 游戏运行时最大堆内存占用 |
| Resource 总数上限 | 2000 | Bevy World 中 Resource 组件总数 |
| Entity 上限 | 50,000 | 单场景最大 Entity 数量 |
| 资源回收阈值 | 128MB | 累计分配超过此阈值时触发资源回收 |

内存监控系统应每 N 帧执行一次，检查 Entity 数量和内存占用是否接近上限。

> **优化来源**: docs/其他/60.md（内存预算管理建议）

---

### 规则10：瓶颈检测与优化

常见瓶颈模式：

| 瓶颈类型 | 表现 | 解决方案 |
|---------|------|---------|
| AI 评估过重 | AI 回合帧时间 >5ms | 分帧评估、异步计算 |
| 范围计算过频 | 每帧重算移动/攻击范围 | 缓存范围结果 |
| 链式查询 | 多个 System 串行执行 | 合并 System、优化依赖 |
| 大量 Entity 查询 | Entity 数量多时帧时间飙升 | 使用过滤器、分组处理 |
| 变更检测缺失 | 每帧全量更新 UI | 添加 Changed 过滤器 |

优化原则：

- 🟩 先 Profile 再优化，禁止凭直觉优化（宪法 15.0.2：绝对禁止凭直觉进行性能优化）
- 🟩 优先优化热点代码，非热点不优化（宪法 15.0.3：绝对禁止为了性能进行全局重构）
- 🟩 缓存 Key 设计需包含影响范围的所有因素（宪法 15.0.7：所有缓存必须明确定义失效条件）
- 🟩 UI 使用脏标记（dirty flag）避免每帧重绘（宪法 15.0.5 ECS优化）
- 🟥 复杂度优先于性能：大多数独立游戏死于复杂度，而非性能（宪法 15.0.9 + 1.5.1）

> **优化来源**: docs/其他/60.md（常见瓶颈代码级案例建议）

## 6. 流程管线

### 性能预算执行管线

```
预算定义 → 模块分配 → 实时测量 → 告警检测 → 定位修复 → 验证关闭
```

#### Step1：预算定义

输入：帧率目标（60FPS）和设备分级
处理：计算单帧预算和模块级预算
输出：各级预算表
禁止：不定义预算就开始开发

#### Step2：模块分配

输入：模块级预算表
处理：将总预算分配给各业务模块
输出：各模块的帧时间上限
禁止：模块预算之和超过总预算

#### Step3：实时测量

输入：运行中的游戏
处理：使用 Profiler 测量各 System 执行时间
输出：实时性能数据
禁止：不测量就宣称性能达标

#### Step4：告警检测

输入：实时性能数据
处理：对照预算阈值检测告警
输出：告警级别和位置
禁止：忽略告警继续开发

#### Step5：定位修复

输入：告警信息
处理：Profile 定位瓶颈并优化
输出：修复后的代码
禁止：未 Profile 就全局重构

#### Step6：验证关闭

输入：修复后的代码
处理：运行性能基准测试验证
输出：验证报告
禁止：未验证就关闭告警

## 7. 数据结构

### 帧时间预算（FrameBudget）

职责：定义各模块的帧时间上限

结构：
- 模块名称：String — 模块标识
- 预算上限：f64 — 允许的最大帧时间（ms）
- 设备级别：DeviceTier — 适用的设备级别
- 场景类型：SceneType — 适用的场景类型

要求：
- 每个模块必须有对应的预算定义
- 预算值必须在总预算范围内
- 不同设备/场景可以有不同预算

---

### 性能告警（PerformanceAlert）

职责：记录性能超标的告警信息

结构：
- 告警级别：AlertLevel — WARN/ERROR/CRITICAL
- 触发 System：String — 触发告警的 System 名称
- 帧时间：f64 — 实际帧时间（ms）
- 预算上限：f64 — 预算帧时间（ms）
- 触发时间：Timestamp — 告警触发时间
- 处理状态：AlertStatus — 处理状态

要求：
- 告警必须包含足够的定位信息
- 告警必须有对应的处理 SLA
- 告警关闭必须经过验证

---

### 设备分级（DeviceTier）

职责：标识设备性能级别

结构：
- 级别：枚举值 — HighEnd/MidRange/LowEnd
- 目标帧率：u32 — 该级别的目标帧率
- 单帧预算：f64 — 该级别的单帧预算（ms）

要求：
- 设备分级必须在启动时确定
- 不同级别使用不同的预算表
- 低配机必须保证基本可玩性

---

## 8. 禁止事项

🟥 禁止：主循环堆内存分配（宪法 15.0.5 ECS优化：使用预分配或延迟分配）

原因：堆分配导致 GC 压力和帧率波动，破坏流畅性

违反后果：帧率不稳定、GC 卡顿

---

🟥 禁止：无过滤器的全 Entity 查询（宪法 15.0.5 ECS优化：必须使用 Changed/Added/Without 等过滤器）

原因：全量查询浪费计算资源，Entity 数量增长时性能急剧下降

违反后果：帧时间随 Entity 数量线性增长

---

🟥 禁止：寻路每帧重算（宪法 15.0.3 优化热点：缓存范围结果，仅在状态变化时重算）

原因：寻路计算昂贵，每帧重算会超出 2ms 预算

违反后果：寻路模块帧时间超标

---

🟥 禁止：凭直觉进行性能优化（宪法 15.0.2：必须先 Profile，绝对禁止凭直觉进行性能优化）

原因：直觉往往不准确，可能优化非热点代码而忽略真正瓶颈

违反后果：浪费优化精力，性能问题未解决

---

🟥 禁止：未 Profile 就全局重构（宪法 15.0.3：必须优先优化热点代码，绝对禁止为了性能进行全局重构）

原因：全局重构风险高、成本大，应优先优化 Profile 确认的热点

违反后果：引入新 bug、代码质量下降

---

🟥 禁止：新增 System 使帧时间增加 >1ms 而不审批（宪法 15.0.2 测量优先）

原因：未经审批的性能回归会累积，最终导致帧率不达标

违反后果：帧时间累积超标、帧率下降

---

🟥 禁止：为了性能牺牲代码可读性（宪法 15.0.4：除非有 Profile 数据证明该部分是瓶颈）

原因：可读性是长期维护的基础，非瓶颈的性能优化得不偿失

违反后果：代码难维护、后续修改成本高

---

🟥 禁止：在每帧 System 中打印 Info/Debug 日志（宪法 13.1.2：绝对禁止记录技术执行流水）

原因：日志 I/O 是同步操作，会阻塞 System 执行

违反后果：帧时间增加、日志文件膨胀

## 9. AI 修改规则

### 如果新增性能预算规则

允许：
- 在预算表中新增模块的预算定义
- 调整现有模块的预算上限（需说明理由）

禁止：
- 不经审批就增加模块预算
- 将预算上限设置为无限大
- 删除现有模块的预算约束

优先检查：
- 新增预算是否在总预算范围内
- 是否与其他模块预算冲突
- 是否有对应的测量和告警机制

---

### 如果发现性能问题

允许：
- 使用 Profiler 定位瓶颈
- 提出优化方案并评估影响
- 调整预算分配（需 ADR 审批）

禁止：
- 未 Profile 就修改代码
- 凭直觉判断瓶颈位置
- 忽略告警继续开发

优先检查：
- 告警级别和 SLA 要求
- 瓶颈的根因是什么
- 优化方案是否会影响其他模块

---

### 如果修改 ECS 查询模式

允许：
- 添加 Changed/Added 过滤器优化查询
- 拆分大型查询为多个小查询
- 使用 par_iter() 并行处理无依赖查询

禁止：
- 删除已有的变更检测过滤器
- 将小查询合并为大型查询
- 在有共享可变状态时使用并行查询

优先检查：
- 查询组件数量是否 ≤ 6 个
- 是否避免了全 Entity 扫描
- 变更检测是否正确使用

---

### 如果调整内存预算

允许：
- 根据实际测量调整预算值
- 新增内存监控指标
- 优化内存占用后降低预算

禁止：
- 不经审批就增加内存预算
- 删除内存监控机制
- 忽略内存告警

优先检查：
- 调整后的预算是否合理
- 是否有对应的监控和告警
- 是否影响其他模块的内存使用
