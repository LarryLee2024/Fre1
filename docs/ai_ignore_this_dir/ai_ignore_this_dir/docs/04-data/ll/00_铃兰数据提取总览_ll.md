## 三、跨领域依赖关系

```
Tag ──────→ Attribute（Tag决定属性修正是否生效）
  └──────→ Trigger（Tag作为触发条件）
  └──────→ Stacking（Tag决定互斥/叠加规则）
  └──────→ Effect（Tag决定驱散/净化目标）

Attribute ──→ Modifier（属性修正通过Modifier表达）
  └──────→ Execution（属性值作为伤害/治疗计算输入）

Modifier ──→ Attribute（Modifier改变属性值）
  └──────→ Stacking（Modifier叠加方式由Stacking策略决定）

Ability ──→ Effect（Ability引用Effect列表）
  └──→ Targeting（Ability引用目标选择规则）
  └──→ Trigger（反应技通过Trigger触发）

Effect ──→ Modifier（Effect可产生Modifier）
  └──→ Tag（Effect可添加/移除Tag）
  └──→ Cue（Effect触发表现）
  └──→ Trigger（Effect触发事件）

Trigger ──→ Effect（Trigger触发Effect执行）
  └──→ Ability（反应技是特殊Ability）

Execution ──→ Attribute（执行算式读取属性值）
  └──→ Modifier（执行算式应用Modifier）

Stacking ──→ Modifier（Stacking决定Modifier叠加方式）
  └──→ Effect（Stacking决定Effect叠加行为）

Pipeline ──→ 全部Core Domain（Pipeline定义执行时序）
Replay ──→ Pipeline（Replay依赖Pipeline确定性）
```

## 五、Data Laws 合规检查

| Data Law | 铃兰设计是否合规 | 说明 |
|----------|-----------------|------|
| 001 Definition/Instance分离 | ✅ 合规 | 基础属性(Definition)与战斗修正(Instance)严格分层 |
| 002 Rule/Content分离 | ✅ 合规 | 技能、Buff、效果均可配置化，公式在代码中 |
| 003 配置只引用ID | ✅ 合规 | 技能引用效果列表ID，不重复定义 |
| 004 Ability不拥有行为 | ⚠️ 部分合规 | 反应技归属Trigger领域，但原文档中反应技仍归在技能分类下 |
| 005 Effect是唯一业务执行入口 | ✅ 合规 | 所有业务结果（伤害/治疗/Buff/位移）都经过Effect |
| 006 Modifier不拥有业务逻辑 | ✅ 合规 | Modifier只改变数值，无事件处理逻辑 |
| 007 Duration属于Effect | ✅ 合规 | Buff持续时间统一在Effect中定义 |
| 008 堆叠行为归属Stacking | ✅ 合规 | 所有叠加规则统一管理，不散落 |
| 009 表现经过Cue | ✅ 合规 | Cue 是表现层唯一入口，所有 Effect 结果通过 CueEvent 广播（详见 12_Cue_ll.md） |

## 六、国际化支持（ADR-017）

基于 `ADR-017-国际化架构决策.md`，所有玩家可见文本均采用本地化Key模式：

### 6.1 Key命名规范

```
<namespace>.<permanent_id>.<field>
```

| 命名空间 | 领域 | Key示例 |
|----------|------|---------|
| `attr` | Attribute | `attr.a_001.name` |
| `tag` | Tag | `tag.t_001.name` |
| `buff` | Effect | `buff.b_001.name` |
| `skill` | Ability | `skill.s_1001.name` |

### 6.2 字段替换映射

| 原字段 | 新字段 | 适用领域 |
|--------|--------|----------|
| `name` | `name_key` | Attribute、Tag、Effect、Ability |
| `description` | `desc_key` | Effect、Ability |

### 6.3 FTL文件组织

```text
assets/localization/
├── zh-CN/
│   ├── attr.ftl    # 属性文本
│   ├── tag.ftl     # 标签文本
│   ├── buff.ftl    # Buff文本
│   ├── skill.ftl   # 技能文本
│   ├── ui.ftl      # 界面文本
│   └── battle.ftl  # 战斗文本
├── en-US/
│   └── ...
└── ja-JP/
    └── ...
```

### 6.4 语言回退链

```
请求语言 → zh-CN（项目默认）→ en-US（兜底）→ Key本身(Debug)/空字符串(Release)
```

### 6.5 禁止事项

| 禁止项 | 理由 |
|--------|------|
| 硬编码文本 | 违反宪法 §17.2.2 |
| 语义化Key（如`skill.fireball.name`） | 改名后批量替换灾难 |
| 无意义编号Key（如`text_001`） | 维护时无法识别内容 |
| Core层直接调用LocalizationService | Core只存Key，解析在上层 |

## 七、提取方法论

1. **实体识别**：从文档中提取所有可独立存在的数据实体（属性、技能、Buff、Tag等）
2. **字段提取**：对每个实体提取其数据字段、类型、约束
3. **关系梳理**：识别实体间的引用、依赖、互斥关系
4. **层级判定**：将每个数据元素归入 Definition/Instance/Runtime/Persistence 层
5. **Schema草案**：为每个领域输出可落地的配置Schema结构
6. **合规校验**：对照 Data Laws 001-010 逐条检查

## 八、代码实现映射（v1.0 · 2026-06-16 新增）

### 加载链路

```
content/*.ron（磁盘）
    │ RON 反序列化
    ↓
*Def 结构体（Definition 层，不可变）
    │ From<Def> for Data
    ↓
*Data 结构体（Definition 层，不可变）
    │ 存入 Registry（Resource）
    ↓
Registry.get(&id) → &Data（只读查询）
    │ spawn 时从 Template 复制
    ↓
ECS Component（Instance 层，可变）
```

### 数值约定

| 约定 | 说明 | 示例 |
|------|------|------|
| 整数为主 | 游戏数值统一 `i32` | HP=100, ATK=5 |
| 万分比 | 百分比用万分比 | 50% = 5000, 150% = 15000 |
| 向下取整 | 伤害最终向下取整 | 37.8 → 37 |
| 最小值 | 伤害 ≥ 1，治疗 ≥ 0 | — |
| 封顶值 | 暴击 ≤ 95%，闪避 ≤ 80%，减伤 ≤ 90% | — |
