# 我建议直接制定一份《可见性宪法》

## 第一原则

默认：

```rust
private
```

不是：

```rust
pub
```

---

优先级：

```text
private
↓
pub(super)
↓
pub(crate)
↓
pub
```

能不用 pub 就不用 pub。

---

# 最终推荐（适合你的项目）

记住这个矩阵即可：

| 场景            | 可见性              |
| ------------- | ---------------- |
| 当前文件          | private          |
| 同模块           | private          |
| 同领域           | pub(crate)       |
| 领域内 src/tests | pub(crate)       |
| 领域外 tests     | pub              |
| 同 Crate       | pub(crate)       |
| 跨 Crate       | pub              |
| 测试工具          | #[cfg(test)]     |
| 测试共享库         | test_utils crate |
| 跨领域调用         | Facade + pub     |

你的项目如果按这个规则执行，最终大概会变成：

```text
private
≈ 70%

pub(crate)
≈ 25%

pub
≈ 5%
```

这是我见过最适合大型 Rust/Bevy DDD 项目的比例。

如果一个领域出现：

```text
pub
超过20%
```

基本就意味着领域边界已经开始腐化，需要重构了。


是的，对于你这种 **Workspace + 多领域 Crate + DDD** 的大型项目，`pub(in path)` 系列其实非常有价值，但不要滥用。

先说结论：

```text
private        ★★★★★ 70%
pub(super)     ★★★★☆
pub(crate)     ★★★★★ 25%
pub(in path)   ★★★☆☆ 3%
pub            ★★☆☆☆ 2%
```

大多数情况：

```rust
private
pub(crate)
```

就够了。

# pub(in path) 是干什么的

例如：

```rust
character/
├── model/
├── service/
├── facade/
└── mod.rs
```

Character：

```rust
// model/character.rs

pub(in crate::service) struct Character {
    hp: i32,
}
```

意味着：

```rust
crate::service
```

可以访问。

但是：

```rust
crate::facade
```

不能访问。

---

这比：

```rust
pub(crate)
```

更严格。

# 什么时候值得用 pub(in)

我认为只有一种情况：

## 分层架构

例如你的 Ability 系统：

```text
ability/

src/
├── l0_atom/
├── l1_base/
├── l2_execution/
├── l3_entity/
└── facade/
```

你希望：

```text
l2
可以访问 l1

l3
可以访问 l2

facade
不能访问内部细节
```

此时：

```rust
pub(in crate::l2_execution)
```

是合理的。

---

例如：

```rust
// l1_base

pub(in crate::l2_execution)
struct AbilityContext;
```

这样：

```rust
l2_execution
```

可以用。

而：

```rust
l3_entity
facade
```

都看不到。

---

这种叫：

```text
Layer Boundary Enforcement
```

层边界约束。

---

# 但对于领域建模

例如：

```text
character
ability
effect
battle
```

这种领域模块。

不要写：

```rust
pub(in crate::ability)
```

这种东西。

因为：

```rust
ability
```

根本不应该知道：

```rust
character
```

内部结构。

---

正确做法：

```rust
pub struct CharacterFacade;
```

或者：

```rust
pub trait CharacterReadApi;
```

暴露给外部。

# 对你的项目的最终建议

如果我是你，会制定下面规则：

```rust
// 默认
private

// 同领域共享
pub(crate)

// 分层约束（极少数）
pub(in crate::l2_execution)

// 对外API
pub
```

然后在 AI 宪法里写：

```text
禁止新增 pub(in crate::xxx)

除非：
1. 用于层级边界控制
2. ADR明确批准
3. code-review通过

否则统一使用 pub(crate)
```

这样几十万行以后，你不会看到：

```rust
pub(in crate::a)
pub(in crate::b)
pub(in crate::c)
pub(in crate::d)
```

这种维护噩梦。

对于你的 14领域 + Bevy SRPG，我预计：

```text
private        70%
pub(crate)     25%
pub            4%
pub(in ...)    1%
```

这是长期维护最舒服的状态。
