比 Gameplay 测试更重要。

推荐三层：

### Widget Test

测试：

```text
按钮

列表
```

---

### Screen Test

测试：

```text
BattleScreen
InventoryScreen
```

---

### Snapshot Test

```text
UI树结构
```

是否变化。

# 最后一个真正大型项目才会做的东西

## UI Schema

不要让 AI 自己猜 Widget。

建立：

```text
docs/ui_schema/
```

例如：

```yaml
SkillPanel:
  input:
    SkillPanelVm

  output:
    UiAction

  children:
    SkillButton
    SkillTooltip
```

---

然后：

```text
AI写UI
```

必须遵守 Schema。

---

这样到 50 万行以后：

```text
UI不是一堆Node

而是：
Screen
→ Widget
→ Contract
→ VM
→ Action
```

形成完整体系。

这是我见过最接近大型商业项目（UE UMG、Unity UIToolkit、React大型前端）的做法，也最适合你这种长期维护的 Bevy SRPG。
对于你这个架构，我的答案非常明确：

> **UI 不应该放进 Core、Infra、Content、Tools、Modding 任何一个现有目录。**
>
> UI 应该成为和它们平级的第 8 个顶层模块：
>
> ```text
> src/
> ├── app/
> ├── content/
> ├── core/
> ├── infra/
> ├── modding/
> ├── shared/
> ├── tools/
> └── ui/      ← 新增
> ```

这是我对大型项目最强烈的建议之一。

# 为什么不能放 Core

很多人第一反应：

```text
BattleUI
InventoryUI
QuestUI
```

属于：

```text
combat
inventory
quest
```

所以放：

```text
core/domains/
```

---

这是错的。

因为：

```text
Quest
≠
QuestUI
```

---

Quest 属于：

```text
业务规则
```

---

QuestUI 属于：

```text
表现层
```

---

如果放进去：

```text
core/
└── quest/
    ├── domain
    └── ui
```

几年后一定出现：

```rust
QuestUiSystem
    Query<Quest>
```

这种东西。

---

然后：

```text
领域开始依赖表现
表现开始依赖领域
```

边界彻底消失。

---

DDD里面：

```text
UI永远不是Domain
```

---

# 为什么不能放 Infra

很多团队会这么做：

```text
infra/
├── rendering/
├── input/
├── ui/
```

---

这是 Unity 思维。

---

问题：

UI不是技术实现。

---

例如：

```text
Tooltip
InventoryScreen
BattleHud
ShopScreen
```

这些不是：

```text
Rendering
Audio
Network
Save
```

这种基础设施。

---

它们属于：

```text
Application Presentation
```

---

而不是：

```text
Infrastructure
```

---

# 为什么不能放 App

很多项目：

```text
app/
├── battle_ui
├── menu_ui
├── inventory_ui
```

---

前期很舒服。

---

20万行以后：

```text
app/
    变成垃圾场
```

---

因为：

```text
启动逻辑
插件注册
状态机
UI
```

全部混一起。

---

App 应该保持：

```text
Composition Root
```

身份。

---

唯独缺：

我会这样改：


```text

Projection
Widget
Screen

Theme

依赖:

Infra/Input
```

---

如果是我评审这个架构文档。

我会直接写一个 Review Comment：

```text
当前架构缺失 Presentation Layer。

UI 不属于 Domain(Core)，
不属于 Infrastructure(Infra)，
不属于 Composition Root(App)。

建议新增 src/ui 作为独立顶层模块，
对应 DDD 的 Presentation Layer。

这是未来 50 万行项目避免
“UI与业务逻辑耦合”
最重要的结构边界之一。
```

这会比把 UI 塞进 `infra/ui` 或 `app/ui` 多撑至少几年。
