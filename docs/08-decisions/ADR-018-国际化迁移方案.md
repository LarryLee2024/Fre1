# ADR-018: 国际化迁移方案

## 状态

Proposed

## 背景

项目当前存在大量硬编码中文文本，需要迁移到国际化 Key 模式。迁移涉及三个维度：

1. **RON 配置**：所有 `content/` 下的 RON 文件包含硬编码 `name:` 和 `description:` 字段
2. **UI 代码**：21 处 `Text::new("...")` 硬编码中文字符串
3. **测试数据**：235+ 处 `name: "..."` 硬编码（大部分在测试中）

`src/infrastructure/localization/` 当前为空壳模块（2行注释），需要从零实现。

## 引用的领域规则

- `docs/01-architecture/i18n-design.md` — §4 Content 层 Key 驱动架构
- `docs/03-technical/localization-rules.md` — 不变量1：Content 配置永远只存 Key
- `docs/01-architecture/content-pipeline.md` — RON → Def → Data → Registry 流程

### 2. 迁移策略：分阶段渐进迁移

#### Phase 1：基础设施搭建（Day 1-2）

**目标**：实现 LocalizationService 核心，不修改任何现有代码。

```
src/infrastructure/localization/
├── mod.rs              # 模块入口
├── plugin.rs           # LocalizationPlugin（空注册）
├── locale.rs           # Locale 枚举（zh-CN, en-US, ja-JP）
├── service.rs          # LocalizationService（fluent-rs 封装）
├── ftl_loader.rs       # 自定义 AssetLoader
├── component.rs        # LocalizedText 组件
├── systems.rs          # resolve_localized_texts 系统
├── cache.rs            # LocalizedTextCache
└── font_fallback.rs    # FontFallbackChain
```

**验证**：`cargo build` 通过，`cargo test` 通过，现有功能不受影响。

#### Phase 2：FTL 文件创建（Day 2-3）

**目标**：为所有现有硬编码文本创建 FTL 翻译文件。

```
assets/localization/
├── zh-CN/
│   ├── ui.ftl          # 从 UI 代码提取
│   ├── battle.ftl      # 从战斗日志提取
│   ├── skill.ftl       # 从技能 RON 提取
│   ├── buff.ftl        # 从 Buff RON 提取
│   ├── character.ftl   # 从角色 RON 提取
│   └── system.ftl      # 系统文本
└── en-US/
    ├── ui.ftl
    ├── battle.ftl
    ├── skill.ftl
    ├── buff.ftl
    ├── character.ftl
    └── system.ftl
```

**Key 分配**：为每个现有文本分配永久唯一 ID。

示例（技能）：
```ftl
# skill.ftl
# s_1001: basic_attack（普通攻击）
skill.s_1001.name = 普通攻击
skill.s_1001.desc = 基础物理攻击

# s_1002: charge（冲锋）
skill.s_1002.name = 冲锋
skill.s_1002.desc = 猛冲敌人，造成1.5倍伤害

# s_1003: pierce（穿透箭）
skill.s_1003.name = 穿透箭
skill.s_1003.desc = 远程穿透射击，无视50%防御

# s_1004: fireball（火球）
skill.s_1004.name = 火球
skill.s_1004.desc = 发射火球，造成1.8倍伤害并附加灼烧

# s_1005: heal（治疗）
skill.s_1005.name = 治疗
skill.s_1005.desc = 恢复友方单位8点HP

# s_1006: cleanse_skill（净化）
skill.s_1006.name = 净化
skill.s_1006.desc = 移除友方单位所有减益效果
```

示例（UI）：
```ftl
# ui.ftl
ui.main_menu.title = 回合制战棋
ui.main_menu.version = v0.1.0
ui.level_select.back = ← 返回
ui.level_select.enter_battle = 进入战斗

ui.turn_indicator.round = 第 { $round } 回合
ui.combat_log.title = 战斗日志
```

#### Phase 3：RON 配置迁移（Day 3-5）

**目标**：将所有 RON 配置的 `name:` 替换为 `name_key:`，`description:` 替换为 `desc_key:`。

**迁移映射表**：

| RON 文件 | 旧字段 | 新字段 | Key 格式 |
|---------|--------|--------|---------|
| `skills/fireball.ron` | `name: "火球"` | `name_key: "skill.s_1004.name"` | `skill.{永久ID}.name` |
| `skills/fireball.ron` | `description: "..."` | `desc_key: "skill.s_1004.desc"` | `skill.{永久ID}.desc` |
| `characters/player_warrior.ron` | `name: "战士"` | `name_key: "char.c_001.name"` | `char.{永久ID}.name` |
| `buffs/burn.ron` | `name: "灼-2"` | `name_key: "buff.b_001.name"` | `buff.{永久ID}.name` |

**永久 ID 分配规则**：
- 技能：`s_1001` ~ `s_9999`
- 角色：`c_001` ~ `c_999`
- Buff：`b_001` ~ `b_999`
- 装备：`e_001` ~ `e_999`
- 物品：`i_001` ~ `i_999`
- 地形：`t_001` ~ `t_999`
- 关卡：`st_001` ~ `st_999`

**RON 结构变更**：

```ron
// 迁移前
(
    id: "fireball",
    name: "火球",
    description: "发射火球，造成1.8倍伤害并附加灼烧",
    ...
)

// 迁移后
(
    id: "fireball",
    name_key: "skill.s_1004.name",
    desc_key: "skill.s_1004.desc",
    ...
)
```

**向后兼容**：`name_key` 和 `desc_key` 使用 `#[serde(default)]`，旧配置（无 key 字段）仍可反序列化，回退到 `name` 字段。

#### Phase 4：UI 代码迁移（Day 5-7）

**目标**：将所有 `Text::new("...")` 替换为 `LocalizedText` 组件。

**迁移模式**：

```rust
// 迁移前
Text::new("回合制战棋")

// 迁移后
Text::new("")
// + LocalizedText { key: "ui.main_menu.title".into(), args: None }
```

**注意**：`Text::new("")` 作为占位符，系统会在下一帧通过 `resolve_localized_texts` 替换为实际文本。

#### Phase 5：插件注册与集成（Day 7-8）

**目标**：将 `LocalizationPlugin` 注册到 App 启动流程。

```rust
// src/app/plugins.rs 或 src/main.rs
app.add_plugins(LocalizationPlugin);
```

**验证**：游戏启动时加载 FTL 文件，UI 文本正确显示。

### 5. 迁移顺序与依赖

```
Phase 1: 基础设施（无依赖）
    ↓
Phase 2: FTL 文件（无依赖）
    ↓
Phase 3: RON 配置迁移（依赖 Phase 2 的 Key 分配）
    ↓
Phase 4: UI 代码迁移（依赖 Phase 1 的 LocalizedText 组件）
    ↓
Phase 5: 插件注册（依赖 Phase 1-4 全部完成）
```

每个 Phase 完成后必须：`cargo build` 通过 + `cargo test` 通过 + 现有功能不受影响。

## Module Design

### SkillDef 结构变更

```rust
// 迁移前
pub struct SkillDef {
    pub id: String,
    pub name: String,
    pub description: String,
    // ...
}

// 迁移后
pub struct SkillDef {
    pub id: String,
    #[serde(default)]
    pub name: String,           // 旧字段，兼容
    #[serde(default)]
    pub description: String,    // 旧字段，兼容
    #[serde(default)]
    pub name_key: Option<String>,   // 新字段：本地化 Key
    #[serde(default)]
    pub desc_key: Option<String>,   // 新字段：本地化 Key
    // ...
}
```

**向后兼容**：`name_key` 和 `desc_key` 为 `Option<String>`，旧 RON 配置（无 key 字段）仍可反序列化。运行时优先使用 `name_key`，回退到 `name`。

### 运行时解析流程

```
SkillData.name_key
    ↓
LocalizationService.resolve(key, locale)
    ↓
FluentBundle.get_message() + format()
    ↓
显示文本
```

## Communication Design

### 数据流

```
RON (name_key) → SkillDef → SkillData → UI 查询 → LocalizationService → 显示文本
                                     ↑
                              LocalizedText 组件
```

### Message

| Message | 发送方 | 接收方 | 时机 |
|---------|--------|--------|------|
| `LanguageChangedEvent` | 设置系统 | resolve_localized_texts | 语言切换时 |

## 边界定义

- 允许：`core/` 模块存储 `name_key` 字段（纯数据）
- 允许：`infrastructure/localization/` 被 `content/` 和 `ui/` 依赖
- 禁止：`core/` 直接调用 `LocalizationService`（Core 只存 Key）
- 禁止：`infrastructure/localization/` 依赖 `core/`（Infrastructure 不依赖 Core）

## Forbidden（禁止事项）

- 🟥 禁止：在迁移过程中破坏现有功能 — 理由：每个 Phase 必须保持编译通过和测试通过
- 🟥 禁止：一次性修改所有文件 — 理由：必须分阶段渐进迁移，每阶段独立验证
- 🟥 禁止：删除旧的 `name` 字段 — 理由：需要向后兼容，旧 RON 配置仍需反序列化
- 🟥 禁止：在测试代码中迁移硬编码 — 理由：测试数据不影响运行时，可后续按需迁移
- 🟥 禁止：跳过 Phase 直接修改 UI — 理由：必须先有 LocalizedText 组件才能替换
- 🟥 禁止：修改已有 Key 的命名 — 理由：Key 一旦创建绝对不允许修改
- 🟥 禁止：在 FTL 文件中硬编码数值 — 理由：数值应通过 Fluent 变量注入

## Definition / Instance Design

- Definition：FTL 文件（翻译表）、RON 配置（name_key 字段）
- Instance：FluentBundle（运行时缓存）、LocalizedText（组件）、CurrentLocale（Resource）

## 后果

### 正面
- 所有文本集中管理，新增语言零代码修改
- MOD 自带翻译，主工程无感知
- 平衡调整时描述中的变量自动更新
- 迁移后代码库符合宪法 §17.2.2

### 负面
- 35 个 RON 文件需要迁移（一次性成本）
- 21 处 UI 代码需要替换（一次性成本）
- 235+ 处测试数据中的硬编码暂不迁移（技术债）
- 永久 ID 分配表需要人工维护（但 ID 一旦分配不再变化）

## 替代方案

| 方案 | 优点 | 缺点 | 为何放弃 |
|------|------|------|----------|
| 全量一次性迁移 | 简单 | 风险高，容易遗漏 | 必须分阶段渐进 |
| 只迁移 UI 不迁移 RON | 快速 | Content 仍有硬编码 | 违反宪法 §17.2.2 |
| 只迁移 RON 不迁移 UI | 快速 | UI 仍有硬编码 | 违反宪法 §17.2.2 |
| 使用语义化 Key | 直观 | 改名后批量替换灾难 | 宪法明确禁止 |
| 不迁移，保持现状 | 零成本 | 无法多语言，违反宪法 | 违反 §17.2.2 |
