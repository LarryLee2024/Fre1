# ADR-029: 数据模型完全重构总纲

## 状态
Accepted（2026-06-15）

## 背景

当前项目数据模型经历了从早期架构到七层架构（v4.0+）、再到 SRPG Lite-GAS 对齐（ADR-026）的多次迁移，累计了大量技术债务：

1. **Buff 模块（3030 行/9 文件）** 已标记废弃但代码未删除，仍被其他模块引用
2. **Execution 双路径并行** — 旧的 `calculate_damage_from_effect()` 与新的 `Execution` trait 同时存在
3. **22 个 Registry 使用不一致** — 仅 15/22 使用 `RegistryLoader` trait，其余采用手写加载
4. **ID 系统重复定义** — `BuffId`/`ItemId` 在 `shared/ids/` 和 `core/` 有重复副本
5. **无 `define_id!` 宏** — 8+ 个 ID 类型手写相同样板代码
6. **targeting/resolver.rs 为占位符**（3 行注释）
7. **Movement 模块为空壳**
8. **旧 RON 格式与 Linglan 模型不兼容**

### 重构目标

以 `docs/04-data/ll/`（铃兰之剑数据模型）为唯一目标蓝图，采用**完全重构、零兼容、零技术债务**策略：
- 不保留任何废弃代码路径
- 不提供向后兼容层
- 所有模块一次性重写到位
- 旧模块直接删除，不保留"迁移期间共存"状态

## 决策

### 重构范围

13 个领域全覆盖（与 Linglan 模型完全对齐）：

| 领域 | ADR | 当前状态（KLoC） | 操作 |
|------|-----|-----------------|------|
| ID/Registry | ADR-030 | 跨层 | 完全重写 — define_id! + 统一 Registry |
| Attribute | ADR-031 | 1.5 | 完全重写 — Linglan 5+6 属性模型 |
| Tag | ADR-031 | 0.8 | 完全重写 — 4 分类强制 |
| Effect | ADR-032 | 1.3 | 完全重写 — 新 Pipeline |
| Modifier | ADR-032 | 0.6 | 完全重写 — 8 种操作 |
| Execution | ADR-032 | 0.6 | 完全重写 — 4 段公式 |
| Stacking | ADR-032 | 0.4 | 完全重写 — 8 类型 |
| Ability | ADR-033 | 2.0 | 完全重写 — 5 分类+5 阶段 |
| Trigger | ADR-033 | 1.0 | 完全重写 — 5 大类事件 |
| Targeting | ADR-033 | 0.3 | 完全重写 — 填充占位符 |
| Cue | ADR-034 | 0.4 | 完全重写 — 全覆盖信号 |
| Replay | ADR-034 | 0（需新建） | 全新实现 |
| I18n | ADR-034 | 0（需新建） | 全新实现 |
| **Buff（删除）** | ADR-035 | 3.0（9 文件） | **删除** |
| **Movement（填充）** | ADR-035 | 0.03（空壳） | 完全重写 |

### 执行策略

**5 阶段顺序执行**，每阶段完成后必须通过编译+测试门禁：

```
Phase 1: 基础设施重写 (ADR-030) → Shared 层 + ID + Registry
Phase 2: 基础数据重写 (ADR-031) → Attribute + Tag
Phase 3: 核心管线重写 (ADR-032) → Effect/Execution/Stacking/Modifier + 删除 Buff
Phase 4: 能力系统重写 (ADR-033) → Ability/Trigger/Targeting
Phase 5: 表现层实现 (ADR-034) → Cue/Replay/I18n
Phase 6: 清理收尾 (ADR-035) → Buff 删除确认、Movement 填充、全量回归
```

### 重写原则

| 原则 | 说明 |
|------|------|
| **一次性替换** | 不渐进迁移，不保留旧代码路径 |
| **零兼容模式** | 不允许 `#[allow(deprecated)]`、兼容重导出、条件编译兼容分支 |
| **删除 > 注释** | 废弃代码必须删除，不得保留为注释 |
| **编译门禁** | 每 Phase 完成后 `cargo build` 零错误、`cargo test` 零失败 |
| **类型级约束** | ID 必须是强类型 newtype，禁止 `String` 作为 Registry key |
| **数据驱动** | 公式值在 RON 配置中，代码只承载规则 |
| **确定性优先** | 所有计算路径必须确定（无 f32、无系统时间、无外部熵） |

## Module Design

```
src/
├── shared/
│   ├── ids/                  # define_id! macro + 所有强类型 ID（Phase 1）
│   └── registry/             # 统一 Registry trait（Phase 1）
├── core/
│   ├── attribute/            # Linglan 5+6 属性模型（Phase 2）
│   ├── tag/                  # Linglan 4 分类标签（Phase 2）
│   ├── effect/               # 新 Effect Pipeline（Phase 3）
│   │   ├── types.rs          # EffectDef + EffectType + DurationDef
│   │   ├── handler.rs        # EffectHandler trait（替代 942 行旧文件）
│   │   └── ...               # 每种 Effect 独立文件
│   ├── execution/            # Execution trait（Phase 3）
│   │   ├── damage.rs         # 4 段伤害公式
│   │   ├── heal.rs           # 治疗公式
│   │   └── shield.rs         # 护盾公式
│   ├── stacking/             # 8 类型堆叠策略（Phase 3）
│   ├── modifier/             # Modifier 管线（Phase 3）
│   ├── ability/              # 新 Ability 系统（Phase 4）
│   ├── trigger/              # 统一 Trigger 系统（Phase 4，吸收旧 buff/trigger.rs）
│   ├── targeting/            # TargetingDefinition + resolver（Phase 4）
│   ├── cue/                  # Cue 事件总线（Phase 5）
│   ├── replay/               # 回放系统（Phase 5，新建）
│   └── pipeline/             # 全链路调度（Phase 3 → Phase 5）
├── infrastructure/
│   └── localization/         # Fluent 国际化（Phase 5，新建）
└── content/                  # 所有 RON 文件按新 Schema 重写（Phase 2-5）
```

## Communication Design

### Phase 间依赖

```
Phase 1 (ID/Registry) → 无上游依赖
Phase 2 (Attr/Tag) → 依赖 Phase 1
Phase 3 (Pipeline) → 依赖 Phase 1 + 2
Phase 4 (Ability) → 依赖 Phase 1 + 2 + 3
Phase 5 (Cue/Replay/I18n) → 依赖 Phase 1-4
Phase 6 (Cleanup) → 依赖所有 Phase
```

### 跨模块通信

| 通信方式 | 用途 | 示例 |
|----------|------|------|
| Message | 跨 Feature 广播 | `CharacterDied`, `TurnEnded` |
| Observer | 同 Feature 响应 | Buff 过期自动清理 |
| Hook | Component 固有行为 | `Dead` → `remove_moveable` |
| Cue | 表现信号 | `CueDamage`, `CueDeath` |

## 边界定义

| 规则 | 允许 | 禁止 |
|------|------|------|
| Core → Infrastructure | 通过共享事件 | 直接调用 LocalizationService |
| Core → UI | Cue 事件 | 直接修改 UI 组件 |
| Shared → 其他 | 无 | `shared/` 引入外部依赖 |
| Content → Core | 通过 Registry | RON 中包含游戏规则 |

## Forbidden（禁止事项）

- 🟥 **禁止** 保留旧 `Buff` 模块的任何代码 — 必须在 Phase 3 开始时整个目录删除
- 🟥 **禁止** 创建"兼容模式"数据路径 — 新旧系统不共存
- 🟥 **禁止** 使用 `#[allow(deprecated)]`、`#[allow(warnings)]` 绕过编译错误
- 🟥 **禁止** `HashMap<String, T>` 作为 Registry 内部存储 — 必须使用 `HashMap<StrongId, T>`
- 🟥 **禁止** Phase 跨越时的临时兼容导出 — 每个 Phase 完成后所有旧类型必须不可访问
- 🟥 **禁止** `f32` 在核心战斗逻辑中出现 — 使用 `i32`/`i64` 整数计算
- 🟥 **禁止** `Box<dyn EffectHandler>` 动态分发 — 使用 `ExecutionRegistry` + 具体类型
- 🟥 **禁止** 在 `content/*.ron` 中硬编码玩家可见文本 — 必须使用 `name_key`/`desc_key`
- 🟥 **禁止** 引入任何新的第三方程碑式依赖 — 优先使用 Bevy 原生能力

## Definition / Instance Design

所有 13 个领域必须严格遵循 Definition/Instance 分离（Data Law 001）：

| 领域 | Definition（不可变） | Instance（运行时） |
|------|---------------------|-------------------|
| Attribute | `AttributeDefinition`（RON） | `Attributes` Component |
| Tag | `TagDefinition`（RON） | `GameplayTags` Component |
| Modifier | `ModifierDefinition`（RON） | `ModifierInstance` Component |
| Effect | `EffectDefinition`（RON） | `EffectInstance` Component |
| Ability | `AbilityDefinition`（RON） | `AbilityInstance` Component |
| Trigger | `TriggerDefinition`（RON） | `TriggerInstance` Component |
| Targeting | `TargetingDefinition`（RON） | —（纯函数） |
| Execution | `ExecutionDefinition`（RON 公式） | `ExecutionResult`（Runtime） |
| Stacking | `StackingDefinition`（RON） | `StackingState` Component |
| Cue | `CueDefinition`（RON） | `CueEvent`（Runtime） |
| Pipeline | `PipelineConfig`（RON） | —（纯调度） |
| Replay | — | `ReplayEvent`（Persistence） |
| I18n | FTL 文件 | `FluentBundle` Resource |

## 后果

### 正面
- 消除所有技术债务（旧 Buff 模块、双路径执行、占位符文件）
- 与 SRPG Lite-GAS 架构完全对齐
- 所有 Registry 使用一致的加载/查询接口
- 所有 ID 使用宏生成，消除样板代码
- 确定性执行 + Replay 兼容性根植于数据模型设计

### 负面
- 一次性重构成本高（涉及 30+ 文件、~31K 行代码重写）
- 重构期间项目不可用（需 Feature Flag 隔离或独立分支）
- 团队需要重新熟悉所有模块
- 已有的 RON 配置文件（41 文件）全部需要迁移到新 Schema

### 风险缓解
- 每 Phase 独立编译+测试门禁，避免大爆炸式集成风险
- 每 Phase 完成后可独立回滚（Git Tag）
- 第 1 Phase 仅影响 Shared 层，风险最低
- 使用 `--features dev` 在开发期间保持 Debug 工具可用

## 替代方案（已拒绝）

| 方案 | 拒绝原因 |
|------|----------|
| 渐进式迁移 | 会延续当前"新旧共存"的混乱状态，累计更多兼容性代码 |
| 仅修改 RON 格式 | 不解决 ID 重复、Registry 不一致、Buff 模块残留等基础设施问题 |
| 先 Refactor 后 Re-architect | 当前架构不完全性已影响开发效率，需一次性根治 |
| 保留旧模块作参考 | 旧模块本身就是垃圾来源，保留只会误导后续开发者 |
