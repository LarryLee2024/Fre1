# ADR-013: 技能数据模型与配置规范

## 状态

Proposed

## 背景

技能系统是 SRPG 战斗的核心数据模型之一。当前项目已有 6 个技能（basic_attack, charge, pierce, fireball, heal, cleanse_skill）通过 RON 配置驱动，但缺少正式的数据模型架构决策。本 ADR 定义技能的 Definition/Instance 分离策略、RON 配置契约、字段规范和版本管理。

## 引用的领域规则

- `docs/02-domain/skill/skill-rules.md` — 技能系统领域规则（定义、冷却、释放管线）
- `docs/01-architecture/skill-buff-abstraction.md` — 技能/Buff/Effect 统一抽象模型
- `docs/01-architecture/README.md` — 七层架构、Definition/Instance 分离原则

## 决策

### 1. 双类型模式（Def/Data 分离）

遵循宪法 1.1.2（Definition/Instance 分离）和宪法 1.1.3（Rule/Content 分离）：

```
SkillDef（RON 反序列化用）
  ├─ 使用 TagName 字符串（RON 友好）
  ├─ 包含 version 字段（配置版本管理）
  └─ impl From<SkillDef> for SkillData

SkillData（运行时用）
  ├─ 使用 GameplayTag 位掩码（O(1) 查询）
  └─ 从 SkillDef 转换后注册到 SkillRegistry
```

### 2. SkillDef RON 格式契约

```ron
(
    id: "fireball",              // 必填，技能唯一标识
    version: 1,                  // 必填，配置版本号
    name: "火球",                // 必填，显示名称
    description: "发射火球",     // 可选，默认空字符串
    cost_mp: 10,                 // 必填，MP 消耗
    range: 3,                    // 必填，技能范围（0=使用基础攻击范围）
    targeting: SingleEnemy,      // 必填，目标类型
    effects: [                   // 必填，效果列表
        Damage(multiplier: 1.8, ignore_def_percent: 0.0),
        ApplyBuff(buff_id: "burn", duration: 2),
    ],
    tags: [FIRE, SKILL_ACTIVE],  // 必填，标签列表
    conditions: [],              // 可选，使用条件
    cooldown: 3,                 // 必填，冷却回合数
    priority: 20,                // 必填，AI 优先级
)
```

### 3. 字段规范

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `id` | String | 是 | — | 全局唯一标识 |
| `version` | u32 | 是 | — | 配置版本号（0=旧配置兼容） |
| `name` | String | 是 | — | 显示名称 |
| `description` | String | 否 | `""` | 描述文本 |
| `cost_mp` | i32 | 是 | — | MP 消耗 |
| `range` | u32 | 是 | — | 技能范围（0=基础攻击范围） |
| `targeting` | SkillTargeting | 是 | — | 目标类型枚举 |
| `effects` | Vec\<EffectDef\> | 是 | — | 效果列表 |
| `tags` | Vec\<TagName\> | 是 | — | 分类标签 |
| `conditions` | Vec\<SkillConditionDef\> | 否 | `[]` | 使用条件 |
| `cooldown` | u32 | 是 | — | 冷却回合数 |
| `priority` | u32 | 是 | — | AI 优先级 |

### 4. SkillTargeting 枚举

```rust
pub enum SkillTargeting {
    SingleEnemy,    // 单体敌方
    SingleAlly,     // 单体友方
    SelfOnly,       // 自身
    AoeEnemies,     // 范围敌方
    AoeAllies,      // 范围友方
    NoTarget,       // 无目标（自身生效）
}
```

`requires_target_selection()` 仅 `SingleEnemy` 和 `SingleAlly` 返回 true。

### 5. EffectDef 枚举

```rust
pub enum EffectDef {
    Damage { multiplier: f32, ignore_def_percent: f32 },
    Heal { amount: i32 },
    ApplyBuff { buff_id: String, duration: u32 },
    Cleanse,
}
```

每个变体必须实现 `type_name()` 方法，与 EffectHandler 的 type_name 对应。

### 6. SkillConditionDef（条件定义）

```rust
pub enum SkillConditionDef {
    MpCost(i32),                    // MP 消耗检查
    RequireTag(TagName),            // 施法者需要标签
    TargetRequireTag(TagName),      // 目标需要标签
    HpBelow(f32),                   // 施法者 HP 低于百分比
    HpAbove(f32),                   // 施法者 HP 高于百分比
}
```

### 7. 版本管理

- 所有 SkillDef 必须包含 `version` 字段
- `#[serde(default)]` 保证旧配置（无 version）兼容
- 版本号递增规则：小版本 +1（新增可选字段），大版本 +1（删除/类型变更）

## Module Design

### 文件组织

```
src/core/skill/
├── mod.rs                    # SkillPlugin 入口
├── domain/
│   ├── mod.rs               # SkillRegistry + RegistryLoader 实现
│   ├── types.rs             # SkillDef, SkillData, SkillTargeting, SkillCondition, SkillUseError, can_use()
│   ├── skill_error.rs       # SkillError 枚举（ADR-011 迁移后）
│   └── defaults.rs          # 内置默认技能注册
├── slots.rs                 # SkillSlots + SkillCooldowns 组件
├── preview.rs               # 技能效果预览 + SkillExecutionContext
└── id.rs                    # 技能 ID（从 shared/ids/ 引用）
```

### Definition / Instance 分离

| 概念 | Definition（不可变） | Instance（运行时） |
|------|---------------------|-------------------|
| 技能 | SkillDef / SkillData | SkillSlots + SkillCooldowns |
| 效果 | EffectDef | PendingEffectData |
| 条件 | SkillConditionDef | SkillCondition |
| 目标 | SkillTargeting | 运行时 Query 结果 |

## Communication Design

### Message

| Message | 发送方 | 接收方 | 用途 |
|---------|--------|--------|------|
| `shared::event::skill::SkillActivated` | Cast 阶段 | UI、日志、回放（参见 ADR-012） | 技能已释放通知 |

### 函数调用

- `SkillData.can_use()` — 纯函数验证，无副作用
- `SkillRegistry.get()` — 按 ID 查询技能定义
- `SkillCooldowns.tick()` — 回合结束时递减冷却

## 边界定义

- 允许：`core/skill/` 依赖 `core/effect/`（EffectDef 类型）、`core/tag/`（GameplayTag）、`core/attribute/`（Attributes）
- 允许：`core/skill/` 依赖 `shared/ids/`（SkillId）、`shared/event/`（SkillActivated）
- 禁止：`core/skill/` 依赖 `ui/`、`infrastructure/`、`debug/`

## Forbidden（禁止事项）

- 🟥 禁止：在 Rust 代码中硬编码技能属性值（如 `const FIREBALL_DAMAGE: i32 = 120`）— 理由：违反 Rule/Content 分离
- 🟥 禁止：为新增技能修改 Rust 代码 — 理由：新增内容 = 新增 RON 文件
- 🟥 禁止：修改 SkillDef 中的字段含义而不更新所有 RON 配置 — 理由：破坏向后兼容
- 🟥 禁止：在 SkillData 中使用 String 查询标签（必须用 GameplayTag 位掩码）— 理由：性能和类型安全
- 🟥 禁止：跳过 `version` 字段 — 理由：配置版本管理是长期维护的基础
- 🟥 禁止：SkillDef 包含运行时状态（如当前冷却）— 理由：Definition/Instance 分离

## Definition / Instance Design

- Definition：SkillDef（RON 反序列化）、SkillData（运行时查询）、EffectDef（效果定义）
- Instance：SkillSlots（技能槽位组件）、SkillCooldowns（冷却追踪组件）

## 后果

### 正面
- 新增技能只需新增 RON 文件，零代码修改
- 配置版本管理支持长期内容迭代
- 双类型模式保证 RON 友好和运行时性能

### 负面
- SkillDef → SkillData 转换增加少量初始化开销
- 版本管理增加配置文件复杂度

## 替代方案

| 方案 | 优点 | 缺点 | 为何放弃 |
|------|------|------|----------|
| 单一 SkillData 类型 | 简单 | RON 中使用位掩码不友好 | TagName 字符串更直观 |
| 运行时反射加载 | 灵活 | 性能差、类型不安全 | 违反宪法 19.0.1 |
| JSON 配置 | 工具链丰富 | Rust 生态不匹配 | RON 是 Rust 原生格式 |
