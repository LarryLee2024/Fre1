# ADR-016: 技能系统扩展点设计

## 状态

Proposed

## 背景

技能系统需要支持新增技能类型、效果类型、条件类型和目标类型，而不修改核心代码。当前项目已实现 EffectHandler trait 分发和 SkillCondition 枚举，但缺少完整的扩展点架构决策。本 ADR 定义四个扩展维度的标准化机制。

## 引用的领域规则

- `docs/02-domain/skill/skill-rules.md` — 规则5：数据驱动技能定义、规则7：标签驱动技能过滤
- `docs/01-architecture/01-battle-gas/skill-buff-abstraction.md` — §3.3 EffectHandler trait 分发、§4 十大补充系统
- `docs/02-domain/attribute-modifier/attribute-modifier-rules.md` — 规则4：EffectHandler trait 分发

## 决策

### 1. 四个扩展维度

```
维度1: 新增效果类型（EffectDef + EffectHandler）
维度2: 新增条件类型（SkillConditionDef + SkillCondition）
维度3: 新增目标类型（SkillTargeting 枚举）
维度4: 新增技能模板（SkillDef RON 配置）
```

### 2. 维度1：新增效果类型

当前 4 种效果：Damage, Heal, ApplyBuff, Cleanse

扩展步骤：
1. 在 `EffectDef` 枚举中新增变体
2. 实现 `EffectHandler` trait（type_name / generate / preview / execute）
3. 注册到 `EffectHandlerRegistry`
4. 新增对应的 `PendingEffectData` 变体
5. 新增对应的 `EffectResultData` 变体

```rust
// 扩展示例：新增 Shield 效果
// Step 1: EffectDef 新增变体
pub enum EffectDef {
    // ... 现有
    Shield { amount: i32, duration: u32 },
}

// Step 2: 实现 EffectHandler
pub struct ShieldHandler;
impl EffectHandler for ShieldHandler {
    fn type_name(&self) -> &'static str { "Shield" }
    fn generate(&self, def: &EffectDef, ctx: &GenerateContext) -> Option<PendingEffectData> { ... }
    fn preview(&self, def: &EffectDef, ctx: &PreviewContext) -> Option<EffectPreview> { ... }
    fn execute(&self, effect: &PendingEffect, ctx: &mut ExecuteContext) -> Option<ExecuteOutput> { ... }
}

// Step 3: 注册
registry.register(Box::new(ShieldHandler));
```

关键约束：
- `type_name()` 必须与 `EffectDef::type_name()` 一致
- `generate()` 返回 `None` 表示类型不匹配
- `execute()` 返回 `None` 表示类型不匹配
- 未注册的效果类型输出 warn 日志并跳过

### 3. 维度2：新增条件类型

当前 5 种条件：MpCost, RequireTag, TargetRequireTag, HpBelow, HpAbove

扩展步骤：
1. 在 `SkillConditionDef` 枚举中新增变体（RON 反序列化用）
2. 在 `SkillCondition` 枚举中新增变体（运行时用）
3. 实现 `From<SkillConditionDef> for SkillCondition`
4. 在 `SkillData.can_use()` 中添加检查分支

```rust
// 扩展示例：新增 HasWeapon 条件
pub enum SkillConditionDef {
    // ... 现有
    HasWeapon(TagName),
}

pub enum SkillCondition {
    // ... 现有
    HasWeapon(GameplayTag),
}

// can_use() 中添加检查
SkillCondition::HasWeapon(tag) => {
    if !source_tags.has(*tag) {
        return Err(SkillUseError::MissingTag { tag: *tag });
    }
}
```

关键约束：
- 条件检查顺序固定（冷却→消耗→标签→HP阈值）
- 新条件必须在正确的位置插入检查顺序
- 缺失参数时条件永不触发（安全默认值）

### 4. 维度3：新增目标类型

当前 6 种目标：SingleEnemy, SingleAlly, SelfOnly, AoeEnemies, AoeAllies, NoTarget

扩展步骤：
1. 在 `SkillTargeting` 枚举中新增变体
2. 实现 `requires_target_selection()` 方法
3. 实现 `label()` 方法
4. 在目标选择系统中添加处理逻辑

```rust
// 扩展示例：新增 EmptyTile 目标
pub enum SkillTargeting {
    // ... 现有
    EmptyTile,  // 空地选择（传送、陷阱放置）
}

impl SkillTargeting {
    pub fn requires_target_selection(&self) -> bool {
        match self {
            // ... 现有
            Self::EmptyTile => true,  // 需要选择空地
        }
    }
}
```

### 5. 维度4：新增技能模板

最简单的扩展——只需新增 RON 文件：

```ron
// content/skills/shield_bash.ron
(
    id: "shield_bash",
    version: 1,
    name: "盾牌猛击",
    description: "用盾牌猛击敌人，造成伤害并晕眩",
    cost_mp: 8,
    range: 1,
    targeting: SingleEnemy,
    effects: [
        Damage(multiplier: 1.2, ignore_def_percent: 0.0),
        ApplyBuff(buff_id: "stun", duration: 1),
    ],
    tags: [MELEE, SKILL_ACTIVE],
    conditions: [RequireTag(WARRIOR)],
    cooldown: 3,
    priority: 15,
)
```

无需修改任何 Rust 代码，SkillRegistry 自动加载。

### 6. 扩展优先级

| 扩展类型 | 当前状态 | 优先级 | 说明 |
|---------|---------|--------|------|
| 新增技能模板 | ✅ 已支持 | 高 | 只需 RON 文件 |
| 新增效果类型 | ✅ 已支持（4种） | 中 | 需实现 EffectHandler |
| 新增条件类型 | ✅ 已支持（5种） | 中 | 需修改 can_use |
| 新增目标类型 | ✅ 已支持（6种） | 低 | 需修改目标选择系统 |
| 新增修饰规则 | ✅ 已支持 | 高 | 只需 RON 文件 |
| 新增触发时机 | ❌ 待建设 | 中 | 需 TriggerRegistry |
| 新增持续策略 | ❌ 待建设 | 中 | 需 DurationPolicy |
| 新增叠层策略 | ❌ 待建设 | 中 | 需 StackPolicy |

## Module Design

### 扩展点注册位置

```
src/core/effect/handler.rs      → EffectHandler trait + EffectHandlerRegistry
src/core/skill/domain/types.rs  → SkillConditionDef/SkillCondition
src/core/skill/domain/mod.rs    → SkillRegistry + can_use()
src/core/modifier_rule.rs       → ModifierRule + ModifierCalculatorRegistry
```

### 新增效果类型的文件变更

| 文件 | 变更 |
|------|------|
| `core/effect/types.rs` | EffectDef 新增变体、PendingEffectData 新增变体 |
| `core/effect/handler.rs` | 新增 Handler 实现、注册到 Registry |
| `core/effect/mod.rs` | 无变更（模块结构不变） |

## Communication Design

### 扩展点注册流程

```
新增效果类型：
  1. 修改 EffectDef（types.rs）
  2. 新增 Handler（handler.rs）
  3. 注册到 EffectHandlerRegistry（handler.rs）

新增条件类型：
  1. 修改 SkillConditionDef（types.rs）
  2. 修改 SkillCondition（types.rs）
  3. 修改 can_use()（types.rs）

新增技能模板：
  1. 新增 RON 文件（content/skills/）
  2. 无需修改代码
```

## 边界定义

- 允许：新增 EffectHandler 实现（通过 trait 扩展）
- 允许：新增 SkillConditionDef 变体（通过枚举扩展）
- 允许：新增 SkillTargeting 变体（通过枚举扩展）
- 禁止：修改 Effect Pipeline 调度代码（generate.rs / modify.rs / execute.rs）
- 禁止：修改 can_use() 的检查顺序
- 禁止：在 Execute 阶段使用 match 硬编码执行逻辑

## Forbidden（禁止事项）

- 🟥 禁止：修改 Effect Pipeline 调度代码添加新效果类型 — 理由：破坏管线稳定性
- 🟥 禁止：在 execute_effects 中添加 match 分支 — 理由：必须通过 EffectHandler trait 分发
- 🟥 禁止：修改 can_use() 的条件检查顺序（冷却必须最先）— 理由：错误信息准确性
- 🟥 禁止：为新效果类型编写独立的 execute_xxx() 函数 — 理由：必须通过 trait 分发
- 🟥 禁止：修改 ModifierCalculator trait 的 calculate 方法签名 — 理由：破坏扩展性
- 🟥 禁止：在 Guard 中直接查询 ECS 世界 — 理由：性能下降 10-100 倍
- 🟥 禁止：事件监听器直接修改 BattleFsm Component — 理由：破坏单向数据流

## Definition / Instance Design

- Definition：EffectDef（效果定义）、SkillConditionDef（条件定义）、ModifierRuleDef（修饰规则定义）
- Instance：EffectHandlerRegistry（处理器注册表）、SkillRegistry（技能注册表）、ModifierRuleRegistry（修饰规则注册表）

## 后果

### 正面
- 新增技能模板零代码修改
- 新增效果类型只需实现 trait + 注册
- 扩展点清晰，开发者知道在哪里添加新功能

### 负面
- EffectHandler trait 方法较多（4个），实现成本中等
- SkillCondition 枚举增长后 can_use() 复杂度增加

## 替代方案

| 方案 | 优点 | 缺点 | 为何放弃 |
|------|------|------|----------|
| match 硬编码分发 | 简单 | 每次新增需修改核心代码 | 违反开闭原则 |
| 插件系统动态加载 | 灵活 | 过度设计、性能开销 | 当前需求不需要 |
| 策略模式替代枚举 | 灵活 | 失去模式匹配的类型安全 | 枚举更适合有限集合 |
