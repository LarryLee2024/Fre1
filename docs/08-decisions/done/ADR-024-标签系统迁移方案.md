# ADR-024: 标签系统迁移方案

## 状态

Proposed

## 背景

当前标签系统存在三重数据源重复（`src/core/tag.rs` 常量 + `src/core/tag_def.rs` register_defaults() + `content/definitions/tags.ron`）、RON 覆盖不完整（仅 16/30+ 标签有 RON 元数据）、`TagCategory` 分类缺失、`label()` 硬编码中文等问题。ADR-023 定义了目标架构，本 ADR 制定分阶段迁移方案，确保每阶段编译通过 + 测试通过。

### 迁移范围

涉及文件：

| 类别 | 文件 | 改动类型 |
|------|------|----------|
| 标签定义 | `src/core/tag.rs` | 添加 TagName 新变体、废弃 label()、添加 rebuild_tags() |
| 标签注册表 | `src/core/tag_def.rs` | 删除 register_defaults()、扩展 TagCategory、添加校验 |
| RON 定义 | `content/definitions/tags.ron` | 扩展为完整标签列表 |
| 单位生成 | `src/core/character/spawn.rs` | 替换手工位运算为 rebuild_tags() |
| 所有 label() 调用 | 全域 30+ 处 | 替换为 TagRegistry::display_name() |

## 决策

### 迁移阶段总览

```
Phase 1（核心稳固） → Phase 2（RON 扩展） → Phase 3（API 迁移） → Phase 4（清理）
```

**步骤 1.2：TagCategory 扩展**

在 `src/core/tag_def.rs` 中扩展 `TagCategory`：

```rust
pub enum TagCategory {
    Element,
    Status,
    Weapon,
    WeaponType,
    Class,
    Movement,
    SkillType,
    BuffType,
    ItemType,
    EquipmentAttribute,
}
```

影响范围：仅 `TagDefPlugin` 和 `TagRegistry` 的分类查询方法。

**步骤 1.3：删除 register_defaults()**

将 `TagRegistry::register_defaults()` **标记为废弃**，添加编译告警。`TagDefPlugin` 行为改为：

```
1. 尝试从 content/definitions/tags.ron 加载
2. 成功 → 使用 RON 数据（register_defaults 不再调用）
3. 失败 → 从 TagName 枚举自动生成默认 TagDefinition
```

`TagRegistry::from_enum_defaults()` 新函数实现：

```rust
impl TagRegistry {
    /// 从 TagName 枚举生成默认元数据（RON 缺失时的回退方案）
    pub fn from_enum_defaults() -> Self {
        let mut registry = Self::default();
        for name in TagName::ALL {
            let tag = name.to_tag();
            registry.definitions.insert(tag, TagDefinition {
                version: 0,
                tag: *name,
                display_name: tag.label().to_string(),
                description: String::new(),
                display_name_key: None,
                desc_key: None,
                category: TagCategory::default_for(name),
            });
        }
        registry
    }
}
```

- [x] 编译通过
- [x] 所有现有测试通过
- [x] RON 文件覆盖所有 TagName 枚举变体
- [x] 无老代码被破坏

### Phase 2：API 迁移（渐进式，兼容旧 API）

**目标**：弃用旧 API，引入新 API，提供迁移路径。

**步骤 2.1：废弃 label()**

`GameplayTag::label()` 标记为 `#[deprecated(since = "0.1.0", note = "请使用 TagRegistry::display_name()")]`。

**步骤 2.2：引入 rebuild_tags()**

在 `src/core/tag.rs` 中添加：

```rust
/// 从 PersistentTags 重建 GameplayTags
/// 统一标签合并方式，替代 spawn.rs 中的手动位运算
pub fn rebuild_tags(persistent: &PersistentTags) -> GameplayTags {
    GameplayTags(persistent.from_traits.0 | persistent.from_equipment.0)
}
```

**步骤 2.3：替换 spawn.rs 中的手动位运算**

`src/core/character/spawn.rs` 中第 189-194 行的手动位运算替换为：

```rust
// 使用统一的 rebuild_tags()
gameplay_tags = rebuild_tags(&persistent_tags);
```

**Phase 2 验证门禁**：
- [x] label() 调用处产生废弃告警（仍可编译）
- [x] rebuild_tags() 引入并替代 spawn.rs 手动位运算
- [x] 所有现有测试通过

### Phase 3：全域 label() 替换（分批次，每批不破坏编译）

**目标**：逐步替换所有 label() 调用为 TagRegistry::display_name()。

**步骤 3.1：查找所有 label() 调用**

```bash
grep -r "\.label()" src/core/ --include="*.rs"
```

**步骤 3.2：分批替换计划**

| 批次 | 文件 | 替换目标 | 优先度 |
|------|------|----------|--------|
| Batch A | tag.rs 自身（active_tags） | 内部 label 保留用于回退 | 低 |
| Batch B | debug/ viewers | panel.rs, buff_viewer.rs, equipment_viewer.rs 等 | P1 |
| Batch C | UI | view_models.rs, command_handler.rs, highlight.rs | P1 |
| Batch D | 业务代码 | skill/slots.rs, buff/resolve.rs, equipment/definition.rs | P0 |

**步骤 3.3：代码替换模式**

```rust
// 旧代码
GameplayTag::FIRE.label()

// 新代码
// 注入 TagRegistry Resource
fn some_system(registry: Res<TagRegistry>, ...) {
    registry.display_name(GameplayTag::FIRE)
}
```

在命令 handler 和 UI ViewModel 中，通过已持有的 `TagRegistry` 资源访问。

**Phase 3 验证门禁**：
- [x] 编译通过（deprecated 告警可控）
- [x] 测试通过
- [x] UI 显示与之前一致

### Phase 4：清理

**目标**：移除所有已废弃代码。

**步骤 4.1：删除 label()**

在所有调用方迁移完毕后，从 `GameplayTag` 中删除 `label()` 方法。

**步骤 4.2：移除 register_defaults() 废弃代码**

删除 `TagRegistry::register_defaults()` 及所有相关代码。

**步骤 4.3：添加标签覆盖验证测试**

在 `src/core/tag_def.rs` 中添加验证测试：

```rust
#[test]
fn ron_coverage_all_tags() {
    let registry = TagRegistry::load_from_file("content/definitions/tags.ron");
    let missing = registry.ron_missing_tags();
    assert!(
        missing.is_empty(),
        "以下标签在 RON 中缺少定义: {:?}",
        missing
    );
}

#[test]
fn tag_category_coverage_all_tags() {
    let registry = TagRegistry::load_from_file("content/definitions/tags.ron");
    // 每类至少有一个标签
    for category in &ALL_CATEGORIES {
        assert!(
            !registry.tags_by_category(*category).is_empty(),
            "TagCategory {:?} 没有对应的标签定义",
            category
        );
    }
}
```

**Phase 4 验证门禁**：
- [x] 编译通过（无 deprecated 告警）
- [x] 所有测试通过
- [x] 新增标签覆盖测试通过

## 迁移风险与回退

### 风险矩阵

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| RON 文件缺失导致启动失败 | 低 | 高 | `from_enum_defaults()` 回退方案 |
| label() 调用方遗漏 | 中 | 低 | deprecated 告警，编译不阻断 |
| TagCategory 新增破坏下游查询 | 低 | 中 | 新变体不影响已有查询方法 |
| rebuild_tags() 行为变更 | 低 | 高 | 与旧手工位运算行为完全一致 |

### 回退策略

每阶段完成后创建 Git Tag：
```
v0.1-tag-phase1
v0.1-tag-phase2
v0.1-tag-phase3
```

若某阶段出现问题，通过 `git checkout` 回退到上一阶段 Tag。

## 分阶段依赖关系

```
Phase 1 ──→ Phase 2 ──→ Phase 3 ──→ Phase 4
 无依赖      依赖 Phase 1  依赖 Phase 2  依赖 Phase 3
```

每阶段完成后可独立发布。Phase 1 修复最紧急的数据一致性问题，可独立优先执行。

## Module Design

### 迁移后的文件清单

```
src/core/tag.rs
├── GameplayTag(u64)              ← 位掩码常量（不变）
├── GameplayTags                  ← 实体组件（不变）
├── GameplayTags::label()         ← [已删除]
├── TagName                       ← 枚举（不变）
├── TagName::ALL                  ← [新增] 所有 TagName 变体列表
├── PersistentTags                ← 标签来源管理（不变）
└── rebuild_tags()                ← [新增] 统一标签重建

src/core/tag_def.rs
├── TagCategory                   ← [扩展] 10 个变体
├── TagDefinition                 ← 标签元数据（不变）
├── TagRegistry                   ← 注册表（不变）
├── TagRegistry::register_defaults()  ← [已删除]
├── TagRegistry::from_enum_defaults() ← [新增] 回退方案
├── TagRegistry::ron_missing_tags()   ← [新增] RON 覆盖检查
└── TagDefPlugin                  ← 加载逻辑调整

content/definitions/tags.ron       ← [扩展] 完整标签列表
```

## 后果

### 正面
- 分阶段迁移，每阶段可独立验证和发布
- 风险可控，每阶段有明确回退策略
- 旧 API 保持兼容（deprecated 而非立即删除）
- 替换 label() 的 30+ 处调用可分批并行处理

### 负面
- 迁移需要跨 4 个阶段，周期较长
- Phase 3 的 label() 替换涉及 UI 层，需要视觉确认
- 迁移期间会有 deprecated 告警，需要团队一致接受
