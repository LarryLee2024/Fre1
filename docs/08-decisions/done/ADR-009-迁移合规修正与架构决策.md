# ADR-009: 迁移合规修正与架构决策

## 状态: Accepted

**版本**: 1.0
**日期**: 2026-06-14
**范围**: 解决 ADR-001 至 ADR-008 迁移计划集中发现的 8 项合规性问题
**来源**: `docs/reviews/adr_migration_review.md`

---

## 背景

对全部 8 份迁移 ADR 的合规审查（`docs/reviews/adr_migration_review.md`）发现了 8 项问题，包括跨 ADR 矛盾、引用过期文件、章节编号不匹配、路径失效、内部自相矛盾、架构文档缺失以及范围蔓延风险。

本 ADR 以首席架构师身份对每项问题做出最终决策，并输出具体修正指令。

## 分项决策

---

### 决策 A：`assets/rules/*.ron` 目标路径

**来源**：ADR-001 §Phase 5 与 ADR-004 §3.2 冲突

**裁定**：ADR-004 为权威来源。`assets/rules/*.ron` 的目标路径为 `content/modifiers/*.ron`。

**理由**：
1. ADR-001 第 1070 行自身注明"→ 参见 ADR-004 for authoritative content paths"，表明 ADR-001 主动将权威性让渡给 ADR-004。
2. `assets/rules/` 中的文件（`element_interactions.ron`）本质上是修饰规则（Modifier Rules），应归属 `modifiers/` 而非 `formulas/`。
3. `content/formulas/` 是另一类概念（公式定义），未来可能有独立的 RON 文件，但与当前 `assets/rules/` 的迁移无关。

**修正指令**：
1. ADR-001 §Phase 5 迁移对照表（第 1070 行）：`content/formulas/` → `content/modifiers/`
2. ADR-001 §5.4 全局替换步骤（第 1099 行）：`content/formulas/` → `content/modifiers/`
3. ADR-003 Content-to-Core 映射表（第 823 行）：`content/formulas/` 条目标记为"未来规划"，不参与当前迁移

---

### 决策 D：失效路径修正

**来源**：ADR-002（第 41 行）、ADR-006（第 42、992 行）引用 `.trae/rules/` 路径

**裁定**：修正为 `docs/` 下的实际文件路径。

**映射表**：

| 旧路径 | 新路径 | 说明 |
|--------|--------|------|
| `.trae/rules/coding_rules.md` | `docs/coding_rules.md` | 编码执行规范 |
| `.trae/rules/test_spec精简版.md` | `docs/test_spec.md` | 测试规范 |

**理由**：
- `.trae/` 目录不存在；项目使用 `.qoder/` 存放 Agent 定义
- 编码规则和测试规范的实际文件位于 `docs/` 目录
- 修正后所有引用指向真实存在的文件

### 决策 E：`attribute_def.rs` / `tag_def.rs` 位置

**来源**：ADR-001（第 292-293 行）保留在 `core/`，ADR-002（Debt 4，第 344-345 行）迁至 `content/`

**裁定**：以 ADR-002 Debt 4 为权威方案，`attribute_def.rs` 和 `tag_def.rs` 迁至 `content/`。

**理由**：
1. ADR-002 Debt 4 是对 `core/` 职责超载的专项分析，方案更详细、更权威
2. `attribute_def.rs` 和 `tag_def.rs` 属于"配置桥接"（RON 加载 → Registry），符合 Content 层的三问判断
3. ADR-001 Phase 2 的 Core 结构写于 Debt 4 之前，未反映 Debt 4 的分析结果
4. 七层架构要求 Core 只包含纯游戏规则，配置注册属于 Content 层

**修正指令**：
1. ADR-001 Phase 2 Core 目标结构（第 292-293 行）：移除 `attribute_def.rs` 和 `tag_def.rs`，标注"迁至 content/"
2. `docs/01-architecture/README.md` Core 模块列表不再列出 `attribute_def`/`tag_def`

### 决策 F：Phase 编号澄清

**来源**：ADR-001 内部 Phase 计数矛盾

**裁定**：采用以下 Phase 编号方案：

| Phase | 名称 | 工期 | 性质 |
|-------|------|------|------|
| Phase 0 | 脚手架搭建 | 1 天 | 准备阶段 |
| Phase 1 | Shared 层 | 4 天 | 迁移阶段 |
| Phase 2 | Core 重组 | 7 天 | 迁移阶段 |
| Phase 3 | 错误迁移 | 3 天 | 迁移阶段 |
| Phase 4 | Infrastructure 层 | 4 天 | 迁移阶段 |
| Phase 5 | Content 层 | 7 天 | 迁移阶段 |
| Phase 6 | App 层 | 4 天 | 迁移阶段 |
| Phase 7 | Modding 层 | 未来 | 未来阶段 |
| Phase 8 | Tools 层 | 未来 | 未来阶段 |

**计数规则**：

- **全部计划阶段**（Phase 0-8）：9 个阶段

**修正指令**：ADR-001 第 8、71 行：将"8 个 Phase"修正为"7 个执行阶段（Phase 0-6）+ 2 个未来阶段（Phase 7-8）"

- 未来新增模块时需在 `project-structure.md` 统一添加，避免再次出现 ADR 间不一致

## 替代方案

**不修复直接迁移**：否决。问题 A（路径冲突）会导致 `element_interactions.ron` 被放入 `content/formulas/` 而非 `content/modifiers/`，后续所有代码中的路径引用都会错位。修复成本低（改几行文本），不修复的隐患高。

**拆分为多个 ADR（问题 A 单独一个、其余合并一个）**：否决。所有 8 个问题都是同一轮合规审查的发现，且相互关联（宪法引用影响所有 ADR），合并到一个 ADR 便于整体追踪和验证。
