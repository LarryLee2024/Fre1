# ADR-017: 国际化架构决策

## 状态

Proposed

## 背景

项目当前所有 RON 配置（技能、角色、Buff、装备等）和 UI 代码中硬编码了中文文本。随着项目规模增长和潜在的多语言需求，需要建立正式的国际化（i18n）架构。当前 `src/infrastructure/localization/` 为空壳模块，无 FTL 文件、无翻译资源、无 `LocalizedText` 组件。

## 引用的领域规则

- `docs/01-architecture/08-i18n-modding-collaboration/i18n-design.md` — 国际化系统架构（v2.1，798行详细设计）
- `docs/03-technical/localization-rules.md` — 本地化领域规则（v1.1，859行）
- `docs/01-architecture/README.md` — 七层架构，Infrastructure 层职责
- `docs/00-governance/ai-constitution-complete.md` — 宪法 §17.2.2：禁止硬编码玩家可见文本

## 决策

### 1. 技术选型：Mozilla Fluent (.ftl)

采用 **Fluent**（`fluent-rs` + `intl-memoizer`，**自封装** Bevy 适配层）。

🟥 **绝对禁止**：依赖第三方 `bevy_fluent` 插件。Bevy 版本迭代极快（0.15~0.18 生态变动巨大），社区第三方插件极大概率年久失修。自封装代码量仅几百行（AssetLoader + Resource + System），完全掌控 ECS 生命周期。

Fluent 优势：
- 变量插值：`{ $damage }`
- 复数支持：`{ $count -> [one] ... [other] ... }`
- 术语复用：`-fire-damage = 火焰伤害`
- 条件文本：`{ $gender -> [male] ... [female] ... }`

### 2. Content 数据只存 Key，不存文本

**核心架构决策**：所有 Content RON 配置中的 `name` 和 `description` 字段替换为 `name_key` 和 `desc_key`，存储本地化 Key 而非实际文本。

当前模式（反模式）：
```ron
// content/skills/fireball.ron
(name: "火球", description: "发射火球，造成1.8倍伤害并附加灼烧")
```

目标模式：
```ron
// content/skills/fireball.ron
(name_key: "skill.s_1001.name", desc_key: "skill.s_1001.desc")
```

### 3. Key 命名规范：永久唯一 ID

🟥 **禁止使用语义化名称作为 Key**（如 `skill.fireball.name`）。一旦策划改名，Key 维护陷入批量替换灾难。

使用「命名空间 + 永久唯一 ID」：
- ✅ `skill.s_1001.name`（s_1001 是火球术的永久 ID）
- 🟥 `skill.fireball.name`（与实际技能名耦合）

Key 一旦创建，即使文案/技能名改了，Key 绝对不允许修改。

### 4. FTL 文件组织

```text
assets/localization/
├── en-US/
│   ├── ui.ftl          # 界面文本
│   ├── battle.ftl      # 战斗文本
│   ├── skill.ftl       # 技能文本
│   ├── quest.ftl       # 任务文本
│   └── system.ftl      # 系统文本
├── zh-CN/
│   ├── ui.ftl
│   ├── battle.ftl
│   ├── skill.ftl
│   ├── quest.ftl
│   └── system.ftl
└── ja-JP/
    └── ...
```

拆分规则：按业务域拆分，每个 `.ftl` 文件不超过 500 条翻译条目。

### 5. LocalizedText 组件

```rust
#[derive(Component)]
pub struct LocalizedText {
    pub key: String,
    pub args: Option<FluentArgs>,
}
```

系统仅在 `LanguageChangedEvent` 或组件刚添加时执行解析，平时静默。

### 6. 语言回退链

```
请求语言 → 项目默认语言(zh-CN) → 英语(en-US) → Key 本身(Debug) / 空字符串(Release)
```

MOD 语言回退链：`MOD_zh-CN → 本体_zh-CN → 本体_en-US`

### 7. 性能策略

- FluentBundle 启动时预解析并缓存
- 运行时仅语言切换时重新格式化
- `LocalizedTextCache` 按 Key + 参数哈希缓存（LRU，上限 2048 条）
- 大体积 FTL 文件（>100KB）异步加载

## Module Design

### 模块位置

```
src/infrastructure/localization/
├── mod.rs                  # 模块入口
├── plugin.rs               # LocalizationPlugin
├── locale.rs               # Locale 类型
├── service.rs              # LocalizationService（封装 fluent-rs）
├── ftl_loader.rs           # 自定义 AssetLoader
├── component.rs            # LocalizedText 组件
├── systems.rs              # 文本解析系统
├── cache.rs                # LocalizedTextCache
└── font_fallback.rs        # FontFallbackChain
```

### 依赖关系

```
infrastructure/localization/
    → infrastructure/assets/     （加载 FTL 文件）
    → shared/error/              （错误工具）
    ← content/                   （Content 层调用 resolve）
    ← ui/                        （UI 层使用 LocalizedText）
    ← modding/                   （MOD 加载本地化文件）
```

## Communication Design

### Message

| Message | 发送方 | 接收方 | 时机 |
|---------|--------|--------|------|
| `LanguageChangedEvent` | 设置/UI | 所有监听系统 | 语言切换时 |

### 函数调用

- `LocalizationService.resolve(key, locale, args)` — Key 解析
- `CurrentLocale` Resource — 当前语言状态

## 边界定义

- 允许：`infrastructure/localization/` 依赖 `infrastructure/assets/`（加载 FTL）
- 允许：`content/` 和 `ui/` 依赖 `infrastructure/localization/`（解析文本）
- 禁止：`core/` 直接调用 `LocalizationService`（Core 只存 Key，解析在 Content/UI 层）
- 禁止：运行时修改 FTL 翻译表（Definition 不可变）

## Forbidden（禁止事项）

- 🟥 禁止：依赖 `bevy_fluent` 等第三方 Bevy 插件 — 理由：Bevy 版本迭代快，插件极大概率断更
- 🟥 禁止：Content 配置中硬编码文本（`name: "火球"`）— 理由：违反宪法 §17.2.2
- 🟥 禁止：UI 中使用 `Text::new("硬编码")` — 理由：语言切换时无法自动刷新
- 🟥 禁止：单个巨大 FTL 文件 — 理由：协作冲突、按需加载困难
- 🟥 禁止：使用语义化名称作为 Key（`skill.fireball.name`）— 理由：改名后批量替换灾难
- 🟥 禁止：使用无意义编号 Key（`text_001`）— 理由：维护时无法识别内容
- 🟥 禁止：Core 层直接调用 LocalizationService — 理由：Core 只存 Key，解析在上层
- 🟥 禁止：每帧解析 Fluent AST — 理由：CPU 开销导致帧率暴跌
- 🟥 禁止：MOD 翻译覆盖原版 Key — 理由：破坏原版文本完整性
- 🟥 禁止：UI 容器固定像素宽度承载本地化文本 — 理由：德语/俄语比中文长 30%~50%

## Definition / Instance Design

- Definition：FTL 文件（翻译表）、Locale（语言标识）
- Instance：FluentBundle（运行时缓存）、LocalizedText（组件）、CurrentLocale（Resource）

## 后果

### 正面
- 新增语言只需添加 FTL 文件，零代码修改
- MOD 自带翻译，主工程无感知
- 平衡调整时描述中的变量自动更新

### 负面
- 所有现有 RON 配置需要迁移到 Key 模式（一次性成本）
- 所有 UI 代码需要替换 `Text::new("...")` 为 `LocalizedText`
- 自封装 Fluent 适配层需要维护（但代码量仅几百行）

## 替代方案

| 方案 | 优点 | 缺点 | 为何放弃 |
|------|------|------|----------|
| bevy_fluent 插件 | 开箱即用 | Bevy 版本断更风险高 | 宪法明确禁止 |
| JSON 存储翻译 | 简单 | 无复数/条件文本/变量插值 | 功能不足 |
| RON 存储翻译 | Rust 原生 | 无复数/条件文本 | 功能不足 |
| 硬编码不迁移 | 零成本 | 无法多语言，违反宪法 | 违反 §17.2.2 |
