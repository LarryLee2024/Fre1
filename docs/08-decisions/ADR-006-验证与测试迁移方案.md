# ADR-006: 验证与测试迁移方案

## 状态: Proposed

**版本**: 1.0
**日期**: 2026-06-14
**范围**: 迁移过程的质量保障体系——验证门禁、测试迁移、CI 流水线、架构合规检查
**依赖**: ADR-001（迁移计划）、ADR-003（分层契约与依赖迁移）、`docs/01-architecture/migration-roadmap.md`

---

## 背景

### 问题

项目正处于从扁平结构到七层架构的迁移过程中（参见 `docs/01-architecture/migration-roadmap.md`，共 8 个 Phase）。迁移涉及 157 个 Rust 文件（33,771 行）的重组，任何一步出错都可能导致：

1. **编译断路**：模块移动后 `use` 引用未更新，导致编译失败
2. **测试断路**：测试依赖的路径变更，测试无法定位被测代码
3. **行为回归**：迁移过程中业务逻辑被意外修改，战斗行为不再一致
4. **架构退化**：迁移引入跨层依赖、错误类型放错位置等架构违规
5. **CI 失效**：测试目录结构变化后 CI 命令找不到测试文件

### 决策目标

定义一套**完整的迁移验证体系**，确保：

- 每个 Phase 迁移后，项目仍可编译、可测试、可运行
- 测试从源文件内嵌（`#[cfg(test)] mod tests`）安全迁移到五层测试目录结构
- CI 流水线在迁移过程中持续有效
- 架构合规性有自动化检查守护

### 架构规范引用

| 文档 | 路径 | 关系 |
|------|------|------|
| 测试体系架构 | `docs/01-architecture/testing_architecture.md` | 五层测试层级定义 |
| 全局校验规则 | `docs/01-architecture/validation_rules.md` | 游戏状态校验框架 |
| 迁移路线图 | `docs/01-architecture/migration-roadmap.md` | Phase 0-7 迁移计划 |

| AI 开发宪法 | `docs/AI开发宪法完整版.md` | 第 13 部分（测试）、第 14 部分（可观测性） |
| 测试宪法 | `docs/test_spec.md` | 测试分类与优先级 |

## 决策

**建立四层验证体系：Pre-Migration Baseline → Per-Phase Gate → Test Suite Migration → CI Pipeline，贯穿迁移全过程。**

## 1. Pre-Migration Baseline（迁移前基线）

迁移开始前，必须建立可量化的质量基线。所有后续 Phase 的验证都以基线为参照。

### 1.1 基线采集步骤

```bash
# Step 1: 记录编译状态
cargo check 2>&1 | tee baseline/check.log
cargo check --all-features 2>&1 | tee baseline/check-all-features.log

# Step 2: 记录所有警告
cargo build 2>&1 | grep "warning" | tee baseline/warnings.log
# 目标：迁移前修复所有警告，达到 zero-warning 状态

# Step 3: 运行全量测试
cargo test 2>&1 | tee baseline/test-results.log

# Step 4: 记录测试数量
cargo test 2>&1 | grep -E "test result:" | tee baseline/test-count.log

# Step 5: 快照游戏行为（Battle Replay 回归基线）
cargo run --release -- --record-battle baseline/battle_replay_001.ron

# Step 6: 记录架构状态
grep -rn "use crate::" src/ > baseline/import-graph.txt
```

### 1.2 基线内容清单

| 基线项 | 文件 | 用途 |
|--------|------|------|
| 编译状态 | `baseline/check.log` | 迁移后对比编译结果 |
| 警告列表 | `baseline/warnings.log` | 确保迁移不引入新警告 |
| 测试结果 | `baseline/test-results.log` | 确保迁移不丢失测试 |
| 测试数量 | `baseline/test-count.log` | 量化测试覆盖变化 |
| 战斗回放 | `baseline/battle_replay_001.ron` | 回归检测基准 |
| 依赖图 | `baseline/import-graph.txt` | 迁移后对比依赖变化 |

### 1.3 基线采集执行命令

```bash
# 一键采集基线
mkdir -p baseline
cargo check 2>&1 | tee baseline/check.log
cargo build 2>&1 | tee baseline/build.log
cargo test 2>&1 | tee baseline/test-results.log
grep -c "^test " baseline/test-results.log > baseline/test-count.log
grep -rn "use crate::" src/ > baseline/import-graph.txt
echo "Baseline采集完成: $(date)" > baseline/timestamp.log
```

## 2. Per-Phase Verification Gates（逐阶段验证门禁）

每个 Phase 完成后必须通过对应的验证门禁，**任何一项失败都必须修复后才能进入下一个 Phase**。

### 2.1 验证门禁矩阵

| Phase | 必须通过 | 不得破坏 | 验证命令 |
|-------|---------|---------|---------|
| **0: 准备** | `cargo check` | — | `cargo check` |
| **1: Shared** | `cargo build`, `cargo test` | 所有现有测试 | `cargo build && cargo test` |
| **2: Core** | `cargo build`, `cargo test`, 游戏可运行 | 战斗流程、技能执行 | `cargo test && cargo run --release -- --test-battle` |
| **3: Error** | `cargo build`, `cargo test` | 所有错误路径 | `cargo test` |
| **4: Infra** | `cargo build`, `cargo test` | 日志、审计 | `cargo test` + 手动日志检查 |
| **5: Content** | `cargo build`, `cargo test`, 游戏可运行 | 所有内容加载 | `cargo test && cargo run` |
| **6: App** | `cargo build`, `cargo test`, 游戏可运行 | Plugin 注册、启动 | `cargo run` |
| **7-8: Modding/Tools** | `cargo build`, `cargo test` | — | `cargo test` |

### 2.2 每 Phase 验证脚本

```bash
#!/bin/bash
# verify_phase.sh — Phase 验证脚本
# 用法: ./verify_phase.sh <phase_number>

PHASE=$1
set -e

echo "=== Phase $PHASE 验证开始 ==="

# 通用验证（所有 Phase）
echo "[1/4] 编译检查..."
cargo check --all-features 2>&1 | tee "baseline/phase${PHASE}_check.log"

echo "[2/4] Clippy 检查..."
cargo clippy -- -D warnings 2>&1 | tee "baseline/phase${PHASE}_clippy.log"

echo "[3/4] 运行测试..."
cargo test 2>&1 | tee "baseline/phase${PHASE}_test.log"

# 对比测试数量
BASELINE_COUNT=$(cat baseline/test-count.log)
CURRENT_COUNT=$(grep -c "^test " "baseline/phase${PHASE}_test.log")
if [ "$CURRENT_COUNT" -lt "$BASELINE_COUNT" ]; then
    echo "❌ 测试数量减少! 基线: $BASELINE_COUNT, 当前: $CURRENT_COUNT"
    exit 1
fi
echo "✅ 测试数量: $CURRENT_COUNT (基线: $BASELINE_COUNT)"

# Phase 特定验证
case $PHASE in
    2|5|6)
        echo "[4/4] 游戏运行测试..."
        cargo run --release -- --test-battle 2>&1 | tee "baseline/phase${PHASE}_runtime.log"
        ;;
    *)
        echo "[4/4] 跳过运行时测试"
        ;;
esac

echo "=== Phase $PHASE 验证完成 ==="
```

# 验证 Shared 层无外部依赖

## 3. Test Suite Migration（测试套件迁移）

### 3.1 当前状态

当前测试以内嵌方式存在于源文件中：

```
src/xxx.rs 中的 #[cfg(test)] mod tests { ... }
```

无独立测试文件，无 `tests/` 目录。

### 3.2 目标：五层测试目录结构

```
tests/
├── unit/                     # 领域单元测试（从源文件迁出）
│   ├── core/
│   │   ├── battle/           # 战斗数值公式测试
│   │   ├── skill/            # 技能验证测试
│   │   ├── buff/             # Buff 堆叠规则测试
│   │   ├── character/        # 属性计算测试
│   │   ├── equipment/        # 装备系统测试
│   │   ├── inventory/        # 背包系统测试
│   │   ├── ai/               # AI 决策测试
│   │   ├── map/              # 寻路算法测试
│   │   └── turn/             # 回合状态机测试
│   └── shared/
│       ├── ids/              # 强类型 ID 测试
│       └── math/             # 数学工具测试
├── integration/              # 领域集成测试
│   ├── battle_flow/          # 战斗生命周期测试
│   ├── skill_buff/           # 技能 + Buff 交互测试
│   └── equipment_inventory/  # 装备 + 背包交互测试
├── system/                   # 系统集成测试（Bevy 无头模式）
│   ├── plugin_init/          # Plugin 初始化验证
│   └── event_flow/           # 事件路由验证
├── e2e/                      # 端到端回归测试
│   ├── main_flow/            # 主流程测试
│   ├── save_load/            # 存档兼容测试
│   └── mod_compat/           # MOD 兼容测试
└── content/                  # 内容数据校验
    ├── schema/               # RON Schema 校验
    └── references/           # 引用完整性校验
```

### 3.3 迁移策略

**原则**：测试迁移与代码迁移同步进行，每个 Phase 完成后，对应模块的测试迁移到新位置。

#### 迁移映射表

| 当前位置（内嵌测试） | 目标位置 | 迁移时机 | 迁移方式 |
|---------------------|---------|---------|---------|
| `src/battle/*.rs` 中的 `#[cfg(test)]` | `tests/unit/core/battle/` | Phase 2（Core 重组） | 提取到独立文件 |
| `src/skill/*.rs` 中的 `#[cfg(test)]` | `tests/unit/core/skill/` | Phase 2 | 提取到独立文件 |
| `src/buff/*.rs` 中的 `#[cfg(test)]` | `tests/unit/core/buff/` | Phase 2 | 提取到独立文件 |
| `src/character/*.rs` 中的 `#[cfg(test)]` | `tests/unit/core/character/` | Phase 2 | 提取到独立文件 |
| `src/equipment/*.rs` 中的 `#[cfg(test)]` | `tests/unit/core/equipment/` | Phase 2 | 提取到独立文件 |
| `src/inventory/*.rs` 中的 `#[cfg(test)]` | `tests/unit/core/inventory/` | Phase 2 | 提取到独立文件 |
| `src/ai/*.rs` 中的 `#[cfg(test)]` | `tests/unit/core/ai/` | Phase 2 | 提取到独立文件 |
| `src/map/*.rs` 中的 `#[cfg(test)]` | `tests/unit/core/map/` | Phase 2 | 提取到独立文件 |
| `src/turn/*.rs` 中的 `#[cfg(test)]` | `tests/unit/core/turn/` | Phase 2 | 提取到独立文件 |
| `src/core/id/*.rs` 中的 `#[cfg(test)]` | `tests/unit/shared/ids/` | Phase 1（Shared 迁移） | 提取到独立文件 |
| `src/core/error/*.rs` 中的 `#[cfg(test)]` | 分散到各领域 | Phase 3（Error 重组） | 按错误类型归属迁移 |
| 新增：跨模块交互 | `tests/integration/` | Phase 2-3 | 新建集成测试 |
| 新增：Plugin 初始化 | `tests/system/plugin_init/` | Phase 6（App 层） | 新建系统测试 |
| 新增：主流程回归 | `tests/e2e/main_flow/` | Phase 5-6 | 新建 E2E 测试 |

### 3.4 测试提取规则

从源文件提取测试到独立文件时，遵循以下规则：

```rust
// BEFORE: 内嵌测试（在 src/core/battle/damage.rs 中）
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn physical_damage_respects_armor() {
        // ...
    }
}

// AFTER: 独立测试文件（tests/unit/core/battle/damage_calculation.rs）
// 不使用 #[cfg(test)]，直接是 #[test]
// 需要显式 use 引用被测模块
use a1::core::battle::damage::calculate_physical_damage;
// 或使用 crate 级别的 re-export

#[test]
fn physical_damage_respects_armor() {
    // 原测试逻辑不变
}
```

#### 关键约束

| 约束 | 说明 |
|------|------|
| 🟥 领域单元测试禁止引入 Bevy 依赖 | 纯 Rust 函数测试，毫秒级执行 |
| 🟥 领域单元测试不依赖 content 配置数据 | 测试用例硬编码入参 |
| 🟩 使用 `shared/testing/` 中的公共测试工具 | fixtures、assertions、deterministic |
| 🟩 每条领域规则至少对应一组单元测试 | 从源文件提取时检查覆盖 |
| 🟩 测试函数名保持不变 | 迁移不改测试逻辑 |
| 🟩 使用标准测试单位 | Unit_001(战士), Unit_002(法师), Unit_003(坦克) |

### 3.5 shared/testing 公共测试库

迁移过程中同步搭建 `src/shared/testing/`，提供全项目复用的测试工具：

```
src/shared/testing/
├── mod.rs              # 公共导出
├── fixtures.rs         # 标准测试夹具（create_test_character, standard_1v1_battle）
├── assertions.rs       # 自定义断言（assert_approx_eq, assert_game_state_valid）
├── deterministic.rs    # 确定性基础设施（DeterministicRng, seed=42）
└── time.rs             # 模拟时间工具
```

### 3.6 测试迁移执行步骤

```bash
# Step 1: 创建目标目录结构
mkdir -p tests/unit/core/{battle,skill,buff,character,equipment,inventory,ai,map,turn}
mkdir -p tests/unit/shared/{ids,math}
mkdir -p tests/integration/{battle_flow,skill_buff,equipment_inventory}
mkdir -p tests/system/{plugin_init,event_flow}
mkdir -p tests/e2e/{main_flow,save_load,mod_compat}
mkdir -p tests/content/{schema,references}

# Step 2: 对于每个需要迁移的测试文件
# 2a. 复制测试函数到目标文件
# 2b. 添加必要的 use 引用
# 2c. 移除 #[cfg(test)] mod tests { ... } 包装
# 2d. 在原文件中保留 #[cfg(test)] mod tests { ... } 并添加
#     #[allow(dead_code)] 标注（过渡期保留旧位置）

# Step 3: 验证迁移后的测试
cargo test --test unit::core::battle::damage_calculation

# Step 4: 清理原文件中的旧测试
# 确认新位置测试全部通过后，删除原文件中的 #[cfg(test)] mod tests
```

## 4. CI Pipeline Design（CI 流水线设计）

### 4.1 流水线结构

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

env:
  CARGO_TERM_COLOR: always
  RUST_BACKTRACE: 1

jobs:
  # ===== 编译检查 =====
  check:
    name: 编译检查
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
      - uses: Swatinem/rust-cache@v2
      - name: cargo check
        run: cargo check --all-features
      - name: cargo clippy
        run: cargo clippy -- -D warnings

  # ===== 单元测试 =====
  test-unit:
    name: 单元测试
    needs: check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
      - uses: Swatinem/rust-cache@v2
      - name: 运行单元测试
        run: cargo test --lib -- --test-threads=4

  # ===== 集成测试 =====
  test-integration:
    name: 集成测试
    needs: check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
      - uses: Swatinem/rust-cache@v2
      - name: 运行集成测试
        run: cargo test --test integration:: -- --test-threads=2

  # ===== 系统集成测试 =====
  test-system:
    name: 系统集成测试
    needs: check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
      - uses: Swatinem/rust-cache@v2
      - name: 运行系统测试
        run: cargo test --tests system:: -- --nocapture

  # ===== 内容数据校验 =====
  test-content:
    name: 内容数据校验
    needs: check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
      - uses: Swatinem/rust-cache@v2
      - name: 运行内容校验
        run: cargo run --bin content_validator

  # ===== E2E 回归测试 =====
  test-e2e:
    name: E2E 回归测试
    needs: [test-unit, test-integration, test-system]
    runs-on: ubuntu-latest
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
      - uses: Swatinem/rust-cache@v2
      - name: 运行 E2E 测试
        run: cargo test --tests e2e:: -- --test-threads=1

  # ===== 性能基准测试 =====
  benchmark:
    name: 性能基准
    needs: check
    runs-on: ubuntu-latest
    if: github.event_name == 'pull_request'
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
      - uses: Swatinem/rust-cache@v2
      - name: 运行基准测试
        run: cargo bench -- --output-format bencher | tee benchmark-results.txt
      - name: 上传基准结果
        uses: actions/upload-artifact@v4
        with:
          name: benchmark-results
          path: benchmark-results.txt

  # ===== 架构合规检查 =====
  architecture-check:
    name: 架构合规
    needs: check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: 层间导入检查
        run: |
          echo "=== 检查 Core 不依赖 Infra ==="
          ! grep -rn "use crate::infrastructure" src/core/
          echo "=== 检查 Shared 不依赖任何层 ==="
          ! grep -rn "use crate::core\|use crate::infrastructure\|use crate::ui\|use crate::content" src/shared/
          echo "✅ 层间导入检查通过"

      - name: Feature-First 检查
        run: |
          echo "=== 检查非 Shared 目录无技术类型文件 ==="
          # 检查是否有 components.rs、systems.rs、utils.rs 在非 shared 目录
          ! find src/core/ src/infrastructure/ src/content/ src/ui/ -name "components.rs" -o -name "systems.rs" -o -name "utils.rs" 2>/dev/null | grep .
          echo "✅ Feature-First 检查通过"

      - name: Plugin 注册顺序检查
        run: |
          echo "=== 检查 Plugin 注册顺序 ==="
          cargo test test_plugin_registration_order
          echo "✅ Plugin 注册顺序检查通过"

      - name: 错误位置检查
        run: |
          echo "=== 检查领域错误不在 core/error/ ==="
          ! test -d src/core/error/
          echo "✅ 错误位置检查通过"

      - name: Content 路径检查
        run: |
          echo "=== 检查 assets/ 无 RON 配置 ==="
          ! find assets/ -name "*.ron" | grep .
          echo "✅ Content 路径检查通过"
```

### 4.2 测试失败处理策略

| 测试类型 | 失败处理 | 理由 |
|---------|---------|------|
| 单元测试失败 | 🟥 阻塞合并 | 核心规则验证，不可妥协 |
| 集成测试失败 | 🟥 阻塞合并 | 跨模块协作验证 |
| 系统测试失败 | 🟥 阻塞合并 | 架构落地验证 |
| E2E 测试失败 | 🟥 阻塞合并 | 主流程回归验证 |
| 内容校验失败 | 🟥 阻塞合并 | 数据完整性验证 |
| Clippy 警告 | 🟥 阻塞合并 | 代码质量门禁 |
| 性能基准回归 >10% | 🟩 标记警告，不阻塞 | 性能需持续关注但不阻塞开发 |

### 4.3 迁移期间的 CI 适配

## 5. Architecture Compliance Checks（架构合规检查）

### 5.1 自动化检查脚本

```bash
#!/bin/bash
# scripts/check_architecture.sh — 架构合规检查脚本
set -e

echo "=========================================="
echo "  架构合规检查"
echo "=========================================="

ERRORS=0

# Check 1: 层间导入检查
echo ""
echo "[Check 1] 层间导入检查"
echo "  Core → Infra: $(grep -rn 'use crate::infrastructure' src/core/ 2>/dev/null | wc -l) 处违规"
echo "  Core → UI: $(grep -rn 'use crate::ui' src/core/ 2>/dev/null | wc -l) 处违规"
echo "  Core → Content: $(grep -rn 'use crate::content' src/core/ 2>/dev/null | wc -l) 处违规"
echo "  Shared → Core: $(grep -rn 'use crate::core' src/shared/ 2>/dev/null | wc -l) 处违规"
echo "  Shared → Infra: $(grep -rn 'use crate::infrastructure' src/shared/ 2>/dev/null | wc -l) 处违规"
if grep -rn "use crate::infrastructure" src/core/ 2>/dev/null | grep -q .; then
    echo "  ❌ Core 层存在 Infrastructure 依赖"
    ERRORS=$((ERRORS + 1))
fi
if grep -rn "use crate::core\|use crate::infrastructure\|use crate::ui\|use crate::content" src/shared/ 2>/dev/null | grep -q .; then
    echo "  ❌ Shared 层存在外部依赖"
    ERRORS=$((ERRORS + 1))
fi

# Check 2: Feature-First 检查
echo ""
echo "[Check 2] Feature-First 检查"
BAD_FILES=$(find src/core/ src/infrastructure/ src/content/ src/ui/ \
    -name "components.rs" -o -name "systems.rs" -o -name "utils.rs" 2>/dev/null)
if [ -n "$BAD_FILES" ]; then
    echo "  ❌ 发现技术类型文件:"
    echo "$BAD_FILES"
    ERRORS=$((ERRORS + 1))
else
    echo "  ✅ 无技术类型文件"
fi

# Check 3: 错误位置检查
echo ""
echo "[Check 3] 错误位置检查"
if [ -d src/core/error/ ]; then
    echo "  ❌ src/core/error/ 仍然存在"
    ERRORS=$((ERRORS + 1))
else
    echo "  ✅ 旧错误目录已清理"
fi

# Check 4: Content 路径检查
echo ""
echo "[Check 4] Content 路径检查"
RON_COUNT=$(find assets/ -name "*.ron" 2>/dev/null | wc -l)
if [ "$RON_COUNT" -gt 0 ]; then
    echo "  ⚠️ assets/ 中仍有 $RON_COUNT 个 RON 文件"
    # 迁移完成后应为 error
    # ERRORS=$((ERRORS + 1))
else
    echo "  ✅ assets/ 无 RON 配置"
fi

# Check 5: Plugin 注册顺序
echo ""
echo "[Check 5] Plugin 注册顺序"
cargo test test_plugin_registration_order 2>/dev/null && \
    echo "  ✅ Plugin 注册顺序正确" || \
    echo "  ⚠️ Plugin 注册顺序测试未找到或失败"

# Check 6: ID 类型位置
echo ""
echo "[Check 6] ID 类型位置"
if [ -d src/shared/ids/ ]; then
    echo "  ✅ ID 类型在 Shared 层"
else
    echo "  ⚠️ src/shared/ids/ 不存在（可能尚未迁移）"
fi

# 汇总
echo ""
echo "=========================================="
if [ "$ERRORS" -eq 0 ]; then
    echo "  ✅ 架构合规检查通过"
else
    echo "  ❌ 发现 $ERRORS 个违规"
    exit 1
fi
echo "=========================================="
```

### 5.2 架构合规检查项目

| 检查项 | 工具 | 检测内容 | 严重性 |
|--------|------|---------|--------|
| 层间导入检查 | grep 脚本 | Core 导入 Infra、Shared 导入 Core 等 | 🟥 阻塞 |
| Feature-First 检查 | find 脚本 | 非 Shared 目录出现 `components.rs`、`systems.rs`、`utils.rs` | 🟥 阻塞 |
| Plugin 注册顺序 | cargo test | Plugin 注册不按层顺序 | 🟥 阻塞 |
| 错误位置检查 | test 脚本 | 领域错误不在领域目录 | 🟥 阻塞 |
| Content 路径检查 | find 脚本 | RON 配置仍在 assets/ 而非 content/ | 🟩 警告 |
| ID 类型位置 | test 脚本 | 强类型 ID 不在 src/shared/ids/ | 🟥 阻塞 |
| Shared 门卫检查 | grep 脚本 | Shared 包含业务逻辑 | 🟥 阻塞 |

### 5.3 检查执行频率

| 检查项 | 本地执行 | CI 执行 | 时机 |
|--------|---------|---------|------|
| 层间导入检查 | 每 Phase | 每次 PR | 迁移核心检查 |
| Feature-First 检查 | 每 Phase | 每次 PR | 架构规范检查 |
| Plugin 注册顺序 | 每 Phase | 每次 PR | App 层检查 |
| 错误位置检查 | Phase 3 后 | 每次 PR | Error 重组检查 |
| Content 路径检查 | Phase 5 后 | 每次 PR | Content 迁移检查 |
| ID 类型位置 | Phase 1 后 | 每次 PR | Shared 迁移检查 |
| Shared 门卫检查 | 每次向 Shared 添加代码 | 每次 PR | 持续守护 |

## 6. Regression Test Strategy（回归测试策略）

### 6.1 回归检测方法

| 方法 | 用途 | 执行时机 | 工具 |
|------|------|---------|------|
| Battle Replay 对比 | 战斗行为一致性 | 每个 Phase | `replay_validator` |
| 确定性测试套件 | 数值计算一致性 | 每次提交 | seed=42 + 固定输入 |
| Snapshot Testing | 关键输出一致性 | 每个 Phase | `insta` crate |
| Golden File Testing | 参考输出精确匹配 | 每个 Phase | 自定义脚本 |

### 6.2 Battle Replay 回归对比

```bash
# 迁移前：录制基线回放
cargo run --release -- --record-battle baseline/battle_replay_001.ron

# 每个 Phase 后：重放并对比
cargo run --release -- --replay-battle baseline/battle_replay_001.ron \
    --compare baseline/battle_replay_001_expected_hash.txt

# 如果哈希不一致，输出差异帧
# 需要检查是否是迁移引入的行为变化
```

### 6.3 确定性测试套件

```rust
// 所有领域测试使用固定 seed，确保可复现
#[test]
fn damage_calculation_deterministic() {
    let mut rng = DeterministicRng::new(42);  // 固定种子
    let attacker = create_test_character(100, 50, 20);  // 标准测试单位
    let defender = create_test_character(200, 30, 30);
    let result = calculate_physical_damage(&attacker, &defender, &mut rng);
    assert_eq!(result.value, 70);  // 迁移前后结果必须一致
}
```

### 6.4 Snapshot Testing

```rust

```

### 7.4 验收执行脚本

```bash
#!/bin/bash
# scripts/verify_migration_complete.sh — 迁移完成验收
set -e

echo "=========================================="
echo "  迁移完成验收检查（15 项）"
echo "=========================================="

PASS=0
FAIL=0

check() {
    local num=$1
    local desc=$2
    local cmd=$3
    if eval "$cmd" > /dev/null 2>&1; then
        echo "  ✅ #$num: $desc"
        PASS=$((PASS + 1))
    else
        echo "  ❌ #$num: $desc"
        FAIL=$((FAIL + 1))
    fi
}

# 结构验收
check 1 "顶层业务模块不存在" "! test -d src/battle/ && ! test -d src/skill/"
check 2 "core/id/ 不存在" "! test -d src/core/id/"
check 3 "core/error/ 不存在" "! test -d src/core/error/"
check 4 "app/ 存在" "test -f src/app/app_plugin.rs"
check 5 "content/ 源码存在" "test -d src/content/"
check 6 "content/ 数据目录存在" "test -d content/skills/"
check 7 "assets/ 无 RON" "! find assets/ -name '*.ron' 2>/dev/null | grep ."

# 架构验收
check 8 "Shared 无业务逻辑" "! grep -rn 'DamageCalculation\|BuffApply\|SkillCast' src/shared/"
check 9 "Core 不依赖 Infra" "! grep -rn 'use crate::infrastructure' src/core/"
check 10 "Shared 无外部依赖" "! grep -rn 'use crate::core\|use crate::infrastructure\|use crate::ui\|use crate::content' src/shared/"

# 功能验收
check 11 "Messages 位置正确" "cargo test test_message_location 2>/dev/null || true"
check 12 "Plugins 顺序正确" "cargo test test_plugin_registration_order 2>/dev/null || true"
check 13 "所有测试通过" "cargo test 2>/dev/null | grep -q 'test result: ok'"
check 14 "零编译警告" "cargo build 2>&1 | grep -c 'warning' | grep -q '^0$'"
check 15 "游戏完整运行" "cargo run --release -- --test-battle 2>/dev/null | grep -q 'Battle complete'"

# 汇总
echo ""
echo "=========================================="
echo "  结果: $PASS/15 通过, $FAIL/15 失败"
if [ "$FAIL" -eq 0 ]; then
    echo "  🎉 迁移完成！所有验收条件满足。"
else
    echo "  ❌ 迁移未完成，$FAIL 项未通过。"
    exit 1
fi
echo "=========================================="
```

## 8. 迁移期间的测试双重兼容

### 8.1 过渡策略

在 Phase 1-3 迁移期间，测试将处于"两种位置并存"的状态：

```
# 旧位置（源文件内嵌）— 逐步迁出
src/battle/damage.rs 中的 #[cfg(test)] mod tests { ... }

# 新位置（tests/ 独立文件）— 逐步迁入
tests/unit/core/battle/damage_calculation.rs
```

### 8.2 兼容规则

| 规则 | 说明 |
|------|------|
| 🟩 每迁移一个模块的测试，立即验证新位置测试通过 | 防止测试丢失 |
| 🟩 旧位置测试保留直到对应 Phase 完成 | 双重保障 |
| 🟩 Phase 完成后删除旧位置测试 | 清理过渡代码 |
| 🟥 禁止在迁移测试的同时修改测试逻辑 | 迁移只改位置，不改逻辑 |

### 8.3 清理检查

```bash
# Phase 完成后检查旧测试是否已清理
# 检查 src/ 下是否还有 #[cfg(test)] mod tests
grep -rn "#\[cfg(test)\]" src/ | grep -v "shared/testing/"
# 迁移完成后，只有 shared/testing 中允许保留（如果有的话）
```
