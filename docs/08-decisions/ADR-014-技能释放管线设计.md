# ADR-014: 技能释放管线设计

## 状态

Proposed

## 背景

技能释放是 SRPG 战斗中最核心的操作流程。当前项目已实现 Effect Pipeline（Generate→Modify→Execute）和 SkillData.can_use() 验证，但缺少从用户输入到效果执行的完整管线架构决策。本 ADR 定义五阶段释放管线、验证逻辑、冷却管理和管线衔接。

## 引用的领域规则

- `docs/02-domain/skill/skill-rules.md` — 技能释放管线（6 步）、验证顺序、冷却管理
- `docs/02-domain/attribute-modifier/attribute-modifier-rules.md` — 效果管线 Generate→Modify→Execute
- `docs/02-domain/turn/turn-rules.md` — 回合阶段、Tick 时机

## 决策

### 1. 五阶段释放管线

所有技能释放必须遵循五阶段固定管线（宪法 11.2.1）：

```
Stage 1: Validate（验证）
  ├─ 调用 SkillData.can_use()
  ├─ 检查：冷却 → 消耗 → 标签 → 目标标签 → HP 阈值
  └─ 失败 → 返回 SkillUseError，UI 提示

Stage 2: Cost（消耗）
  ├─ 扣除 MP（通过 set_vital）
  ├─ 设置冷却（SkillCooldowns.set）
  └─ 失败 → 回滚（不应发生，Validate 已检查）

Stage 3: Cast（施法）
  ├─ 发送 SkillActivated Message
  ├─ 目标选择（通过 SkillTargeting）
  └─ 生成 SkillExecutionContext

Stage 4: Effect（效果执行）
  ├─ 遍历 SkillData.effects
  ├─ 每个 EffectDef 路由到 Effect Pipeline
  ├─ Generate → Modify → Execute
  └─ 收集 EffectResult

Stage 5: Settlement（结算）
  ├─ 更新 BattleRecord
  ├─ 检查死亡（Dead Tag）
  ├─ 触发 Trait 效果（OnAttack/OnKill）
  └─ 通知 UI（飘字、战斗日志）
```

### 2. 验证逻辑（can_use）

`SkillData.can_use()` 是纯函数，检查顺序固定：

```rust
pub fn can_use(
    &self,
    attrs: &Attributes,
    source_tags: &GameplayTags,
    target_tags: Option<&GameplayTags>,
    cooldown_remaining: u32,
) -> Result<(), SkillUseError> {
    // 1. 冷却检查（最先）
    if cooldown_remaining > 0 {
        return Err(SkillUseError::OnCooldown { remaining: cooldown_remaining });
    }
    // 2. MP 消耗检查
    if let Some(SkillCondition::MpCost(required)) = self.conditions.iter().find(...) {
        let current = attrs.get(AttributeKind::Mp);
        if current < required as f32 {
            return Err(SkillUseError::InsufficientMp { required, current: current as i32 });
        }
    }
    // 3. 施法者标签检查
    // 4. 目标标签检查
    // 5. HP 阈值检查
    Ok(())
}
```

### 3. 冷却管理

| 操作 | 时机 | 说明 |
|------|------|------|
| `set(skill_id, cooldown)` | 技能成功释放后 | 进入冷却 |
| `tick()` | `OnEnter(TurnPhase::TurnEnd)` | 递减所有冷却 |
| `get(skill_id)` | 验证阶段 | 查询当前冷却 |
| `set(0)` | 清除冷却 | 移除条目（不是设置为 0） |

冷却数据存储在 `SkillCooldowns` 组件中（`HashMap<String, u32>`）。

### 4. Effect Pipeline 衔接

技能效果通过 Effect Pipeline 执行，不是直接调用：

```
SkillData.effects[]
  ↓
每个 EffectDef
  ↓
EffectHandlerRegistry.find(type_name)
  ↓
handler.generate() → PendingEffectData
  ↓
ModifierRuleRegistry.apply_damage_modifiers()（Modify 阶段）
  ↓
handler.execute() → EffectResult
```

### 5. SkillExecutionContext

封装一次技能释放的全部输入数据，避免执行阶段重新查询 ECS：

```rust
pub struct SkillExecutionContext {
    pub source: Entity,       // TODO(future): 替换为 UnitId（需 Core 层统一携带 UnitId Component）
    pub target: Entity,       // TODO(future): 替换为 UnitId
    pub skill_id: String,     // TODO(future): 替换为 SkillId
    pub source_attrs: Attributes,
    pub target_attrs: Attributes,
    pub source_tags: GameplayTags,
    pub target_tags: GameplayTags,
    pub terrain_defense_bonus: i32,
}
```

> 当前使用 `Entity` 而非 Strong ID，因为 SkillExecutionContext 在 Effect Pipeline 内部传递，
> 不直接序列化到日志。切换到 Strong ID 的时机是 Core 层所有模块统一携带 `UnitId` Component。
> （参见 `TODO(future)` 在 `pipeline/execute.rs`、`preview.rs` 中的标注）

通过 `from_query()` 从 ECS 查询构建，克隆属性数据避免借用冲突。

## Module Design

### 管线执行位置

```
TurnPhase::ExecuteAction
  └─ execute_action_system
       ├─ 构建 SkillExecutionContext
       ├─ 调用 can_use() 验证
       ├─ 扣除资源 + 设置冷却
       ├─ 遍历 effects → Effect Pipeline
       └─ 收集结果 → Settlement
```

### 文件组织

```
src/core/skill/
├── mod.rs                    # SkillPlugin + execute_action_system
├── domain/
│   ├── types.rs             # SkillData.can_use() 验证逻辑 + SkillCondition/SkillTargeting
│   └── skill_error.rs       # SkillUseError 枚举
├── slots.rs                 # SkillSlots + SkillCooldowns 组件
└── preview.rs               # 效果预览（纯函数）+ SkillExecutionContext
```

## Communication Design

### Message

| Message（共享事件类型） | 发送方 | 接收方 | 时机 |
|------------------------|--------|--------|------|
| `shared::event::skill::SkillActivated` | Cast 阶段 | UI、日志、回放 | 技能确认释放后 |
| `shared::event::battle::DamageDealt` | Execute 阶段 | UI 飘字、战斗日志 | 伤害结算后 |
| `shared::event::battle::HealApplied` | Execute 阶段 | UI 飘字、战斗日志 | 治疗结算后 |
| `shared::event::buff::BuffApplied` | Execute 阶段 | UI 图标、战斗日志 | Buff 施加后 |

> 所有可观测事件类型定义在 `shared::event/` 目录（参见 ADR-012）。
> 旧类型（`core::battle::DamageApplied` 等）仅供 BattleRecord/VFX 消费，日志侧统一使用共享事件。

### 函数调用

- `SkillData.can_use()` — 纯函数验证
- `SkillCooldowns.set()` / `tick()` — 冷却管理
- `EffectHandlerRegistry.find()` — 效果处理器查找

## 边界定义

- 允许：`core/skill/` 调用 `core/effect/` 的 EffectHandlerRegistry
- 允许：`core/skill/` 调用 `core/attribute/` 的 Attributes 方法
- 允许：`core/skill/` 发送 `shared/event/skill.rs` 的 SkillActivated Message
- 禁止：`core/skill/` 直接修改目标 HP（必须走 Effect Pipeline）
- 禁止：`core/skill/` 直接施加 Buff（必须走 Effect Pipeline 的 ApplyBuff）

## Forbidden（禁止事项）

- 🟥 禁止：跳过 can_use() 验证直接释放技能 — 理由：破坏战斗公平性
- 🟥 禁止：技能效果绕过 Effect Pipeline 执行 — 理由：跳过修饰、记录等关键步骤
- 🟥 禁止：在 TurnEnd 以外的阶段调用 tick() — 理由：冷却递减必须与回合同步
- 🟥 禁止：验证失败后仍执行效果 — 理由：规则失败是正常不满足，不是错误
- 🟥 禁止：SkillSlots 为空时不回退到 basic_attack — 理由：空槽位必须保证单位仍有基础攻击能力
- 🟥 禁止：预览时修改游戏状态 — 理由：预览是纯函数，用于 UI 展示
- 🟥 禁止：set(0) 后 HashMap 中仍保留条目 — 理由：set(0) 必须移除条目

## Definition / Instance Design

- Definition：SkillData（技能定义）、EffectDef（效果定义）
- Instance：SkillCooldowns（冷却状态）、PendingEffect（待处理效果）、EffectResult（执行结果）

## 后果

### 正面
- 五阶段管线保证技能释放的确定性和可观测性
- 验证逻辑纯函数化，可独立测试
- Effect Pipeline 衔接保证修饰规则和记录完整性

### 负面
- 五阶段管线增加少量代码复杂度
- SkillExecutionContext 的属性克隆增加内存开销

## 替代方案

| 方案 | 优点 | 缺点 | 为何放弃 |
|------|------|------|----------|
| 直接执行效果 | 简单 | 跳过验证、修饰、记录 | 违反宪法 11.2.1 |
| 两阶段（验证+执行） | 简化 | 管线衔接不清晰 | 五阶段更精确 |
| 异步管线 | 并发 | 确定性破坏 | SRPG 需要确定性 |
