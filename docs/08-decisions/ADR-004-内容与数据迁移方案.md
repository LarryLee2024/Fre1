# ADR-004: 内容与数据迁移方案

## 状态: Proposed

**版本**: 1.0
**日期**: 2026-06-14
**范围**: 14 个 RON 配置加载点从 `assets/` 到 `content/` 的完整迁移
**依赖**: ADR-001（迁移计划）、ADR-003（分层契约与依赖迁移）

---

## 1. 背景

### 问题

当前项目中 38 个 RON 配置文件与二进制资源（精灵图、音频）混放在 `assets/` 目录下，导致以下核心问题：

| # | 问题 | 影响 |
|---|------|------|
| 1 | `assets/` 混合二进制资源（sprite、audio）与游戏数据（RON 配置） | 无法独立管理内容与渲染资源 |
| 2 | Content（是什么）与 Assets（渲染资源）无清晰边界 | 违反三棵树分离原则 |
| 3 | `assets/` 前缀硬编码在 14 个加载点中 | 迁移路径不透明，MOD 隔离困难 |
| 4 | 热重载同时影响游戏数据和视觉资源 | 无法实现内容级精确热重载 |
| 5 | MOD 内容无法安全隔离 | MOD 内容需注入 `assets/` 与基础内容混杂 |

### 架构规范引用

| 文档 | 路径 | 关系 |
|------|------|------|
| 内容管线架构 | `docs/01-architecture/content-pipeline.md` | Content/Core 分离、数据流 |
| 内容数据格式 | `docs/01-architecture/content_data_format.md` | RON 格式契约、引用机制 |
| 目标目录结构 | `docs/01-architecture/project-structure.md` | content/ 目录树定义 |
| 七层架构总纲 | `docs/01-architecture/README.md` | 最高优先级架构约束 |

### 决策目标

将所有 RON 游戏配置数据从 `assets/` 迁移到项目根 `content/` 目录，建立 Content/Core 判定标准，实现 MOD 安全隔离与精确热重载。

### 3.3 Content/Core 判定标准

> **Skill 是 Core，Fireball 是 Content。**

| 概念 | 层级 | 位置 | 说明 |
|------|------|------|------|
| 技能规则引擎 | Core | `src/core/skill/` | 怎么释放技能、冷却怎么算 |
| 火球术数据 | Content | `content/skills/fireball.ron` | 伤害数值、范围、标签 |
| Buff 规则引擎 | Core | `src/core/buff/` | 怎么施加 Buff、回合结算 |
| 剧毒数据 | Content | `content/buffs/poison.ron` | 持续伤害、回合数 |
| 关卡规则 | Core | `src/core/stage/` | 胜负条件判定逻辑 |
| 第三关配置 | Content | `content/stages/stage_03.ron` | 地图大小、敌人配置 |

**一句话总结**：**Core 回答"怎么做"，Content 回答"是什么"。**

## 4. 迁移步骤（按内容类型）

### 4.1 通用 5 步迁移流程

对**每种**内容类型，执行以下 5 步：

| 步骤 | 操作 | 验证 |
|------|------|------|
| Step 1 | 创建目标目录：`mkdir -p content/{dir}` | 目录存在 |
| Step 2 | 移动配置文件：`mv assets/{old}/*.ron content/{new}/` | 文件数一致 |
| Step 3 | 更新 RegistryLoader 路径字符串 | 编译通过 |
| Step 4 | 更新所有直接 `load` 调用（如有） | 编译通过 |
| Step 5 | 完整验证：`cargo build && cargo test` | 零错误零警告 |

## 6. RegistryLoader Trait 迁移

### 6.1 当前 RegistryLoader 定义

```rust
// src_待迁移/core/registry_loader.rs
pub trait RegistryLoader: Default {
    type Item: DeserializeOwned;

    fn register_item(&mut self, item: Self::Item);
    fn register_defaults(&mut self);
    fn is_empty(&self) -> bool;
    fn registry_name() -> &'static str;

    fn load_from_dir(dir: &str) -> Self { ... }        // 单条记录文件
    fn load_from_dir_vec(dir: &str) -> Self { ... }    // 数组记录文件
    fn load_from_file(path: &str) -> Self { ... }       // 单文件加载
}
```

### 6.2 过渡期双路径支持

迁移期间，`RegistryLoader` 默认方法增加双路径回退：

```rust
/// 过渡期路径解析：优先新路径，回退旧路径
fn resolve_path(dir: &str) -> String {
    let new_path = format!("content/{}", dir);
    let old_path = format!("assets/{}", dir);

    if std::path::Path::new(&new_path).exists() {
        new_path
    } else if std::path::Path::new(&old_path).exists() {
        bevy::log::warn!(
            target: "content_migration",
            old_path = %old_path,
            new_path = %new_path,
            "使用旧路径加载内容，请迁移到 content/ 目录"
        );
        old_path
    } else {
        new_path // 默认返回新路径（可能为空目录）
    }
}

fn load_from_dir(dir: &str) -> Self {
    let resolved = Self::resolve_path(dir);
    // ... 原有逻辑使用 resolved 而非 dir
}

fn load_from_dir_vec(dir: &str) -> Self {
    let resolved = Self::resolve_path(dir);
    // ... 原有逻辑使用 resolved 而非 dir
}

fn load_from_file(path: &str) -> Self {
    // 文件路径直接更新，不需回退
    // ... 原有逻辑
}
```

### 6.3 过渡期完成后

移除双路径回退，所有加载只使用 `content/` 路径：

```rust
fn load_from_dir(dir: &str) -> Self {
    // 不再回退 assets/，只使用 content/
    // ... 原有逻辑
}
```

## 8. 内容校验管线

### 8.1 三级校验体系

```
Level 1: Schema 校验（加载时）
    ├── RON 语法正确性
    ├── 必填字段存在性（id、version、name）
    ├── 类型正确性
    └── 字段值范围

Level 2: 引用校验（加载后）
    ├── ID 引用存在性（buff_ids → BuffRegistry）
    ├── 标签引用存在性（tags → TagRegistry）
    └── 循环依赖检测

Level 3: 规则校验（运行时 / 开发工具）
    ├── 游戏规则一致性
    ├── 数值平衡合理性
    └── 逻辑矛盾检测
```

### 8.2 校验时机

| 级别 | 时机 | 失败处理 |
|------|------|---------|
| Level 1 | Content Plugin 加载时 | 跳过该配置，`warn!` 日志 |
| Level 2 | 所有 Content 加载完成后（`validate_all_references`） | 报告所有引用错误，不阻断启动 |
| Level 3 | 开发工具运行时（`tools/data_validator/`） | 输出校验报告 |

### 8.3 引用校验规则

| 校验类型 | 级别 | 失败处理 |
|----------|------|---------|
| 必填引用缺失 | 🟥 错误 | 加载失败，报告错误 |
| 可选引用缺失 | ⚠️ 警告 | 跳过该引用，使用默认值 |
| 循环引用 | 🟥 错误 | 加载失败，报告错误 |
| 引用类型不匹配 | 🟥 错误 | 加载失败，报告错误 |

## 9. 风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|---------|
| 配置路径断裂导致游戏无法启动 | 高 | 高 | 过渡期双路径回退：优先 `content/`，回退 `assets/` |
| 迁移过程中编译失败 | 中 | 高 | 每种内容类型独立迁移，迁移后立即 `cargo build && cargo test` |
| MOD 内容与基础内容冲突 | 中 | 中 | `content/` 隔离使 MOD 加载安全；MOD 配置加 MOD 前缀 |
| 热重载失效 | 低 | 中 | 内容热重载使用独立文件监视器，与 assets/ 热重载隔离 |
| 遗漏加载点 | 低 | 高 | 使用 `grep` 扫描全部 `"assets/` 路径字符串，确保零遗漏 |
| settings.ron 迁移后丢失 | 低 | 低 | settings.ron 保留原位，不纳入内容迁移范围 |
| Campaign 加载器未使用 RegistryLoader | 低 | 低 | 迁移路径字符串，后续统一为 RegistryLoader |

## 10. 迁移顺序与依赖

### 10.1 推荐迁移顺序

由于各内容类型之间存在引用依赖，建议按以下顺序迁移：

```
Phase 1: 基础定义（无外部依赖）
    ├── definitions/attributes.ron + tags.ron
    ├── terrains/*.ron
    └── modifiers/*.ron

Phase 2: 核心内容（依赖基础定义）
    ├── skills/*.ron
    ├── buffs/*.ron
    └── ai_behaviors/*.ron

Phase 3: 实体模板（依赖技能/Buff）
    ├── characters/*.ron
    ├── enemies/*.ron (未来)
    └── classes/*.ron

Phase 4: 经济系统（依赖实体模板）
    ├── items/*.ron
    └── equipments/*.ron

Phase 5: 关卡系统（依赖所有实体）
    ├── stages/*.ron
    └── campaigns/*.ron
```
