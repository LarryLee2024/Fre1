# ADR-015: 技能标签与分类体系

## 状态

Proposed

## 背景

技能的分类、筛选和交互规则依赖标签系统。当前项目已定义 17 个 GameplayTag（Element/Status/Weapon/Class/SkillType/BuffType），并通过 ModifierRule 实现标签驱动的伤害修饰。但缺少标签在技能系统中统一使用的架构决策。本 ADR 定义标签在技能分类、修饰规则匹配、AI 决策和装备限制中的驱动作用。

## 引用的领域规则

- `docs/02-domain/skill/skill-rules.md` — 规则6：标签驱动技能过滤
- `docs/02-domain/attribute-modifier/attribute-modifier-rules.md` — 规则3：标签匹配控制修饰规则
- `docs/01-architecture/skill-buff-abstraction.md` — §4.9 Tag 统一使用场景

## 决策

### 1. 标签分类体系

当前 17 个标签按 6 个维度分类：

| 维度 | 标签 | 用途 |
|------|------|------|
| **元素** | FIRE, ICE, POISON | 元素类型、克制关系 |
| **状态** | STUN, BURN, REGEN | 状态效果标记 |
| **武器** | MELEE, RANGED | 攻击距离分类 |
| **职业** | WARRIOR, ARCHER, MAGE | 职业归属 |
| **技能类型** | SKILL_ACTIVE, SKILL_PASSIVE | 主动/被动 |
| **Buff 类型** | BUFF, DEBUFF | 增益/减益 |

### 2. 标签在技能中的驱动作用

#### 2.1 修饰规则匹配（ModifierRule）

标签驱动伤害/治疗修饰的核心机制：

```ron
// content/modifiers/element_interactions.ron
(
    name: "火焰共鸣",
    source_tag: FIRE,    // 攻击方技能标签
    target_tag: FIRE,    // 目标标签
    effect: DamageMultiplier(1.5),
)
```

匹配逻辑：`source_tags.contains(rule.source_tag) AND target_tags.has(rule.target_tag)`

#### 2.2 技能筛选

按标签过滤技能列表：
- 查询所有 `SKILL_ACTIVE` 技能进行冷却缩减
- 查询所有 `FIRE` 技能进行元素伤害加成
- 查询所有 `MELEE` 技能判断近战范围

#### 2.3 AI 决策

AI 通过标签评估目标状态：
- 目标有 `STUN` → 选择物理攻击（不用魔法）
- 目标有 `BUFF` → 优先使用 `DEBUFF` 技能
- 自身有 `FIRE` → 优先使用 `FIRE` 技能（触发火焰共鸣）

#### 2.4 装备限制

装备佩戴条件通过标签校验：
- 需要 `MARTIAL` 标签才能装备军用武器
- 需要 `WARRIOR` 标签才能装备重甲

### 3. 标签命名规范

使用点分层级命名（支持前缀匹配）：

```
Damage.Fire              → 伤害类型.火焰
Damage.Ice               → 伤害类型.冰霜
Unit.Human               → 单位.人类
Unit.Boss                → 单位.Boss
Status.Poison            → 状态.中毒
Weapon.Sword             → 武器.剑
Buff.Defensive           → Buff.防御类
```

当前使用扁平枚举（TagName::Fire），未来可扩展为层级命名。

### 4. 标签查询 API

`GameplayTags` 组件提供位掩码查询：

```rust
pub fn has(&self, tag: GameplayTag) -> bool       // 单标签查询
pub fn has_any(&self, tags: &[GameplayTag]) -> bool // 任一匹配
pub fn has_all(&self, tags: &[GameplayTag]) -> bool // 全部匹配
pub fn add(&mut self, tag: GameplayTag)            // 添加标签
pub fn remove(&mut self, tag: GameplayTag)         // 移除标签
```

### 5. 标签来源管理

标签来自三个来源，通过 `PersistentTags` 组件管理：

| 来源 | 生命周期 | 示例 |
|------|---------|------|
| Trait（种族/职业） | 永久 | WARRIOR, MAGE, FIRE |
| Equipment | 穿戴期间 | MELEE, RANGED |
| Buff | Buff 持续期间 | STUN, BURN |

Trait 变化或装备变化时，从两层合并到 GameplayTags。

## Module Design

### 标签定义位置

```
content/definitions/tags.ron    → 标签定义（RON 配置）
src/core/tag.rs                → GameplayTag 枚举 + GameplayTags 组件
src/core/tag_def.rs            → TagDefRegistry（标签定义注册表）
src/core/character/traits/     → PersistentTags（标签来源管理）
```

### 标签在技能 RON 中的使用

```ron
// 技能标签：定义技能的分类属性
tags: [FIRE, SKILL_ACTIVE]

// 修饰规则标签：定义交互规则
source_tag: FIRE
target_tag: FIRE
```

## Communication Design

### 标签数据流

```
tags.ron → TagDefRegistry → GameplayTag 枚举
                                  ↓
                    SkillDef.tags → GameplayTags 组件
                    BuffDef.tags → GameplayTags 组件
                    EquipmentDef.tags → PersistentTags.from_equipment
                    TraitDef.effects → PersistentTags.from_traits
                                  ↓
                    GameplayTags（合并后）→ ModifierRule 匹配
```

### 函数调用

- `GameplayTag::to_tag()` — TagName → GameplayTag 转换
- `GameplayTags::has()` / `has_any()` / `has_all()` — 标签查询
- `ModifierRuleRegistry::apply_damage_modifiers()` — 标签匹配修饰

## 边界定义

- 允许：`core/tag/` 被所有业务模块依赖（标签是跨模块共享的分类机制）
- 允许：`core/tag_def/` 依赖 `core/registry_loader/`（加载标签定义）
- 禁止：在业务代码中硬编码标签匹配逻辑（如 `if skill_id == FIREBALL`）
- 禁止：运行时使用字符串查询标签（必须使用 GameplayTag 位掩码）

## Forbidden（禁止事项）

- 🟥 禁止：硬编码技能筛选逻辑（如 `if skill_id == FIREBALL`）— 理由：违反数据驱动原则
- 🟥 禁止：使用字符串匹配进行技能分类 — 理由：性能差、类型不安全
- 🟥 禁止：在 RON 中使用位掩码数值（必须使用 TagName 枚举）— 理由：可读性
- 🟥 禁止：运行时使用字符串查询 GameplayTag — 理由：性能（位掩码 O(1)）
- 🟥 禁止：绕过标签匹配直接应用修饰规则 — 理由：破坏修饰规则的通用性
- 🟥 禁止：在 ModifierRule 中使用 match 硬编码分发 — 理由：违反 Calculator trait 扩展原则

## Definition / Instance Design

- Definition：TagName 枚举（RON 配置）、TagDefRegistry（标签定义）
- Instance：GameplayTag（位掩码值）、GameplayTags（组件）、PersistentTags（来源管理）

## 后果

### 正面
- 新增元素/类型只需添加 Tag 定义和 ModifierRule 配置，不改代码
- 标签驱动的交互规则可扩展性强
- 位掩码查询性能 O(1)

### 负面
- 标签数量增长后需要管理命名规范
- 位掩码 u64 限制最多 64 个标签

## 替代方案

| 方案 | 优点 | 缺点 | 为何放弃 |
|------|------|------|----------|
| 字符串匹配 | 简单 | 性能差、类型不安全 | 违反宪法 1.2.1 |
| Enum 索引 | 类型安全 | 位掩码查询更高效 | GameplayTag 更灵活 |
| 独立 Tag 系统 | 灵活 | 过度设计 | 复用 Bevy 原生能力 |
