# ADR-025: 七领域模块化架构设计

## 状态

Proposed

## 背景

项目已完成 SRPG-GAS 架构对齐，将战斗能力系统拆分为七个独立领域：**Tag、Modifier、Buff、Effect、Targeting、Ability、Trigger**。此前这些概念的边界模糊——Effect 散落在 skill 和 modifier 中，Targeting 是 Selector 的别名，Skill 概念过载了 Ability 的职责。领域规则文档已更新（`docs/02-domain/`），现需在架构层将这七个领域正式化为独立模块。

### 当前问题

| 问题 | 当前状态 | 后果 |
|------|---------|------|
| Effect 非一级模块 | EffectDef 在 skill/domain/types.rs 中 | 模块间耦合，新增 Effect 类型需改多处 |
| Skill 概念过载 | Skill 同时承担 Ability + Skill 两个概念 | 职责不单一，管道编排和技能定义混在一起 |
| Selector → Targeting 未架构确认 | Selector 作为模块存在，但概念已演变为 Targeting | 命名与实际业务不符 |
| 模块间依赖不明确 | tag → modifier → effect → buff → ability 的无环依赖链未定义 | 新增代码可能引入循环依赖 |
| Plugin 注册顺序未规范 | 各模块 Plugin 随意注册 | 启动顺序依赖不明确 |

## 决策

### 一、七领域模块目录结构

每个领域为一个独立模块，遵循「定义分离 + 组件实例分离 + 注册表分离」的模式：

```
src/
├── core/
│   ├── tag/                     ← 领域 1：标签系统
│   │   ├── mod.rs               → TagPlugin, 组件 GameplayTags
│   │   ├── types.rs             → GameplayTag(u64), TagName enum, TagCategory
│   │   ├── components.rs        → PersistentTags Component
│   │   └── registry.rs          → TagRegistry Resource, TagDefinition
│   │
│   ├── modifier/                ← 领域 2：属性修饰器（从 modifier_rule.rs + attribute/ 收敛）
│   │   ├── mod.rs               → ModifierPlugin
│   │   ├── types.rs             → ModifierRule, ModifierEntry, ModifierOp, StatTarget
│   │   ├── calculator.rs        → ModifierCalculator, 修饰计算逻辑
│   │   └── registry.rs          → ModifierRuleRegistry Resource
│   │
│   ├── effect/                  ← 领域 3：效果系统（已有，需要调整边界）
│   │   ├── mod.rs               → EffectPlugin（已有）
│   │   ├── types.rs             → EffectDef enum（从 skill/domain/types.rs 移入）
│   │   ├── handler.rs           → EffectHandler trait + EffectHandlerRegistry（已有）
│   │   ├── pipeline.rs          → Generate → Modify → Execute 调度
│   │   └── data.rs              → PendingEffectData, EffectResult, EffectPreview
│   │
│   ├── buff/                    ← 领域 4：Buff 系统（已有）
│   │   ├── mod.rs               → BuffPlugin
│   │   ├── domain/
│   │   │   ├── types.rs         → BuffDef, BuffData, BuffInstance
│   │   │   ├── apply.rs         → 施加逻辑
│   │   │   ├── resolve.rs       → 持续效果结算
│   │   │   └── expire.rs        → 过期处理
│   │   └── registry.rs          → BuffRegistry Resource
│   │
│   ├── targeting/               ← 领域 5：目标选择（从 selector 重构）
│   │   ├── mod.rs               → TargetingPlugin
│   │   ├── types.rs             → TargetingType enum, TargetingContext
│   │   ├── resolver.rs          → calculate_targets() 纯函数
│   │   └── validator.rs         → TargetValidator（检查目标合法性）
│   │
│   ├── ability/                 ← 领域 6：能力系统（从 skill 重构）
│   │   ├── mod.rs               → AbilityPlugin
│   │   ├── domain/
│   │   │   ├── types.rs         → AbilityDef, AbilityData, AbilityCastCommand
│   │   │   ├── pipeline.rs      → Requirement → Cost → Targeting → Effect → Settlement
│   │   │   ├── cost.rs          → 消耗扣除
│   │   │   ├── cooldown.rs      → 冷却管理
│   │   │   └── preview.rs       → 效果预览
│   │   └── registry.rs          → AbilityRegistry Resource
│   │
│   └── trigger/                 ← 领域 7：触发器系统（已有）
│       ├── mod.rs               → TriggerPlugin
│       ├── stack.rs             → ExecutionStack Resource, MAX_STACK_DEPTH=32
│       ├── registry.rs          → TriggerRegistry Resource, TriggerHandler trait
│       └── types.rs             → Trigger enum, TriggerContext
```

### 二、模块依赖图（DAG，禁止循环依赖）

```
                  ┌──────────────────┐
                  │      tag         │ ← 无依赖（最底层）
                  └────────┬─────────┘
                           │
                  ┌────────▼─────────┐
                  │    modifier      │ ← 依赖: tag
                  └────────┬─────────┘
                           │
                  ┌────────▼─────────┐
                  │     effect       │ ← 依赖: tag, modifier
                  └────────┬─────────┘
                      ╱    │    ╲
                     ╱     │     ╲
     ┌───────────────┐ ┌───▼──────┐ ┌───────────────┐
     │     buff      │ │ targeting│ │    trigger    │ ← 依赖: effect, tag
     │ 依赖: effect  │ │ 依赖: tag│ │ 依赖: effect  │
     └───────┬───────┘ └───┬──────┘ └───────┬───────┘
             └──────┬──────┘                │
                    │                       │
             ┌──────▼───────────────────────▼──────┐
             │            ability                  │
             │ 依赖: buff, targeting, trigger,     │
             │        effect, modifier, tag        │
             └─────────────────────────────────────┘
```

**关键规则**：
- tag 是唯一不依赖任何模块的底层模块
- modifier 只依赖 tag（通过标签匹配查找 ModifierRule）
- effect 依赖 tag 和 modifier（Generate 阶段查 Formula，Modify 阶段走 Modifier）
- buff、targeting、trigger 平行，都依赖 effect
- ability 是最上层模块，依赖所有其他六个模块

### 三、Plugin 注册顺序

```rust
// App 构建顺序（禁止颠倒）
App::new()
    // 第 1 层：无依赖的基础设施
    .add_plugins(TagPlugin)
    .add_plugins(ModifierPlugin)
    
    // 第 2 层：依赖 tag 和 modifier
    .add_plugins(EffectPlugin)
    
    // 第 3 层：平行依赖 effect
    .add_plugins(BuffPlugin)
    .add_plugins(TargetingPlugin)
    .add_plugins(TriggerPlugin)
    
    // 第 4 层：依赖所有下层
    .add_plugins(AbilityPlugin)
```

**注册顺序不变量**：任何 Plugin 必须在它依赖的 Plugin 之后注册。违反导致启动 panic。

### 四、各 Plugin 职责

| Plugin | 插入的 Resource | 注册的 Component | 职责 |
|--------|----------------|-----------------|------|
| TagPlugin | TagRegistry | GameplayTags, PersistentTags | 标签定义加载、标签查询 |
| ModifierPlugin | ModifierRuleRegistry | — | ModifierRule 注册、Calculator 计算 |
| EffectPlugin | EffectHandlerRegistry | — | EffectDef 分发、Pipeline 调度 |
| BuffPlugin | BuffRegistry | ActiveBuffs, BuffInstance | Buff 施加/移除/Tick |
| TargetingPlugin | — | — | 纯函数目标计算，无 Resource |
| TriggerPlugin | TriggerRegistry, ExecutionStack | — | Trigger 注册、ExecutionStack 管理 |
| AbilityPlugin | AbilityRegistry | AbilitySlots, SkillCooldowns | Ability 定义加载、执行管线编排 |

### 五、跨模块通信设计

| 通信场景 | 通信方式 | 说明 |
|---------|---------|------|
| Effect Pipeline 需要 Modifier 修饰 | 函数调用 | `ModifierCalculator::apply()` 在 Effect 的 Modify 阶段调用 |
| Ability 执行需要 Targeting 解析 | 函数调用 | `TargetingResolver::resolve()` 在 Ability 的 Targeting 阶段调用 |
| Ability 需要 Effect Pipeline | 函数调用 | `EffectPipeline::execute()` 在 Ability 的 Effect 阶段调用 |
| Buff 需要 Effect Pipeline | 函数调用 | Buff 触发效果时调用 `EffectPipeline::execute()` |
| Trigger 分发需要 Effect Pipeline | 函数调用 | ExecutionStack 弹出 Entry 后调用 `EffectPipeline::execute()` |
| Tag 新增/移除需要通知 | Observer | GameplayTags 变化时触发 TagAdded/TagRemoved Hook |
| Ability 开始/结束 | Message | AbilityCastStarted, AbilityCastFinished → 跨模块广播 |
| Buff 施加/移除 | Message | BuffApplied, BuffRemoved → UI/Trigger 订阅 |
| 伤害/治疗应用 | Message | DamageApplied, HealApplied → UI/Trigger 订阅 |

### 六、现有代码迁移路径

| 当前内容 | 迁移目标 | 迁移方式 |
|---------|---------|---------|
| `src/core/skill/` → ability/ | 重构 | 内部架构不变，概念重命名为 Ability，文件位置不变 |
| `src/core/selector/` → targeting/ | 重构 | 同 ability |
| `src/core/modifier_rule.rs` 独文件 | modifier/types.rs + registry.rs | 拆分为目录模块 |
| `src/core/attribute/` | modifier/calculator.rs | attribute 的 ModifierStack 逻辑并入 modifier |
| `EffectDef` 在 skill/domain/types.rs | effect/types.rs | 移动到 effect 模块 |
| `Trigger` 枚举在 skill/domain/ | trigger/types.rs | 已分离 |

### 七、EffectDef 所有权迁移（关键）

这是最紧迫的迁移。当前 `EffectDef` 定义在 `src/core/skill/domain/types.rs`，但 Effect 是一级领域，EffectDef 应属于 Effect 模块。

迁移步骤：
1. 在 `src/core/effect/types.rs` 中定义 `EffectDef`（新位置）
2. 从 `src/core/skill/domain/types.rs` 中删除 `EffectDef` 定义
3. 所有 `use crate::core::skill::domain::types::EffectDef` → 改为 `use crate::core::effect::EffectDef`
4. 删除 `mod types` 的 pub 层级调整

**依赖影响**：Ability、Buff、Trigger 都引用 EffectDef，迁移后 import 路径变化，逻辑不变。

## Module Design

### 每个模块的通用结构

```
[domain_name]/
├── mod.rs           → Plugin 定义 + pub use 公开 API
├── types.rs         → 核心类型（枚举、结构体）
├── registry.rs      → 注册表 Resource（如有）
├── [logic files]    → 纯函数业务逻辑（无 ECS 依赖的文件）
├── components.rs    → ECS Component（如有）
└── systems.rs       → System 函数（如有，遵循命名规范）
```

### 各模块对外暴露的 API

```
tag::GameplayTag, tag::GameplayTags, tag::TagName, tag::TagRegistry
modifier::ModifierRule, modifier::ModifierCalculator, modifier::ModifierRuleRegistry
effect::EffectDef, effect::EffectHandler, effect::EffectHandlerRegistry, effect::EffectPipeline
buff::BuffDef, buff::BuffData, buff::BuffInstance, buff::ActiveBuffs, buff::BuffRegistry
targeting::TargetingType, targeting::TargetingResolver, targeting::TargetingContext
ability::AbilityDef, ability::AbilityData, ability::AbilitySlots, ability::AbilityRegistry
trigger::Trigger, trigger::TriggerContext, trigger::TriggerHandler, trigger::TriggerRegistry, trigger::ExecutionStack
```

## Communication Design

### 函数调用链（确定性执行）

```
Ability::execute()
  ├── cost::check_and_pay()            ← 函数调用
  ├── targeting::resolver::resolve()   ← 函数调用（纯函数）
  ├── effect::pipeline::execute()      ← 函数调用
  │     ├── handler::generate()        ← 函数调用（纯函数）
  │     ├── modifier::calculator::apply() ← 函数调用（纯函数）
  │     └── handler::execute()         ← 副作用
  │           └── Message (DamageApplied)
  ├── buff::apply()                    ← 如果效果包含 ApplyBuff
  ├── trigger::stack::push()           ← 如果触发链激活
  └── cooldown::set()                  ← 函数调用
```

### Observer/Hook

| 组件 | Hook 事件 | 用途 |
|-----|----------|------|
| GameplayTags | on_add / on_remove | 标签变化时触发 ModifierRule 重匹配 |
| ActiveBuffs | on_add / on_remove | Buff 变化时触发三层标签重建 |
| BuffInstance | on_insert / on_remove | Buff 生命周期日志 |

### Message

| Message | 发送者 | 订阅者 | 用途 |
|---------|-------|--------|------|
| DamageApplied | effect::pipeline | UI 飘字、Trigger 分发、BattleRecord | 伤害通知 |
| HealApplied | effect::pipeline | UI 飘字、BattleRecord | 治疗通知 |
| BuffApplied | buff::apply | UI 图标、Trigger 分发 | Buff 施加通知 |
| BuffRemoved | buff::expire | UI 图标、Trigger 分发 | Buff 移除通知 |
| AbilityCastStarted | ability::pipeline | UI 技能面板、BattleRecord | 技能开始 |
| AbilityCastFinished | ability::pipeline | UI 技能面板、BattleRecord、Turn 系统 | 技能结束 |

## 边界定义

### 允许的跨模块依赖

| 目标模块 | 可以被哪些模块依赖 | 不允许被哪些模块依赖 |
|---------|------------------|-------------------|
| tag | 所有 | 无（它是基础设施） |
| modifier | effect, buff, ability | tag（反向依赖禁止） |
| effect | buff, trigger, ability | tag, modifier（反向依赖禁止） |
| buff | ability | tag, modifier, effect, targeting, trigger |
| targeting | ability | tag, modifier, effect, buff, trigger |
| trigger | ability | tag, modifier, effect, buff, targeting |
| ability | 无（它是顶层） | 所有模块 |

### 禁止的循环依赖

- 🟥 tag → modifier → effect → buff → ability （允许）
- 🟥 tag → modifier → effect → trigger → ability （允许）
- 🟥 ability → effect → ability （禁止：循环依赖）
- 🟥 buff → effect → modifier → buff （禁止：循环依赖）
- 🟥 trigger → effect → tag → trigger （禁止：循环依赖）

### 模块间数据流方向

```
只读数据（Definition, Registry）：
  tag → modifier → effect → buff/trigger/targeting → ability

可变数据（Instance, Component）：
  ability → buff/effect → modifier → tag
  (通过 Observer/Hook 反向传播，不通过直接函数调用)
```

## Forbidden（禁止事项）

- 🟥 **禁止：EffectDef 定义在 ability 模块中** — EffectDef 属于 effect 模块，ability 只是消费它
- 🟥 **禁止：Ability 模块直接操作 Buff 的内部状态** — 必须通过 Effect Pipeline（ApplyBuff Effect）
- 🟥 **禁止：模块间循环依赖** — 依赖图必须是 DAG，禁止 `A→B→A`
- 🟥 **禁止：Tag 模块依赖任何其他领域模块** — tag 是最底层基础设施
- 🟥 **禁止：Targeting 模块有副作用** — targeting 是纯函数，不得修改游戏状态
- 🟥 **禁止：Modifier 模块直接感知 Ability 或 Buff** — modifier 是纯计算，不关心来源
- 🟥 **禁止：Ability 模块直接调用 Modifier 计算** — 必须通过 Effect Pipeline 的 Modify 阶段
- 🟥 **禁止：Trigger 模块跳过 ExecutionStack 直接执行 Effect** — Stack 保证触发链顺序
- 🟥 **禁止：Effect 模块跳过 EffectHandlerRegistry 直接 match 分发** — 新增 Effect 类型必须注册
- 🟥 **禁止：Plugin 注册顺序违反依赖图** — 低层 Plugin 必须先注册，否则启动 panic

## Definition / Instance Design

| 模块 | Definition（不可变配置） | Instance（运行时状态） |
|------|------------------------|----------------------|
| tag | TagDefinition, TagName enum | GameplayTags Component, PersistentTags Component |
| modifier | ModifierRule, ModifierCalculator | (ModifierEntry 记录在 PendingEffectData 中) |
| effect | EffectDef, EffectHandler | PendingEffectData（瞬态），EffectResult（瞬态） |
| buff | BuffDef, BuffData | BuffInstance in ActiveBuffs Component |
| targeting | TargetingType enum | TargetingContext（瞬态） |
| ability | AbilityDef, AbilityData | AbilitySlots Component, SkillCooldowns Component |
| trigger | Trigger enum, TriggerHandler | ExecutionStack Resource, TriggerContext（瞬态） |

## 后果

### 正面
- 七个领域各自独立模块，职责单一，边界清晰
- 依赖图 DAG 无环，杜绝循环依赖
- Effect 不再散落各处，归属 effect 模块
- Plugin 注册顺序明确，启动确定性
- Skill 重命名为 Ability，概念对齐 SRPG-GAS

### 负面
- 迁移期间需要修改大量 import 路径（EffectDef 等）
- `skill/` 和 `ability/` 在一段时间内可能共存
- 新团队需要学习 7 个模块的边界规则
- 部分模块（modifier, targeting）从单文件变为目录模块，复杂度略增

## 替代方案

| 方案 | 优点 | 缺点 | 为何放弃 |
|------|------|------|----------|
| 保持现状（skill 含 EffectDef） | 无迁移成本 | 概念过载、循环依赖风险 | 违反单一职责 |
| 所有 7 个模块合并为一个 CombatSystem | 简单 | God 模块，违反 Feature First | 违反宪法 |
| 将 Trigger 并入 Buff | 减少模块数 | 职责耦合，Buff 膨胀 | 76.md 推荐独立 Trigger |
| 将 Targeting 并入 Ability | 减少模块数 | Targeting 是纯函数，可独立测试和复用 | 降低复用性 |
