# ADR-011: 错误模块实施

## 状态

Accepted

## 背景

项目已有完整的错误架构设计文档 `docs/01-architecture/error-architecture.md`，但当前实现与架构文档之间存在显著差距。

### 当前实现状态

`src/shared/error/` 仅有骨架：

- ✅ `InfrastructureError` 枚举（4 个变体：AssetLoadFailed, SerializationError, IoError, SaveDataCorrupted）
- ✅ `GameResult<T>` 类型别名
- ✅ 单元测试覆盖错误格式和类型可用性
- 🟥 **缺失 1**：`ErrorContext` trait 未实现（架构文档 §3.1）
- 🟥 **缺失 2**：`LogIfError` trait 未实现（架构文档 §3.2）
- 🟥 **缺失 3**：`GameErrorEvent` 事件通道未定义（架构文档 §4）
- 🟥 **缺失 4**：领域错误仍在 `src/core/error/` 中未拆分（架构文档 §8 迁移计划）

### 输入文档

- `docs/01-architecture/error-architecture.md` — 错误架构设计（三层错误模型、失败分类学、GameErrorEvent）
- `docs/01-architecture/infrastructure-design.md` §4.7 — 错误策略：模块独立错误枚举
- `docs/00-governance/ai-constitution-complete.md` §13.9 — 错误体系规范
- `docs/00-governance/ai-constitution-complete.md` §13.10 — 领域事件与审计轨迹

## 引用的领域规则

- 宪法 §13.9.1 — 分领域错误原则，禁止全局 AppError
- 宪法 §13.9.2 — 失败分类学：RuleFailure / DomainError / InfrastructureError / Bug
- 宪法 §13.9.3 — 错误强制要求（带编号、完整上下文）
- 宪法 §13.9.4 — 业务层 Panic 禁令
- 宪法 §1.2.1 — 强类型 ID
- 宪法 §1.1.4 — 逻辑与表现分离
- 宪法 §1.4.1 — 核心领域与引擎解耦

## 决策

### 核心决策

错误体系分三层，每层独立但不重复造轮子：

```
┌─────────────────────────────────────────────────┐
│  Shared Layer：通用错误工具                         │
│  ├── ErrorContext trait     → 任意 Result 增强     │
│  ├── LogIfError trait       → 错误 → DEBUG 日志    │
│  └── InfrastructureError    → 基础设施通用错误枚举   │
│     + GameResult<T>                                  │
└──────────────────────┬──────────────────────────┘
                       │ 被所有层使用
┌──────────────────────▼──────────────────────────┐
│  Core Layer：领域专属错误                          │
│  每个领域定义独立错误枚举 + 结果别名                  │
│  ┌──────────┐ ┌───────┐ ┌──────────┐              │
│  │BattleError│ │BuffErr│ │SkillError│  ...        │
│  │BattleResult│ │...   │ │...       │              │
│  └──────────┘ └───────┘ └──────────┘              │
└──────────────────────┬──────────────────────────┘
                       │ 同一领域内使用
┌──────────────────────▼──────────────────────────┐
│  App Layer：错误跨层通信通道                        │
│  ┌─────────────────────────────────────────────┐  │
│  │ GameErrorEvent (Event) → ErrorMonitor System │  │
│  │   → UI Toast (玩家可见)                       │  │
│  │   → tracing::error! (开发可见)                │  │
│  └─────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────┘
```

**为什么不分三层？**

- Shared 层工具所有层共享（Infrastructure 和 Core 都要用）
- Core 层错误按领域独立（宪法 13.9.1 强制要求）
- App 层事件通道解决"错误被吞掉"的常见问题

## Module Design

### Phase 1：补全 shared/error 工具（Shared 层）

目标：将 `src/shared/error/` 从骨架补全为完整的错误工具模块。

**目录结构变更：**

```
src/shared/error/
├── mod.rs           # 公开导出
├── result.rs        # InfrastructureError + GameResult<T>（保持现有）
├── context.rs       # 新增：ErrorContext trait
└── extensions.rs    # 新增：LogIfError trait
```

**新增内容：**

```rust
// src/shared/error/context.rs
use std::fmt::Display;

/// 为 Result 增加错误上下文信息的 trait
///
/// 适用场景：底层错误需要附加业务上下文时使用。
/// 示例：AssetLoadFailed → with_context("加载技能配置")
///
/// 🟥 禁止用于包装领域错误（领域错误应自带完整上下文）
pub trait ErrorContext<T, E> {
    fn with_context(self, f: impl FnOnce() -> String) -> Result<T, E>;
}

// 为 Result<T, InfrastructureError> 实现
impl<T> ErrorContext<T, crate::shared::error::InfrastructureError> for Result<T, crate::shared::error::InfrastructureError> {
    fn with_context(self, f: impl FnOnce() -> String) -> Result<T, crate::shared::error::InfrastructureError> {
        self.map_err(|e| {
            let context = f();
            // InfrastructureError 本身携带上下文，通过 Display 拼接
            // 实际上 InfrastructureError 的变体已经包含 String 字段
            // 这里仅做日志辅助
            tracing::debug!(error = %e, context = %context, "错误附加上下文");
            e
        })
    }
}
```

```rust
// src/shared/error/extensions.rs
use tracing::debug;

/// 为 Result 添加便捷的日志记录能力
///
/// 适用场景：
/// - 非关键路径的错误只需记录，不需要冒泡处理
/// - 调试阶段需要快速知道哪些路径出了错
///
/// 🟥 禁止用于核心业务路径（核心路径错误必须显式处理）
/// 🟥 仅输出 DEBUG 级别，不影响 INFO 级别日志
pub trait LogIfError<T, E: std::fmt::Display> {
    /// 如果 Result 是 Err，输出 DEBUG 日志并返回 None
    /// 如果 Result 是 Ok，返回 Some(value)
    fn log_if_error(self, context: &str) -> Option<T>;
}

impl<T, E: std::fmt::Display> LogIfError<T, E> for Result<T, E> {
    fn log_if_error(self, context: &str) -> Option<T> {
        match self {
            Ok(val) => Some(val),
            Err(e) => {
                debug!(
                    error = %e,
                    context = %context,
                    "操作失败（已忽略）"
                );
                None
            }
        }
    }
}
```

**关键约束：**
- 🟥 shared/error 禁止定义任何领域错误变体
- 🟥 shared/error 禁止引用任何领域类型（UnitId、SkillId 等）
- 🟥 `LogIfError` 只输出 DEBUG 级别
- 🟩 `ErrorContext` 保持 trait 形式，不绑定具体 Error 类型（仅当需要 InfrastructureError 上下文增强时提供默认实现）

### Phase 2：领域错误拆分（Core 层）

目标：将 `src/core/error/` 中的错误按领域拆分到各 `domain/` 目录。

**当前状态确认：**

先检查 `src/core/error/` 实际有哪些文件：

```
src/core/error/
├── mod.rs
├── battle_error.rs     →  BattleError
├── buff_error.rs       →  BuffError
├── skill_error.rs      →  SkillError
├── inventory_error.rs  →  InventoryError
└── game_result.rs      →  GameResult（已迁移到 shared/error，可能残留）
```

**目标结构：**

```
src/core/battle/domain/battle_error.rs   → BattleError + BattleResult<T>
src/core/buff/domain/buff_error.rs       → BuffError + BuffResult<T>
src/core/skill/domain/skill_error.rs     → SkillError + SkillResult<T>
src/core/inventory/domain/inventory_error.rs → InventoryError + InventoryResult<T>

src/core/error/ → 删除
```

**领域错误规范：**

```rust
// 示例：src/core/battle/domain/battle_error.rs
use thiserror::Error;

/// 战斗领域错误
///
/// 错误码格式：B + 三位序号
/// B001-B009: 通用错误
/// B010-B019: 单位相关
/// B020-B029: 伤害相关
#[derive(Error, Debug)]
pub enum BattleError {
    /// 单位未在战斗中
    #[error("B001: 单位 {unit_id} 未在战斗中")]
    UnitNotInBattle { unit_id: crate::shared::id::UnitId },

    /// 攻击目标无效
    #[error("B010: 攻击目标无效: {target}")]
    InvalidTarget { target: crate::shared::id::UnitId },

    /// 伤害计算时缺少必要属性
    #[error("B020: 单位 {unit_id} 缺少攻击属性")]
    MissingAttackAttribute { unit_id: crate::shared::id::UnitId },
}

/// 战斗领域结果类型
pub type BattleResult<T> = Result<T, BattleError>;
```

**每条规则：**
- 🟩 错误码前缀：Battle=B, Skill=S, Buff=BF, Inventory=INV
- 🟩 每个错误变体必须携带完整上下文（禁止无参数变体）
- 🟩 每个领域定义自己的 `XxxResult<T>` 类型别名
- 🟩 RuleFailure 使用专门的结果枚举（不是 `Result::Err`）
- 🟥 核心业务代码中禁止 `unwrap()`、`expect()`、`panic!()`

```rust
// RuleFailure 示例（非错误，是正常业务不满足）
// 放在同目录下，与 Error 分开
pub enum BattleRuleFailure {
    OutOfRange,          // 超出攻击范围
    InsufficientMp,      // 法力不足
    AlreadyActed,        // 已行动过
    TargetImmune,        // 目标免疫
}
```

**迁移策略——逐个领域迁移：**

| 步骤 | 操作 | 验证 |
|------|------|------|
| 1 | 在目标目录创建 `domain/battle_error.rs` | - |
| 2 | 将旧错误定义复制并转换为新格式（加错误码、补上下文） | - |
| 3 | 逐个文件更新 `use` 引用为新的路径 | `cargo build` |
| 4 | 运行对应领域测试 | `cargo test` |
| 5 | 重复步骤 1-4 直到所有领域完成 | `cargo build && cargo test` 零错误 |

**每个领域迁移后立即编译验证，不强求一次完成。**

**量级估算：**
- 每个领域 1 个新文件（含错误枚举 + 结果别名 + 测试）
- 约 5-7 个领域需要迁移
- 每个错误变体 3-8 个，每个领域 4-10 个变体
- 整体约 10-20 个文件需要更新 import

### Phase 3：GameErrorEvent 事件通道（App 层）

目标：定义统一错误上报通道，解决"System 层错误被吞掉"的常见问题。

**新增文件：**

```rust
// src/app/error_event.rs
use bevy::prelude::*;

/// 游戏错误事件——跨层统一错误上报通道
///
/// 任一 System 遇到不可恢复的异常时发送此事件：
/// 1. ErrorMonitor 消费 → 输出 ERROR 级别日志
/// 2. UI 层消费 → 显示 Toast 通知
///
/// 🟥 禁止携带领域具体类型（使用字符串/Display）
/// 🟩 与宪法要求的"分领域错误"不冲突——领域内用 XxxError，跨层用此事件
#[derive(Event, Debug, Clone)]
pub struct GameErrorEvent {
    /// 错误类型分类
    pub error_type: ErrorType,
    /// 人类可读的错误描述
    pub message: String,
    /// 源模块名（如 "battle"、"buff"、"infrastructure"）
    pub source: &'static str,
    /// 结构化上下文键值对
    pub context: Vec<(&'static str, String)>,
}

/// 错误类型分类（对应宪法 §13.9.2 失败分类学）
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum ErrorType {
    /// 规则失败：业务正常不满足，非程序错误
    /// 例：法力不足、超出范围
    RuleFailure,
    /// 领域错误：领域内预期内的异常
    /// 例：技能配置不存在、BuffID 无效
    DomainError,
    /// 基础设施错误：底层能力异常
    /// 例：资源加载失败、存档 IO 错误
    Infrastructure,
    /// 程序 Bug：非法状态、逻辑断言失败
    /// 例：状态机非法跳转、数据一致性破坏
    Bug,
}

impl GameErrorEvent {
    pub fn new(error_type: ErrorType, message: impl Into<String>, source: &'static str) -> Self {
        Self {
            error_type,
            message: message.into(),
            source,
            context: Vec::new(),
        }
    }

    pub fn with_context(mut self, key: &'static str, value: impl Into<String>) -> Self {
        self.context.push((key, value.into()));
        self
    }
}
```

```rust
// src/app/error_monitor.rs
use bevy::prelude::*;
use tracing::error;

use super::error_event::{ErrorType, GameErrorEvent};

/// 错误监控 System——监听 GameErrorEvent 并处理
pub fn error_monitor(mut reader: EventReader<GameErrorEvent>) {
    for event in reader.read() {
        // 统一输出 ERROR 级别日志
        let type_str = match event.error_type {
            ErrorType::RuleFailure => "RULE",
            ErrorType::DomainError => "DOMAIN",
            ErrorType::Infrastructure => "INFRA",
            ErrorType::Bug => "BUG",
        };

        // 构建结构化上下文日志
        if event.context.is_empty() {
            error!(
                target: event.source,
                error_type = type_str,
                "{}", event.message,
            );
        } else {
            let ctx: Vec<(&str, &str)> = event.context.iter()
                .map(|(k, v)| (*k, v.as_str()))
                .collect();
            error!(
                target: event.source,
                error_type = type_str,
                ?ctx,
                "{}", event.message,
            );
        }
    }
}
```

**错误流完整视图：**

```
System 内部业务调用返回 Err
    │
    ├── 预期内的 RuleFailure（如法力不足）
    │   └── 直接处理，不进入错误通道
    │
    ├── 预期外的 DomainError（如技能配置缺失）
    │   └── 发送 GameErrorEvent { type: DomainError }
    │
    ├── InfrastructureError（如资源加载失败）
    │   └── 发送 GameErrorEvent { type: Infrastructure }
    │
    └── Bug（如状态机非法状态）
        └── 发送 GameErrorEvent { type: Bug }
            │
            ▼
        ErrorMonitor System
            ├── tracing::error! → 日志
            └── EventReader<GameErrorEvent> → UI
```

**约束：**
- 🟥 GameErrorEvent 禁止携带领域具体类型（使用字符串/Display）
- 🟩 UI 层通过 `EventReader<GameErrorEvent>` 消费并展示 Toast
- 🟩 只对 DomainError/Bug 类型展示 Toast；RuleFailure 和 Infrastructure 不打扰玩家
- 🟩 基础设施层同样使用此通道上报错误

## Communication Design

```
Core 层（领域逻辑）
    │
    ├── 返回 Result<T, XxxError>     → 同一领域内调用方处理
    ├── 返回 RuleFailure 枚举         → 调用方直接判断
    └── 遇到不可恢复错误 → 发送 GameErrorEvent
                              │
App 层（编排）                   │
    │                           │
    ├── ErrorMonitor System      ◄─┘
    │   ├── tracing::error!
    │   └── → UI (EventReader)
    │
    └── GameErrorEvent (Event)   — 跨层通道

Shared 层（工具）
    ├── ErrorContext trait        — 任意结果增强
    ├── LogIfError trait          — 错误 → DEBUG 日志
    └── InfrastructureError       — 基础设施通用
       + GameResult<T>
```

### 消息流明细

| 消息 | 发送方 | 接收方 | 用途 |
|------|--------|--------|------|
| `Result<T, XxxError>` | Core System | 同领域 System | 领域内错误传播 |
| `RuleFailure` 枚举 | Core System | 同领域 System | 正常业务不满足 |
| `GameErrorEvent` | 任意 System | ErrorMonitor | ERROR 日志 + UI Toast |
| `LogIfError` 调用 | 任意模块 | tracing 宏 | DEBUG 级别上下文日志 |

## 边界定义

### 层间依赖

```
Shared/error/ — 零外部依赖
  ├── context.rs     — 仅 std
  ├── extensions.rs  — 仅 tracing (debug!)
  └── result.rs      — 仅 thiserror

Core/XxxError → 仅引用 shared/id（强类型 ID）
  └── 定义在领域自身的 domain/ 模块内

App/error_event.rs → 仅 bevy::prelude
App/error_monitor.rs → tracing + GameErrorEvent

禁止的依赖路径：
  🟥 Core → Infrastructure（领域错误不依赖基础设施）
  🟥 Error → UI（GameErrorEvent 用字符串解耦）
```

### 允许

- 🟩 同一领域内函数直接返回 `Result<T, XxxError>`
- 🟩 跨层错误上报通过 `GameErrorEvent` 事件通道
- 🟩 shared/error 工具被所有层使用
- 🟩 基础设施错误上报时附带操作上下文

### 禁止

- 🟥 禁止创建全局 `AppError` 大枚举（宪法 13.9.1）
- 🟥 禁止使用 `anyhow::Error` / `Box<dyn Error>` 作为业务层返回类型（宪法 13.9.1）
- 🟥 禁止将领域错误放在 `shared/` 或 `infrastructure/` 中（宪法 13.9.1）
- 🟥 禁止混淆 RuleFailure 与 DomainError（将规则不满足作为 Err 返回）（宪法 13.9.2）
- 🟥 禁止核心业务代码中使用 `unwrap()`、`expect()`、`panic!()`（宪法 13.9.4）
- 🟥 禁止无上下文的错误变体（如 `InvalidTarget` 不带参数）（宪法 13.9.3）
- 🟥 禁止 `GameErrorEvent` 携带领域具体类型
- 🟥 禁止 `LogIfError` 输出 INFO 及以上级别日志

## Forbidden（禁止事项）

- 🟥 禁止将 Logging 和 Error 合并为同一个模块
- 🟥 禁止在 shared/error 中定义任何领域错误变体
- 🟥 禁止在 Phase 2 中一次性迁移所有错误文件（逐个领域迁移，保持编译通过）
- 🟥 禁止在游戏逻辑路径中使用 `LogIfError` 替代显式错误处理

## Definition / Instance Design

不适用（本 ADR 不涉及 Definition/Instance 分离）。

## 后果

### 正面

- ✅ 错误体系清晰：领域错误在领域内，基础设施错误在基础设施内，共享工具在 shared 内
- ✅ GameErrorEvent 统一了"System 层错误上报"这个长期痛点
- ✅ 类型安全：领域使用专用错误枚举，编译期保证边界
- ✅ 渐进迁移：逐个领域拆分 compile-check，风险可控
- ✅ LogIfError 降低了非关键路径的错误处理噪音

### 负面

- ⚠️ Phase 2 影响 10-20 个文件需要更新 import
- ⚠️ 现有 `src/core/error/game_result.rs` 可能有残留代码需要清理
- ⚠️ Phase 1 新增 `LogIfError` 后，开发者可能过度使用它替代正确错误处理
- ⚠️ GameErrorEvent 跨越了 Core/Infra/UI 三层，需要维护一个公共事件类型

### 风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| Phase 2 迁移导致编译失败 | 中 | 高 | 逐个领域迁移，每个迁移后立即 build |
| LogIfError 被滥用 | 中 | 中 | Forbidden 已明确禁止用于核心路径；review 时重点检查 |
| GameErrorEvent 字段膨胀 | 低 | 中 | 用 Vec<(&str, String)> 动态上下文，不新增字段 |
| 移除 src/core/error/ 后遗漏引用 | 低 | 高 | 删目录前 grep 确认所有引用已迁移 |

## 迁移顺序与时间估算

| Phase | 内容 | 影响范围 | 估时 | 验证标准 |
|-------|------|---------|------|---------|
| 1 | shared/error 补全 ErrorContext + LogIfError | 2 个新文件 | 0.5-1h | `cargo build`，新 trait 可被 use |
| 2 | 领域错误逐个迁移到 domain/ | 5-7 新文件 + 10-20 文件改 import | 2-4h | `cargo build && cargo test` 零错误 |
| 2.5 | 清理旧 src/core/error/ | 删除目录 | 0.2h | `cargo build` 无 dead_code |

**总计**：约 3-6 小时。

## 替代方案

### 方案 A：统一为单一 "Observability" 模块（已拒绝）

**思路**：将 Error 和 Logging 合并为单一可观测性模块。

**拒绝理由**：
1. 违反七层架构——错误属于 Core 层（业务规则），日志属于 Infrastructure 层（技术实现）
2. 领域错误需要放在领域内部（宪法 13.9.1），无法被"可观测性"模块收纳
3. 项目现有架构文档已经严格分离两者

### 方案 B：使用 `miette` 替代 `thiserror`（已拒绝）

**思路**：全项目统一使用 `miette` 做错误。

**拒绝理由**（确认 error-architecture.md §3 决策）：
- `thiserror` 零依赖、轻量、适合库代码
- `miette` 诊断信息丰富但依赖重，仅用于 CLI 工具和内容校验
- 游戏引擎内用 `thiserror`，内容校验工具用 `miette`

### 方案 C：不拆分 src/core/error/，保持大枚举（已拒绝）

**思路**：保持当前单一的 `src/core/error/` 结构，不断增变体。

**拒绝理由**：
- 违反宪法 13.9.1 "分领域错误原则"
- 大枚举随项目膨胀，AI 和人类都难维护
- 领域自治时需要编辑同一个文件
