# ADR-031: 核心属性与标签系统重构

## 状态
Accepted（2026-06-15）

## 背景

当前 Attribute 和 Tag 系统与 Linglan 目标模型存在根本性不匹配：

### Attribute 差距

当前系统定义 24 个 `AttributeKind` 变体（8 核心 + 3 生命/魔法/耐力 + 13 派生），源自早期 DND 风格设计。铃兰模型要求 5 核心 + 6 次级属性的简单模型：

| 对比项 | 当前系统 | Linglan 目标 |
|--------|---------|-------------|
| 核心属性 | Might/Agility/Vitality/Intelligence/Luck/Resistance/Dexterity/Willpower（8 个） | phys_atk/magic_atk/phys_def/magic_def/max_hp（5 个） |
| 次级属性 | Initiative/MoveRange/AttackRange/DodgeChance/HitChance/etc（13 个派生） | crit_rate/crit_dmg/move_range/atk_range/hit_rate/dodge_rate（6 个） |
| HP/MP/Stamina | 3 个独立字段 | 仅 max_hp（MP/Stamina 由 Effect 管理） |
| 取整规则 | 混合 i32/f32 | 纯整数（向下取整，最小为 1） |
| 属性上限/下限 | 无统一管理 | 每个属性有明确的 min/max 边界 |
| 数值边界 | 分散在代码中 | `content/executions/executions.ron` 统一配置 |

### Tag 差距

当前系统使用 10 个 `TagCategory`（elements/status/weapons/classes/movement/item_types/skill_types/buff_types/equipment_attributes/elementalTypes），铃兰要求 4 类：

| 对比项 | 当前系统 | Linglan 目标 |
|--------|---------|-------------|
| 分类 | 10 个 TagCategory | 4 类（damage_type/status/faction/mechanism） |
| 互斥规则 | 无显式互斥表 | 互斥 Tag 对在 RON 中配置 |
| 控制层级 | 无层级定义 | 3 级控制（soft/hard/full） |
| 免疫规则 | 无免疫模型 | 3 种免疫精度（免疫控制/免疫行动限制/免疫眩晕） |
| 控制递减 | 无 | 连续同类控制 → 持续时间衰减 |

## 决策

### 1. Attribute 模型完全重写

#### 1.1 替换 `AttributeKind` 枚举

删除现有的 24 变体 `AttributeKind`，替换为 Linglan 双分类模型：

```rust
/// 核心属性（定义态，运行时不可变）
#[derive(Debug, Clone, Serialize, Deserialize, Hash, Eq, PartialEq)]
pub enum CoreAttribute {
    PhysAtk,    // 物理攻击
    MagicAtk,   // 魔法攻击
    PhysDef,    // 物理防御
    MagicDef,   // 魔法防御
    MaxHp,      // 最大生命值
}

/// 次级属性（运行时由 Modifier 计算）
#[derive(Debug, Clone, Serialize, Deserialize, Hash, Eq, PartialEq)]
pub enum SecondaryAttribute {
    CritRate,   // 暴击率 [0, 0.95]
    CritDmg,    // 暴击伤害 [1.5, ~)
    MoveRange,  // 移动范围 [1, ~)
    AtkRange,   // 攻击范围 [1, ~)
    HitRate,    // 命中率 [0, 1.0]
    DodgeRate,  // 闪避率 [0, 0.8]
}

/// 统一属性标识（将 AttributeKind 替换为此类型）
#[derive(Debug, Clone, Serialize, Deserialize, Hash, Eq, PartialEq)]
pub enum AttributeId {
    Core(CoreAttribute),
    Secondary(SecondaryAttribute),
}
```

#### 1.3 Attributes Component 重写

```rust
#[derive(Component)]
pub struct Attributes {
    /// 核心属性基础值（战斗外固定）
    pub core: EnumMap<CoreAttribute, i32>,
    /// 次级属性当前值（由 Modifier 管线实时计算）
    pub secondary: EnumMap<SecondaryAttribute, i32>,
    /// 当前生命值
    pub current_hp: i32,
    /// Modifier 实例列表
    pub modifiers: Vec<ModifierInstance>,
}
```

#### 1.4 Derived 属性变迁

| 当前派生属性 | 迁移方式 |
|-------------|---------|
| MaxHp = 5 + Vit * 5 | 变为核心属性 max_hp，直接从配置读取 |
| MaxMp = Int * 5 | **删除** — MP 由 Effect 管理（Linglan 无 MP 模型，改用 CP 能量） |
| Attack = Might * 2 | 变为核心属性 phys_atk |
| MagicAttack = Int * 2 | 变为核心属性 magic_atk |
| Defense = Vit | 变为核心属性 phys_def |
| MagicDefense = Will | 变为核心属性 magic_def |
| Accuracy = 80 + Dex * 2 | 变为次级属性 hit_rate |
| Evasion = Agi * 3 | 变为次级属性 dodge_rate |
| CritRate = 5 + Luck | 变为次级属性 crit_rate |
| Initiative = Agi * 2 + Luck | **删除** — Linglan 使用速度排序（由 Execution 层管理） |
| MoveRange = floor(Agi * 0.5) + 2 | 变为次级属性 move_range |
| AttackRange = base | 变为次级属性 atk_range |

**所有派生公式删除** — 属性值直接从 RON 配置读取 + Modifier 管线计算，不再有硬编码派生公式。

### 2. Tag 系统完全重写

#### 2.1 替换 TagCategory

> **领域验证确认 (2026-06-15)**：与 `docs/02-domain/tag/tag-rules.md` 的 10 类 → 5 类折中方案。保留 Equipment 分类以支持装备标签查询（AI 寻路和装备系统依赖），其余 4 类对齐铃兰模型。

```rust
/// 5 分类标签（领域验证折中方案）
#[derive(Debug, Clone, Serialize, Deserialize, Hash, Eq, PartialEq)]
pub enum TagCategory {
    Elemental,   // fire, ice, lightning, dmg_physical, dmg_magical, etc.
    Status,      // buff, debuff, special_state, control_soft, control_hard, control_full
    Class,       // warrior, mage, archer, ally, enemy, boss, summon
    Equipment,   // weapon_type, armor_type, item_type, equipment_attr
    Mechanism,   // flying, grounded, dispellable, undispellable, reflectable
}
```

### 3. 属性转换（Attribute Conversion）

支持 Linglan 的中间层属性转换机制：

```ron
// content/attributes/conversions.ron
(
    conversions: [
        // 防御转攻击
        (source: "phys_def", target: "phys_atk", ratio: 0.5, condition: None),
        // 损失血量转攻击
        (source: "lost_hp", target: "phys_atk", ratio: 0.3, condition: Some("hp_below_50")),
    ],
)
```

转换规则：
- 转换后的属性值**可被后续百分比加成放大**
- 条件型转换**不参与基础属性面板计算**
- 在 Attribute Modifier 管线中作为单独阶段执行

## Module Design

```
src/core/
├── attribute/
│   ├── mod.rs              # Attributes Component + CoreAttribute/SecondaryAttribute 枚举
│   ├── def.rs              # AttributeDefinition + AttributeRegistry (HashMap<AttributeId, Def>)
│   ├── conversion.rs       # 属性转换机制（Linglan 中间层）
│   ├── snapshot.rs         # 战斗内属性快照
│   └── ops.rs              # Attribute 算术操作 + 边界限制
├── tag/
│   ├── mod.rs              # GameplayTag bitmask + GameplayTags Component + TagCategory
│   ├── def.rs              # TagDefinition + TagRegistry + mutual_exclusion 校验
│   ├── control.rs          # 控制层级 + 免疫检查 + 控制递减
│   └── rebuild.rs          # rebuild_tags() 基于 Trait/Equipment/Buff 来源重建
```

## Communication Design

```
AttributeRegistry 加载
  │   (PreStartup)
  ↓
Attributes Component 创建（spawn_unit 时从 UnitTemplate 读取）
  │
  ├──→ add_modifier() / remove_modifier()  (Modifier 管线入口)
  │       │
  │       ↓
  │   Attribute 重新计算（重新遍历 modifiers 计算最终值）
  │       │
  │       ↓
  │   [消息] AttributeModified (core/attribute 内部)
  │
  ├──→ get(attr_id)  → 实时计算值
  │
  └──→ 属性转换阶段（conversion.rs）→ 在 Modifier 管线之后执行
```

## 边界定义

| 规则 | 允许 | 禁止 |
|------|------|------|
| Attribute → Modifier | Modifier 修改 Attribute 值 | Attribute 自行修改值 |
| Tag → Attribute | Tag 决定条件修正是否生效 | Tag 直接修改属性值 |
| Attribute → Execution | 属性值作为计算输入 | Execution 写回 Attribute |
| Tag → Trigger | Tag 作为触发条件 | Tag 直接触发 Effect |

## Forbidden（禁止事项）

- 🟥 **禁止** 保留旧的 24 变体 `AttributeKind` 枚举 — 替换为 `AttributeId`（Core/Secondary 二分类）
- 🟥 **禁止** 硬编码任何派生属性公式 — 所有数值计算移至 Execution 层或从 RON 读取
- 🟥 **禁止** 保留旧的 10 类 `TagCategory` — 替换为 Linglan 4 分类
- 🟥 **禁止** Tag 互斥规则硬编码在 Rust 代码中 — 必须在 `content/tags/tags.ron` 中声明
- 🟥 **禁止** 缺乏数值边界的属性定义 — 每个属性必须有明确的 min/max
- 🟥 **禁止** 运行时无边界检查的属性修改 — `set_attr()` 必须 clamp 到 [min, max]
- 🟥 **禁止** `f32` 在属性存储和计算中使用 — 使用 `i32` + 百分比 i32 表示（百分比 = 万分比，如 50% = 5000）

## Definition / Instance Design

| 层 | Attribute | Tag |
|----|-----------|-----|
| Definition | `AttributeDefinition`（id, name_key, category, default, min, max） | `TagDefinition`（id, name_key, category, priority_weight, dispellable, reflectable, mutual_exclusions） |
| Instance | `Attributes` Component（core: EnumMap, secondary: EnumMap, current_hp, modifiers） | `GameplayTags` Component（bitmask: u64）|
| Runtime | `AttributeConversion`（source→target ratio） | `ControlImmunityResult`（免疫判定结果） |

## 后果

### 正面
- 属性模型从 DND 风 24 属性简化为 SRPG 风格 5+6 属性
- 所有数值边界统一管理，避免属性溢出/越界 Bug
- Tag 系统从 10 类混乱分类收敛为 4 类精确定义
- 控制层级和免疫规则有明确的优先级模型
- 所有属性公式和数据边界在 RON 中配置，无需改代码调整平衡性

### 负面
- 现有 UnitTemplate RON 文件中的旧属性字段需要全部迁移
- Character 模块的 `UnitTemplate` 属性映射逻辑需要重写
- Test fixture 中的属性构造需要全部更新
- 移除派生公式后，某些原本自动计算的属性值（如 Initiative）需要从 RON 显式配置

## 替代方案（已拒绝）

| 方案 | 拒绝原因 |
|------|----------|
| 保留现有 24 属性并在上层做映射 | 只是增加间接层，架构复杂度不变还多了一组转换代码 |
| 逐步替换 AttributeKind 变体 | 24->11 属性删掉一半，渐进式修改会在共存期引入条件分支 |
| 只在 RON 层改为 Linglan 格式，保留内部 AttributeKind | 不解决核心计算路径中 24 变体的 switch/match 复杂度 |
