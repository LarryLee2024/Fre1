# ADR-010: 测试迁移与重整方案

## 状态

Proposed

## 背景

项目已完成七层架构的源码迁移（`src/` 已按 App/Core/Shared/Infrastructure/Content/Modding 组织），但测试体系未同步迁移。

### 测试现状

| 层级 | 形式 | 文件数 | `#[test]` 数量 | 状态 |
|------|------|--------|---------------|------|
| Level 1: 领域单元测试 | 内联 `#[cfg(test)] mod` in `src/` | 78 个块 | **581** | ✅ 正常工作 |
| Level 2: 领域集成测试 | `tests/feature/` (11 文件) | 11 | 68 | ❌ 路径断裂 |
| (Legacy 领域集成) | `tests/legacy/` (7 文件) | 7 | 112 | ❌ 路径断裂 |
| Level 3: 系统集成测试 | `tests/system/` (1 文件) | 1 | ~32 | ❌ 路径断裂 |
| Level 4: E2E 回归测试 | `tests/scenario/` (1 文件) | 1 | ~5 | ❌ 路径断裂 |
| Level 5: 快照回归 | `tests/golden/` (1 文件) | 1 | 3 | ❌ 路径断裂 |
| 属性测试(跨层) | `tests/rule/` (proptest) | 1 | 4 proptest | ❌ 路径断裂 |

**结论**：
- 内联单元测试（Level 1）✅ 路径正确，编译运行正常（`cargo test --lib` 通过）
- 集成测试（Level 2-5）❌ 共 212 个 `#[test]` 因模块路径断裂而失败
- **当前 `cargo test` 无法通过**

### 问题清单

**问题 1：Import 路径断裂**

代码迁移到 `src/core/xxx/` 后，`tests/` 中的集成测试仍使用旧路径：

```rust
// ❌ 旧路径 — 编译错误
use tactical_rpg::buff::{...};
use tactical_rpg::battle::{...};
use tactical_rpg::character::{...};

// ✅ 新路径 — 符合七层架构
use tactical_rpg::core::buff::{...};
use tactical_rpg::core::battle::{...};
use tactical_rpg::core::character::{...};
```

**问题 2：Legacy 与 Feature 测试内容重叠**

| Legacy 文件 | 对应 Feature | 重叠度 | 处理 |
|-------------|-------------|--------|------|
| `legacy/buff_damage.rs` (556 行) | `feature/buff.rs` (505 行) | 高度 | 合并去重 |
| `legacy/buff_lifecycle.rs` (523 行) | `feature/buff.rs` | 高度 | 合并去重 |
| `legacy/combat_pipeline.rs` (651 行) | `feature/death.rs`, `traits.rs` | 中 | 按主题合并 |
| `legacy/skill_system.rs` (562 行) | `feature/skill.rs` (150 行) | 中 | 合并 |
| `legacy/turn_flow.rs` (464 行) | `feature/turn.rs` (159 行) | 中 | 合并 |
| `legacy/terrain_combat.rs` (498 行) | `feature/unified_movement.rs` | 中 | 合并 |
| `legacy/edge_cases.rs` (326 行) | 无对应 | 唯一 | 全部保留 |

**问题 3：测试入口命名与架构层不对齐**

| 当前入口 | 实际作用 | 架构期望命名 |
|---------|---------|-------------|
| `feature.rs` | 领域集成测试 | `integration.rs` |
| `system.rs` | 系统集成测试 | ✅ `system.rs`（匹配） |
| `scenario.rs` | E2E 回归测试 | `e2e.rs` |
| `golden.rs` | 快照回归 | ✅ `golden.rs`（匹配） |
| `rule.rs` | 属性测试 | 不要求改名 |

`testing-rules.md` 已预设 `cargo test --test integration`、`cargo test --test system` 等命令，当前命名只有 `system` 和 `golden` 对齐。

### 无冲突声明

本 ADR 与上述文档的核心一致：
- 五层测试金字塔定义 ✅ 对齐
- 测试命令命名（`--test integration`） ✅ 对齐（将 `feature.rs` → `integration.rs`）
- 领域单元测试内联模式 ✅ 保留（已在 src/ 中正常运行）
- 测试夹具（tests/common/） ✅ 保留
- Golden 快照 ✅ 保留不动

### 差异说明

`testing-rules.md` 描述"领域单元测试位于 `core/[领域模块]/tests/`"，当前实际是所有单元测试使用 `#[cfg(test)] mod` 内联在源文件中。两者功能等价，内联是 Rust 标准实践。本 ADR 不改变此模式。

### 决策 3：Legacy + Feature 合并策略

Legacy 7 文件全部合并到 integration/ 对应文件：

| Legacy 文件 | 合并目标 | 操作说明 |
|-------------|---------|---------|
| `buff_damage.rs` | `integration/buff.rs` | 数据驱动 Buff 定义测试追加到 buff.rs |
| `buff_lifecycle.rs` | `integration/buff.rs` | Buff 生命周期流程追加到 buff.rs |
| `combat_pipeline.rs` | `integration/death.rs` + `integration/traits.rs` | 按主题拆分合并 |
| `skill_system.rs` | `integration/skill.rs` | 冷却测试 + 释放前提测试合并 |
| `turn_flow.rs` | `integration/turn.rs` | 多阵营回合排序合并 |
| `terrain_combat.rs` | `integration/unified_movement.rs` | 地形消耗组合合并 |
| `edge_cases.rs` | `integration/edge_cases.rs` | 全部保留，独立文件 |

合并原则：
- **优先保留内容更全的版本**（按行数 + 场景数量判断）
- **不丢失独特场景**（特别是 edge_cases 的 326 行边界测试）
- **只去重，不删减** — 删掉完全相同的断言序列，保留差异部分

所有集成测试文件的 import 路径从：

### 决策 5：内联单元测试不动

Level 1 单元测试（src/ 中的 78 个 `#[cfg(test)]` 块）不做任何修改。它们：
- 使用 `use super::*` 或 `use crate::core::xxx` 路径，不存在路径断裂问题
- 编译通过，`cargo test --lib` 正常运行
- 数量（506 core + 75 其他 = 581）远超架构目标 500+

### 决策 6：Golden snapshots 保持不变

`tests/golden/snapshots/` 目录保留不动。金文件是 insta 快照测试的资产目录。

### 决策 7：Common 夹具保持 `#[path]` 模式

`tests/common/` 的 `#[path]` inclusion 模式短期保留。改为独立 crate 是另一项工作。

## Communication Design

不适用。测试是编译期资产。

## 边界定义

### 允许
- ✅ 测试文件通过 `#[path]` 引用子模块
- ✅ `tests/common/` 被所有入口通过 `use crate::common::xxx` 共享
- ✅ 内联 `#[cfg(test)]` 单元测试留在 `src/` 中
- ✅ Golden snapshots 留在 `tests/golden/snapshots/`
- ✅ Git `blame` 历史可通过入口文件重命名追溯（git 识别重命名）

### 禁止
- 🟥 禁止修改 `src/` 中的任何代码（测试路径修复只改 `tests/`）
- 🟥 禁止删除 Legacy 测试的唯一覆盖场景（edge_cases 等）
- 🟥 禁止删除测试以消除编译错误
- 🟥 禁止修改 Golden 快照文件
- 🟥 禁止此阶段将集成测试下沉为内联单元测试
- 🟥 禁止修改 Cargo.toml 的 `[[test]]` 节

## Forbidden（禁止事项）

- 🟥 **禁止修改业务代码** — Import 路径修正只改 `tests/`，不改 `src/`
- 🟥 **禁止删除 Legacy 测试** — 必须先确认场景已迁移到 integration/
- 🟥 **禁止改变测试断言逻辑** — 只改路径和目录结构，不改测试判断
- 🟥 **禁止重写 Golden 测试** — 金文件对比必须保持完全一致
- 🟥 **禁止删除 `tests/rule/`** — proptest 属性测试是独特的跨层资产

## 后果

### 正面
1. `cargo test` 恢复通过（当前断裂）
2. 测试命名与架构五层金字塔对齐 → `cargo test --test integration/system/e2e/golden`
3. Legacy + Feature 双重覆盖去重，减少 ~1,500 行重复代码
4. edge_cases.rs 唯一的边界测试被完整保留
5. 为 CI 门禁配置明确的命令：`cargo test --lib && cargo test --test integration && ...`

### 负面
1. 入口文件重命名导致 git blame 历史在不同工具中可能断裂
2. Legacy→Feature 合并需要逐文件审查，耗时约 1-2 个 AI Session
3. `#[path]` 模式保留，common/ 的编译可见性交叉问题仍在

## 替代方案

### 方案 A：只修路径不重构
不合并 legacy，只修复 import 路径。
**被拒理由**：legacy/feature 双重覆盖导致维护负担翻倍，且入口命名与架构规范不一致。

### 方案 B：测试全部重写
删除所有 tests/ 内容，按架构重新编写。
**被拒理由**：212 个测试，8,100 行代码，重写工作量大且易丢失边界覆盖。

## 执行计划

### Phase 1：Import 路径修正

为 `tests/` 中 18 个文件批量替换 import 路径。

```
替换规则：
tactical_rpg::buff::       → tactical_rpg::core::buff::
tactical_rpg::battle::     → tactical_rpg::core::battle::
tactical_rpg::character::  → tactical_rpg::core::character::
tactical_rpg::map::        → tactical_rpg::core::map::
tactical_rpg::skill::      → tactical_rpg::core::skill::
tactical_rpg::turn::       → tactical_rpg::core::turn::
tactical_rpg::equipment::  → tactical_rpg::core::equipment::
tactical_rpg::inventory::  → tactical_rpg::core::inventory::
tactical_rpg::campaign::   → tactical_rpg::core::campaign::
```

覆盖：11 feature/*.rs + 7 legacy/*.rs + 4 common/*.rs

验证：`cargo test --test feature --test legacy_buff --test legacy_combat --test legacy_turn --test legacy_edge --test system --test scenario --test golden --test rule` 编译通过

### Phase 2：目录重组

1. `feature.rs` → `integration.rs`；`feature/` → `integration/`
2. `scenario.rs` → `e2e.rs`；`scenario/` → `e2e/`
3. 创建 `integration/edge_cases.rs`（空壳）
4. 删除 4 个 `legacy_*.rs` 入口文件
5. 删除 `legacy/` 目录（内容已在 Phase 3 合并后删除）

### Phase 3：Legacy 内容合并去重

逐对审查，将 Legacy 独特场景合并到 integration/ 对应文件：

| 优先级 | 对 | 操作 | 预计 |
|--------|-----|------|------|
| P0 | legacy/buff_damage.rs → integration/buff.rs | 合并 buff 数据驱动定义 | 1 session |
| P0 | legacy/buff_lifecycle.rs → integration/buff.rs | 合并 Buff 生命周期 | 同上 |
| P0 | legacy/combat_pipeline.rs → death.rs + traits.rs | 拆分战斗管线 | 1 session |
| P1 | legacy/skill_system.rs → integration/skill.rs | 合并冷却+前提 | 1 session |
| P1 | legacy/edge_cases.rs → integration/edge_cases.rs | 全量搬运 | 1 session |
| P2 | legacy/turn_flow.rs → integration/turn.rs | 合并回合排序 | 1 session |
| P2 | legacy/terrain_combat.rs → integration/unified_movement.rs | 合并地形组合 | 1 session |

### 执行顺序

```
Phase 1 (路径修正) ─────────────────────┐
                                        ├──→ 同时进行
Phase 2 (目录创建 + 入口重命名) ─────────┘
    ↓
Phase 3 (内容合并去重) ─ 可并行处理 P0/P1/P2 组
    ↓
cargo test 全量验证
```

| Phase | 操作 | 范围 | AI Session |
|-------|------|------|------------|
| 1 | Import 路径批量替换 | 18 文件 | 1 |
| 2 | 目录重组 + 入口重命名 | 4 入口 + 子目录 | 1 |
| 3 | Legacy 内容合并去重 | 7 对文件 | 2-3 |
| | **总计** | | **4-5** |

```bash

cargo test --test system                          # Level 3: 通过
cargo test --test e2e                             # Level 4: 通过
cargo test --test golden                          # Level 5: 通过

# 全量测试不退化

# Legacy 目录已清理
! test -d tests/legacy/
! test -f tests/legacy_buff.rs

# 测试数量不减少（去重后合并场景应持平或增加）
# 内联单元测试 581 + 集成测试 >= 212 = 总数 >= 793
```
