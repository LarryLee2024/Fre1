# ADR-008: 核心机制与工程质量迁移

## 状态: Proposed

**版本**: 1.0
**日期**: 2026-06-14
**范围**: 5 项核心机制与工程质量迁移方案——命令总线、确定性执行、战斗状态机、性能预算、存档格式
**依赖**: ADR-001（迁移计划）、ADR-003（分层契约与依赖迁移）、ADR-006（验证与测试迁移方案）

> 本文档补充 ADR-001~007 中未覆盖的核心机制与工程质量迁移方案

## 背景

### 问题

项目从扁平结构迁移到七层架构的过程中（ADR-001，8 个 Phase），以下 5 项核心机制在现有 ADR 中未被覆盖，但对项目质量和可维护性至关重要：

| # | 问题 | 严重度 | 影响范围 |
|---|------|--------|---------|
| 1 | **无命令总线**：UI→Core 和 AI→Core 使用直接消息模式，操作不可回滚/回放 | 高 | 全局通信架构 |
| 2 | **无确定性执行**：使用 Bevy 默认 rand，回放系统无法验证、Bug 无法稳定复现 | 高 | 战斗系统、回放、测试 |
| 3 | **无正式战斗 FSM**：AppState + TurnPhase 工作但无形式化状态机契约 | 中 | 战斗流程控制 |
| 4 | **无性能约束**：无帧预算、无模块级性能指标、无性能门禁 | 中 | 运行时质量 |
| 5 | **存档无版本管理**：core/snapshot.rs 处理序列化但无版本号、无迁移链 | 高 | 存档兼容性 |

### 决策目标

为每项机制定义**完整迁移方案**，包括：当前状态、目标状态、迁移路径、模块设计、通信设计、边界定义、禁止模式。确保这些机制与 ADR-001 的 8 个 Phase 对齐。

## 一、命令总线架构迁移

### 1.1 当前状态

- 无命令总线抽象
- UI→Core 使用 `UiCommand` Message 直接发送意图
- AI→Core 使用函数调用直接执行攻击逻辑
- 操作不可回滚、不可回放、不可批量原子执行
- 校验与执行混在同一逻辑路径中

### 1.2 目标状态

所有玩家和 AI 的操作统一抽象为 Command 对象。每个命令经过校验后执行，确保游戏状态的合法性。

**核心价值**：
- 输入与领域逻辑解耦：UI/AI 只生成命令，不直接操作状态
- 操作可回滚：命令队列支持撤销
- 操作可回放：命令序列可序列化、重放
- 校验与执行分离：校验阶段只读、执行阶段只写

### 1.3 迁移方案

#### Phase 2（Core 层重组）：定义 Command trait

**目标**：在 `core/command/` 下定义命令抽象，实现具体命令变体。

**模块设计**：
```
src/core/command/
├── mod.rs                      # 模块导出
├── command_trait.rs             # Command trait + Result 类型
├── command_context.rs           # CommandContext 结构
├── command_queue.rs             # CommandQueue 资源
├── validation_error.rs          # ValidationError 枚举
├── execution_error.rs           # ExecutionError 枚举
└── commands/
    ├── mod.rs
    ├── cast_skill.rs            # CastSkillCommand
    ├── move_command.rs          # MoveCommand
    ├── use_item.rs              # UseItemCommand
    └── end_turn.rs              # EndTurnCommand
```

**Command trait 核心接口**：
```rust
pub trait Command: Send + Sync + 'static {
    /// 校验命令是否可执行（只读，不修改状态）
    fn validate(&self, world: &World, context: &CommandContext) -> Result<(), ValidationError>;
    /// 执行命令（修改游戏状态）
    fn execute(&self, world: &mut World, context: &CommandContext) -> CommandResult;
    /// 命令描述（用于日志和调试）
    fn description(&self) -> String;
    /// 命令是否支持撤销
    fn is_undoable(&self) -> bool { false }
}
```

**前置消息映射**：
| 现有消息 | 迁移后的 Command | 来源 |
|----------|-----------------|------|
| `UiCommand::CastSkill` | `CastSkillCommand` | ui/events.rs |
| `UiCommand::Move` | `MoveCommand` | ui/events.rs |
| `UiCommand::UseItem` | `UseItemCommand` | ui/events.rs |
| `UiCommand::EndTurn` | `EndTurnCommand` | ui/events.rs |
| AI 直接攻击逻辑 | `CastSkillCommand` | ai/decision.rs |

#### Phase 6（App 层）：CommandHandler 接入

**目标**：将 CommandHandler 接入 App 层，使 UI 和 AI 的命令流经统一管线。

**通信流程**：
```
┌──────────┐     ┌──────────┐     ┌──────────────┐     ┌──────────────┐
│ UI / AI  │────→│Command   │────→│  Command     │────→│  Effect      │
│          │     │Handler   │     │  Queue       │     │  Pipeline    │
│ UiCommand│     │          │     │              │     │              │
│ AiIntent │     │ 生成      │     │ 校验+执行    │     │ generate→    │
│          │     │ Command  │     │              │     │ modify→      │
│          │     │ 对象      │     │              │     │ execute      │
└──────────┘     └──────────┘     └──────────────┘     └──────────────┘
                                       │                       │
                                       │ 领域事件               │ 状态变更
                                       ↓                       ↓
                                 ┌──────────┐          ┌──────────────┐
                                 │ 审计系统  │          │ UI ViewModel │
                                 │ AuditTrail│          │  刷新         │
                                 └──────────┘          └──────────────┘
```

**通信方式**：
| 通信路径 | 方式 | 说明 |
|----------|------|------|
| UI → CommandHandler | Message (UiCommand) | 跨层通信 |
| AI → CommandHandler | 函数调用 | AI 内部直接调用 |
| CommandHandler → CommandQueue | 函数调用 | 同模块内 |
| CommandQueue → Effect Pipeline | 函数调用 + Message | 执行命令触发效果 |
| Effect Pipeline → 审计系统 | Message | 领域事件广播 |
| Effect Pipeline → UI | Message (ViewModel) | 状态变更通知 |

### 1.4 边界定义

#### 模块归属

| 模块 | 层级 | 职责 |
|------|------|------|
| `core/command/` | Core | Command trait 定义、命令变体、CommandQueue |
| `ui/command_handler.rs` | UI | UiCommand → Command 对象转换 |
| `core/ai/ai_command_generator.rs` | AI → Command 转换 | AI 决策 → Command 对象 |
| `app/command_plugin.rs` | App | 注册 CommandHandler 系统 |

#### 层间依赖

```
UI → CommandHandler → CommandQueue → Effect Pipeline
AI → AiCommandGenerator → CommandQueue → Effect Pipeline
```

- UI 和 AI 不直接访问 CommandQueue
- CommandQueue 是 Core 层的 Resource
- Effect Pipeline 不感知命令来源（Player vs AI）

### 1.5 Forbidden（禁止事项）

| # | 禁止事项 | 理由 | 违反后果 |
|---|---------|------|---------|
| F1 | 🟥 跳过校验直接执行命令 | 游戏状态可能非法 | 数值崩坏、逻辑错误 |
| F2 | 🟥 校验阶段修改游戏状态 | 校验必须只读 | 非确定性行为 |
| F3 | 🟥 执行阶段重复校验 | 冗余检查、性能损失 | 校验逻辑分散 |
| F4 | 🟥 UI 直接修改 ECS 组件 | 绕过命令总线 | 操作不可回滚/回放 |
| F5 | 🟥 AI 独立执行攻击逻辑 | 绕过 Effect Pipeline | 伤害计算不一致 |
| F6 | 🟥 命令中包含业务规则 | 命令是操作抽象 | 规则与命令耦合 |
| F7 | 🟥 使用裸 Entity 而非 Strong ID | 不可序列化 | 回放系统无法使用 |

## 二、确定性执行迁移

### 2.1 当前状态

- 无确定性 RNG，使用 Bevy 的默认 `rand::thread_rng()`
- 战斗数值计算无整数约束，部分使用 `f32`
- ECS 查询依赖默认迭代顺序，无显式排序
- 无状态哈希机制，无法检测执行分歧

### 2.2 目标状态

给定相同的输入序列，游戏必须产出完全相同的状态序列。这是回放系统、Bug 复现、自动化测试、未来多人同步的基石。

**确定性保证链**：
```
单一种子 → 确定性 PRNG → 所有随机可复现
整数运算 → 精度可控 → 数值结果一致
显式排序 → 迭代顺序确定 → 处理结果一致
状态哈希 → 分歧可检测 → 问题可定位
```

### 2.3 迁移方案

#### Phase 1（Shared 层）：创建 DeterministicRng

**模块设计**：
```
src/shared/random/
├── mod.rs          # 模块导出
├── deterministic.rs # DeterministicRng 实现（xorshift64*）
└── seed.rs         # Seed 类型定义
```

**目标接口**：
```rust
/// 确定性随机数生成器
pub struct DeterministicRng {
    state: u64,
    seed: u64,
}

impl DeterministicRng {
    pub fn from_seed(seed: u64) -> Self;
    pub fn gen_range(&mut self, range: std::ops::Range<i32>) -> i32;
    pub fn gen_bool(&mut self, probability: f64) -> bool;
    pub fn seed(&self) -> u64;
}
```

#### Phase 2（Core 层）：BattleRng Resource

**模块设计**：
```
src/core/battle/rng.rs  # BattleRng Resource
```

**目标接口**：
```rust
/// 战斗随机数 Resource — 所有战斗随机从此获取
#[derive(Resource)]
pub struct BattleRng {
    inner: DeterministicRng,
}

impl BattleRng {
    pub fn from_seed(seed: u64) -> Self;
    pub fn gen_range(&mut self, range: std::ops::Range<i32>) -> i32;
    pub fn gen_bool(&mut self, probability: f64) -> bool;
}
```

**使用规则**：

| 规则 | 强制等级 | 说明 |
|------|---------|------|
| 所有战斗随机通过 `BattleRng` | 🟥 | 暴击判定、闪避判定、伤害浮动、Buff 触发等 |
| 种子在战斗开始时确定 | 🟥 | 从回放文件或关卡配置读取 |
| 禁止使用 `rand::thread_rng()` | 🟥 | 系统随机不可复现 |
| 禁止在 System 内创建独立 Rng 实例 | 🟥 | 破坏全局状态一致性 |

#### Phase 3：状态哈希计算

**模块设计**：
```
src/core/battle/state_hash.rs  # compute_state_hash()
```

**哈希计算时机**：
| 时机 | 强制等级 | 说明 |
|------|---------|------|
| 每回合结束（TurnEnd） | 🟥 | 回合级一致性校验 |
| 回放验证时 | 🟥 | 与参考哈希对比 |

**哈希内容**：
```rust
fn compute_state_hash(world: &World) -> u64 {
    let mut hasher = DeterministicHasher::new();
    // 1. 所有存活单位的关键属性（HP、MP、攻击力、防御力、位置）
    // 2. 活跃 Buff 列表（ID、剩余回合、层数）
    // 3. 当前回合号
    // 4. 回合阶段
    hasher.finish()
}
```

#### Phase 6：回放系统集成

回放系统通过确定性执行验证一致性：
```
录制时：每个 TurnEnd 计算 state_hash，存入 ReplayFrame
重放时：相同 TurnEnd 计算 state_hash，与 ReplayFrame 中的值对比
不一致时：记录分歧位置，输出差异详情
```

### 2.4 数值精度约束

核心战斗数值**禁止使用浮点数**：

| 类型 | 要求 | 替代方案 |
|------|------|---------|
| HP、MP、伤害值 | 🟥 必须为 `i32` 或 `u32` | — |
| 攻击力、防御力 | 🟥 必须为整数 | — |
| 百分比修饰（如 +15% 攻击） | 🟥 用整数表示（15 = 15%） | 不使用 `f32` |
| 公式计算 | 🟥 先乘后除，避免精度丢失 | `value * multiplier / 100` |

**允许使用浮点数的场景**：世界坐标计算、动画插值、UI 布局、音频音量（不影响战斗逻辑）。

### 2.5 迭代顺序约束

Bevy 的 ECS 查询默认**不保证迭代顺序**，在战斗逻辑中会导致不确定性。

**需要排序的场景**：

| 场景 | 强制等级 | 排序依据 |
|------|---------|---------|
| 行动队列构建 | 🟥 | Initiative 降序 + Entity ID 稳定排序 |
| 同时触发的 Buff 结算 | 🟥 | Buff 注册顺序（InsertionOrder） |
| 同回合多个死亡判定 | 🟥 | Entity ID 升序 |
| 属性 Modifier 栈计算 | 🟥 | ModifierSource 优先级 |
| AOE 伤害目标遍历 | 🟥 | Entity ID 升序 |

### 2.6 边界定义

#### 模块归属

| 模块 | 层级 | 职责 |
|------|------|------|
| `shared/random/` | Shared | DeterministicRng 基础实现 |
| `core/battle/rng.rs` | Core | BattleRng Resource |
| `core/battle/state_hash.rs` | Core | 状态哈希计算 |
| `infrastructure/replay/` | Infra | 回放录制/播放 |

#### 层间依赖

```
Shared (DeterministicRng) → 无外部依赖
Core (BattleRng) → Shared (DeterministicRng)
Infra (Replay) → Core (BattleRng, state_hash)
```

### 2.7 Forbidden（禁止事项）

| # | 禁止事项 | 理由 | 违反后果 |
|---|---------|------|---------|
| F1 | 🟥 使用 `rand::thread_rng()` | 系统随机不可复现 | 回放失败、测试不可靠 |
| F2 | 🟥 使用 `rand::rngs::SmallRng::from_entropy()` | 同上 | 同上 |
| F3 | 🟥 在 System 内创建独立 Rng | 破坏全局状态 | 随机序列不可控 |
| F4 | 🟥 核心战斗数值使用 `f32`/`f64` | 浮点精度不可控 | 伤害值不一致 |
| F5 | 🟥 依赖 Bevy 默认迭代顺序 | 顺序不确定 | 处理结果不一致 |
| F6 | 🟥 跳过状态哈希计算 | 分歧不可检测 | Bug 隐藏 |
| F7 | 🟥 使用 `println!`/`dbg!` 影响执行路径 | 输出是副作用 | 不确定性 |

## 三、战斗 FSM 迁移

### 3.1 当前状态

- `AppState`（MainMenu/LevelSelect/InGame）+ `TurnPhase`（14 个状态）工作但无形式化 FSM 契约
- 状态转换通过手动 `NextState` 调用，无转换表约束
- 无非法转换检测机制
- 无进入/退出钩子的标准化接口

### 3.2 目标状态

完整的战斗生命周期有限状态机：PreBattle → RoundStart → PlayerPhase → EnemyPhase → TurnEnd → VictoryCheck → RoundEnd → PostBattle

**两层 Bevy State 架构**：
```
AppState::InGame
  └── BattlePhase (SubState)
        ├── PreBattle
        ├── RoundStart
        ├── PlayerPhase     ← TurnPhase SubState 在此激活
        ├── EnemyPhase      ← TurnPhase SubState 在此激活
        ├── TurnEnd
        ├── VictoryCheck
        ├── RoundEnd
        └── PostBattle
```

### 3.3 迁移方案

#### Phase 2（Core 层）：形式化 BattlePhase

**模块设计**：
```
src/core/turn/battle_phase.rs   # BattlePhase State 定义
src/core/turn/transitions.rs    # 转换表 + 非法转换检测
```

**BattlePhase 枚举**：
```rust
#[derive(States, Debug, Clone, PartialEq, Eq, Hash)]
pub enum BattlePhase {
    PreBattle,
    RoundStart,
    PlayerPhase,
    EnemyPhase,
    TurnEnd,
    VictoryCheck,
    RoundEnd,
    PostBattle,
}
```

**合法转换表**：

| 从 | 到 | 条件 | 触发系统 |
|----|-----|------|---------|
| PreBattle | RoundStart | 关卡初始化完成 | `init_battle_system` |
| RoundStart | PlayerPhase | 回合开始完成 | `round_start_system` |
| PlayerPhase | EnemyPhase | 玩家方所有单位行动完毕或 ForceEndTurn | `phase_transition_system` |
| EnemyPhase | TurnEnd | 敌方所有单位行动完毕 | `phase_transition_system` |
| TurnEnd | VictoryCheck | 回合结算完成 | `turn_end_system` |
| VictoryCheck | RoundEnd | 未达到终态 | `victory_check_system` |
| VictoryCheck | PostBattle | 终态达成（Victory/Defeat） | `victory_check_system` |
| RoundEnd | RoundStart | 新回合开始 | `round_end_system` |

**非法转换（禁止）**：

| 从 | 到 | 禁止原因 |
|----|-----|---------|
| PreBattle | PlayerPhase | 跳过 RoundStart，行动队列未初始化 |
| PlayerPhase | TurnEnd | 跳过 EnemyPhase，敌方未行动 |
| PlayerPhase | VictoryCheck | 跳过回合结算 |
| PlayerPhase | PostBattle | 跳过所有后续阶段 |
| RoundEnd | PlayerPhase | 跳过 RoundStart，队列未重建 |
| TurnEnd | RoundStart | 跳过 VictoryCheck，胜负未判定 |
| PostBattle | 任意 | 终态不可逆 |

#### Phase 2-3：OnEnter/OnExit 钩子

每个 BattlePhase 状态定义 `OnEnter` 和 `OnExit` 系统：

**OnEnter 钩子**：

| 状态 | OnEnter 系统 | 职责 | 禁止 |
|------|-------------|------|------|
| PreBattle | `init_turn_order` | 初始化行动队列 | 检查胜负条件 |
| RoundStart | `send_turn_started` | 发送 TurnStarted 消息 | 修改单位状态 |
| PlayerPhase | `highlight_player_units` | 高亮可行动单位 | 执行攻击逻辑 |
| EnemyPhase | `start_ai_decision` | 启动 AI 决策 | 直接执行攻击 |
| TurnEnd | `turn_end_cleanup` | 回合结算 | 执行攻击逻辑 |
| VictoryCheck | `check_victory_conditions` | 五步胜负检查管线 | 修改战斗状态 |
| RoundEnd | `send_turn_ended` | 发送 TurnEnded 消息 | 执行业务逻辑 |
| PostBattle | `send_level_completed` | 发送 LevelCompleted 消息 | 继续战斗逻辑 |

**OnExit 钩子**：

| 状态 | OnExit 系统 | 职责 |
|------|-----------|------|
| PlayerPhase | `cleanup_player_highlights` | 清理高亮标记 |
| EnemyPhase | `cleanup_ai_state` | 清理 AI 临时状态 |
| PostBattle | `cleanup_battle_resources` | 清理战斗资源 |

#### Phase 6：扩展点注册

插件可以在以下位置注入行为：

| 扩展点 | 位置 | 允许行为 | 禁止行为 |
|--------|------|---------|---------|
| OnEnter(RoundStart) | 回合开始 | 发送消息、记录日志、触发 Trait 效果 | 修改单位状态 |
| OnExit(TurnEnd) | 回合结束 | Buff 过期检查、持续效果结算 | 执行攻击逻辑 |
| OnEnter(VictoryCheck) | 胜负判定前 | 准备判定数据 | 修改判定结果 |
| OnEnter(PostBattle) | 战斗结束 | 播放结算动画、保存结果 | 继续战斗逻辑 |

### 3.4 边界定义

#### 模块归属

| 模块 | 层级 | 职责 |
|------|------|------|
| `core/turn/battle_phase.rs` | Core | BattlePhase State 定义 |
| `core/turn/transitions.rs` | Core | 转换表 + 非法转换检测 |
| `core/turn/` 各阶段系统 | Core | OnEnter/OnExit 钩子实现 |
| `app/game_state.rs` | AppState | AppState::InGame 映射 |

#### 不变量

1. **不变量1**：PreBattle 必须先于所有战斗阶段
2. **不变量2**：RoundStart 必须在每个新回合执行
3. **不变量3**：TurnEnd 必须在 VictoryCheck 之前
4. **不变量4**：终态后不可逆
5. **不变量5**：PlayerPhase 和 EnemyPhase 互斥
6. **不变量6**：每个回合必须完整走完流程

### 3.5 Forbidden（禁止事项）

| # | 禁止事项 | 理由 | 违反后果 |
|---|---------|------|---------|
| F1 | 🟥 手动设置 BattlePhase 而不经过 NextState | 跳过 OnEnter/OnExit 生命周期 | 钩子未触发、状态不一致 |
| F2 | 🟥 在 OnEnter 中执行跨阶段跳转 | 破坏状态机的确定性 | 循环跳转、栈溢出 |
| F3 | 🟥 跳过 TurnEnd 直接开始新回合 | acted 未重置、队列未重建 | 单位无法行动 |
| F4 | 🟥 在 FSM 内部包含业务逻辑 | 违反状态机职责单一 | 逻辑耦合、难以维护 |
| F5 | 🟥 循环状态转换（A→B→A→B 无终止） | 无限循环 | 游戏卡死 |
| F6 | 🟥 在 PlayerPhase 内执行敌方逻辑 | 职责混淆 | AI 和玩家逻辑耦合 |

## 四、性能预算迁移

### 4.1 当前状态

- 无性能约束，无帧预算
- 无模块级性能指标
- 无性能门禁，无 CI 性能回归检测
- 主循环中可能存在堆内存分配
- ECS 查询可能无过滤器全量扫描

### 4.2 目标状态

60 FPS（16.67ms 单帧预算），各模块有明确的性能预算。

**帧时间构成**：
```
单帧 16.67ms
├── 输入处理          ≤1ms
├── ECS 调度/执行     ≤9ms（游戏逻辑）
├── 渲染              ≤5ms
├── 系统开销          ≤1.67ms
└── 余量              ~0ms
```

**模块预算**：

| 模块 | 预算 | 说明 |
|------|------|------|
| 战斗逻辑（Effect Pipeline） | <5ms | 伤害计算、Buff 结算 |
| AI 决策 | <2ms | 策略选择、目标评估 |
| 寻路计算 | <2ms | BFS + 地形消耗 |
| 回合管理 | <1ms | 状态机转换、队列操作 |
| 属性计算 | <1ms | Modifier Stack 求值 |
| UI 渲染 | <3ms | 面板更新、飘字动画 |
| 动画系统 | <2ms | 精灵动画播放 |
| 音频系统 | <1ms | 音效触发 |

### 4.3 迁移方案

#### Phase 4（Infrastructure 层）：诊断与分析模块

**模块设计**：
```
src/infrastructure/diagnostics/
├── mod.rs              # DiagnosticsPlugin
├── frame_monitor.rs    # 帧时间监控
└── system_profiler.rs  # System 执行时间分析

src/infrastructure/profiler/
├── mod.rs              # ProfilerPlugin
└── benchmark.rs        # 基准测试框架
```

**帧时间监控**：
```rust
fn frame_time_monitor(time: Res<Time>, diagnostics: Res<FrameTimeDiagnosticsPlugin>) {
    let avg = diagnostics.average_frame_time().unwrap_or_default();
    if avg > Duration::from_millis(16.67) {
        warn!(
            avg_frame_ms = avg.as_secs_f64() * 1000.0,
            "Frame time exceeds budget"
        );
    }
}
```

#### Phase 4-6：性能门禁集成

**CI 集成**：在 ADR-006 的 CI 流水线中添加 benchmark job：


- 帧时间回归超过 10% 时自动标记
- 关键 System 的执行时间监控

#### Phase 4-6：堆内存分配规则

🟥 **主更新循环中禁止堆内存分配**。

**规则**：
- 🟥 禁止在 `Update` Schedule 的 System 中使用 `Vec::new()`、`HashMap::new()`、`String::from()` 等堆分配操作
- 🟥 禁止在 System 中调用 `Box::new()`、`Rc::new()` 等堆分配
- 🟥 禁止在高频 System（每帧执行）中创建临时集合

**允许的替代方案**：

| 场景 | 禁止 | 允许 |
|------|------|------|
| 临时收集结果 | `Vec::new()` + push | 预分配的 `Local<Vec<T>>` |
| 字符串拼接 | `format!()` 创建新 String | 预分配的 `Local<String>` |
| 错误消息 | `format!()` 构造错误 | 使用 `&'static str` 错误描述 |
| 日志消息 | `format!()` 构造日志 | 结构化字段 + `tracing` 宏 |

#### Phase 4-6：ECS 查询优化规则

🟥 **禁止大型多组件查询**。

**规则**：
- 🟩 单个 System 的查询组件数不超过 6 个
- 🟩 优先使用 `Changed<T>` 和 `Added<T>` 过滤器
- 🟩 将大型查询拆分为多个小查询
- 🟥 禁止每帧查询所有 Entity（使用过滤器缩小范围）

#### 模块归属

| 模块 | 层级 | 职责 |

| `infrastructure/diagnostics/` | Infra | 帧时间监控、System 分析 |

```
  ↓

  ↓
```

| # | 禁止事项 | 理由 | 违反后果 |

## 五、存档格式迁移

### 5.1 当前状态

- `core/snapshot.rs` 处理序列化，无版本号
- 无迁移链，旧存档无法加载
- 无备份策略，迁移失败可能损坏存档
- 存档内容规则未定义（可能保存 Definition 数据）

### 5.2 目标状态

SemVer 版本号的存档格式，支持迁移链（V1→V2→V3），最近 3 个 MAJOR 版本兼容。

**存档格式**：
```rust
pub struct SaveData {
    pub version: SemVer,          // 存档版本号
    pub timestamp: u64,           // 保存时间戳
    pub instances: InstanceData,  // 运行时状态（只保存 Instance）
    pub metadata: SaveMetadata,   // 存档元信息
}
```

**版本管理规则**：
| 版本变更 | 含义 | 示例 |
|---------|------|------|
| MAJOR | 不兼容变更，需要迁移 | 1.0.0 → 2.0.0 |
| MINOR | 向后兼容的新增字段 | 1.0.0 → 1.1.0 |
| PATCH | Bug 修复，不影响格式 | 1.0.0 → 1.0.1 |

### 5.3 迁移方案

#### Phase 4（Infrastructure 层）：迁移框架

**模块设计**：
```
src/infrastructure/persistence/
├── mod.rs
├── plugin.rs              # PersistencePlugin
├── save.rs                # 存档保存
├── load.rs                # 存档加载
├── validate.rs            # 存档验证
└── migration/
    ├── mod.rs
    ├── registry.rs        # MigrationRegistry
    ├── v1_to_v2.rs        # V1 → V2 迁移函数
    └── v2_to_v3.rs        # V2 → V3 迁移函数
```

**迁移链**：
```rust
pub fn migrate(mut data: SaveData, target: SemVer) -> Result<SaveData, MigrationError> {
    while data.version < target {
        data = match (data.version.major, data.version.minor) {
            (1, 0) => migrate_v1_0_to_v1_1(data)?,
            (1, 1) => migrate_v1_1_to_v2_0(data)?,
            (2, 0) => migrate_v2_0_to_v2_1(data)?,
            _ => return Err(MigrationError::UnsupportedVersion(data.version)),
        };
    }
    Ok(data)
}
```

**迁移函数规范**：
```rust
pub fn migrate_v1_0_to_v1_1(mut data: SaveData) -> Result<SaveData, MigrationError> {
    // 1. 验证当前版本
    assert_eq!(data.version, SemVer::new(1, 0, 0));
    // 2. 执行数据转换
    //    - 新增字段：使用默认值填充
    //    - 删除字段：忽略
    //    - 字段重命名：映射到新名称
    // 3. 更新版本号
    data.version = SemVer::new(1, 1, 0);
    // 4. 验证迁移后数据合法性
    validate_save_data(&data)?;
    Ok(data)
}
```

#### Phase 4：备份策略

**迁移失败处理**：
```rust
fn migrate_with_rollback(
    data: SaveData,
    target: SemVer,
) -> Result<SaveData, MigrationError> {
    // 1. 备份原始数据
    let backup = data.clone();
    // 2. 尝试迁移
    match migrate(data, target) {
        Ok(migrated) => Ok(migrated),
        Err(e) => {
            // 3. 迁移失败，回滚到原始数据
            error!(
                from_version = %backup.version,
                to_version = %target,
                error = %e,
                "Migration failed, rolling back"
            );
            // 4. 保持原始存档文件不变
            Err(e)
        }
    }
}
```

**备份规则**：
- 🟥 迁移失败时保持原始存档文件不变
- 🟥 不修改原文件内容
- 🟥 不删除原文件
- 🟩 迁移前创建临时备份文件
- 🟩 迁移成功后删除备份
- 🟩 迁移失败后保留备份供用户恢复

#### Phase 5：存档内容规则

🟥 **存档只保存 Instance 数据，禁止保存 Definition**。

**必须保存**：
- 单位属性（Attributes）
- Buff 列表（ActiveBuffs）
- 装备槽位（EquipmentSlots）
- 技能冷却（SkillCooldowns）
- 背包物品（Inventory）
- 回合状态（TurnState）
- 地图状态（地图 ID、单位位置）

**禁止保存**：
- 技能定义（SkillDef）
- Buff 定义（BuffDef）
- 装备定义（EquipmentDef）
- 物品定义（ItemDef）
- 角色模板（UnitTemplate）

**Definition 恢复**：加载存档时，通过 ID 从 Registry 恢复 Definition。

### 5.4 最低支持版本

🟥 **支持最近 3 个 MAJOR 版本的存档**。

| 当前版本 | 支持的最旧版本 |
|---------|--------------|
| 3.0.0 | 1.0.0 |
| 4.0.0 | 2.0.0 |
| 5.0.0 | 3.0.0 |

**不支持的版本处理**：
- 🟥 拒绝加载
- 🟥 记录 ERROR 日志
- 🟥 提示用户更新游戏或重新开始
- 🟥 不尝试加载（避免数据损坏）

### 5.5 存档验证

存档加载后必须执行验证：
- 🟥 版本号在支持范围内
- 🟥 时间戳合理（不早于游戏发布日期）
- 🟥 所有 ID 引用的 Definition 存在于 Registry
- 🟥 属性值在合法范围内（HP ≥ 0，MP ≥ 0）
- 🟥 Buff 数量不超限
- 🟥 单位位置在地图范围内

### 5.6 边界定义

#### 模块归属

| 模块 | 层级 | 职责 |
|------|------|------|
| `infrastructure/persistence/` | Infra | 存档保存/加载/迁移 |
| `infrastructure/persistence/migration/` | Infra | 迁移函数注册表 |
| `core/save_domain/` | Core | 存档领域模型（SaveData 结构定义） |
| `core/snapshot.rs` | Core | 场景快照（保留） |

#### 层间依赖

```

```

### 5.7 Forbidden（禁止事项）

| # | 禁止事项 | 理由 | 违反后果 |
|---|---------|------|---------|
| F1 | 🟥 破坏存档格式而不提供迁移路径 | 旧存档无法加载 | 用户数据丢失 |
| F2 | 🟥 删除旧版本的迁移函数 | 迁移链断裂 | 旧存档无法迁移 |
| F3 | 🟥 跳过迁移直接加载旧存档 | 数据格式不兼容 | 程序崩溃 |

| F5 | 🟥 存档保存 Definition 数据 | Definition 不可变 | 配置污染 |
| F6 | 🟥 存档格式无版本号 | 无法检测版本 | 无法迁移 |
| F7 | 🟥 迁移函数不验证输入输出合法性 | 数据损坏不可检测 | 静默错误 |
| F8 | 🟥 迁移函数依赖外部状态 | 不可测试 | 迁移不可靠 |
| F9 | 🟥 迁移在战斗中执行 | 性能问题 | 游戏卡顿 |

| 机制 | 预期收益 |
