# ADR-003: 分层契约与依赖迁移

## 状态: Proposed

**版本**: 1.0
**日期**: 2026-06-14
**范围**: 7 层 + 2 跨层的完整迁移契约定义
**依赖**: ADR-001（迁移计划）

---

## 背景

### 问题

当前 `src_待迁移` 目录采用扁平 Feature-First 结构，157 个 Rust 文件（33,771 行）散布在 14 个顶层模块中。虽然每个模块内部职责清晰，但**层间边界未定义**：

1. `core/` 同时承担 Shared + Content + Infra 职责（超载）
2. ID 类型错误地放在 `core/id/`（应在 `shared/ids/`）
3. 错误类型集中放在 `core/error/`（应回归各领域 `domain/`）
4. `assets.rs` 放在根目录（应在 `infrastructure/assets/`）
5. `input.rs` 放在根目录（应在 `ui/input.rs`）
6. 无 App 层：`main.rs` 直接注册所有 Plugin
7. 无 Content 层：38 个 RON 配置散落在 `assets/` 中

### 决策目标

为每层定义**完整的迁移契约**，包括：职责、判断标准、模块清单、依赖规则、通信协议、禁止模式。确保任何 AI Agent 或开发者在迁移时都有**唯一正确的参考**。

### 架构规范引用

本 ADR 遵循以下文档的最高优先级约束：

| 文档 | 路径 | 关系 |
|------|------|------|
| 七层架构总纲 | `docs/architecture.md` | 最高优先级 |
| 层边界定义 | `docs/architecture/layer-contracts.md` | 本 ADR 的详细展开 |
| 目标目录结构 | `docs/architecture/project-structure.md` | 模块归属的最终参考 |
| Plugin 注册顺序 | `docs/architecture/plugin-design.md` | 第八节 Plugin 契约 |
| 错误体系架构 | `docs/architecture/error-architecture.md` | 错误归属规则 |
| 内容管线架构 | `docs/architecture/content-pipeline.md` | Content/Core 分离 |

## 决策

**采用逐层迁移契约，每层定义 10 项标准，确保迁移过程可验证、可追溯、可自动化检查。**

## Module Design — 逐层迁移契约

---

### 第一层：App — 游戏启动与装配

#### 1. Layer Purpose

App 层是游戏的**唯一组装者**——负责注册所有 Plugin、定义状态机、管理启动/关闭生命周期，**但不包含任何业务逻辑**。

#### 2. Three-Question Test

> **这是把游戏"启动起来"的代码吗？**

- 如果代码是注册 Plugin → App 层
- 如果代码是定义 AppState/TurnPhase → App 层
- 如果代码是定义 Schedule/SystemSet → App 层
- 如果代码包含任何 `if` 判断游戏逻辑 → **不是 App 层**

#### 3. In-Scope Modules

```
src/app/
├── app_plugin.rs       # 主 Plugin，注册所有子 Plugin
├── game_state.rs       # AppState 定义
├── schedules.rs        # Schedule 定义
├── sets.rs             # SystemSet 定义
├── startup.rs          # 启动逻辑（加载初始资源）
├── shutdown.rs          # 关闭逻辑（清理资源）
└── plugins.rs          # Plugin 汇集注册（22 个 Plugin 的 add_plugins 调用）
```

#### 4. Allowable Dependencies

```
App → 所有层 ✅（唯一允许全局视野的层）
```

App 是唯一可以知道所有模块存在的层。但它只是**组装**，不执行业务逻辑。

#### 5. Forbidden Dependencies

无（App 可依赖所有层）。

#### 6. Communication Contract

- **App 内部**：`plugins.rs` 按顺序调用 `add_plugins()`，使用元组批量注册
- **App → 其他层**：通过 Plugin 注册（`add_plugins()`），不直接创建 Entity
- **其他层 → App**：无（其他层不感知 App 的存在）

#### 7. Current Modules Mapped Here

| 当前位置 | 目标位置 | 迁移说明 |
|----------|---------|---------|
| `src_待迁移/main.rs` | `src/app/app_plugin.rs` | 重构为 AppPlugin |
| `src_待迁移/lib.rs` | `src/lib.rs` | 保持，但重构为 re-export |

#### 8. New Modules to Create

| 模块 | 说明 |
|------|------|
| `src/app/app_plugin.rs` | 主 Plugin（从 main.rs 重构） |
| `src/app/game_state.rs` | AppState + TurnPhase SubState 定义 |
| `src/app/schedules.rs` | Update/Startup Schedule 定义 |
| `src/app/sets.rs` | SystemSet 定义（BattleSet, TurnSet 等） |
| `src/app/startup.rs` | 启动逻辑（初始资源加载） |
| `src/app/shutdown.rs` | 关闭逻辑（资源清理） |
| `src/app/plugins.rs` | 所有 Plugin 汇集注册（22 个） |

#### 9. Migration Verification

```bash

grep -rn "DamageCalculation\|BuffApply\|SkillCast\|AIDecision" src/app/

# 2. 验证 Plugin 注册顺序
cargo test test_plugin_registration_order

# 3. 验证 AppState/TurnPhase 定义完整
cargo test test_state_machine_coverage
```

#### 10. Forbidden Patterns

| 禁止模式 | 原因 | 替代方案 |
|----------|------|----------|
| App 层包含业务逻辑 | 违反单一职责 | 业务逻辑放在对应 Core 模块 |
| App 层直接创建 Entity | App 只组装，不操作 | Entity 创建在各业务模块的 startup 中 |
| App 层硬编码游戏数值 | 数值应数据驱动 | 放在 content/ RON 文件中 |

### 第二层：Core — 游戏规则

#### 1. Layer Purpose

Core 层是项目的**心脏**——纯游戏规则。所有战斗伤害计算、Buff 施加与结算、属性修饰、回合管理、AI 决策、技能效果、装备穿脱、背包操作、寻路算法、胜负判定都在这里。

#### 2. Three-Question Test

> **如果明天把 Bevy 删了，换成 Godot / Unity / UE / 服务器模拟器，这个逻辑还存在吗？**

- 如果**存在** → Core 层
- 如果**不存在**（如 AssetServer 加载） → 不是 Core 层

```
Core → Shared    ✅ 允许

Core → App       ❌ 禁止
```

#### 5. Forbidden Dependencies

| 禁止依赖 | 原因 |
|----------|------|
| `Core → infrastructure/` | 规则不依赖技术实现 |
| `Core → content/` | 规则不依赖数据加载 |
| `Core → ui/` | 规则不依赖表现层 |
| `Core → modding/` | 规则不依赖 MOD |
| `Core → any business module` | 模块间通过 Message 通信 |

#### 6. Communication Contract

- **Core 内部**：通过 **Message** 广播（跨模块），通过函数调用（同模块内）
- **Core → UI**：通过 **ViewModel** 间接通信
- **UI → Core**：通过 **UiCommand** Message
- **Core → Infra**：通过 **Shared 事件**间接通信（Infra 的 Observer 监听）

| skill → buff | 只能通过 `SkillCastStarted` / `SkillCastFinished` Message |
| equipment → buff | 只能通过 `EquipItem` / `ItemEquipped` Message |
| character → battle | 只能通过 `CharacterDied` / `UnitMoved` Message |
| ai → battle | 只能通过 `CombatIntent` Message |
| ai → movement | 只能通过 `MoveIntent` Message |

### Core 层 — 内部模块边界（Effect Pipeline 契约）

#### Effect Pipeline 合约

Effect Pipeline 是 Core 层的核心管线，定义了战斗效果的完整生命周期：

```
CombatIntent → Generate → Modify → Execute
```

| 步骤 | 模块 | 职责 | 参与的 Message |
|------|------|------|---------------|
| **Intent** | `core/battle/pipeline/intent.rs` | 收集战斗意图（攻击、技能、道具） | `CombatIntent` |
| **Generate** | `core/battle/pipeline/generate.rs` | 根据意图生成原始效果 | `EffectsGenerated` |
| **Modify** | `core/battle/pipeline/modify.rs` | 应用 Modifier 规则修改效果 | `EffectsModified` |
| **Execute** | `core/battle/pipeline/execute.rs` | 执行最终效果，修改 ECS 状态 | `DamageApplied`, `HealApplied`, `BuffApplied` |

**管线约束**：

- 🟥 每个步骤必须在独立的 SystemSet 中执行
- 🟥 步骤间必须通过 Message 传递数据
- 🟥 Execute 阶段是唯一允许修改 ECS 状态的阶段
- 🟩 Generate 和 Modify 阶段应是纯函数（无副作用）

#### Core 内部模块通信规则

```
┌─────────────────────────────────────────────────────────────┐
│                    Core Layer Internal                       │
│                                                             │
│  ┌──────────┐    Message    ┌──────────┐                    │
│  │ battle   │──────────────→│ skill    │                    │
│  └──────────┘               └──────────┘                    │
│       │                         │                           │
│       │ Message                 │ Message                   │
│       ↓                         ↓                           │
│  ┌──────────┐               ┌──────────┐                    │
│  │ buff     │←──────────────│equipment │                    │
│  └──────────┘   Message     └──────────┘                    │
│       │                                                   │
│       │ Message                                           │
│       ↓                                                   │
│  ┌──────────┐               ┌──────────┐                    │
│  │character │──────────────→│   ai     │                    │
│  └──────────┘   Message     └──────────┘                    │
│                                                             │
│  ┌──────────┐    函数调用   ┌──────────┐                    │
│  │ map      │──────────────→│ pathfind │                    │
│  └──────────┘               └──────────┘                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**通信规则**：

| 通信类型 | 适用场景 | 示例 |
|----------|---------|------|
| **Message** | 跨模块广播 | `battle → skill` 通过 `DamageApplied` |
| **函数调用** | 同模块内部 | `map/pathfinding/` 内部调用 |
| **Shared Event** | 跨层通信 | Core 发布事件 → Infra 的 Observer 监听 |

### 第三层：Shared — 基础能力

#### 1. Layer Purpose

Shared 层提供**所有模块都能复用的基础工具**——它既不是游戏规则，也不是技术实现，而是"如果没有任何游戏逻辑，这个模块依然有意义"的东西。

#### 2. Three-Question Test

> **三个问题必须全部回答 YES：**

1. **如果删掉所有游戏逻辑，这个模块还有用吗？** → YES
2. **这个模块包含任何业务逻辑吗？** → NO（不包含）
3. **这个模块依赖任何业务类型吗？** → NO（不依赖）

**任何一个 NO → 不放 Shared。**

#### 3. In-Scope Modules

```
src/shared/
├── ids/             # 强类型 ID（UnitId, SkillId, BuffId, ...）
├── error/           # GameResult<T> + 错误转换 trait
├── events/          # 跨模块领域事件（公共基础设施）
├── audit/           # 审计轨迹基础设施
├── random/          # 确定性随机数
├── math/            # 数学工具（距离计算等）
├── time/            # 时间工具
├── collections/     # 通用集合类型
├── validation/      # 校验工具
├── constants/       # 全局常量
├── traits/          # 核心 trait（DamageSource, Healable 等）
├── macros/          # 过程宏
├── testing/         # 测试工具（spawn_test_battle 等）
└── versioning/      # 版本管理工具
```

#### 4. Allowable Dependencies

```
Shared → 无依赖     ✅ Shared 不依赖任何其他层
其他层 → Shared     ✅ 所有层都可以依赖 Shared
```

**Shared 是依赖图的叶子节点——它只提供工具，不消费任何业务逻辑。**

#### 5. Forbidden Dependencies

| 禁止依赖 | 原因 |
|----------|------|
| `Shared → core/` | 基础不依赖规则 |
| `Shared → infrastructure/` | 基础不依赖技术实现 |
| `Shared → ui/` | 基础不依赖表现层 |
| `Shared → content/` | 基础不依赖数据加载 |

#### 6. Communication Contract

- **Shared 内部**：模块间无通信（每个模块独立）
- **Shared → 其他层**：无（Shared 是被动提供工具）
- **其他层 → Shared**：直接 `use shared::xxx`

### Registration Order Rules

- 🟥 **表现层插件禁止在数据层之前注册**
- 🟥 **逻辑层插件禁止在核心层之前注册**
- 🟥 **禁止跳过任何注册顺序**
- 🟩 **每个分组内用元组批量注册**（`add_plugins((A, B, C))`）

### Anti-Pattern: Plugin build() Executing Business Logic

```rust
// ❌ 绝对禁止：Plugin::build() 中执行业务逻辑
impl Plugin for BattlePlugin {
    fn build(&self, app: &mut App) {
        // 错误：在 build 中执行业务逻辑
        let mut world = app.world_mut();
        let entity = world.spawn(BattleState::new()).id();
        
        // 错误：在 build 中修改 Resource
        world.insert_resource(EffectQueue::default());
    }
}

// ✅ 正确：Plugin::build() 只注册
impl Plugin for BattlePlugin {
    fn build(&self, app: &mut App) {
        app
            // 注册 Message
            .add_message::<DamageApplied>()
            .add_message::<HealApplied>()
            .add_message::<CharacterDied>()
            
            // 注册 Resource（初始化，不修改）
            .init_resource::<EffectQueue>()
            
            // 注册 System（业务逻辑在 System 中执行）
            .add_systems(Update, (
                generate_effects,
                modify_effects,
                execute_effects,
            ).chain().run_if(in_state(AppState::InGame)))
    }
}
```

### Plugin Anti-Patterns

| 禁止模式 | 原因 | 替代方案 |
|----------|------|----------|
| Plugin::build() 执行业务逻辑 | 只注册不执行 | 业务逻辑放在 System 中 |
| Plugin 间循环依赖 | 编译错误、难以维护 | 重新设计模块边界 |
| UI Plugin 注册 Core 系统的 Message | 跨层注册 | 跨层 Message 在 App 层注册 |
| 跳过 Plugin 直接注册 System | 破坏模块边界 | 通过 Plugin 注册 |
| 为单个实现创建 Plugin | 过度拆分 | 按业务领域拆分 |
|表现层插件在数据层之前注册 | 依赖顺序错误 | 遵循 Group 1-8 顺序 |
| 逻辑层插件在核心层之前注册 | 依赖顺序错误 | 遵循 Group 1-8 顺序 |

## Forbidden — 三个垃圾桶警告

在大型项目中，以下三个目录最容易退化为垃圾桶：

### 1. shared/ 垃圾桶

**症状**：所有不知道放哪里的东西都往 shared 塞。

**防治**：每次往 shared 添加模块时，回答三个问题（Gatekeeper Rules）：
1. 如果删掉所有游戏逻辑，这个模块还有用吗？
2. 这个模块包含任何业务逻辑吗？
3. 这个模块依赖任何业务类型吗？

如果 1=否 或 2=是 或 3=是 → **不放 Shared**。

### 2. common/ 垃圾桶

**症状**：和 shared 一样，只是换了个名字。

**防治**：不创建 common/ 目录。如果已经存在，合并到 shared/ 或拆分到业务模块。

### 3. utils/ 垃圾桶

**症状**：把小函数全部塞进 utils.rs。

**防治**：
- 🟥 禁止创建 `utils.rs`、`helpers.rs` 垃圾桶文件
- 🟩 通用工具按功能拆分到 shared/ 对应模块
- 🟩 领域工具放在对应业务模块内部

## 后果

### 正面后果

1. **可验证性**：每层 10 项标准，迁移过程可自动化检查
2. **可追溯性**：每个模块有明确的归属层和迁移映射
3. **可独立开发**：4 个并行 feature-developer agents 可按层独立工作
4. **可替换性**：Infrastructure 层文档化了替换方案
5. **可扩展性**：Content 层的桥接契约确保新增内容不修改 Rust 代码

### 负面后果

1. **迁移成本**：157 个文件需要逐层迁移，预计 30 个工作日
2. **学习曲线**：团队需要理解 10 项标准和 8 个 Plugin 注册组
3. **过度规范化**：某些简单模块可能被过度拆分

### 风险缓解

| 风险 | 缓解措施 |
|------|----------|
| 迁移过程中项目无法编译 | 每个 Phase 保持可编译、可运行 |
| 模块归属判断错误 | 使用 Three-Question Test 重新判断 |
| Plugin 注册顺序错误 | 使用 `cargo test test_plugin_registration_order` 验证 |
