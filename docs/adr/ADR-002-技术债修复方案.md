## 状态: Proposed

**版本**: 1.0

**范围**: 10 项技术债务的系统性修复

## 背景

### 当前状态

`src_待迁移` 目录包含 157 个 Rust 文件、33,771 行代码。在 ADR-001 的迁移分析中，识别出 10 项独立于目录迁移的技术债务。这些债务即使迁移完成后仍然存在，必须在各自对应的 Phase 中一并修复。

### 技术债务总览

| # | 债务名称 | 严重度 | 所在文件 | 迁移 Phase |
|---|---------|--------|---------|-----------|
| 1 | Unit God Component | 高 | character/components.rs | Phase 2 |
| 2 | 循环依赖 (character → ui::events) | 高 | character/mod.rs | Phase 2 |

| 10 | 审计轨迹无消费者 | 低 | infrastructure/audit/ | Phase 4+Future |

## 决策

### 核心决策

**采用「随迁随修」策略：每项技术债在其对应的迁移 Phase 中修复，不单独创建修复分支。**

理由：
1. 迁移本身就要修改文件，顺手修复比二次返工更高效
2. 部分债务（如 core/ 超载）是迁移的前置条件，必须先修
3. 每个 Phase 保持可编译可运行，修复风险可控

### 修复顺序与依赖关系

```
Phase 1 (Shared)    → Debt 4 部分（ID 拆分）+ Debt 8（强类型 ID 定义）
Phase 2 (Core)      → Debt 4 部分（core 重组）+ Debt 1 + Debt 2 + Debt 5 + Debt 6 + Debt 8（ID 使用）
Phase 3 (Error)     → Debt 7（错误迁移）
Phase 4 (Infra)     → Debt 3（Double-Event）+ Debt 10（审计消费）
Phase 6 (App)       → Debt 9（插件注册顺序）
```

### Debt 1: Unit God Component

**文件**: `character/components.rs` (第 27-44 行)

#### 问题描述

`Unit` 组件通过 `#[require(...)]` 自动插入 10 个组件：`Attributes`, `SkillSlots`, `SkillCooldowns`, `ActiveBuffs`, `GameplayTags`, `PersistentTags`, `TraitCollection`, `EquipmentSlots`, `Container`, `GridPosition`。这创建了虚假耦合——Unit "知道"每个子系统的存在。

```rust
// 当前代码（character/components.rs:27-44）
#[derive(Component, Reflect)]
#[reflect(Component)]
#[require(
    Attributes,
    SkillSlots,
    SkillCooldowns,
    ActiveBuffs,
    GameplayTags,
    PersistentTags,
    TraitCollection,
    EquipmentSlots,
    Container,
    GridPosition
)]
pub struct Unit {
    pub faction: Faction,
    pub acted: bool,
}
```

#### 宪法违反

- **1.1.6 组合绝对优先于继承**：Unit 通过 `#[require]` 硬编码了所有子系统的依赖，违反组合原则
- **2.1.2 Component 只存数据**：虽然 Unit 本身只存数据，但 `#[require]` 创建了 Unit 与所有子系统的编译期强耦合
- **2.3.2 依赖关系表达**：宪法允许 `#[require]`，但要求依赖关系反映真实业务需求，而非技术便利

#### 修复策略

将 Unit 改为标记组件，仅保留 `Faction` 和 `acted` 状态。10 个 required 组件由各自 Plugin 的 spawn hook 在单位生成时按需添加。

#### 迁移 Phase

Phase 2（Core 重组）

#### 代码变更

**Before**（当前）：
```rust
#[derive(Component, Reflect)]
#[require(
    Attributes, SkillSlots, SkillCooldowns, ActiveBuffs,
    GameplayTags, PersistentTags, TraitCollection,
    EquipmentSlots, Container, GridPosition
)]
pub struct Unit {
    pub faction: Faction,
    pub acted: bool,
}
```

**After**（目标）：
```rust
/// 战斗单位标记组件（仅身份与状态）
/// 各子系统的组件由各自 Plugin 的 spawn hook 添加
#[derive(Component, Reflect)]
pub struct Unit {
    pub faction: Faction,
    pub acted: bool,
}

/// 单位生成时的组件装配（在 spawn 系统中调用）
pub fn assemble_unit(entity: Entity, commands: &mut Commands) {
    commands.entity(entity).insert((
        Attributes::default(),
        SkillSlots::default(),
        SkillCooldowns::default(),
        ActiveBuffs::default(),
        GameplayTags::default(),
        PersistentTags::default(),
        TraitCollection::default(),
        EquipmentSlots::default(),
        Container::default(),
        GridPosition::default(),
    ));
}
```

#### 验证方法

1. 编译通过：移除 `#[require]` 后所有使用 `Unit` 的代码仍能编译
2. 测试通过：`unit_auto_inserts_required_components` 测试改为验证 `assemble_unit` 后组件存在
3. 运行验证：游戏启动正常，单位生成后所有组件就位

### Debt 2: 循环依赖 (character → ui::events)

**文件**: `character/mod.rs` (第 126 行)

#### 问题描述

`CharacterPlugin` 注册了 `crate::ui::events::MovementIntent`——一个 UI 消息被 character 模块注册。这创建了循环依赖：character 依赖 ui::events。

```rust
// 当前代码（character/mod.rs:125-126）
.add_message::<crate::ui::events::MovementIntent>()
```

#### 宪法违反

- **3.0.3 跨模块通信规范**：跨模块只能通过 Message、Observer、Command 通信，但 character 不应注册 UI 层定义的消息
- **1.1.4 逻辑与表现强制分离**：character 是逻辑层，ui::events 是表现层，逻辑层不应依赖表现层定义

#### 修复策略

将 `MovementIntent` 定义从 `ui/events/` 迁移到 `core/movement/events.rs`（移动关注点归 movement 模块）。CharacterPlugin 只注册位于 core 层的消息。

#### 迁移 Phase

Phase 2（Core 重组）

#### 代码变更

**Before**（当前）：
```rust
// character/mod.rs
.add_message::<crate::ui::events::MovementIntent>()
```

**After**（目标）：
```rust
// core/movement/events.rs
#[derive(Message, Debug, Clone)]
pub struct MovementIntent {
    pub entity: Entity,
    pub path: Vec<IVec2>,
}

// character/mod.rs
.add_message::<crate::core::movement::events::MovementIntent>()
```

#### 验证方法

1. 编译通过：character 模块不再 `use crate::ui`
2. 依赖检查：`cargo tree -i ui` 不显示 character 依赖 ui
3. 测试通过：移动执行系统测试正常

### Debt 3: Double-Event 日志模式

**文件**: `infrastructure/logging/events.rs`, `infrastructure/logging/observer.rs`

#### 问题描述

日志基础设施定义了 12 个领域事件类型（`BuffApplied`, `EquipmentEquipped`, `ItemUsed` 等），这些事件与实际的领域级 Message 平行存在。造成双重事件系统：领域事件触发后，日志事件再次触发。

当前 `events.rs` 定义了 12 个日志专用事件：
- `ConfigLoaded`, `BuffApplied`, `BuffRemoved`, `BuffExpired`
- `SkillActivated`, `LevelCompletedEvent`, `EquipmentEquipped`, `EquipmentUnequipped`
- `ItemUsed`, `ItemTransferred`, `UnitMoved`, `SnapshotCreated`

而 `observer.rs` 同时监听两类事件：领域事件（`DamageApplied`, `HealApplied` 等 6 个）和日志事件（上述 12 个）。

#### 宪法违反

- **2.2.5 领域事件是唯一业务事实源**：日志事件与领域事件并存，违反「一次事件触发，全链路复用」原则
- **14.8.1 核心架构模式**：日志必须是领域事件的消费者，禁止为打日志单独创建事件
- **14.8.2 强制适用范围**：复用现有业务事件，绝对禁止为了打日志单独创建事件

#### 修复策略

删除 `infrastructure/logging/events.rs` 中的 12 个日志专用事件。`LogObserver` 改为直接监听领域层的原始 Message（`battle::BuffApplied`, `equipment::ItemEquipped` 等）。

#### 迁移 Phase

Phase 4（Infrastructure）

#### 代码变更

**Before**（当前）：
```rust
// infrastructure/logging/events.rs
pub struct BuffApplied {
    pub target: Entity,
    pub target_name: String,
    pub buff_id: String,
    pub source: Option<Entity>,
    pub remaining_turns: u32,
}

// infrastructure/logging/observer.rs
pub fn log_buff_applied(mut reader: MessageReader<super::events::BuffApplied>) { ... }
```

**After**（目标）：
```rust
// infrastructure/logging/events.rs — 删除全部 12 个事件类型
// 文件删除或仅保留模块说明注释

// infrastructure/logging/observer.rs
pub fn log_buff_applied(
    mut reader: MessageReader<crate::buff::BuffApplied>,
    units: Query<&UnitName>,
) {
    for msg in reader.read() {
        let name = units.get(msg.target).map(|n| n.0.as_str()).unwrap_or("?");
        bevy::log::info!(
            target: "buff",
            event = "buff_applied",
            target = ?msg.target,
            target_name = %name,
            buff_id = %msg.buff_id,
            "Buff 已施加"
        );
    }
}
```

#### 验证方法

1. 编译通过：删除 12 个事件类型后所有引用处更新
2. 日志输出验证：游戏运行时 buff 相关日志仍正常输出
3. 事件数量检查：`grep -r "Message" src/ | wc -l` 减少 12

### Debt 4: core/ 职责超载

**文件**: `core/mod.rs`

#### 问题描述

`core/` 同时包含：
- **Shared 层内容**：`id/`（BuffId, ItemId, SkillId, UnitId）— 通用强类型 ID
- **Core 层内容**：`attribute/`, `effect/`, `modifier_rule.rs`, `tag.rs` — 游戏规则
- **Content 层内容**：`attribute_def.rs`, `tag_def.rs`, `registry_loader.rs` — 配置桥接
- **Infrastructure 层内容**：`snapshot.rs` — 场景快照

```rust
// 当前 core/mod.rs 结构
pub mod attribute;        // Core: 属性计算规则
pub mod attribute_def;    // Content: 属性定义注册表
pub mod effect;           // Core: 效果管线
pub mod error;            // 应在各领域 domain/ 内
pub mod id;               // Shared: 强类型 ID
pub mod modifier_rule;    // Core: 修饰规则
pub mod registry_loader;  // Content: RON 加载器
pub mod snapshot;         // Infrastructure: 快照
pub mod tag;              // Core: 标签系统
pub mod tag_def;          // Content: 标签定义注册表
```

#### 宪法违反

- **architecture.md §七层架构**：Core 只应包含游戏规则，Shared 只应包含通用工具，Content 只应包含配置桥接
- **3.0.5 路径命名规范**：宪法禁止 `core/` 等纯技术命名（已有宪法豁免，但内部职责仍需分离）
- **architecture.md §层间依赖规则**：Core → Shared ✅，但 Core 不应包含 Shared 内容

#### 修复策略

拆分 `core/` 为三个目标位置：

| 当前位置 | 目标位置 | 层级 |
|---------|---------|------|
| `core/id/` | `shared/ids/` | Shared |
| `core/tag.rs`（GameplayTag 枚举） | `shared/tag.rs` | Shared |
| `core/attribute/types.rs`（AttributeKind） | `shared/attribute_types.rs` | Shared |
| `core/attribute/`（计算规则） | `core/attribute/` | Core（不变） |
| `core/effect/` | `core/effect/` | Core（不变） |
| `core/modifier_rule.rs` | `core/modifier_rule.rs` | Core（不变） |
| `core/attribute_def.rs` | `content/attribute_def.rs` | Content |
| `core/tag_def.rs` | `content/tag_def.rs` | Content |
| `core/registry_loader.rs` | `content/registry_loader.rs` | Content |
| `core/snapshot.rs` | `infrastructure/snapshot.rs` | Infrastructure |

#### 迁移 Phase

Phase 1（Shared 定义）+ Phase 2（Core 重组）

#### 代码变更

**Before**（当前）：
```rust
// core/mod.rs
pub mod attribute;
pub mod attribute_def;
pub mod effect;
pub mod error;
pub mod id;
pub mod modifier_rule;
pub mod registry_loader;
pub mod snapshot;
pub mod tag;
pub mod tag_def;
```

**After**（目标）：
```rust
// shared/mod.rs
pub mod ids;
pub mod tag;          // GameplayTag 枚举
pub mod attribute_types; // AttributeKind

// core/mod.rs（仅保留游戏规则）
pub mod attribute;    // 属性计算规则
pub mod effect;       // 效果管线
pub mod modifier_rule; // 修饰规则

// content/mod.rs
pub mod attribute_def;
pub mod tag_def;
pub mod registry_loader;

// infrastructure/mod.rs
pub mod snapshot;
```

#### 验证方法

1. 编译通过：所有 `use crate::core::xxx` 更新为新路径
2. 依赖检查：`cargo tree` 显示 Core 不依赖 Content/Infrastructure
3. 层级验证：三问判断法验证每个文件归层正确

### Debt 5: Entity 占位符

**文件**: `battle/pipeline/intent.rs:149,202`, `equipment/equip.rs:554`, `character/movement_execution.rs:131`, `ai/targeting.rs` (10 处)

#### 问题描述

`Entity::PLACEHOLDER` 在 4 个文件中使用 15 处。`String::new()` 用于 `caster_name`/`target_name` 代替查询 UnitName 组件。这些是运行时数据 Bug 的隐患。

```rust
// 当前代码（battle/pipeline/intent.rs:149-150）
log_skill_writer.write(SkillActivated {
    caster: entity,
    caster_name: _name.0.clone(),
    skill_id: skill_id.to_string(),
    target: Entity::PLACEHOLDER, // TODO: 从 CombatIntent 获取目标
    target_name: String::new(),
});
```

#### 宪法违反

- **1.2.2 对外输出必须使用可读标识**：`Entity::PLACEHOLDER` 和空字符串在日志/事件中不可读，违反可读标识要求
- **2.1.1 Entity 只是 ID**：将 Entity 当作占位符使用，而非真实业务标识
- **14.9.3 错误强制要求**：事件/日志中的 Entity 字段应携带完整上下文

#### 修复策略

1. 查询 `UnitName` 组件获取可读名称（替代 `String::new()`）
2. 通过 `CombatIntent.target_coord` + 占位网格查询解析目标 Entity（替代 `Entity::PLACEHOLDER`）
3. 对于测试代码中的 `Entity::PLACEHOLDER`，保留但添加注释说明

#### 迁移 Phase

Phase 2（Core 重组）

#### 代码变更

**Before**（当前）：
```rust
log_skill_writer.write(SkillActivated {
    caster: entity,
    caster_name: String::new(),
    skill_id: skill_id.to_string(),
    target: Entity::PLACEHOLDER,
    target_name: String::new(),
});
```

**After**（目标）：
```rust
// 查询 UnitName 获取可读名称
let caster_name = unit_names.get(entity)
    .map(|n| n.0.clone())
    .unwrap_or_else(|| format!("{:?}", entity));

// 通过 target_coord 解析目标
let target_entity = target_coord
    .and_then(|coord| resolve_entity_at(coord, &occupancy_grid))
    .unwrap_or(entity); // 兜底使用自身
let target_name = unit_names.get(target_entity)
    .map(|n| n.0.clone())
    .unwrap_or_else(|| format!("{:?}", target_entity));

log_skill_writer.write(SkillActivated {
    caster: entity,
    caster_name,
    skill_id: skill_id.to_string(),
    target: target_entity,
    target_name,
});
```

#### 验证方法

1. 编译通过：所有 `Entity::PLACEHOLDER` 替换为真实 Entity 查询
2. 日志验证：游戏运行时技能日志显示真实目标名称
3. 测试通过：`cargo test` 全部通过

### Debt 6: 手动资源重置

**文件**: `turn/mod.rs` (第 119-132 行)

#### 问题描述

`cleanup_ingame` 系统手动将 13 个 Resource 重置为 Default。如果新增 Resource 而忘记在此处重置，会残留上一局的状态。

```rust
// 当前代码（turn/mod.rs:119-132）
commands.insert_resource(TurnState::default());
commands.insert_resource(TurnOrder::default());
commands.insert_resource(GameOverState::default());
commands.insert_resource(NeedsResolve::default());
commands.insert_resource(AiTimer::default());
commands.insert_resource(CombatIntent::default());
commands.insert_resource(PrevPosition::default());
commands.insert_resource(SelectedUnitView::default());
commands.insert_resource(CombatPreviewView::default());
commands.insert_resource(HoveredEntity::default());
commands.insert_resource(CombatLogCollapsed::default());
commands.insert_resource(ActionMenuEntity::default());
commands.insert_resource(UiFocusState::default());
```

#### 宪法违反

- **architecture.md §回合状态机**：状态切换必须透明，手动重置列表不透明
- **17.0.3 副作用透明**：状态切换时绝对禁止隐藏副作用，遗漏重置即为隐藏副作用
- **1.1.7 只解决当前复杂度**：当前 13 个手动重置已超出简单范围，需要机制化

#### 修复策略

创建 `GameScoped` 标记组件或 `CleanupOnGameEnd` trait，让所有游戏局域 Resource 自动注册到清理列表。`cleanup_ingame` 系统查询所有标记了该 trait 的 Resource 并自动重置。

#### 迁移 Phase

Phase 2（Core 重组）

#### 代码变更

**Before**（当前）：
```rust
fn cleanup_ingame(mut commands: Commands, ...) {
    // ... entity despawn ...
    commands.insert_resource(TurnState::default());
    commands.insert_resource(TurnOrder::default());
    // ... 11 more manual resets ...
}
```

**After**（目标）：
```rust
/// 游戏局域 Resource 必须实现的 trait
/// 实现后自动参与 cleanup_ingame 清理
pub trait GameScopedResource: Resource + Default + 'static {}

// 各 Resource 实现
impl GameScopedResource for TurnState {}
impl GameScopedResource for TurnOrder {}
impl GameScopedResource for CombatIntent {}
// ...

fn cleanup_ingame(
    mut commands: Commands,
    // ... entity queries ...
) {
    // ... entity despawn ...

    // 自动重置所有 GameScopedResource
    macro_rules! reset_game_scoped {
        ($($ty:ty),* $(,)?) => {
            $(commands.insert_resource(<$ty as Default>::default());)*
        };
    }
    reset_game_scoped!(
        TurnState, TurnOrder, GameOverState, NeedsResolve, AiTimer,
        CombatIntent, PrevPosition, SelectedUnitView, CombatPreviewView,
        HoveredEntity, CombatLogCollapsed, ActionMenuEntity, UiFocusState,
    );
}
```

> **注意**：Bevy 当前不支持运行时遍历所有 Resource 类型，因此 macro 方案是务实选择。与手动列表相比，macro 至少保证声明与使用在同一处，且编译期检查类型存在性。

#### 验证方法

1. 编译通过：所有 Resource 实现 `GameScopedResource`
2. 测试通过：从 InGame 退出再进入，所有 Resource 恢复默认值
3. 新增 Resource 检查：新增 Resource 时必须同时在 macro 中注册，否则无清理

### Debt 7: 错误放置错误

**文件**: `core/error/` (battle_error.rs, buff_error.rs, skill_error.rs, inventory_error.rs, game_result.rs)

#### 问题描述

所有领域错误集中在 `core/error/` 目录，而非各自领域模块的 `domain/` 子目录。

```
当前结构：
core/error/
├── battle_error.rs     # 应在 core/battle/domain/
├── buff_error.rs       # 应在 core/buff/domain/
├── skill_error.rs      # 应在 core/skill/domain/
├── inventory_error.rs  # 应在 core/inventory/domain/
├── game_result.rs      # 应在 shared/error/
└── mod.rs
```

#### 宪法违反

- **architecture.md §错误架构**：领域错误放领域内部（`core/skill/domain/skill_error.rs`）
- **14.9.1 分领域错误原则**：每个领域定义独立错误枚举，放领域内部
- **architecture.md §停止条件**：领域错误放在 shared/ 或 infrastructure/ 必须停止

#### 修复策略

按 `error-architecture.md` 定义的三层模型迁移：
- 领域错误 → 各领域 `domain/` 子目录
- 共享工具 → `shared/error/`
- 基础设施错误 → `infrastructure/` 内部

#### 迁移 Phase

### Debt 8: 缺少强类型 ID 使用

**文件**: 多个文件使用原始 Entity 或 String 作为实体引用

#### 问题描述

部分 Message 仍使用原始类型而非 Strong ID。例如 `CombatIntent.skill_id` 使用 `Option<String>`，`SkillActivated` 的 `skill_id` 使用 `String`。

```rust
// 当前代码（battle/pipeline/intent.rs:23）
pub struct CombatIntent {
    pub source_entity: Option<Entity>,
    pub target_coord: Option<IVec2>,
    pub skill_id: Option<String>,  // 应为 Option<SkillId>
}

// infrastructure/logging/events.rs:44
pub struct SkillActivated {
    pub skill_id: String,  // 应为 SkillId
    // ...
}
```

#### 宪法违反

- **1.2.1 核心领域必须使用强类型 ID**：所有业务实体必须定义专属强类型标识，禁止直接使用裸 Entity 或 String 作为业务标识跨模块传递
- **1.2.2 对外输出必须使用可读标识**：日志/事件中的 String ID 不如强类型 ID 可读

#### 修复策略

1. Phase 1：在 `shared/ids/` 定义所有 Strong ID（已有 `core/id/`，直接迁移）
2. Phase 2：更新所有 Message 和 Resource 使用 Strong ID
3. 添加 `From<Entity>` / `Into<Entity>` 转换

#### 迁移 Phase

### Debt 9: 插件注册顺序错误

**文件**: `main.rs` (第 67-94 行)

#### 问题描述

当前插件注册顺序为：
1. 数据层（SkillPlugin, BuffPlugin, AiBehaviorPlugin, EquipmentPlugin, InventoryPlugin）
2. 核心层（EffectPlugin, ModifierRulePlugin, AttributeDefPlugin, TagDefPlugin）
3. 基础设施（LogPlugin, AuditPlugin）
4. 逻辑层（AssetsPlugin, TurnPlugin, MapPlugin, CharacterPlugin, BattlePlugin, AiPlugin）
5. 表现层（UiPlugin, InputPlugin, DebugPlugin）

数据层在核心层之前注册，违反七层依赖规则。

```rust
// 当前代码（main.rs:67-80）
// 数据层插件（在核心层之前 ❌）
.add_plugins((
    SkillPlugin,
    BuffPlugin,
    AiBehaviorPlugin,
    EquipmentPlugin,
    InventoryPlugin,
))
// 核心层插件（在数据层之后 ❌）
.add_plugins((
    EffectPlugin,
    ModifierRulePlugin,
    AttributeDefPlugin,
    TagDefPlugin,
))
```

#### 宪法违反

- **architecture.md §插件注册顺序**：必须按 核心层 → 数据层 → 逻辑层 → 表现层 顺序注册
- **architecture.md §层间依赖规则**：Content 依赖 Core，Core 不依赖 Content

#### 修复策略

按七层架构重新排列插件注册顺序。需要创建 App 层作为统一装配点。

#### 迁移 Phase

Phase 6（App 层）

#### 代码变更

**Before**（当前）：
```rust
// main.rs — 扁平注册

    // 基础设施

    // 逻辑层
```

**After**（目标）：

```rust
            // Layer 3: Shared（零依赖）
```

### Debt 10: 审计轨迹无消费者

**文件**: `infrastructure/audit/` (mod.rs, trail.rs, event.rs, whitelist.rs)

#### 问题描述

审计基础设施已实现（`AuditTrail`, `AuditEvent`, `EventWhitelist`），但没有下游消费者实际使用它。Replay 系统、测试验证、调试面板都未接入审计数据。
