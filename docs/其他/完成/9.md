# DND背包的本质

DND真正管理的不是：

```text
物品
```

而是：

```text
资源(Resource)
```

例如：

```text
背包空间
重量
金币
箭矢
药水
法术材料
```

本质都是资源。

---

所以我推荐：

```text
Inventory
    ↓
Container
    ↓
ItemInstance
    ↓
ItemDefinition
```

四层模型。

---

# 第一层：Definition / Instance分离

永远不要：

```rust
struct Item {
    name: String,
    count: u32,
}
```

---

应该：

```rust
ItemDefinition
```

```yaml
id: potion_healing

name: 治疗药水

tags:
  - consumable

effects:
  - heal 50
```

---

实例：

```rust
ItemInstance {
    guid: Uuid,
    definition: potion_healing,
}
```

---

原因：

未来：

```text
耐久
品质
附魔
绑定
签名
任务状态
```

都在实例层。

---

# 第二层：Container系统

这是绝大多数游戏没做好的地方。

不要：

```rust
Inventory {
    items: Vec<ItemInstance>
}
```

---

应该：

```rust
Container
```

统一。

例如：

```text
角色背包

仓库

宝箱

尸体

商店

邮件

掉落袋
```

本质都是：

```text
Container
```

---

统一接口：

```rust
trait ContainerOps {
    add_item()
    remove_item()
    transfer_item()
}
```

---

未来扩展极舒服。

# 第三层：Slot背包还是Weight背包？

DND推荐：

```text
Weight
```

而不是：

```text

```

例如：

```yaml

```

---

角色：

```yaml

strength × 15
```

---

然后：

```text

```

这种体验。

这比：

```text
20格满了
```

更符合RPG。

```rust
Container
```
```text
    ↓

    ↓
```

```text
    ↓
```

然后：

```text
角色背包

尸体

```

全部都是：


这样未来做到 5000+ 物品、几十个系统联动，也不会失控。

真正的铁律可以浓缩成一句：

> **不要设计“背包系统”，而要设计“Container（容器）系统”；背包只是容器的一种表现形式。** 这是 DND、CRPG（如《博德之门3》）和大型 RPG 最容易被忽略、但长期收益最高的设计。
