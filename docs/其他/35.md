一、 🌟 核心亮点（为什么这是一份优秀的设计？）

哲学极其正确：“App 只是装配器”
   明确禁止 App 层包含业务逻辑和硬编码数值。这保证了 main.rs 永远干净，所有业务都被封装在独立的 Plugin 中，为未来的热插拔、MOD 加载、单元测试打下了完美基础。
精准利用 Bevy 最新特性：SubState
   将 TurnPhase 设计为 AppState::InGame 的 SubState，这是 Bevy 0.15+ 最优雅的状态机用法。它完美解决了“只有在战斗中才存在回合阶段”的逻辑嵌套问题，避免了用一堆布尔值（is_in_game && is_selecting_unit）去控制系统的反模式。
严格的 SystemSet 数据流分层
   Input → Command → Logic → Effect → ViewModel → UI 的单向数据流设计，是大型游戏避免“状态竞态条件（Race Condition）”的黄金法则。UI 永远只读 ViewModel，Logic 永远只处理 Command，这会让后期的 Bug 排查难度降低 80%。

二、 ⚠️ 深度技术审查与“避坑”优化（关键！）

尽管架构完美，但在 Bevy 引擎的实际运行中，以下几个细节如果不调整，会导致性能暴跌或逻辑死锁：

关于 .chain() 的致命性能陷阱
文档描述“TurnPhase 内的系统必须通过 .chain() 保证顺序”
潜在灾难：在 Bevy 中，.chain() 会强制系统串行执行，彻底破坏 Bevy 的多线程并行优势。如果你在 ExecuteAction 阶段把 combat_intent、effect_generate、effect_execute 全部 .chain() 起来，当同屏有 20 个单位同时结算 Buff 时，帧率会暴跌。
优化方案：
    放弃无脑 .chain()，改用 after() / before() 约束。Bevy 调度器会在满足依赖关系的前提下，自动将无冲突的系统并行化。
    对于必须严格串行的逻辑（如效果管线的生成->修饰->执行），不要放在普通的 Update 里，建议自定义一个专属的 Schedule（例如 EffectPipelineSchedule），在 Update 的特定节点通过 run_schedule 手动触发。

SRPG 核心痛点：逻辑与动画的“时间轴撕裂”
文档描述WaitAction 阶段等待动画/结算完成。
潜在灾难：SRPG 最怕“逻辑已经算完了（单位已死），但死亡动画还在播，此时玩家狂点鼠标触发了下一个回合”。用 Bevy 原生的 State 去等待动画，很容易造成状态机卡死。
优化方案（命令队列模式）：
    在 LogicSet 和 EffectSet 中，不要直接修改 Transform 或播放动画。
    逻辑层只产出 GameEvent（如 UnitMoved, UnitDamaged）。
    引入一个 ActionTimeline（行动时间轴/命令队列）Resource。UI 和动画层从时间轴中消费事件并播放动画。
    只有当 ActionTimeline 为空时，TurnPhase 才会从 WaitAction 推进到 TurnEnd。

OnExit 清理的“内存泄漏”刺客
文档描述cleanup_battle_resources() 清理战斗资源。
潜在灾难：在 Bevy 中，如果手动写系统去 despawn 战斗 Entity，极容易漏掉子 Entity（如血条、特效挂载点），导致内存泄漏或下一局游戏出现“幽灵单位”。
优化方案：
    定义一个标记组件：#[derive(Component)] struct BattleEntity;
    所有在 InGame 状态生成的 Entity（包括 UI 节点），必须带上这个组件。
    在 OnExit(AppState::InGame) 中，只需一个通用系统：
            fn cleanup_battle_entities(mut commands: Commands, query: Query<Entity, With<BattleEntity>>) {
          for entity in query.iter() { commands.entity(entity).despawn_recursive(); }
      }
      
    绝对不要在 OnExit 里去手动清理特定的业务 Resource，让 Bevy 的 Res 自然覆盖或在 OnEnter 时重置。

插件依赖的“隐式炸弹”
文档描述App 层按顺序 add_plugins。
潜在灾难：随着 Plugin 增多，如果某天你把 SkillPlugin 移到了 EffectPlugin 前面，运行时可能会因为 Resource 未初始化而 Panic。
优化方案：
    利用 Bevy 的 require_plugins 机制（Bevy 0.15+ 支持）。
    在 SkillPlugin 的 build 方法中声明：app.require_plugins(EffectPlugin);
    这样即使 App 层注册顺序乱了，Bevy 也会在启动时给出明确的依赖缺失报错，而不是运行时崩溃。

三、 架构微调建议（针对目录与文件）

你设计的 src/app/ 目录结构很清晰，但我建议做一点微调，使其更符合 Rust 的模块化习惯：

src/
├── main.rs               # 极简，只有 App::new().add_plugins(AppPlugin).run()
├── app/
│   ├── mod.rs            # 导出 AppPlugin
│   ├── plugin.rs         # AppPlugin 的 build 实现（装配线）
│   ├── states.rs         # 合并 game_state.rs (AppState + TurnPhase 定义)
│   ├── sets.rs           # SystemSet 定义 (InputSet, LogicSet 等)
│   └── schedules.rs      # 自定义 Schedule 定义 (如 EffectPipelineSchedule)
│
├── shared/               # 零依赖工具 (数学、随机数、扩展 trait)
├── infrastructure/       # 外部交互 (存档、日志、音频、本地化 Fluent)
├── core/                 # 纯领域逻辑 (属性计算、效果管线、寻路算法)
├── content/              # 数据定义与加载 (技能、Buff、关卡配置的 AssetLoader)
├── logic/                # 业务 System (回合控制、AI 决策、输入处理)
└── ui/                   # 视图层 (Egui/BevyUI 渲染、ViewModel 同步)
注：将 startup.rs 和 shutdown.rs 的职责直接下放给各个业务 Plugin 的 OnEnter/OnExit，App 层不需要专门的 startup/shutdown 文件，保持 App 层的绝对“空心化”。

## 一、核心设计亮点（全都是踩过坑后的正确决策）
### 1. 「App = 装配器」的哲学定位：从根源避免大泥球
这是整套设计最核心的正确判断，也是90% Bevy项目最先踩的坑。
- 绝大多数中小项目会把`main.rs`写成“启动+配置+业务逻辑+资源初始化”的混合体，项目规模上来后彻底失控；而你明确了App层的唯一职责是「组装」，不包含任何业务逻辑，总装线的类比非常精准。
- 配套的“禁止事项”（不创建Entity、不修改业务状态、不硬编码数值）形成了强约束，把业务逻辑全部下沉到各层Plugin中，保证App层永远是薄的、可预测的。
- 最终`main.rs`只保留一行插件注册，极致简洁，后续所有功能扩展都通过新增Plugin完成，完全符合开闭原则。

### 2. 分层插件注册顺序：严格遵循依赖倒置原则
插件的注册顺序「Shared → Infrastructure → Core → Content → Logic → UI → Debug」是教科书级的分层依赖设计：
- 每层只依赖比自己更底层的模块，上层依赖下层，下层绝不感知上层，从架构上杜绝了循环依赖的可能。
- 分层粒度非常合适：Shared是纯工具无依赖，Core是纯领域逻辑无UI，Content是数据配置，Logic是业务流程，UI是纯表现，Debug是外挂式工具，和你之前的七层架构完全对齐。
- 特别值得肯定的是**Debug插件条件编译**的设计，通过feature gate隔离开发工具，正式构建零冗余，也避免了调试代码污染生产逻辑。

### 3. 双层状态机设计：粒度精准，适配SRPG特性
`AppState`宏观流程 + `TurnPhase`局内子状态的双层设计，完美匹配回合制SRPG的流程特性，避开了状态机设计的两个极端：
- 避免了“单一层级状态爆炸”：不会把主菜单、回合阶段、结算都塞到同一个状态枚举里，导致状态转换混乱。
- 避免了“状态嵌套过深”：两层是复杂度的黄金平衡点，再深就会出现状态转换地狱。
- 配套的「子状态仅在InGame激活」「通过NextState驱动」「禁止OnEnter跨阶段跳转」规则，堵死了状态机最容易被滥用的口子，保证状态流转的可追溯性。

### 4. Schedule + SystemSet流水线：彻底解决系统顺序混乱
`InputSet → CommandSet → LogicSet → EffectSet → ViewModelSet → UISet`的执行流水线设计，是这套架构里最体现工业级功底的部分：
- 严格划分了每一层的职责边界，明确了“哪一层能改什么数据”：比如PreUpdate只做输入转换，PostUpdate只做UI刷新，禁止反向修改状态，从根源上避免了“一帧内逻辑和UI互相覆盖”的经典bug。
- 引入`ViewModelSet`作为逻辑层和UI层的中间层，是非常有远见的设计：逻辑层只更新视图模型数据，UI层只读取视图模型渲染，两者完全解耦。未来换UI方案、做多端适配、做单元测试的时候，逻辑层完全不用动。
- 支持子Set嵌套（比如LogicSet下拆TurnSet、CombatSet），既保证了顶层顺序的一致性，又给下层留出了细化空间。

### 5. 完整的合约与边界规则：适配团队/AI协作
每个文件、每个状态、每层都明确了「职责+禁止项」，而不是只写“做什么”，不写“不能做什么”。
- 这对于长期项目、多人协作、AI辅助开发至关重要：没有边界的架构会在迭代中快速腐化，而明确的禁止规则相当于给代码加了护栏，无论是新人还是AI写代码，都不容易越界。
- 依赖规则里**「没有任何层可以依赖App」**的约束非常关键：App是顶层组装者，一旦被下层依赖，立刻就会出现循环依赖和架构腐化，这条规则守住了分层的底线。

## 二、进阶补强：中长期一定会遇到的工业级细节
整套架构大框架已经非常完善，下面是千级代码量、多模块、MOD扩展场景下，建议提前预留的设计点：

### 1. 补充状态切换的加载中间态
当前`AppState`是直接跳转，但是SRPG切换关卡、进入战斗时会有大量资产加载和数据初始化，建议新增`Loading`状态作为所有重状态切换的过渡：
```
MainMenu → Loading → InGame
LevelSelect → Loading → InGame
InGame → Loading → GameOver
```
- 统一管理加载进度、加载动画，避免状态切换时画面冻结；
- 把资源加载、数据初始化逻辑放到Loading状态的OnEnter里，失败时可以兜底回退到主菜单，不会直接崩溃；
- 支持异步加载，配合Bevy的资产句柄做进度追踪。

### 2. 插件的模块化与可配置化
当前插件是全量注册，未来支持MOD、多平台、DLC时，需要更灵活的插件管理：
- 引入**Plugin Group**概念：比如`CorePlugins`、`LogicPlugins`都是插件组，支持按feature开关启用子插件，比如移动端禁用某些高级特效插件；
- 预留MOD插件加载槽位：在`ContentPlugin`之后、`LogicPlugin`之前预留MOD加载入口，由App层统一扫描并加载启用的MOD插件，保证MOD只能扩展内容，不能修改核心逻辑；
- 插件配置化：通过`AppSettings`资源控制插件行为，比如是否开启日志、是否启用热重载，而不是硬编码在插件里。

### 3. 细化FixedUpdate的Set划分
当前设计只覆盖了`Update` Schedule的Set排序，对于SRPG里的动画、物理、确定性结算逻辑，`FixedUpdate`同样需要明确的Set和顺序：
- 建议同步在FixedUpdate里划分`LogicFixedSet`、`PhysicsSet`、`AnimationFixedSet`，保证固定步长的逻辑也有统一的执行顺序；
- 明确规则：**确定性战斗结算必须放在FixedUpdate里**，Update只处理输入和UI表现，避免帧率波动影响战斗数值，这是SRPG网络同步、战斗回放功能的基础。

### 4. 资产加载分级与生命周期管理
启动序列里的资产加载目前是分散的，建议按生命周期分级管理：
- **App生命周期资产**：字体、全局UI素材、通用配置，启动时加载，全程不卸载；
- **状态生命周期资产**：主菜单UI、关卡地图、角色模型，进入状态时加载，退出状态时卸载，控制内存峰值；
- App层统一管理资产加载的生命周期，各业务Plugin只声明自己需要的资产，不直接控制加载卸载。

### 5. 全局错误兜底与可观测性
App层作为全局入口，需要承担崩溃兜底和可观测性的职责：
- 注册全局panic hook，捕获异常后输出日志、保存现场、弹窗提示，而不是直接闪退；
- Infrastructure层的日志、审计系统，在App层统一初始化，全链路埋点；
- 状态切换异常兜底：比如进入InGame失败，自动回退到MainMenu并提示错误，保证程序不会卡死。

## 三、落地避坑：最容易走偏的几个架构陷阱
设计规则写得很清楚，但实际落地中很容易慢慢突破边界，这里提前预警几个最常见的腐化路径：

### 1. 警惕AppPlugin的“功能 creep”
最常见的腐化路径：一开始AppPlugin只注册插件，后来慢慢加个全局系统、加个资源初始化、加个状态切换逻辑，最后AppPlugin变成了新的大泥球。
- **铁律**：`app_plugin.rs`里只允许出现三类代码：`add_plugins`、`init_resource`（全局配置类）、`configure_sets/configure_schedules`，绝对不允许出现`add_systems`的业务系统。
- 所有系统必须归属到对应层级的Plugin里，哪怕只有一行代码的系统，也要放到对应的模块中。

### 2. SystemSet不要过度拆分
现在的6层顶层Set粒度非常合适，但落地时很容易陷入“每个系统都建一个Set”的误区，反而导致排序规则爆炸、维护成本飙升。
- **原则**：只有需要明确排序先后的系统才需要分到不同的Set；同一层级、无先后依赖的系统放在同一个Set里即可，不需要过度细分。
- 子Set只在大模块内部使用，顶层Set永远保持6~8个的量级，保证全局执行流水线清晰可读。

### 3. 状态机不要无限嵌套
双层状态机是黄金平衡点，不要为了“严谨”再加第三层、第四层子状态（比如在ActionMenu里再加子状态）。
- 更细粒度的流程控制，用**组件标记**或者**资源标志位**实现，而不是不断嵌套State。比如动作菜单的子选项，用一个`CurrentMenuSelection`资源控制就足够，没必要上升到状态机层面。

### 4. 用工具强制依赖约束
只靠文档约定的依赖规则，长期一定会被突破。建议落地时加上强制校验：
- 用`cargo-modules`或者`cargo-depgraph`在CI里检查模块依赖，比如禁止`core`模块引用`ui`模块，禁止任何模块引用`app`模块；
- 对于AI生成的代码，可以在提示词里明确依赖规则，配合静态检查兜底。
