# MOD 支持架构设计文档点评
这份 SRPG 项目的 MOD 支持架构文档是一份**结构完整、目标明确、落地性强**的设计方案，既覆盖了游戏 MOD 系统的核心诉求，也兼顾了安全性、扩展性和可维护性，以下从「核心优势」「可优化/补充点」「整体评价」三个维度展开点评：

## 一、核心优势
### 1. 目标与原则锚定精准
设计目标紧扣“内容扩展”核心（策划/社区仅需改 RON 文件），同时守住“安全可控”底线（Rule/Content 分离），既满足内部快速迭代需求，也适配社区 MOD 生态的开放性，核心原则（如禁止绕过核心战斗规则、隔离 ECS 直接操作）从根源上保障游戏体验的一致性。

### 2. 架构分层清晰且职责闭环
模块化的分层设计（API → Registry → Loaders → Validators → Sandbox → Compatibility）职责边界明确：
- API 层聚焦“稳定扩展入口”，仅暴露内容扩展接口，降低 MOD 作者学习成本；
- Registry/Loaders 层解决“依赖/加载/合并”核心问题；
- Validators/Sandbox 层守住“安全/校验”底线；
- Compatibility 层解决生态最核心的“版本/冲突”问题。
各层形成“扩展-加载-校验-兼容”的完整闭环，可维护性和扩展性强。

### 3. 生命周期与流程设计严谨
- **目录结构**：区分 official/community/dev 目录，兼顾官方维护、社区贡献、开发调试场景；
- **清单格式**：manifest.ron 覆盖 ID/版本/依赖/冲突/优先级/内容声明，核心字段（如 conflicts、priority）直击 MOD 生态的痛点；
- **加载流程**：从扫描→解析→依赖→加载→校验→合并的步骤逻辑严密，既保证加载顺序可控，也通过校验管线规避非法内容。

### 4. 安全性与兼容性设计落地性强
- **安全边界**：明确的“禁止/允许”操作清单，结合沙箱机制（隔离 ECS World/Commands），从权限层面杜绝 MOD 破坏核心规则；
- **冲突与兼容**：拓扑排序解析依赖、优先级加载、命名空间防 ID 冲突、版本语义化校验，全方位解决 MOD 生态的兼容痛点，避免“一装 MOD 就崩”的常见问题。

### 5. 内容组织规范且符合直觉
MOD 内容目录（skills/buffs/characters 等）按游戏核心内容维度划分，配合 RON 格式的结构化配置，既便于策划/MOD 作者编写，也便于游戏侧解析加载，降低协作成本。

## 二、可优化/补充的方向
### 1. 未来目标的落地细节需补充
文档提到“运行时热加载”“脚本支持”等 Phase 2 目标，但未提及核心技术挑战和初步方案：
- 热加载：补充 ModdableRegistry 的可变设计细节（如内容注册的回滚日志、已实例化实体的状态同步策略）；
- 脚本支持：明确 Lua/WASM 脚本的可操作边界（如仅允许扩展数值逻辑，还是可自定义技能触发规则）、脚本与 Effect Pipeline 的交互方式（如何避免绕过核心规则）。

### 2. 性能与大规模场景考量不足
未提及大量 MOD 加载时的性能问题：
- 依赖解析（拓扑排序）、校验管线、内容合并在数百个社区 MOD 场景下的性能瓶颈；
- 可补充优化方向：如 manifest 解析缓存、增量校验（仅校验变更 MOD）、异步加载非核心内容、Registry 内存优化。

### 3. 覆盖（override）机制的管控需强化

- override 操作的审计日志（便于排查冲突）。

### 4. 资源（美术/音频）管理细节缺失
仅提及 assets 目录，但未说明：

### 5. 调试与排查机制缺失
未考虑 MOD 作者的开发调试需求：

- 提供 MOD 状态查询 API（如已加载内容、依赖关系、冲突项）；
- 内置调试工具（如 MOD 加载模拟器、冲突检测预览）。

### 6. 权限分级可补充
当前所有 MOD 权限一致，可补充分级策略：
- 官方 MOD：可有限访问扩展接口（如轻度自定义战斗逻辑）；
- 社区 MOD：严格沙箱，仅开放基础内容扩展接口；
- 测试 MOD：开发环境专属权限，便于调试。

## 三、整体评价
这份文档是一份**高质量、可落地的 MOD 架构设计方案**，核心优势在于：
1. 守住“核心规则不被破坏”的底线，同时最大化开放内容扩展能力；
2. 覆盖 MOD 生态从开发、加载、运行到兼容的全生命周期；
3. 设计贴合游戏开发的实际场景（策划用 RON 快速扩展、社区 MOD 安全隔离）。

可优化点均为“补充细节”而非“架构重构”，整体不影响核心方案的可行性。落地后既能支撑内部策划快速迭代内容，也能构建健康的社区 MOD 生态，是 SRPG 类游戏 MOD 系统的优秀设计范本。

3. 模块化架构的专业设计
六层架构分离：API层、Registry层、Loaders层、Validators层、Sandbox层、Compatibility层，关注点分离极致化
命名空间防冲突：base:/mod_x:前缀方案从根本上解决了MOD间ID冲突问题
依赖解析完备：拓扑排序 + 循环依赖检测 + 版本冲突处理，这是大型MOD生态的基础设施

4. 工程化实践的高度成熟

impl Plugin for ModdingPlugin {
    fn build(&self, app: &mut App) {
        // 注册MOD专属AssetLoader
        app.init_asset_loader::<ModContentLoader>();
        
        // 将MOD资源目录添加到AssetServer

            .watch_for_changes()
            .add_path("mods/community/");
    }
}

这种设计完美利用了Bevy的资产加载系统，避免了重复造轮子。

Effect Pipeline保护机制
// MOD API的核心约束
pub fn register_skill(ctx: &mut ModContext, skill: SkillDef) -> ModResult<()> {
    // 1. Schema校验
    validate_skill_schema(&skill)?;
    
    // 2. 引用完整性校验
    validate_references(&skill, &ctx.registry)?;
    
    // 3. 核心保护：技能效果必须通过Effect Pipeline
    for effect in &skill.effects {
        if !is_valid_effect_type(effect) {
            return Err(ModError::InvalidEffectType);
        }
    }
    
    // 4. 安全注册
    ctx.registry.register_skill(skill)
}

这种四重校验机制确保了MOD无法破坏游戏最核心的战斗规则。
