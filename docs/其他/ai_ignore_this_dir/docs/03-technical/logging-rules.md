# 日志体系领域规则

Version: 1.0
Status: Draft
Applies To: 所有使用 tracing 宏的日志代码、日志配置、日志输出策略

> **优化来源**: `docs/01-architecture/logging_design.md`

---

## 1. 统一术语

| 术语 | 定义 | 职责边界 |
|------|------|----------|
| 结构化日志 | 使用 tracing crate 的键值对日志，不使用字符串拼接 | 负责：记录带结构化字段的事件；不负责：格式化输出 |
| 五级日志体系 | TRACE/DEBUG/INFO/WARN/ERROR 五级分类 | 负责：区分日志重要程度；不负责：控制日志输出 |
| 审计日志 | 记录游戏事件供回放/分析的日志，级别 INFO/WARN/ERROR | 负责：永久保留游戏事实；不负责：调试开发 |
| 调试日志 | 辅助开发调试的日志，级别 TRACE/DEBUG | 负责：开发调试；不负责：生产环境使用 |
| 批量输出 | 高频事件累计 N 条后统一打印，避免日志洪流 | 负责：减少日志开销；不负责：控制日志级别 |
| 采样策略 | 高频事件按固定比例采样输出 | 负责：控制日志量；不负责：保证完整性 |
| 日志降级 | 输出失败时降级到 stdout 或静默丢弃 | 负责：保证程序不崩溃；不负责：保证日志完整性 |
| LogObserver | 监听领域事件生成日志记录的观察者 | 负责：事件→日志转换；不负责：事件生产 |
| AuditTrail | 战斗中所有审计记录的时序集合 | 负责：收集事件供回放；不负责：日志输出 |

> 🟩 对应宪法第十三部分：可观测性宪法（日志 + 错误 + 调试）

---

## 2. 领域边界

### 2.1 负责范围

- 日志级别分类与使用规范
- 结构化字段格式
- 日志输出目标管理
- 日志文件滚动与清理
- 日志降级策略
- 数据脱敏规则
- 采样与批量输出策略
- 审计日志与调试日志区分

### 2.2 不负责范围

- 具体业务逻辑的日志内容（由各子系统领域决定）
- Feature Flag 的日志级别编译控制（由 feature_flag 领域负责）
- 遥测数据收集（由 telemetry 模块负责）
- 崩溃报告（由 crash_report 领域负责）
- 调试面板展示（由 debug_ui 领域负责）

## 3. 生命周期

> **优化来源**: docs/01-architecture/logging_design.md — 日志系统从设计到运维的完整生命周期

```
1. 设计阶段
   → 确定系统需要记录的日志事件
   → 为每个事件分配日志级别
   → 定义结构化字段
   → 确定输出目标（stdout/文件/遥测）
    ↓
2. 实现阶段
   → 使用 tracing crate 统一日志
   → 使用结构化字段，禁止字符串拼接
   → 注册 LogObserver 监听领域事件
   → 配置日志初始化（输出目标、格式、级别）
    ↓
3. 测试阶段
   → 使用 tracing-test 断言业务事件触发正确级别日志
   → 测试日志文件滚动
   → 测试降级策略
   → 验证性能影响在预算内
    ↓
4. 发布阶段
   → 配置生产环境日志级别（INFO 及以上）
   → 启用 JSON 结构化输出
   → 配置日志文件滚动（7天、10MB/文件）
   → 可选接入 OpenTelemetry
    ↓
5. 运维阶段
   → 监控日志文件大小
   → 清理过期日志
   → 处理降级场景（磁盘满、权限不足）
```

## 4. 不变量（Invariants）

### INV-LOG-01: tracing 统一

统一使用 `tracing` crate，禁止 `println!`、`dbg!`、`log` crate。

- 所有日志必须通过 tracing 宏输出
- 禁止使用标准输出宏（println、eprintln）
- 禁止使用调试宏（dbg!）
- 禁止使用 log crate

> **验证方法**: 搜索代码中的 `println!`、`dbg!`、`eprintln!`，结果必须为空。

### INV-LOG-02: 结构化字段

> 🟥 对应宪法 13.1.2：日志必须记录业务事件事实，绝对禁止记录技术执行流水
> 🟥 对应宪法 13.3.1：所有 INFO 级别日志必须携带 `event` 字段，值与事件名完全一致
> 🟥 对应宪法 13.3.2：日志 `target` 必须与所属 Feature 目录名完全一致

> **优化来源**: docs/01-architecture/logging_design.md §2.2 — 结构化字段规范

日志必须使用结构化字段，不使用字符串拼接。

- 禁止：`error!("Unit {} died", id)`
- 要求：`error!(unit_id = id.0, "Unit died")`
- 结构化字段便于日志过滤、查询、分析

> **验证方法**: 搜索日志宏中的字符串拼接模式（`{}`、`{:?}`），结果必须为空。

### INV-LOG-03: 每帧系统日志限制

> **优化来源**: docs/01-architecture/logging_design.md §3.4 — 每帧系统日志规则

每帧系统中**仅允许 ERROR 级别日志**。

- 禁止在每帧系统中使用 INFO/DEBUG/TRACE
- 每帧系统中的日志必须是致命错误
- 高频日志应使用采样或批量输出

> **验证方法**: 检查标记为每帧执行的系统，确认日志级别均为 ERROR。

### INV-LOG-04: 循环内日志禁止

> **优化来源**: docs/01-architecture/logging_design.md §3.2, §3.3 — 批量输出与采样策略

循环内禁止直接输出日志。

- 应使用批量输出（累计 N 条后统一打印）
- 或使用采样策略（每 N 帧采样 1 次）
- 循环内日志会导致日志洪流

> **验证方法**: 检查 for/while 循环内的日志调用，确认使用了采样或批量输出。

### INV-LOG-05: 数据脱敏

> **优化来源**: docs/01-architecture/logging_design.md §7.2 — 数据脱敏规则

日志和遥测数据必须脱敏，避免泄露敏感信息。

- 玩家 UID：哈希脱敏（SHA256 取前 8 位）
- 设备信息：不记录
- IP 地址：部分掩码（`192.168.***.***`）
- 密钥/Token：绝对禁止记录
- 存档路径：相对路径化
- MOD 配置：仅记录 ID

> **验证方法**: 搜索日志中的敏感数据模式，确认已脱敏。

### INV-LOG-06: 日志文件滚动

日志文件必须有滚动和清理策略。

- 按天滚动
- 保留最近 7 天日志
- 单个日志文件最大 10MB
- 超过大小限制时压缩旧日志

> **验证方法**: 检查日志目录，确认无超过 10MB 的文件，确认有 7 天前的清理。

### INV-LOG-07: 日志降级

日志写入失败时必须有降级策略，避免程序崩溃。

- 磁盘满：降级到 stdout 输出 + WARN 标记 `log_degraded=true`
- 文件权限不足：降级到 stdout 输出 + WARN 标记 `log_degraded=true`
- 格式化失败：输出原始消息（无结构化字段）+ WARN 标记 `format_degraded=true`
- 全部输出失败：静默丢弃（不阻塞业务逻辑）+ 内存计数器记录丢弃数量

> **验证方法**: 模拟磁盘满场景，确认日志仍可输出（降级到 stdout）。

## 5. 业务规则

### BR-LOG-01: 日志级别使用规范

> 🟩 对应宪法 13.2.1：五级日志体系

| 级别 | 用途 | 典型场景 | Release 构建行为 |
|------|------|---------|----------------|
| TRACE | 极细粒度算法细节 | A*节点探索、单帧循环内部逻辑、Modifier计算过程 | 编译出 |
| DEBUG | 开发调试辅助信息 | 寻路路径长度、Modifier计算过程、资源加载详情 | 编译出 |
| INFO | 重要事件 | 战斗开始、关卡完成、角色升级 | 保留 |
| WARN | 可恢复异常 | 资源加载失败（已降级）、配置校验警告 | 保留 |
| ERROR | 致命错误 | 数据损坏、状态机非法转换、校验失败 | 保留 |

### BR-LOG-02: 必须记录的事件

> 🟩 对应宪法 13.3.1：所有 INFO 级别日志必须携带 `event` 字段
> 🟩 对应宪法 13.3.2：日志 `target` 必须与所属 Feature 目录名完全一致
> 🟥 对应宪法 13.4.5：绝对禁止业务代码直接调用 `info!()` 输出核心业务事件（必须走领域事件链路）

| 事件类型 | 日志级别 | 结构化字段 |
|---------|---------|-----------|
| 战斗开始/结束 | INFO | stage, player_count, enemy_count |
| 角色死亡 | INFO | unit_id, killer_id, position |
| 技能释放 | INFO | caster_id, skill_id, target_id |
| Buff 施加/移除 | INFO | target_id, buff_id, source |
| 装备穿脱 | INFO | unit_id, equipment_id, slot |
| 状态转换 | DEBUG | from_state, to_state |
| 校验违规 | WARN/ERROR | violation_type, current_value, expected |
| 资源加载失败 | WARN | path, fallback_used |

### BR-LOG-03: 采样规则

- TRACE/DEBUG：每帧系统中**禁止**（已通过 Feature Gate 编译出）
- INFO：高频事件采样 1/100，低频事件全量输出
- WARN/ERROR：全量输出，不采样

### BR-LOG-04: 日志输出目标管理

| 构建类型 | 输出目标 | 格式 | 级别 |
|---------|---------|------|------|
| 开发构建 | stdout | 彩色格式化 | debug |
| 测试构建 | stdout | 紧凑格式 | info |
| 发布构建 | 文件 | JSON 结构化 | info |
| Release 构建 | 文件 + 可选遥测 | JSON 结构化 | info |

### BR-LOG-05: 审计日志 vs 调试日志

> 🟩 对应宪法 13.5：战斗日志使用独立事件体系，不依赖 tracing
> 🟩 对应宪法 13.8：日志架构 = 领域事件 → Log Observer → tracing 日志

| 特征 | 审计日志 | 调试日志 |
|------|---------|---------|
| 目的 | 记录游戏事件供回放/分析 | 辅助开发调试 |
| 级别 | INFO/WARN/ERROR | TRACE/DEBUG |
| 持久化 | 永久保留 | 按天滚动 |
| 格式 | 结构化 JSON | 彩色格式化 |
| 性能影响 | 低（批量写入） | 中（实时格式化） |

### BR-LOG-06: 多环境配置

| 环境 | RUST_LOG 值 | 输出目标 | 格式 |
|------|------------|---------|------|
| 开发环境 | debug | stdout | 彩色 |
| 测试环境 | info | stdout | 紧凑 |
| 生产环境 | info | 文件 | JSON |
| 调试模式 | bevy_srpg::battle=debug | stdout | 彩色 |

### BR-LOG-07: 遥测接入

Release 构建可选接入 OpenTelemetry，实现分布式追踪和远程监控。

- 通过 `telemetry` Feature Flag 控制
- 遥测数据必须经过脱敏处理
- 遥测关闭时零运行时开销

## 6. 流程管线

### 6.1 日志事件流程

> 🟩 对应宪法 13.8.1：日志是领域事件的消费者，而非业务代码主动调用的功能
> 🟩 对应宪法 13.8.2：所有 INFO 级别的核心业务事件，必须通过触发领域事件的方式生成日志
> 🟩 对应宪法 13.8.4：Battle Replay、战斗履历UI、成就系统、任务系统与日志共用同一套领域事件源

```
Core 系统执行业务逻辑
  → 产生 DomainEvent（事实记录）
  → LogObserver 监听事件
  → 生成日志记录（INFO/WARN/ERROR）
  → 输出到配置的目标（stdout/文件/遥测）
```

### 6.2 日志降级流程

```
日志写入尝试
  ├── 成功 → 正常输出
  └── 失败
      ├── 磁盘满 → 降级到 stdout + WARN 标记
      ├── 权限不足 → 降级到 stdout + WARN 标记
      ├── 格式化失败 → 输出原始消息 + WARN 标记
      └── 全部失败 → 静默丢弃 + 内存计数器
```

### 6.3 日志级别决策流程

```
需要记录日志
  → 是否为每帧系统？
      ├── 是 → 仅 ERROR 级别
      └── 否 → 事件是否为高频？
          ├── 是 → 使用采样（1/100）或批量输出
          └── 否 → 根据事件重要性选择级别
              ├── 致命错误 → ERROR
              ├── 可恢复异常 → WARN
              ├── 重要事件 → INFO
              ├── 系统执行 → DEBUG
              └── 每帧细节 → TRACE
```

## 7. 数据结构

### 7.1 日志级别

| 级别 | 数值 | 用途 | Release 行为 |
|------|------|------|-------------|
| TRACE | 0 | 每帧细节 | 编译出 |
| DEBUG | 1 | 系统执行 | 编译出 |
| INFO | 2 | 重要事件 | 保留 |
| WARN | 3 | 可恢复异常 | 保留 |
| ERROR | 4 | 致命错误 | 保留 |

### 7.2 日志文件结构

```
logs/
├── game_2026-06-14.log    # 当天日志（活跃）
├── game_2026-06-13.log    # 昨天日志
├── game_2026-06-12.log.gz # 前天日志（已压缩）
└── ...
```

### 7.3 降级状态标记

| 标记 | 含义 | 触发条件 |
|------|------|---------|
| log_degraded=true | 日志降级到 stdout | 磁盘满或权限不足 |
| format_degraded=true | 格式化失败，输出原始消息 | 格式化错误 |

## 8. 禁止事项

> 🟥 对应宪法 13.4：五条日志禁令（绝对禁止）

| 编号 | 禁止操作 | 宪法条款 | 原因 |
|------|---------|----------|------|
| L-01 | 使用 `println!` 或 `eprintln!` | 13.1.1 🟥 | 必须使用 tracing 宏 |
| L-02 | 使用 `dbg!` 宏 | 13.1.1 🟥 | 必须使用 tracing 宏 |
| L-03 | 使用 `log` crate | 13.1.1 🟥 | 必须使用 `tracing` crate |
| L-04 | 每帧系统中打印 INFO/DEBUG 日志 | 13.4.2 🟥 | Release 版本绝对禁止 |
| L-05 | 循环内日志 | 13.4.3 🟥 | 应批量输出或采样 |
| L-06 | 通过堆砌日志进行调试 | 13.4.4 🟥 | 应使用 Inspector/Replay/Debug Panel |
| L-07 | 日志中包含 PII（个人身份信息） | — | 数据安全 |
| L-08 | 日志中包含完整 Entity 数据每个 tick | — | 性能杀手 |
| L-09 | 日志文件无限增长 | — | 必须有滚动和清理策略 |
| L-10 | 日志格式化开销影响帧率 | — | 应使用结构化字段 |
| L-11 | 字符串拼接日志 | 13.1.2 🟥 | 应使用结构化字段 |
| L-12 | 记录函数进入、退出、系统执行等技术流水账 | 13.4.1 🟥 | 日志应记录领域事件，非技术执行 |
| L-13 | 业务代码直接调用 `info!()` 输出核心业务事件 | 13.4.5 🟥 | 必须走领域事件链路（13.8） |

## 9. AI 修改规则

### 9.1 允许的修改

- 添加新的日志事件（需遵循 BR-LOG-02）
- 修改日志级别（需遵循 BR-LOG-01）
- 更新日志配置（需遵循 BR-LOG-04）
- 添加采样策略（需遵循 BR-LOG-03）
- 更新脱敏规则（需遵循 INV-LOG-05）

### 9.2 禁止的修改

- 在 Core 层添加 `println!`/`dbg!`（违反 INV-LOG-01）
- 在每帧系统中添加 INFO 日志（违反 INV-LOG-03）
- 移除日志文件滚动策略（违反 INV-LOG-06）
- 移除日志降级策略（违反 INV-LOG-07）
- 在日志中添加未脱敏的敏感数据（违反 INV-LOG-05）

### 9.3 必须遵守的流程

1. 新增日志事件前，必须先确定日志级别
2. 新增日志事件后，必须使用结构化字段
3. 修改日志配置后，必须验证多环境配置
4. 修改脱敏规则后，必须验证敏感数据不再泄露
5. 修改降级策略后，必须测试降级场景
