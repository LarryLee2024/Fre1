✅ 卓越的设计亮点

1. Bevy ECS架构的深度契合
Resource驱动设计：RangeCache和PathfindingStats作为Resource管理，完美契合Bevy的资源系统
System隔离原则：寻路计算封装在独立组件中，业务系统只需调用接口，避免逻辑污染
生命周期管理：缓存失效规则与游戏事件（单位移动、地形变化等）解耦，通过事件驱动更新

2. 算法抽象的极致优雅
Trait-based设计：PathFinder和RangeCalculator traits提供了完美的算法替换能力
策略模式应用：BFS、A*等算法实现完全解耦，未来可无缝替换
参数化配置：RangeParams结构体统一了所有范围计算参数，避免了重复代码

3. 性能优化的全面考量
分层缓存策略：移动范围（不可缓存）vs 攻击/技能范围（可缓存）的区分极其精准
严格的性能预算：单次计算≤2ms、总开销≤5ms/frame，这是经过实战验证的硬指标
内存控制：17KB的缓存预算设计体现了对移动平台的深刻理解

4. 工程化实践的典范
不变量定义：5条核心不变量（如"所有范围计算必须通过统一接口"）是架构稳定的基石
禁止事项清单（🟥标记）比最佳实践更有指导价值，8条明确红线防止架构腐化
交叉引用完善：与地形规则、回合规则、战斗规则的文档关联，体现系统思维

🔧 潜在的优化建议

1. Bevy Schedule集成
当前缺口：未明确寻路系统在Schedule中的位置
建议：将缓存清理系统放在PreUpdate，范围计算放在Update，结果应用放在PostUpdate
示例：

这份文档是针对 SRPG 核心的寻路与范围计算模块的**高质量基础设施设计**，既解决了性能瓶颈和重复造轮子的核心痛点，又兼顾了扩展性、可维护性和落地性。以下从「核心优势」「可优化/补充点」「整体总结」三个维度展开点评：

## 一、核心优势
### 1. 抽象设计贴合开闭原则，扩展性极强
- **算法层抽象（PathFinder）**：通过 Trait 封装 BFS/A*/Dijkstra 等算法的核心能力（可达范围计算、路径回溯），上层业务无需关心算法细节，后续替换/新增算法（如适配大地图的分层寻路）只需实现 Trait，完全符合「对扩展开放、对修改关闭」。
- **范围计算层抽象（RangeCalculator）**：移动/攻击/技能范围统一接口，避免了“每种技能单独写范围逻辑”的重复造轮子问题，是文档核心痛点的精准解决。

### 2. 分层解耦清晰，职责边界明确
- 严格区分「寻路基础设施（怎么算）」和「地形业务规则（算什么）」，与 `map_terrain_rules.md` 的联动关系明确，避免业务规则侵入底层算法；
- 阻挡规则（UnitBlocker）、地形成本（TerrainCostCalculator）、范围计算（RangeCalculator）各司其职，单一职责原则落地到位，后续修改地形成本无需改动寻路算法。

### 3. 性能管控体系完整，可落地、可量化
- **性能预算明确**：为单次计算、总帧开销、缓存命中率设定具体数值（如移动范围≤2ms、缓存命中率≥80%），而非泛泛而谈“优化性能”；
- **优化策略具体**：缓存优先、增量更新、提前终止等策略贴合实际场景（如 BFS 移动力耗尽即停止）；
- **监控可落地**：设计 PathfindingStats 统计资源，能量化监控缓存命中率、计算耗时，便于后续性能调优。

### 4. 工程化约束到位，降低维护成本
- **不变量+禁止事项**：明确禁止硬编码算法、无失效缓存、忽略阻挡等常见坑，相当于为团队制定了开发规范，避免后期重构；
- **缓存策略闭环**：缓存键、失效条件、内存预算、禁止项（无上限缓存）全覆盖，既利用缓存提升性能，又规避内存泄漏、数据过期等风险。

### 5. 场景化设计，贴合 SRPG 业务实际
- 算法选择策略匹配场景（BFS 算范围、A* 算单目标路径），而非盲目追求“高端算法”；
- 范围类型（十字/菱形/方形/直线）覆盖 SRPG 常见的攻击/技能范围需求，参数结构（base_range/bonus_range）支持装备/Buff 加成，贴合游戏业务逻辑。

## 二、可优化/补充的细节点
### 1. 参数封装提升可读性
PathFinder::find_reachable_tiles 的参数列表过长（start/move_points/terrain/occupancy/cost_calculator/blocker），建议封装为 `PathFindingContext` 结构体：
```rust
pub struct PathFindingContext<'a> {
    pub start: IVec2,
    pub move_points: u32,
    pub terrain: &'a TerrainGrid,
    pub occupancy: &'a OccupancyGrid,
    pub cost_calculator: &'a dyn TerrainCostCalculator,
    pub blocker: &'a dyn UnitBlocker,
}

// 简化后接口
fn find_reachable_tiles(&self, ctx: &PathFindingContext) -> HashMap<IVec2, u32>;
```

### 2. 缓存策略可更精细化
- 移动范围缓存键 `(Entity, GridPosition, MovePoints)` 可拆分：地形/阻挡无变化时，仅移动力变化需失效，可设计“双层缓存”（基础地形层 + 移动力层），进一步提升命中率；
- 缓存失效可补充「软失效」策略（如缓存超过 N 帧未使用自动失效），避免冷门缓存长期占用内存。

### 3. 特殊场景补充细节
- **视线检测**：技能范围的 Line 类型仅提及“视线检测”，未明确规则（如是否穿透地形/单位、视线判断的距离公式），建议补充 `LineOfSightChecker` 抽象 Trait 或默认实现规则；
- **飞行单位**：不变量4仅说明移动消耗，未明确飞行单位的阻挡规则（如是否可穿过敌方单位），建议补充飞行单位对应的 UnitBlocker 实现；
- **大地图兜底**：当前性能预算针对 20×20 地图，大地图（40×40+）可补充「分帧计算」「异步计算」策略，避免单帧卡顿。

### 4. 测试与落地补充
- 未提及测试策略：建议补充单元测试（如 PathFinder 不同地形/阻挡下的可达范围验证）、性能基准测试（benchmark）的设计；
- 增量更新细节：“只重算受影响的单位范围”未说明“如何识别受影响单位”（如单位移动后，哪些相邻单位需重算），补充后更易落地。

### 5. 结果结构体扩展性
RangeResult 仅包含 positions/paths，若后续需返回“剩余移动力”“地形防御加成”等信息，可增加扩展字段：
```rust
pub struct RangeResult {
    pub positions: HashSet<IVec2>,
    pub paths: HashMap<IVec2, Vec<IVec2>>,
    // 扩展字段，兼容后续业务需求
    pub metadata: HashMap<IVec2, RangeMetadata>,
}

pub struct RangeMetadata {
    pub remaining_move_points: Option<u32>, // 仅移动范围有效
    pub terrain_defense_bonus: u32,        // 地形防御加成
}
```

这份设计文档是**工业化级别的基础设施设计**，不仅解决了 SRPG 寻路模块的核心痛点（性能、复用），还通过抽象层、约束规则、性能管控构建了完整的工程化体系。

- 对于中小规模 SRPG 项目：当前设计可直接落地，仅需补充测试用例和大地图兜底策略；
- 对于大规模/高复杂度项目：仅需在抽象层（如视线检测、异步计算）做少量扩展，核心架构无需改动。
