# 第一部分：83 大架构专业评审

## 1.1 致命缺陷（🟥 必须修复）

### F1. Core 内部 5 层能力体系存在依赖矛盾

**问题**：5 层（atom/rule/runtime/model/frame）的依赖声明与实际需求自相矛盾。

| 层 | 声明依赖 | 实际需求 | 矛盾 |
|---|---------|---------|------|
| L0 atom | shared | ✅ 无矛盾 | — |
| L1 rule | L0 + shared | ✅ 无矛盾 | — |
| L2 runtime | L0 + L1 + shared | ✅ 无矛盾 | — |
| **L3 model** | **L0 + shared** | **需要 L1 的 attribute/tag/modifier 类型来描述角色属性集合、标签集合** | **声明不依赖 L1，但数据结构必须引用 L1 类型** |
| **L4 frame** | **L3~L0 + shared** | **需要 L2 的 pipeline/registry 来定义执行框架** | **声明不依赖 L2，但框架接口必须引用 L2 抽象** |

**后果**：
- 要么违反自己声明的依赖规则（L3 偷偷引用 L1）
- 要么被迫做"穿透传递"（L0 重新定义 L1 的类型子集，导致重复定义）
- 50 万行代码中，这种矛盾会扩散到每个使用 model 类型的 Domain

**根因**：model 层（纯数据结构）和 atom 层（原子数据）的边界不清晰。character_data 的"属性集合"已经是 L1 attribute 的类型了，不可能退回 L0。

### F2. Domain 间"只能通过全局事件通信"对战斗核心路径不可行

**问题**：16 个 Domain 全部通过事件通信，但 D&D/BG3 级规则的战斗核心路径要求同步执行。

**典型场景**：一次伤害判定的跨域调用链

```
combat 释放技能
  → terrain 检查高地加成（Modifier）
  → reaction 检查机会攻击（同步中断）
  → spell 检查法术反应（Counterspell，同步中断）
  → combat 执行伤害
  → reaction 检查援护格挡（同步替换）
  → character 检查死亡
  → inventory 检查吸血装备触发
```

如果全部走异步事件：
- **时序无法保证**：Bevy Event 是下一帧消费，战斗逻辑需要同步返回结果
- **调试灾难**：一次伤害涉及 6+ 个 Domain 的事件处理，分布在 6 个不同目录
- **性能瓶颈**：每次伤害 6+ 次事件序列化/反序列化/分发

**D&D 反应机制的本质**：反应是伤害判定的同步子流程，不是异步通知。BG3 的 Counterspell 必须在施法完成前同步决定是否使用，不能"下一帧再处理"。

### F3. mod_api 放在 Core 内部违反分层原则

**问题**：`core/mod_api/` 包含 combat_gateway/character_gateway 等接口，意味着 Core 层必须感知 Modding 的需求。

**后果**：
- Core 内部 5 层能力体系变动时，Mod API 不稳定
- Core 开发者需要同时考虑"这个改动会不会破坏 Mod API"
- 违反了"Core 只依赖 Shared"的依赖规则

### F4. 无编译策略

**问题**：50 万行 Rust 代码在单 crate 中意味着增量编译 30 秒以上。文档提到 `crates/` 目录"可选，初期可合并到 src"，但没有给出何时拆分、如何拆分的路线图。

**后果**：这是长期项目最致命的技术风险。编译时间每增加 10 秒，开发效率下降约 15%。50 万行单 crate 的增量编译可能超过 60 秒。

## 1.2 严重设计缺陷（🟨 强烈建议修复）

### S1. Presentation 层被埋在 Infrastructure 内部

**问题**：rendering/ui/audio/animation/particles/platform 全部在 `infra/presentation/` 下。

**为什么不合理**：
- 大型项目中 Presentation 代码量通常占 30-40%，不应被埋在 Infrastructure 的子目录中
- UI 系统有自己的架构模式（ViewModel/UiCommand/CQRS），与"技术实现"的定位不符
- 渲染/动画/粒子是独立的关注点，与存档/网络/本地化没有技术共性
- 违反了文档自身声明的"Logic/Presentation 分离"原则——如果 Presentation 是 Infrastructure 的一部分，那它和 Logic 的分离就不彻底

### S2. Content 层的 schema/ 子目录违反 Rule/Content 分离

**问题**：`content/schema/` 包含 attribute_def.rs/effect_def.rs/ability_def.rs 等数据结构定义。

**为什么不合理**：
- Def 类型是领域规则的定义（"属性怎么计算"），属于 Core
- Content 层的职责是"加载→校验→注册"，不应该定义数据结构
- 如果 Def 类型在 Content 层，Core 的 Domain 怎么引用它？Core 不能依赖 Content

### S3. Domain 强制统一内部结构

**问题**：所有 Domain 必须有 plugin/components/systems/events/error/rules/integration 7 个文件。

**为什么不合理**：
- faction（阵营关系）和 economy（经济）是简单 Domain，7 文件模板迫使他们创建空壳
- combat 是复杂 Domain，7 文件模板不够用（需要 preview/、sub-domains/）
- 与"只解决当前复杂度"原则冲突
- 大型项目最佳实践：模块结构应随复杂度弹性伸缩

### S4. Shared 层包含领域语义

**问题**：`shared/time/game_time.rs` 和 `shared/time/turn_count.rs` 是游戏领域概念。

**为什么不合理**：
- game_time（游戏内时间流逝）和 turn_count（回合计数）是 SRPG 的核心领域概念
- 如果 Shared 包含领域语义，那"零业务语义"的边界就会被持续侵蚀
- 50 万行代码中，每次"不知道放哪"的东西都会往 Shared 塞，最终退化为垃圾桶

### S5. frame 层与 Domain 的边界极度模糊

**问题**：frame/quest 定义"任务系统通用框架"，domains/quest 定义"任务子系统"。框架和子系统的边界在哪里？

**为什么不合理**：
- 如果 Domain 实现了 frame 的接口，那 Domain 必须理解 frame 的抽象——但 frame 的抽象本身就是对 Domain 的归纳
- 这是一种"先有鸡还是先有蛋"的问题：在单人开发中，你不可能先设计出完美的框架再实现 Domain
- 大型项目最佳实践：框架从 Domain 的重复模式中提炼，而非预先设计

## 1.3 中等问题（🟦 建议修复）

### M1. Infrastructure 包含大量不需要的模块

| 模块 | 适用场景 | 本项目需要？ |
|------|---------|------------|
| anti_cheat | 多人在线游戏 | ❌ 单机 SRPG |
| encryption | 保护敏感数据 | ❌ 单机 SRPG |
| cloud | 云存档/云服务 | ❌ 单机 SRPG |
| navmesh | 3D 开放世界 | ❌ SRPG 用 Grid |
| threading | 自定义线程池 | ❌ Bevy 已有 |
| memory | 自定义内存管理 | ❌ Bevy 已有 |
| physics | 物理碰撞 | ❌ SRPG 通常不需要 |

**影响**：增加架构认知负担，暗示不存在的需求，给 AI 协作时产生误导。

### M2. 编辑器架构缺失

**问题**：tools/ 下列出了 map_editor/ability_editor 等，但：
- 没有编辑器状态机设计
- 没有编辑器-游戏数据共享机制
- 没有撤销/重做架构
- 没有实时预览设计
- 编辑器被标记为"永不进入发布构建"，但用户需求明确编辑器是产品的一部分

### M3. 数值模拟架构缺失

**问题**：battle_simulator/ 只有 mod.rs，没有设计：
- 模拟器如何复用 Core 纯函数？
- 如何做蒙特卡洛批量验证？
- 如何 Headless 运行（无 Bevy 渲染）？
- 如何做敏感度分析？

### M4. 缺少 Domain 成熟度分级

**问题**：16 个 Domain 被同等对待，但 combat 和 economy 的复杂度差 10 倍。单人开发无法同时维护 16 个同等复杂度的模块。

# 第二部分：v5.0 完整架构

## 2.1 设计哲学

### 核心原则（继承）

- Feature First、Definition/Instance 分离、Rule/Content 分离、Logic/Presentation 分离
- 四级通信机制（Hook/Trigger/Observer/Message）
- 属性管线统一、数据驱动优先、组合优先于继承
- 只解决当前复杂度、官方能力优先

### 新增/修订原则

| 原则 | 说明 | 理由 |
|------|------|------|
| **渐进式 Crate 化** | 代码量达 5 万行前保持单 crate，超过后按编译热点拆分 | 避免过早 crate 化的维护成本，同时预留编译逃生通道 |
| **Domain 分级治理** | Domain 分 Core/Stable/Experimental 三级 | 单人无法同时维护 16+ 同等复杂度的模块 |
| **紧密域内直调** | 同一 Cluster 内的 Domain 允许函数调用，跨 Cluster 走 Message | D&D 反应机制要求同步执行，异步事件无法保证时序 |
| **编辑器一等公民** | 编辑器与游戏共享 Core，通过 Feature Gate 切换模式 | 编辑器是长期项目的核心资产 |
| **模拟即测试** | 数值模拟复用 Core 纯函数，Headless 模式批量验证 | D&D/BG3 级规则必须可离线验证 |
| **Mod API 版本化** | Mod API 通过 Facade 隔离 Core 内部变动 | 10 年维护意味着 Core 必然重构，API 必须稳定 |
| **Presentation 独立** | UI/渲染/音频从 Infrastructure 中独立为顶层模块 | Presentation 占代码量 30-40%，不应被埋在子目录 |

## 2.2 七层架构总览

```
┌──────────────────────────────────────────────────────────────┐
│  Layer 1: App              启动装配（Composition Root）       │
├──────────────────────────────────────────────────────────────┤
│  Layer 2: Core             游戏规则（纯领域逻辑）              │
│         ├── Primitive      原子层（L0）                       │
│         ├── Mechanism      机制层（L1）                       │
│         ├── Framework      框架层（L2）                       │
│         └── Domains        业务子系统（Cluster 组织）         │
├──────────────────────────────────────────────────────────────┤
│  Layer 3: Shared           基础能力（零业务语义）              │
├──────────────────────────────────────────────────────────────┤
│  Layer 4: Infrastructure   技术实现                          │
├──────────────────────────────────────────────────────────────┤
│  Layer 5: Content          内容桥接（加载→校验→注册）          │
├──────────────────────────────────────────────────────────────┤
│  Layer 6: Presentation     表现层（UI/渲染/音频/动画）         │
├──────────────────────────────────────────────────────────────┤
│  Layer 7: Modding          MOD 支持（Facade 隔离 Core）       │
├──────────────────────────────────────────────────────────────┤
│  Layer 8: Tools            开发工具 + 编辑器 + 模拟器          │
└──────────────────────────────────────────────────────────────┘
```

### 依赖规则

```
App          → 任意层                     ✅（仅注册，不含逻辑）
Core         → Shared                     ✅（唯一允许的外部依赖）
Shared       → 无                         ✅（叶子节点，零外部依赖）
Infra        → Core, Shared               ✅
Content      → Core, Infra, Shared         ✅
Presentation → Core(只读ViewModel), Shared  ✅
Modding      → Core(Facade), Shared, Infra, Content  ✅
Tools        → Core, Shared, Infra        ✅

严格禁止：
Core → Infra              🟥
Core → Content            🟥
Core → Presentation       🟥
Core → Modding            🟥
Shared → Core             🟥
Presentation → Content    🟥（表现层不直接访问内容加载）
```

### 三问判断法

- **Core 问题**：删掉 Bevy 换引擎，逻辑还在吗？→ 存在 → `core/`
- **Infrastructure 问题**：规则不变，能换实现吗？→ 能 → `infrastructure/`
- **Shared 问题**：不是规则也不是实现，所有模块都用的工具吗？→ 是 → `shared/`
- **Presentation 问题**：这是玩家看到/听到的东西吗？→ 是 → `presentation/`

## 2.3 Core 层内部架构

### 2.3.1 从 5 层到 3 层：为什么简化

| 83 大架构 5 层 | v5.0 3 层 | 合并理由 |
|---------------|----------|---------|
| L0 atom | **L0 Primitive** | 保留，原子数据结构 |
| L1 rule | **L1 Mechanism** | 保留，通用规则机制 |
| L2 runtime | **L2 Framework** | 合并 runtime + frame |
| L3 model | **合并入 L0 Primitive** | model 的"纯数据结构"就是 primitive 的"原子数据"，强行分开导致依赖矛盾 |
| L4 frame | **合并入 L2 Framework** | frame 的"编排框架"就是 runtime 的"执行框架"，强行分开导致依赖矛盾 |

**关键改进**：
- 消除 L3 model 不依赖 L1 但需要 L1 类型的矛盾
- 消除 L4 frame 不依赖 L2 但需要 L2 抽象的矛盾
- 认知交叉点从 5×16=80 降至 3×5=15

### 2.3.3 Cluster 通信规则

```
Cluster 内部（允许函数调用）：
  combat_cluster: combat/reaction/terrain/summon → 同步直调
  character_cluster: character/progression/spell/party → 同步直调
  world_cluster: map_play/faction/economy → 同步直调
  narrative_cluster: quest/dialogue → 同步直调
  survival_cluster: inventory/crafting/camp_rest → 同步直调

Cluster 之间（通过 Message 通信）：
  combat_cluster → character_cluster → DamageApplied / CharacterDied
  combat_cluster → world_cluster → UnitMoved / TerrainEffectTriggered
  combat_cluster → survival_cluster → ItemDestroyed (装备损坏)
  character_cluster → narrative_cluster → LevelUp / ClassChanged
  narrative_cluster → world_cluster → QuestCompleted / FactionChanged
  survival_cluster → character_cluster → ItemEquipped / BuffFromItem
```

**为什么 Cluster 内允许直调**：
- D&D 反应机制（Counterspell/机会攻击/援护）必须在伤害判定的同步路径中执行
- BG3 的战斗流程是：施法→反应检查→效果执行→死亡检查，这是一个同步链
- 异步事件无法保证时序，且调试成本极高

### 2.3.4 Domain 内部结构（弹性模板）

取代 83 大架构的强制 7 文件模板，采用弹性模板：

**最小 Domain**（如 faction、economy）：
```

```

**标准 Domain**（如 inventory、quest）：

├── plugin.rs
├── components.rs
├── events.rs
├── error.rs           # 本系统专属错误枚举

├── systems/
│   ├── mod.rs
│   └── xxx_system.rs
└── rules/             # 纯业务规则（优先纯函数）
    ├── mod.rs
    └── formulas.rs
```

**复杂 Domain**（如 combat、character）：
```
domain_name/

├── components.rs

### 2.3.5 Domain 成熟度分级

| 级别 | Domain | 约束 | 修改门槛 |
|------|--------|------|---------|
| **Core** | combat, character, attribute, tag, effect | 完整测试覆盖 + ADR 审批 | 最高 |
| **Stable** | inventory, equipment, skill, turn, map, ai | 基础测试 + 不破坏兼容 | 高 |
| **Experimental** | spell, crafting, economy, camp_rest, dialogue, quest | 快速迭代，允许重构 | 低 |

## 2.11 编译策略（50 万行+ 逃生通道）

### 渐进式 Crate 化路线

| 阶段 | 代码量 | 策略 | 目标增量编译 |
|------|--------|------|------------|
| Phase 1 | 0-5 万行 | 单 crate，src/ 内模块组织 | < 5s |
| Phase 2 | 5-15 万行 | 拆出 fre-shared crate | < 8s |
| Phase 3 | 15-30 万行 | 拆出 fre-core crate | < 10s |
| Phase 4 | 30 万行+ | Workspace + 按编译热点拆分 | < 12s |

### Crate 拆分原则

| 原则 | 说明 |
|------|------|
| 编译热点优先 | 优先拆出修改频率最低的 crate（shared → core → infra） |
| 依赖方向不变 | crate 间依赖方向与七层架构一致 |
| Feature Gate 控制 | 每个 crate 内部用 feature flag 控制可选功能 |
| 单 crate 上限 | 单个 crate 超过 8 万行时评估拆分 |

### 目标 Workspace 结构（Phase 4）

```
fre/
├── Cargo.toml                    # workspace 根
├── crates/
│   ├── fre-shared/               # Shared 层（最先拆出，零依赖）
│   ├── fre-core/                 # Core 层（纯领域逻辑，无 Bevy 依赖）
│   ├── fre-infra/                # Infrastructure 层
│   ├── fre-content/              # Content 层
│   ├── fre-presentation/         # Presentation 层
│   ├── fre-modding/              # Modding 层

│   ├── fre-editor/               # 编辑器独立二进制
│   ├── fre-simulator/            # 模拟器独立二进制
│   └── fre-validator/            # 校验器独立二进制
└── tests/
    ├── fre-integration-tests/    # 集成测试
    └── fre-replay-tests/         # 回放测试
```

## 2.12 性能优化策略

### 编译性能

| 阶段 | 代码量 | 策略 | 目标增量编译 |
|------|--------|------|------------|
| Phase 1 | 0-5 万行 | 单 crate + 增量编译 | < 5s |
| Phase 2 | 5-15 万行 | 拆出 fre-shared crate | < 8s |
| Phase 3 | 15-30 万行 | 拆出 fre-core crate | < 10s |
| Phase 4 | 30 万行+ | Workspace + 按需拆分 | < 12s |

### 运行时性能

| 模块 | 预算 | 策略 |
|------|------|------|
| 战斗结算 | ≤ 5ms/回合 | Effect Pipeline 批处理 + 纯函数 + Cluster 内直调 |
| 寻路 | ≤ 2ms/次 | BFS + 缓存软失效（120 帧） |
| 属性计算 | ≤ 0.5ms/次 | Modifier Stack 增量计算 |
| UI 刷新 | ≤ 1ms/帧 | ViewModel Change Detection |
| AI 决策 | ≤ 3ms/单位 | 策略缓存 + 剪枝 |

### 内存管理

| 场景 | 预算 | 策略 |
|------|------|------|
| 主菜单 | 64MB | 最小资源加载 |
| 战斗场景 | 256MB | 按需加载技能特效 |
| 编辑器 | 512MB | 全量加载 + 热重载 |

## 2.16 v5.0 与 83 大架构对比总览

| 维度 | 83 大架构 | v5.0 | 改进理由 |
|------|-----------|------|---------|
| Core 内部层级 | 5 层（atom/rule/runtime/model/frame） | 3 层（primitive/mechanism/framework） | 消除 L3/L4 依赖矛盾，降低 80% 认知交叉点 |
| Domain 组织 | 16 扁平 Domain，全事件通信 | 5 Cluster，Cluster 内直调/间事件 | D&D 反应机制要求同步执行 |
| mod_api 位置 | core/mod_api/ | modding/facade/ | Core 不感知 Modding，API 稳定性保障 |
| Presentation | infra/presentation/ | 独立顶层模块 | 代码量占比大，独立关注点 |
| Infrastructure | 含 anti_cheat/encryption/cloud/navmesh/physics/threading/memory | 精简为必要模块 | 移除单机不需要的模块 |
| Content schema/ | content/schema/ 定义 Def 类型 | Def 类型在 Core，Content 只做加载 | Rule/Content 分离 |
| Domain 模板 | 强制 7 文件 | 弹性模板（最小/标准/复杂） | 简单 Domain 不应被迫膨胀 |
| Shared 层 | 含 game_time/turn_count | 严格零业务语义 | 防止 Shared 退化为垃圾桶 |
| 编辑器 | tools/ 下的空壳目录 | 一等公民，完整架构设计 | 编辑器是产品核心资产 |
| 数值模拟 | battle_simulator/ 空壳 | Headless + 蒙特卡洛 + 敏感度分析 | D&D 级规则必须可离线验证 |
| 编译策略 | 无 | 渐进式 Crate 化路线 | 50 万行单 crate 编译灾难 |
| Domain 成熟度 | 无分级 | Core/Stable/Experimental 三级 | 单人无法同时维护 16+ 同等复杂度模块 |

## 2.17 版本修订说明

**v5.0**（当前版本）：

**致命缺陷修复**：
- Core 内部从 5 层简化为 3 层，消除 L3 model/L4 frame 的依赖矛盾
- Domain 按 Cluster 分组，Cluster 内允许函数调用，解决战斗核心路径的事件风暴
- Mod API 从 Core 迁移到 Modding Facade，Core 不再感知 Modding
- 新增编译策略，渐进式 Crate 化路线

**严重缺陷修复**：
- Presentation 从 Infrastructure 独立为顶层模块
- Content 层移除 schema/ 定义，Def 类型回归 Core
- Domain 弹性模板取代强制 7 文件模板
- Shared 层严格零业务语义，移除 game_time/turn_count

**新增架构**：
- 编辑器架构（一等公民，与游戏共享 Core，Feature Gate 切换）
- 数值模拟架构（Headless + 蒙特卡洛 + 敏感度分析）
- Domain 成熟度分级（Core/Stable/Experimental）
- Domain Cluster 通信规则

**精简**：
- Infrastructure 移除 anti_cheat/encryption/cloud/navmesh/physics/threading/memory
