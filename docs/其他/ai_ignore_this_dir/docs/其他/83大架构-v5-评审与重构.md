# Fre SRPG 项目架构 v5.0 — 长期连载架构设计

> **适用场景**：单人开发 + AI 协作 | 50 万行+ 代码 | 10-20 年维护 | Mod 系统 | 内置编辑器 | 数值模拟 | D&D/BG3/铃兰之剑级规则复杂度
>
> **架构宪法**：本文档优先级高于任何代码实现。当代码与本文档冲突时，优先认为代码错误。

## 一、83 大架构评审：冲突、问题与技术风险

### 1.1 核心冲突

| # | 冲突点 | 83 大架构 | 现有架构 v4.2 | 风险等级 |
|---|--------|-----------|--------------|----------|
| C1 | **Core 内部结构** | 5 层能力体系（L0 atom → L4 frame）+ 横向 Domain 矩阵 | 扁平业务模块（battle/skill/buff/character） | 🟥 致命 |
| C2 | **mod_api 放在 Core** | `core/mod_api/` 包含 gateway | Core 只依赖 Shared，Modding 是 Layer 6 | 🟥 致命 |
| C3 | **Domain 数量爆炸** | 16+ Domain（combat/ai/quest/dialogue/inventory/progression/map_play/faction/spell/party/camp_rest/crafting/summon/economy/terrain/reaction） | 12 个 Core 模块，按需扩展 | 🟨 高 |
| C4 | **Shared 层越界** | 包含 game_time、turn_count 等领域语义 | Shared 零业务语义，纯工具 | 🟨 高 |
| C5 | **Content 层职责膨胀** | 包含 schema/ 定义（attribute_def.rs 等） | Schema 定义在 Core，Content 只做加载 | 🟨 高 |

### 1.2 设计不合理之处

| # | 问题 | 说明 | 影响 |
|---|------|------|------|
| D1 | **5 层能力体系过度设计** | atom/rule/runtime/model/frame 对单人开发认知负担过重，与 "只解决当前复杂度" 原则冲突 | 开发效率下降 40%+ |
| D2 | **纵×横矩阵结构** | 纵向 5 层 × 横向 16 Domain = 80+ 交叉点，导航困难 | AI 协作时上下文爆炸 |
| D3 | **Infra 包含反作弊/加密/云服务** | 单机 SRPG 不需要 anti_cheat/encryption/cloud | 虚假复杂度 |
| D4 | **无编译策略** | 50 万行单 crate = 10 分钟+ 增量编译 | 开发体验灾难 |
| D5 | **无编辑器架构** | tools/ 下只有 mod.rs，编辑器是"正式产品"但无设计 | 核心资产缺失 |
| D6 | **无数值模拟架构** | battle_simulator/ 只有 mod.rs，无设计 | 核心需求缺失 |
| D7 | **Domain 标准结构强制统一** | 所有 Domain 强制 plugin/components/systems/events/error/rules/integration 7 文件 | 简单 Domain 被迫膨胀 |
| D8 | **Domain 间只允许事件通信** | 对紧密耦合的 combat+reaction+terrain 过于严格 | 性能瓶颈 |

### 1.3 技术风险

| # | 风险 | 触发条件 | 后果 |
|---|------|---------|------|
| R1 | **编译时间爆炸** | 代码量超过 10 万行且单 crate | 增量编译 > 30s，开发效率崩溃 |
| R2 | **Core 内部循环依赖** | L0-L4 层间 + Domain 间双向需求 | 编译失败或架构边界名存实亡 |
| R3 | **Mod API 稳定性** | Core 内部结构频繁变动导致 Mod API 不稳定 | Mod 生态无法建立 |
| R4 | **Domain 间事件风暴** | 16 Domain 全部通过事件通信 | 调试困难，性能瓶颈 |
| R5 | **单人维护 16+ Domain** | 每个 Domain 平均 3 万行 | 认知过载，迭代速度极慢 |
| R6 | **Content Schema 与 Core 类型不同步** | Content 层独立定义 schema | 运行时类型不匹配 |

## 二、架构设计原则（v5.0 修订）

### 2.1 继承的原则（不变）

- Feature First、Definition/Instance 分离、Rule/Content 分离、Logic/Presentation 分离
- 四级通信机制（Hook/Trigger/Observer/Message）
- 属性管线统一、数据驱动优先、组合优先于继承
- 只解决当前复杂度、官方能力优先

### 2.2 新增/修订的原则

| 原则 | 说明 | 理由 |
|------|------|------|
| **渐进式 Crate 化** | 代码量达 5 万行前保持单 crate，超过后按编译热点拆分 | 避免过早 crate 化的维护成本，同时预留编译逃生通道 |
| **Domain 分级治理** | Domain 分 Core/Stable/Experimental 三级，不同级不同约束 | 单人无法同时维护 16+ 同等复杂度的模块 |
| **紧密域内直调** | 同一 Cluster 内的 Domain 允许函数调用，跨 Cluster 走 Message | 避免紧密耦合域的事件风暴 |
| **编辑器一等公民** | 编辑器与游戏共享 Core，通过 Feature Gate 切换模式 | 编辑器是长期项目的核心资产 |
| **模拟即测试** | 数值模拟复用 Core 纯函数，Headless 模式批量验证 | D&D/BG3 级规则必须可离线验证 |
| **Mod API 版本化** | Mod API 通过 Facade 隔离 Core 内部变动 | 10 年维护意味着 Core 必然重构，API 必须稳定 |

## 三、七层架构（v5.0）

### 3.1 层级总览

```
┌──────────────────────────────────────────────────────────────┐
│  Layer 1: App              启动装配（Composition Root）       │
├──────────────────────────────────────────────────────────────┤
│  Layer 2: Core             游戏规则（纯领域逻辑）              │
│         ├── Capabilities    通用机制骨架（3 层，非 5 层）      │
│         └── Domains         业务子系统（Cluster 组织）         │
├──────────────────────────────────────────────────────────────┤
│  Layer 3: Shared           基础能力（零业务语义）              │
├──────────────────────────────────────────────────────────────┤
│  Layer 4: Infrastructure   技术实现                          │
├──────────────────────────────────────────────────────────────┤
│  Layer 5: Content          内容桥接（加载→校验→注册）          │
├──────────────────────────────────────────────────────────────┤
│  Layer 6: Modding          MOD 支持（Facade 隔离 Core）       │
├──────────────────────────────────────────────────────────────┤
│  Layer 7: Tools            开发工具 + 编辑器 + 模拟器          │
└──────────────────────────────────────────────────────────────┘

跨层：
  UI    → 只读 ViewModel，只输出 UiCommand
  Debug → 只读业务数据
```

### 3.2 依赖规则

```
App      → 任意层                    ✅（仅注册，不含逻辑）
Core     → Shared                    ✅（唯一允许的外部依赖）
Shared   → 无                        ✅（叶子节点，零外部依赖）
Infra    → Core, Shared              ✅
Content  → Core, Infra, Shared       ✅
Modding  → Core(Facade), Shared, Infra, Content  ✅
Tools    → Core, Shared, Infra      ✅

严格禁止：
Core → Infra              🟥
Core → Content            🟥
Core → UI                 🟥
Core → Modding            🟥
Shared → Core             🟥
Shared → Infra            🟥
```

### 3.3 三问判断法（不变）

- **Core 问题**：删掉 Bevy 换引擎，逻辑还在吗？→ 存在 → `core/`
- **Infrastructure 问题**：规则不变，能换实现吗？→ 能 → `infrastructure/`
- **Shared 问题**：不是规则也不是实现，所有模块都用的工具吗？→ 是 → `shared/`

## 四、Core 层内部架构（v5.0 核心变更）

### 4.1 设计哲学

83 大架构的 5 层能力体系（atom/rule/runtime/model/frame）对单人开发过度设计。v5.0 采用 **3 层能力 + Cluster 组织**：

- **能力层**从 5 层简化为 3 层：`Primitive`（原子）→ `Mechanism`（机制）→ `Framework`（框架）
- **Domain** 按 Cluster 分组，紧密耦合的 Domain 归入同一 Cluster，Cluster 内允许函数调用
- **Cluster 间**通过 Message 通信

### 4.3 能力层 vs 83 大架构对比

| 维度 | 83 大架构（5 层） | v5.0（3 层） | 理由 |
|------|-------------------|-------------|------|
| 层级数 | L0 atom / L1 rule / L2 runtime / L3 model / L4 frame | L0 primitive / L1 mechanism / L2 framework | model 层（纯数据结构）合并入 primitive；frame 层（抽象框架）合并入 framework |
| 认知负担 | 5 层 × 16 Domain = 80 交叉点 | 3 层 × 5 Cluster = 15 交叉点 | 降低 80% |
| Domain 组织 | 扁平列表，全部通过事件通信 | Cluster 分组，组内直调，组间事件 | 避免紧密域的事件风暴 |
| mod_api 位置 | `core/mod_api/` | `modding/facade/` | Core 不感知 Modding |

### 4.4 Cluster 通信规则

```
Cluster 内部：
  combat_cluster 内 combat/reaction/terrain/summon → 允许函数调用
  character_cluster 内 character/progression/spell/party → 允许函数调用

Cluster 之间：
  combat_cluster → character_cluster → 通过 Message（DamageApplied/CharacterDied）
  combat_cluster → world_cluster → 通过 Message（UnitMoved）
  narrative_cluster → world_cluster → 通过 Message（QuestCompleted）
  survival_cluster → character_cluster → 通过 Message（ItemEquipped）
```

### 4.6 Domain 成熟度分级

| 级别 | Domain | 约束 | 修改门槛 |
|------|--------|------|---------|
| **Core** | combat, character, attribute, tag, effect | 完整测试覆盖 + ADR 审批 | 最高 |
| **Stable** | inventory, equipment, skill, turn, map, ai | 基础测试覆盖 + 不破坏兼容 | 高 |
| **Experimental** | spell, crafting, economy, camp_rest, dialogue, quest | 快速迭代，允许重构 | 低 |

## 五、编译策略（50 万行+ 逃生通道）

### 5.1 渐进式 Crate 化路线

```
Phase 1（0-5 万行）：单 crate，src/ 内模块组织
Phase 2（5-15 万行）：拆出 shared crate（编译最频繁的零依赖层）
Phase 3（15-30 万行）：拆出 core crate（纯领域逻辑，无 Bevy 依赖）
Phase 4（30 万行+）：按编译热点拆分，每个 Cluster 可选独立 crate
```

### 5.2 Crate 拆分原则

| 原则 | 说明 |
|------|------|
| **编译热点优先** | 优先拆出修改频率最低的 crate（shared → core → infra） |
| **依赖方向不变** | crate 间依赖方向与七层架构一致 |
| **Feature Gate 控制** | 每个 crate 内部用 feature flag 控制可选功能 |
| **单 crate 上限** | 单个 crate 超过 8 万行时评估拆分 |

### 5.3 目标 Workspace 结构（Phase 4）

```
fre/
├── Cargo.toml                    # workspace 根
├── crates/
│   ├── fre-shared/               # Shared 层（最先拆出）
│   ├── fre-core/                 # Core 层（纯领域逻辑）
│   ├── fre-infra/                # Infrastructure 层
│   ├── fre-content/              # Content 层
│   ├── fre-modding/              # Modding 层
│   └── fre-app/                  # App 层（依赖所有 crate）
├── tools/
│   ├── fre-editor/               # 编辑器独立二进制
│   ├── fre-simulator/            # 模拟器独立二进制
│   └── fre-validator/            # 校验器独立二进制
└── tests/
    ├── fre-integration-tests/    # 集成测试
    └── fre-replay-tests/         # 回放测试
```

## 十二、性能优化策略

### 12.1 编译性能

| 阶段 | 代码量 | 策略 | 目标增量编译 |
|------|--------|------|------------|
| Phase 1 | 0-5 万行 | 单 crate + 增量编译 | < 5s |
| Phase 2 | 5-15 万行 | 拆出 fre-shared crate | < 8s |
| Phase 3 | 15-30 万行 | 拆出 fre-core crate | < 10s |
| Phase 4 | 30 万行+ | Workspace + 按需拆分 | < 12s |

### 12.2 运行时性能

| 模块 | 预算 | 策略 |
|------|------|------|
| 战斗结算 | ≤ 5ms/回合 | Effect Pipeline 批处理 + 纯函数 |
| 寻路 | ≤ 2ms/次 | BFS + 缓存软失效（120 帧） |
| 属性计算 | ≤ 0.5ms/次 | Modifier Stack 增量计算 |
| UI 刷新 | ≤ 1ms/帧 | ViewModel Change Detection |
| AI 决策 | ≤ 3ms/单位 | 策略缓存 + 剪枝 |

### 12.3 内存管理

| 场景 | 预算 | 策略 |
|------|------|------|
| 主菜单 | 64MB | 最小资源加载 |
| 战斗场景 | 256MB | 按需加载技能特效 |
| 编辑器 | 512MB | 全量加载 + 热重载 |

## 十五、迁移路线（从当前 v4.2 到 v5.0）

### Phase 1：Core 内部重组（2 周）

- 将现有 `core/` 扁平模块迁入 3 层能力体系（primitive/mechanism/framework）
- 创建 5 个 Cluster 目录结构
- 将现有 Domain 迁入对应 Cluster
- 保持所有现有测试通过

### Phase 2：Mod API 迁移（1 周）

- 将 `core/mod_api/` 迁移到 `modding/facade/`
- Core 不再感知 Modding
- 更新 Mod API 文档

### Phase 3：Infrastructure 精简（1 周）

- 移除 anti_cheat/encryption/cloud/navmesh/threading/memory
- 将 game_time/turn_count 从 shared 移至 core
- 合并重复模块

### Phase 4：Content 层清理（1 周）

- 将 schema/ 定义移回 Core
- Content 层只保留加载逻辑
- 更新 Content Pipeline 文档

### Phase 5：编辑器与模拟器（持续）

- 创建 tools/editor/ 和 tools/simulator/ 基础结构
- 实现战斗模拟器的 Headless 模式
- 实现编辑器的基础框架

### Phase 6：Crate 化（代码量达 5 万行时）

- 拆出 fre-shared crate
- 更新 CI 流程
