# 推荐日志层级

## ERROR

程序已经出问题。

例如：

```rust
error!(
    unit = ?unit_id,
    "Attack target not found"
);
```

出现原则：

```text
理论上不应该发生
```

例如：

* Entity丢失
* 配置缺失
* 状态非法
* 存档损坏

---

## WARN

还能继续运行。

例如：

```rust
warn!(
    skill_id = ?skill_id,
    "Skill config missing icon"
);
```

例如：

* 配置不完整
* 数据异常
* 回退逻辑触发

---

## INFO

最重要。

记录业务事件。

例如：

```rust
info!(
    unit = ?unit_id,
    "Unit moved"
);
```

或者：

```rust
info!(
    attacker = ?attacker,
    target = ?target,
    damage = damage,
    "Attack executed"
);
```

---

## DEBUG

开发调试。

例如：

```rust
debug!(
    path_len = path.len(),
    "Path calculated"
);
```

发布版关闭。

---

## TRACE

极细粒度。

例如：

```rust
trace!(
    node = ?node,
    cost = cost,
    "A* exploring node"
);
```

只在算法调试时开启。

---

# 不要这样写

很多教程：

```rust
info!("attack");
info!("move");
info!("turn");
```

这是垃圾日志。

半年后：

```text
attack

attack
attack
```

根本不知道发生了什么。

# 第一原则：日志不是领域

很多人会建：

```text
domains/
└── log/
```

这是错误的。

日志不是领域。

日志是横切关注点（Cross Cutting Concern）。

和：

```text
错误处理
监控
埋点
权限
```

属于一类。

所以不要出现：

```text
plugins/
└── logging/
```

然后到处调用。

---

正确做法

```text
battle
character
skill
buff
quest
```

各领域负责产生日志。

日志框架负责输出。

---

例如：

```rust
BattleStarted
```

属于：

```text
battle领域
```

而不是：

```text
log领域
```

---

# 第三原则：日志写在领域服务边界

很多新人会：

```rust

```

里面疯狂打日志。

这是错误的。

---

例如：

```rust

```

```rust

```

不记录。

---

真正记录的位置：

```rust

```

这里。

因为：

```text

```

了。

---

例如：

```rust

    ↓

```

记录：

```rust
info!(
    attacker=?attacker,
    target=?target,
    damage=damage,
    "UnitDamaged"
);
```

即可。

# 第四原则：一个业务动作最多一条INFO

这是最容易踩坑的。

```rust

info!("damage result");

info!("apply hp");

info!("attack end");
```

后期根本看不了。

推荐：

```rust
info!(
    attacker=?attacker,
    target=?target,
    damage=43,
    crit=true,
    "UnitAttacked"
);
```

一条。

---

DEBUG层可以细。

INFO层必须精。
