这是个非常好的问题。

很多人学习 Bevy 时关注：

```text
ECS
Plugin
Component
Event
```

但真正决定项目能不能做到：

```text
5万行
10万行
20万行
```

的是编码习惯。

实际上：

> 一个写得烂的 Bevy 项目，2万行就开始崩。
>
> 一个写得好的 Bevy 项目，20万行仍然可以维护。

下面这些是我认为 **Rust + Bevy 大型游戏项目** 最值得遵守的最佳实践。

---

# 1. 函数不要超过一个屏幕

我的经验：

```text
20~50行
优秀

50~100行
正常

100~150行
开始危险

200+
必须拆
```

例如：

坏例子：

```rust
fn attack_system(...)
{
    // 找目标

    // 检查距离

    // 检查状态

    // 计算命中

    // 计算暴击

    // 计算伤害

    // 应用伤害

    // 播放动画

    // 发事件
}
```

300行。

---

正确：

```rust
fn attack_system(...)
{
    let target = find_target(...)?;

    validate_attack(...)?;

    let result = calculate_damage(...);

    apply_damage(...);

    send_attack_event(...);
}
```

---

函数应该像目录。

而不是小说。

---

# 2. 一个函数只做一件事

不要：

```rust
fn level_up(...)
```

里面：

```text
升级

回血

学技能

播放特效

写日志

刷新UI
```

---

拆：

```rust
apply_level_up()

grant_skill()

restore_hp()

emit_level_up_event()
```

---

# 3. 系统函数越薄越好

Bevy 最大坑。

很多人：

```rust
fn battle_system(...)
```

1000行。

---

正确：

```rust
fn battle_system(...)
{
    process_turn(...);

    process_damage(...);

    process_death(...);
}
```

---

或者：

```rust
add_systems(Update,
(
    process_turn,
    process_damage,
    process_death,
));
```

---

让 ECS 调度器替你拆。

---

# 4. 避免超级 Struct

不要：

```rust
struct Character {
    hp,
    mp,
    level,
    exp,

    skills,
    inventory,
    equipment,

    buffs,

    ai,

    ...
}
```

500行。

---

拆：

```rust
CharacterIdentity

CharacterProgression

CharacterStats

CharacterSkills

CharacterEquipment
```

---

ECS 本来就是为拆分设计的。

# 5. 避免 bool 地狱

坏例子：

```rust
struct Character {
    is_dead: bool,
    is_poisoned: bool,
    is_stunned: bool,
    is_burning: bool,
}
```

---

正确：

```rust
Dead

Poisoned

Stunned

Burning
```

---

查询：

```rust
Without<Dead>
```

特别舒服。

# 6. 少用 Option<Option<Option>>

看到：

```rust
Option<Option<T>>
```

基本就该警觉了。

---

例如：

```rust

```

已经够了。

不要：

```rust
Option<Option<Weapon>>
```

# 7. 避免魔法数字

不要：

```rust

```

---

应该：

```rust
const CRIT_MULTIPLIER: f32 = 1.5;
```

---

或者：

```rust
GameBalance
```

不要：

应该：

---

# 20. 游戏项目最重要的一条

很多独立开发者失败在这里：

不要为了“优雅”而写代码。

---

例如：

```rust
trait TraitFactoryBuilderProvider
```

这种东西。

---

你未来半年后看到：

```rust
什么鬼？
```

---

最佳实践是：

```text
简单 > 优雅

明确 > 聪明

数据驱动 > 硬编码

组合 > 继承

事件 > 直接调用

模块 > 巨文件

小函数 > 大函数
```

---

如果专门针对 **Bevy 战棋 RPG（铃兰之剑、火焰纹章、FFT）**，我会再补充一个特别重要的原则：

```text
Rule（规则）
Content（内容）
Presentation（表现）
```

三者永远分离。

例如一次攻击：

```text
Rule
    计算伤害

Content
    火球术倍率150%

Presentation
    播放火球动画
```

千万不要把这三件事写在同一个函数里。

这是大型游戏项目后期还能持续扩展职业、技能、地图、剧情而不崩盘的核心习惯。
