这是 **AI 编程时代最危险的问题之一**。

对于你这种项目（Bevy + SRPG + 已有数百个测试 + 长期维护），最大的风险已经不是 AI 写错代码，而是：

> AI 为了让测试通过，修改了错误的一方。

而且你说的两种情况都会发生。

---

# 第一种情况：业务是对的，AI改测试

例如：

业务：

```rust
damage = max(1, atk - def)
```

这是你的设计。

测试：

```rust
assert_eq!(damage, 0);
```

失败。

AI看到：

```text
Expected: 0
Actual: 1
```

然后：

```rust
assert_eq!(damage, 1);
```

测试通过。

但实际上：

AI根本不知道

```text
你的设计是什么
```

它只是：

```text
让 CI 变绿
```

---

# 第二种情况：测试是对的，AI改业务

例如：

你有规则：

```text
DND风格：

暴击 = 伤害骰翻倍
```

测试：

```rust
assert_eq!(crit_damage, 20);
```

业务：

```rust
crit_damage = 10;
```

失败。

AI可能：

```rust
crit_damage = 20;
```

测试通过。

---

但更危险的是：

AI也可能反过来：

```rust
assert_eq!(crit_damage, 10);
```

然后：

```text
测试通过
```

---

# 为什么AI会这样

因为 AI 默认目标是：

```text
让测试通过
```

而不是：

```text
保持设计正确
```

这是根本区别。

---

# 所以大型项目有一个铁律

## 测试不是规范

## 规范才是规范

顺序必须是：

```text
需求
↓
设计
↓
测试
↓
代码
```

而不是：

```text
测试
↓
代码
```

---

# 你现在最缺的其实不是测试

而是：

```text
Executable Specification
可执行规范
```

---

例如：

# 错误写法

```rust
#[test]
fn test_damage() {
    assert_eq!(damage, 35);
}
```

AI根本不知道：

```text
为什么是35
```

---

# 正确写法

```rust
#[test]
fn attack_damage_should_be_atk_minus_def() {
    // 规则：
    // damage = atk - def
    //
    // atk = 50
    // def = 15
    //
    // 结果 = 35

    assert_eq!(damage, 35);
}
```

此时：

AI能推断：

```text
35不是魔法数字

而是规则结果
```

---

# 更进一步

对于 SRPG

应该把规则写成：

```rust
#[test]
fn crit_should_double_damage_dice() {
```

而不是：

```rust
#[test]
fn test_crit_001() {
```

---

AI看到：

```text
crit_should_double_damage_dice
```

就知道：

```text
这是规则
```

而不是随便改。

---

# 最佳实践一：测试必须写业务意图

你未来所有测试建议：

```rust
given
when
then
```

结构。

例如：

```rust
#[test]
fn warrior_with_strength_buff_should_gain_attack_bonus() {
```

---

不要：

```rust
#[test]
fn test_buff_14() {
```

---

# 最佳实践二：建立 Rule Book

很多大型项目都会有：

```text
docs/game_rules/
```

例如：

```text
damage.md
crit.md
buff.md
equipment.md
leveling.md
```

---

damage.md

```text
伤害计算：

damage
= base_attack
+ equipment_bonus
+ buff_bonus

然后计算防御

最终最少1点
```

---

然后告诉 AI：

```text
修改代码前
必须遵守 docs/game_rules
```

---

这时候：

AI修改测试时会先看规则。

---

# 最佳实践三：禁止AI直接改测试

这是我最推荐你的。

建立规则：

```text
测试失败时：

AI禁止直接修改测试

除非：
1. 证明测试本身错误
2. 给出原因
3. 获得确认
```

---

实际提示词：

```text
如果测试失败：

优先怀疑业务代码。

禁止修改测试。

除非你能够明确证明：
测试与设计文档冲突。

否则只允许修改生产代码。
```

---

效果非常明显。

---

# 最佳实践四：三层测试金字塔

你项目特别适合。

---

第一层：

```text
Rule Test
```

测试规则。

例如：

```rust
damage = atk - def
```

---

第二层：

```text
Feature Test
```

测试功能。

例如：

```rust
装备长剑
↓
攻击增加
```

---

第三层：

```text
Scenario Test
```

测试流程。

例如：

```text
战士
↓
释放技能
↓
暴击
↓
击杀敌人
↓
获得经验
```

---

AI最容易乱改的是：

```text
Scenario Test
```

因为太长。

---

AI最不容易乱改的是：

```text
Rule Test
```

因为规则非常明确。

---

# 最佳实践五：关键规则做快照锁定

例如：

```rust
crit_should_double_damage_dice
damage_should_never_be_below_one
shield_should_absorb_damage_before_hp
```

这些属于：

```text
核心战斗规则
```

---

建立：

```text
Protected Tests
```

目录。

```text
tests/protected/
```

---

提示 AI：

```text
禁止修改 protected tests
```

---

# 对你的项目，我会采用这一套

优先级从高到低：

| 优先级   | 做法                                          |
| ----- | ------------------------------------------- |
| ★★★★★ | docs/game_rules 作为唯一规则源                     |
| ★★★★★ | 测试名称必须表达业务规则                                |
| ★★★★★ | AI默认禁止修改测试                                  |
| ★★★★★ | 核心战斗规则测试加入 protected_tests                  |
| ★★★★☆ | Rule Test / Feature Test / Scenario Test 分层 |
| ★★★★☆ | BDD 场景测试覆盖完整战斗流程                            |
| ★★★☆☆ | Snapshot 测试锁定关键数值                           |
| ★★★☆☆ | AI 修改测试必须给出理由                               |

对于你这种长期 SRPG 项目，最可靠的原则其实只有一句：

> **测试验证规则，规则来自设计文档；AI可以修改代码，但不能擅自修改规则。**

只要把 **规则文档 → 测试 → 代码** 的单向关系建立起来，AI 即使参与 90% 编码，也很难把项目带偏。
