对于你这种项目（Bevy 0.18 + SRPG + 长期连载 + 已有377个单测 + 大量Feature模块），代码评审千万不要搞成：

* 看代码风格
* 看命名
* 看格式

这种都是末位指标。

真正的大型项目代码评审，应该按照：

> 架构正确性 → 可维护性 → 扩展性 → 调试性 → 测试性 → 性能 → 风格

排序。

---

# 一级：架构评审（最重要）

权重：★★★★★

这是决定项目能不能活5年的东西。

## 1. 是否符合 Feature First

### 好

```text
battle/
 ├─ damage/
 ├─ turn/
 ├─ skill/
```

### 坏

```text
components/
systems/
events/
```

检查：

* 是否按业务拆分
* 是否出现技术分类目录
* 是否出现超级模块

---

## 2. 是否违反模块边界

检查：

```rust
battle
  ↓
inventory
```

是否直接修改：

```rust
inventory.gold -= 100;
```

而不是：

```rust
EquipItemMessage
```

或者：

```rust
inventory::equip()
```

---

评审问题：

> Battle 是否知道 Inventory 内部实现？

如果知道：

扣分

---

## 3. 是否出现 God Object

危险信号：

```rust
BattleManager
GameManager
CharacterManager
DataManager
```

超过：

```rust
1000+
```

行基本必炸。

---

# 二级：ECS评审

权重：★★★★★

---

## 4. Component是否纯数据

好：

```rust
struct Health {
    current: i32,
    max: i32,
}
```

坏：

```rust
impl Health {
    fn apply_damage(...)
}
```

---

问题：

> 行为是不是开始跑进Component？

---

## 5. 是否滥用Resource

危险：

```rust
Resource
 ├─ CharacterManager
 ├─ InventoryManager
 ├─ SkillManager
```

这已经退化回OOP。

---

合理：

```rust
SelectedUnit
HoveredTile
GameConfig
```

---

## 6. Hook / Observer 是否正确使用

检查：

### Hook

是否用于：

```rust
Dead
Equipment
Buff
```

这种组件生命周期。

---

### Observer

是否用于：

```rust
死亡动画
任务更新
UI刷新
```

---

危险：

```rust
位置变化
AP变化
```

每帧触发 Observer

容易风暴。

---

## 7. Query职责是否清晰

看到：

```rust
Query<(
A,B,C,D,E,F,G,H,I,J,K,L
)>
```

先警觉。

---

再问：

职责是否单一？

如果：

```rust
AttackExecutionSystem
```

可能合理。

---

如果：

```rust
GameSystem
```

直接扣分。

# 三级：SRPG领域评审

权重：★★★★★

这是你项目最核心的部分。

---

## 8. 是否出现重复规则系统

检查：

装备加攻击

Buff加攻击

职业加攻击

天赋加攻击

是否都走：

```rust
Modifier Pipeline
```

---

危险：

```rust
equipment.rs
buff.rs
talent.rs
```

各算各的。

后期必崩。

---

## 9. Definition / Instance是否分离

检查：

### Definition

```rust
KnightDefinition
```

### Instance

```rust
CharacterStats
```

---

危险：

```rust
job.attack += 5;
```

运行时修改职业配置。

---

## 10. Trait体系是否统一

检查：

种族

职业

装备

Buff

天赋

是否进入：

```rust
Trait
Modifier
```

统一体系。

---

如果：

```rust
RaceSystem
JobSystem
TalentSystem
BuffSystem
```

五套独立机制

扣大分。

# 最终评审优先级（推荐）

按照这个顺序审：

| 顺序 | 评审内容      | 权重    |
| -- | --------- | ----- |
| 1  | 架构边界      | ★★★★★ |
| 2  | ECS设计     | ★★★★★ |
| 3  | SRPG规则统一性 | ★★★★★ |
| 4  | 扩展性       | ★★★★★ |
| 5  | 测试体系      | ★★★★☆ |
| 6  | 调试能力      | ★★★★☆ |
| 7  | 性能        | ★★★☆☆ |
| 8  | 命名与格式     | ★★☆☆☆ |

对于你的项目，我会额外增加一个专属评审项：

| 项目                  | 问题                                  |
| ------------------- | ----------------------------------- |
| Trait体系             | 是否已经统一职业/装备/Buff/天赋                 |
| Modifier体系          | 是否所有属性修改都走同一管线                      |
| BattleRecord        | 是否已成为战斗真相来源（Single Source of Truth） |
| Definition/Instance | 是否彻底隔离                              |
| Message/Observer    | 是否形成清晰的战斗链路                         |
| 数据驱动                | 新增职业/技能是否无需改核心代码                    |

这6项比代码风格、命名、格式加起来都重要。对于长期维护的 SRPG，它们决定的是项目后期会不会演变成“规则地狱”。
