第一处优化
增加术语定义区（Glossary）

例如：

# 术语定义

## CombatIntent

表示攻击意图。

不代表已经发生攻击。

仅表示：

谁
使用什么技能
攻击谁

---

## PendingEffect

表示待执行效果。

不代表已经生效。

必须经过：

Modify
Execute

阶段。

---

## EffectQueue

战斗唯一效果缓冲区。

禁止直接执行效果。

所有效果必须进入队列。

原因：

AI最喜欢犯的错误：

把：

CombatIntent

当成：

DamageEvent

处理。

实际上完全不是一回事。

大型项目一定要有：

术语定义

章节。

第二处优化
增加状态机章节

例如：

# 战斗生命周期

Idle

↓

IntentGenerated

↓

EffectsGenerated

↓

EffectsModified

↓

EffectsExecuted

↓

MessagesSent

↓

Completed

AI非常擅长：

乱跳阶段。

例如：

Generate 阶段直接扣血。

Modify 阶段发送死亡消息。

如果没有状态机：

AI经常犯这种错误。

第三处优化
增加 Invariants（不变量）

这是领域文档最重要的部分。

很多人完全没有。

例如：

# 不变量

## EffectQueue

任意时刻：

Execute结束后

pending必须为空。

---

## HP

HP允许小于0。

死亡判断：

依赖 DeadTag。

禁止：

HP <= 0
直接删除实体。

---

## Damage

所有伤害：

必须经过：

Generate
Modify
Execute

禁止绕过管线。

这部分价值极高。

AI重构最怕：

偷偷绕过规则。

# 禁止事项

禁止：

禁止：

禁止：

直接修改DerivedStat

# Battle Domain Boundary

第六处优化
代码示例降级

你现在：

pub struct EffectQueue {
    pub pending: Vec<PendingEffect>,
}

这种代码写太多。

其实未来最容易过期。

推荐：

EffectQueue

职责：

保存待执行效果。

结构：

pending: PendingEffect集合

要求：

Execute阶段必须消费完毕。

因为：

代码会变。

规则不会变。

这个是 AI 项目特有的。

# AI 修改规则

禁止：

---

优先检查：

Domain Rules

Implementation

最后：

Test
