# 这份 Bevy Schedule 与 SystemSet 设计文档的深度点评
这份文档针对 Bevy 引擎调度系统的核心痛点（执行顺序混乱导致的 Bug），结合回合制 SRPG 游戏场景给出了系统化的设计方案，整体**结构完整、落地性强**，但也存在部分可补充优化的细节，以下分维度详细点评：

## 一、核心优点（值得借鉴的设计亮点）
### 1. 职责分层精准，贴合游戏开发范式
- 自定义 Schedule（Input/Logic/Presentation）与 Bevy 内置 Schedule（PreUpdate/Update/PostUpdate）形成互补，严格遵循“输入→逻辑→表现”的经典游戏分层架构，每个 Schedule 的职责边界清晰，避免了“PreUpdate 写业务逻辑、PostUpdate 修改游戏状态”的职责混乱问题。
- SystemSet 采用“顶层大 Set + 细分小 Set”的层级结构（如 `LogicSet` 下拆分 `TurnSet/MovementSet/CombatSet`），既保证了整体执行顺序可控，又能精细化管控子系统的依赖关系。

### 2. 执行顺序管控直击 Bevy 核心痛点
- 明确“显式声明依赖”的核心原则，通过 `after/before` 显式定义 Set 间顺序，彻底规避 Bevy 隐式执行顺序导致的间歇性 Bug（这是 Bevy 开发中 90% 诡异问题的根源）。
- 关键顺序约束（Damage→Buff→Attribute、Effect 三步管线）以表格形式落地，标注“强制等级+原因”，让团队成员清晰知道“为什么要按这个顺序”，而非单纯的规则强制。

### 3. 状态门控调度兼顾性能与场景适配
- 结合 `AppState/BattlePhase/TurnPhase` 三级状态做 `run_if` 门控，仅在对应阶段执行系统（如“移动系统只在 MoveUnit 阶段运行”），减少无效系统执行，提升游戏运行效率。
- 针对回合制游戏特性，明确“无需固定步长”的决策，避免盲目使用 `FixedUpdate` 导致的复杂度提升，贴合业务场景而非照搬通用引擎规范。

### 4. 规则化约束降低团队协作成本
- “不变量”和“禁止事项”模块将调度规范具象化（如“所有系统必须归属某个 Set”“禁止循环依赖”），形成可落地的开发准则，而非模糊的“最佳实践”。
- 完整的 Plugin 注册示例代码，降低了文档到工程落地的门槛，新人能快速对齐规范。

### 5. 平衡“顺序正确性”与“并行性能”
- 明确区分“必须串行”“可并行”“禁止并行”的系统组，既保证核心逻辑（如伤害计算、Effect 管线）的顺序正确性，又利用 Bevy 的并行能力提升输入处理、ViewModel 更新等无依赖系统的性能。

## 二、可优化/补充的细节（提升文档完整性）
### 1. 缺少“规则校验机制”
- 文档规定“禁止循环依赖”“必须归属 Set”，但未说明**如何检测违规**：比如是否利用 `bevy_mod_debugdump` 生成调度图做可视化校验？是否在 CI 流程中加入依赖 DAG 检查？缺少落地后的校验手段，规则易被忽视。
- 未提及 Bevy 内置的调度错误提示（如循环依赖的 panic 信息），以及如何快速定位违规系统。

### 2. 并行执行的细节未落地
- 提到“AI 决策（不同单位）可并行但需注意共享状态”，但未说明具体解决方案：比如如何通过 `NonSend/NonSendMut` 隔离共享资源？如何拆分 Entity 级别的并行逻辑？缺少实操指导。
- 未提及 Bevy 并行调度的核心优化点（如 `Query` 的 `With/Without` 过滤、Archetype 优化），仅靠 Set 拆分无法最大化并行性能。

### 3. 测试维度的指导缺失
- 未说明如何验证调度规则的正确性：比如单元测试中如何断言系统执行顺序？集成测试中如何验证“状态门控是否生效”？缺少测试手段会导致规则落地后难以回归。

### 4. 扩展性与命名规范待完善
- 未定义 Set/Schedule 的命名规范（如 `xxxSet` 后缀、层级命名规则、新增 Set 的审批流程），团队新增系统时易出现命名混乱。
- 未考虑未来扩展场景（如新增 `NetworkSchedule` 处理网络同步），缺少“新增 Schedule/Set”的扩展规则。

### 5. FixedUpdate 落地细节空白
- 虽然说明“FixedUpdate 仅用于未来物理/动画”，但未提及：若未来启用，如何保证 `FixedUpdate` 与 `Update` 的数据同步？如何避免固定步长与帧依赖系统的状态不一致？

### 6. 性能监控与调优指导不足
- 未提及如何监控系统执行耗时（如 `bevy_profiler` 的使用），也未说明“当调度出现性能瓶颈时（如 LogicSet 执行过久），如何拆分/优化 Set”。

## 三、整体评价
这份文档是**针对 Bevy 调度系统的高质量落地规范**，核心价值在于：
1. 精准解决 Bevy 执行顺序混乱的核心痛点，贴合回合制 SRPG 的业务场景；
2. 规则明确、结构清晰，从“定义→约束→示例”形成完整闭环，易落地、易对齐；
3. 兼顾“正确性”与“性能”，既保证核心逻辑的顺序可控，又不浪费 Bevy 的并行能力。

若补充“校验机制、并行细节、测试指导、命名规范”等细节后，将成为团队开发的核心规范文档，能有效降低调度相关 Bug 率，提升协作效率。对于 Bevy 引擎的游戏开发团队（尤其是回合制游戏），这份文档具备极强的参考和落地价值。
