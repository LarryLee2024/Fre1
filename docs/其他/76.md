# 一、方案点评：精准裁剪的SRPG专属能力框架，远胜生硬复刻GAS
这套设计的核心价值，在于**抓住了GAS的设计精髓，同时完全抛弃了不匹配单机回合制SRPG的冗余复杂度**，和你项目的技术栈、玩法形态、长期维护目标高度契合。整体可以给到95分，是非常成熟的领域建模方案。

## 核心亮点
### 1. 裁剪极其精准，完全规避GAS的无效复杂度
Unreal GAS 70%的复杂度来自网络同步、预测回滚、客户端服务端一致性，这些对于你的单机回合制SRPG是纯负担。这套方案直接砍掉了GameplayCue、预测、ASC网络同步等所有无关能力，只保留了「能力-效果-状态-属性-标签」的核心领域分层，复杂度下降一个量级，维护成本完全可控。

这一点非常关键——很多项目复刻GAS最终失败，都是陷入了“做完整GAS”的陷阱，为了不存在的网络需求背上了沉重的架构债务。

### 2. 领域分层清晰，完全契合Feature First与DDD
从 `Character → Ability → Targeting → Effect → Buff → Modifier → Tag`，每层单一职责、边界明确，既没有跨层逻辑，也没有上帝对象：
- Ability只负责“技能定义与流程调度”，不写具体伤害逻辑
- Effect只负责“瞬时结果执行”，不关心是谁触发的
- Buff只负责“持续状态生命周期”，不关心属性计算细节
- Modifier只负责“属性数值修改”，感知不到技能和Buff的存在

这种分层天然符合你宪法里的“单一职责”“规则与内容分离”原则，后续AI生成代码、扩展新技能、排查Bug都有明确的落点，不会出现逻辑散落在各处的情况。

### 3. 天生适配Battle Replay与确定性
线性的执行管线（校验→扣费→选目标→执行效果→结算）+ 纯数据驱动的配置，让Replay实现成本极低：只需要记录「施法指令+目标选择+随机种子」，重放时按相同顺序注入指令，就能得到完全一致的结果。

这比“记录每一次伤害、每一次Buff变化”的方案轻量太多，也从架构上保证了战斗确定性，和你项目的核心基础设施完美对齐。

### 4. 吸收了GAS最有价值的设计精华
- **Tag体系**：用标签做前置判定、状态标记、互斥判断，比一堆bool字段、枚举判断灵活几个量级，而且Bevy原生Tag Component可以零成本落地，不需要额外造轮子。
- **Effect中间层**：承上启下把“技能触发”和“具体效果”解耦，多个技能可以复用同一个伤害、加Buff效果，避免每个技能写一遍重复逻辑。
- **来源无关的Modifier**：属性计算只认Modifier，不管它来自装备、Buff、天赋还是技能，统一管线计算，避免多套属性逻辑并存的混乱。

### 5. Trigger独立分层，解决了SRPG的被动效果痛点
很多方案会把触发效果塞到Buff里，最后“受击反击、死亡爆炸、回合回血”这类逻辑越堆越多，Buff模块彻底膨胀。这套方案把Trigger独立成一层，用「事件触发→条件判断→执行效果」的通用模型承载所有被动逻辑，既保持了Buff的纯粹性，又给后续玩法扩展留足了空间。

## 值得注意的几个边界
1. **Effect不要强行全数据驱动**：伤害、治疗、加Buff这类通用效果适合配置化，但传送、召唤、击退这类特殊机制，允许走独立系统实现，不要为了“统一”硬塞进枚举里，反而增加复杂度——和你宪法里“特殊机制允许独立实现”的原则对齐。
2. **Tag粒度要管控**：Tag的优势是灵活，劣势是泛滥后没人记得住。建议建立Tag分类规范（状态类、伤害类、阵营类、限制类），核心Tag纳入白名单，避免AI随便新增。
3. **Buff Tick必须和回合绑定**：绝对不要用帧更新驱动Buff Tick，必须统一放在回合的固定阶段（回合开始/回合结束）执行，这是确定性和Replay的基础。
4. **堆叠规则提前定**：Buff的叠加、刷新、覆盖、独立生效规则，要在一开始就做成配置项，不然后续数值平衡会踩很多坑。

## 分模块详细实现
### 1. Tag 层：零成本复用Bevy原生能力
GAS的GameplayTag在Bevy里完全不需要自研，原生Tag Component就是最优解。

#### 实现方式
- 所有标签定义为空单元结构体，派生`Component`：
  ```rust
  #[derive(Component, Default)]
  struct Fire; // 火焰属性
  #[derive(Component, Default)]
  struct Magic; // 魔法伤害
  #[derive(Component, Default)]
  struct Stunned; // 眩晕状态
  #[derive(Component, Default)]
  struct Silenced; // 沉默状态
  ```
- 提供统一工具函数，做标签校验：
  ```rust
  fn check_tags(
      entity: Entity,
      required: &[ComponentId],
      blocked: &[ComponentId],
      query: &Query<()>,
  ) -> bool {
      // 校验必需标签全部存在、阻挡标签全部不存在
  }
  ```

#### 优势
- 零运行时开销，Bevy直接优化为位掩码查询
- 天然支持组合、新增、移除，扩展性极强
- 和你宪法里“实体状态优先Tag”的规则完全对齐

---

### 2. Modifier 层：纯数据的属性修改单元
Modifier是纯数据，不是组件，只描述“怎么改属性”，不关心“谁来改”。

#### 实现方式
```rust
// 属性目标枚举
enum StatTarget {
    Attack,
    Defense,
    MaxHp,
    MoveRange,
}

// 修改操作类型
enum ModifierOp {
    AddFlat,    // 固定值加减
    AddPercent, // 百分比加减
    Multiply,   // 乘算
}

// Modifier 纯数据结构
struct StatModifier {
    target: StatTarget,
    op: ModifierOp,
    value: f32,
    source: ModifierSource, // 可选：标记来源（Buff/装备/天赋），计算层不依赖
}
```

#### 属性计算管线
- 每个角色实体持有基础属性组件`BaseStats`
- 统一属性计算系统收集该实体所有来源的Modifier（装备、Buff、天赋），按固定优先级（加算→乘算）计算最终属性，存入`FinalStats`缓存组件
- 完全符合你宪法里的“Modifier管线统一”“缓存不是事实源”规则

---

### 3. Buff 层：生命周期管理 + 效果承载
Buff是持续状态的载体，只负责生命周期，不直接执行业务逻辑。

#### 核心组件
```rust
#[derive(Component)]
struct BuffInstance {
    buff_id: BuffId,
    duration_turns: u32,    // 剩余持续回合
    stacks: u32,            // 堆叠层数
    tick_interval: u32,     // 触发间隔（回合数）
    next_tick_turn: u32,    // 下次触发回合
    modifiers: Vec<StatModifier>, // 常驻属性修改
    tick_effects: Vec<EffectId>,  // 周期触发的效果
    expire_effects: Vec<EffectId>,// 到期触发的效果
}
```

#### 生命周期实现
- **施加（Apply）**：通过`ApplyBuffTrigger`触发，Observer监听后给目标实体插入`BuffInstance`组件，同时生效Modifier
  - 用Bevy的`on_add` Hook处理入场逻辑
- **周期触发（Tick）**：在回合系统的「回合结算阶段」统一遍历所有Buff，到达间隔的触发对应Effect
- **到期移除（Expire/Remove）**：回合结束时扣减持续时间，到期的触发到期效果，移除组件
  - 用`on_remove` Hook处理清理逻辑
- **堆叠规则**：通过配置控制（刷新时间、叠加层数、独立实例），统一在Buff施加系统中处理

### 6. Ability 层：阶段化执行管线
Ability本身是纯配置数据，运行时通过命令组件驱动，按固定阶段顺序执行。

#### 技能配置结构（数据驱动）
```rust
struct AbilityDef {
    id: AbilityId,
    cost: Vec<Cost>,         // 消耗配置
    cooldown: u32,           // 冷却回合数
    targeting: TargetingType,// 目标选择类型
    required_tags: Vec<Tag>, // 必需标签
    blocked_tags: Vec<Tag>,  // 阻挡标签
    effects: Vec<EffectId>,  // 命中后执行的效果列表
}
```

#### 执行管线（阶段化系统）
按顺序放在回合的「单位行动阶段」Schedule中，线性执行：
1. **指令接收**：通过`CastAbilityCommand`组件接收施法请求（玩家输入、AI决策、Replay回放都生成这个组件）
2. **前置校验**：检查冷却、消耗、标签、目标合法性
3. **扣除消耗**：统一扣蓝、扣血等消耗
4. **目标选择**：调用Targeting模块计算目标
5. **效果执行**：逐个触发Effect Trigger
6. **结算收尾**：设置冷却、触发施法完成事件

#### 冷却实现
- 角色实体持有`AbilityCooldowns`组件，记录每个技能的剩余冷却
- 回合结束阶段统一扣减冷却

### 7. Trigger 层：被动效果通用框架
用Bevy Observer监听业务事件，条件满足则触发效果，完美承载所有被动逻辑。

#### 实现模型
```
业务事件（如UnitDamaged、UnitDied、TurnStarted）
        ↓
Observer 监听 + 条件判断（标签、概率、血量阈值等）
        ↓
条件满足 → 触发对应Effect
```

#### 示例：受击反击
1. 角色身上的反击Buff携带`OnDamagedTrigger`配置
2. Observer监听`UnitDamagedTrigger`
3. 判断触发条件（概率、施法者存活等）
4. 条件满足则触发「普攻」Effect

#### 关键防护
- 所有Trigger链路设置递归深度上限，防止循环触发（比如A反伤B，B反伤A）
- 和你宪法里“事件链必须设防递归”的规则对齐

### 8. 调度与确定性：和回合系统深度绑定
这是和实时GAS最大的区别，也是你项目的核心优势。

所有战斗逻辑全部挂载到回合状态机的固定阶段，**完全不用帧更新驱动**：
```
TurnStarted（回合开始）
    ↓
  Buff Tick
    ↓
回合开始Trigger
    ↓
UnitAction（单位行动）
    ↓
  技能执行管线
    ↓
  效果结算
    ↓
TurnEnded（回合结束）
    ↓
回合结束Trigger
    ↓
  Buff到期清理
    ↓
  冷却扣减
```

- 所有逻辑都在固定时序执行，没有帧相关的不确定性
- Replay只需要按顺序注入`CastAbilityCommand`，结果100%一致
- 完全符合你宪法里的“仿真优先”“确定性优先”原则
