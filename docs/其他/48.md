但在 Bevy ECS 的多线程调度器和 Rust 语言特性的实际碰撞中，文档中有几个 “隐性的性能与崩溃陷阱” 必须通过架构手段提前规避。以下是深度点评与避坑指南：

一、 🌟 核心亮点（为什么这套规范能撑起工业级 SRPG？）

极其清醒的“浮点禁令”与“先乘后除”
    明确禁止核心战斗数值使用 f32/f64，并强制 value * multiplier / 100。这是跨平台（PC/移动端/Web）和跨架构（x86/ARM）确定性的绝对底线。很多项目死于不同 CPU 对浮点指令的微小舍入差异，这份文档在 Day 1 就封死了这条路。
ECS 迭代顺序的“显式排序”铁律
    敏锐地指出了 Bevy Query 默认迭代顺序不确定的问题，并给出了基于 Entity ID 和 Initiative 的稳定排序方案。这是保证“同回合多个死亡判定”和“AOE 伤害遍历”结果一致性的唯一正解。
状态哈希（State Hash）的“分歧检测”机制
    在 TurnEnd 计算包含所有关键属性的哈希，并与 Replay 文件对比。这使得“回放失败”不再是一个黑盒，而是能精确定位到“第几回合、哪个属性”出现了分歧，极大降低了 QA 和程序的调试成本。
SystemSet 的严格时序约束
    明确划分了 Input -> Command -> Logic -> Effect -> ViewModel -> UI 的执行管线，并禁止同一 Set 内的隐式依赖。这保证了逻辑的纯粹性和 UI 渲染的稳定性。

二、 ⚠️ Bevy 落地时的 4 个致命陷阱与重构方案

🟥 致命陷阱：BattleRng 的 ResMut 借用冲突导致“多线程瘫痪”
文档现状：BattleRng 作为全局唯一的 Resource，所有战斗随机（暴击、闪避、Buff 触发）必须通过 &mut BattleRng 获取。
Bevy 陷阱：在 Bevy 中，如果多个 System 都需要 ResMut<BattleRng>，它们绝对无法并行执行。在 SRPG 后期的 AOE 结算或海量 Buff 触发时，如果每个伤害判定 System 都要独占写锁，整个战斗逻辑会被迫完全串行化，彻底丧失 ECS 的多线程性能优势。
重构方案：引入“随机流（Random Stream）”或“派生种子（Derived Seed）”
    方案 A（预生成随机流，推荐）：在回合开始或 Command 生成时，一次性从 BattleRng 生成该阶段所需的所有随机数（如 Vec<u32>），存入 Res<RandomStream>。战斗 System 只需要 Res<RandomStream>（只读锁）和当前的 ActionIndex 来获取随机数，从而实现完美并行。
    方案 B（派生局部种子）：使用可并行的 PRNG（如 rand_chacha）。为每个 Entity 或每个 Action 分配一个基于全局种子派生的局部种子（如 local_seed = hash(global_seed, entity_id, action_index)）。System 在本地实例化 PRNG，无需共享全局状态。

🟥 致命陷阱：Rust 整数运算的“溢出 Panic”
文档现状：规定使用 i32 或 u32，公式为 let damage = (attack * 100 / defense).max(1);。
Rust 陷阱：在 Rust 的 Debug 模式下，整数溢出会直接 Panic（崩溃）！即使在 Release 模式下也会发生 Wrap around（环绕截断）。如果后期数值膨胀（如 attack = 50,000），attack * 100 * crit_multiplier 极易突破 i32::MAX (21亿)，导致游戏在开发期频繁崩溃，或在 Release 期出现“伤害变成负数”的诡异 Bug。
重构方案：中间变量强制提升为 i64
    在核心公式计算时，必须将中间变量提升为 i64，计算完成后再截断回 i32。
    // ✅ 正确：安全的防溢出计算
        let damage = ((attack as i64 * 100 * crit_mult as i64) / defense as i64).max(1) as i32;
        
    或者引入 checked_mul / saturating_mul 来保证数值边界安全。

Entity ID 排序的“跨平台/跨存档”不确定性
文档现状：排序依据大量依赖 Entity ID（如“相同 Initiative 时按 Entity ID 升序”）。
Bevy 陷阱：Bevy 的 Entity ID 是由内部计数器生成的（包含 Index 和 Generation）。它不是确定性的！ 如果玩家 A 和玩家 B 加载同一个存档，但由于加载 Asset 的异步时序微小差异，导致 Entity 生成的顺序不同，他们的 Entity ID 就会不同。此时依赖 Entity ID 排序会导致两人的行动队列不一致，直接引发状态分歧（Desync）。
重构方案：引入“确定性逻辑 ID (Logical ID)”
    为每个参与战斗的 Entity 分配一个确定性的、随存档/Replay 保存的 LogicalId（如 u32，在战斗开始时按加载顺序从 0 递增分配，或基于 Unit 的数据库 ID）。
    所有排序（行动队列、死亡判定、AOE 遍历）必须使用 LogicalId 而不是 Bevy 的 Entity。

状态哈希（State Hash）的“遗漏与顺序”陷阱
文档现状：compute_state_hash 遍历 query.iter() 和 buff_query.iter() 来写入哈希。
Bevy 陷阱：
    顺序不一致：如前所述，query.iter() 的顺序是不确定的。如果两次计算哈希时，Entity 的遍历顺序不同，即使数据完全一样，算出的哈希值也会不同！
    遗漏隐藏状态：如果漏掉了某个影响战斗的 Component（如 Cooldown、GridPosition），会导致“哈希一致但实际状态分歧”的假阳性。
重构方案：排序后哈希 + 宏驱动生成
    在计算哈希前，必须先将查询结果收集到 Vec 中并按 LogicalId 排序，然后再遍历写入 Hasher。
    使用 Rust 的派生宏（如 #[derive(DeterministicHash)]）自动为所有战斗相关的 Component 生成哈希代码，防止人工遗漏字段。

三、 针对 SRPG 项目的特殊补充
规范维度   SRPG 专项建议   理由
寻路 (Pathfinding)   寻路算法（如 A*）在遇到“代价相同的多条路径”时，必须引入确定性的 Tie-breaker（平局打破机制）。   如果 A* 在 (x+1, y) 和 (x, y+1) 之间犹豫，不同平台的实现可能会选出不同的路径，导致单位走位分歧。必须规定“优先 X 轴”或“优先 Y 轴”。

空间查询 (Spatial Query)   范围技能（AOE）的目标获取，禁止依赖物理引擎（如 bevy_xpbd）的 Broad-phase 返回顺序。   物理引擎的碰撞检测顺序通常是不确定的。必须在获取 AOE 范围内的 Entity 后，强制按 LogicalId 排序再执行伤害结算。

浮点坐标 (Transform)   明确区分“逻辑网格坐标 (GridCoord: IVec2)”和“渲染世界坐标 (Transform: Vec3)”。   战斗逻辑（移动、射程）绝对只能使用 GridCoord。Transform 仅用于表现层插值。禁止用 Transform.translation.distance() 来计算射程！

四、 文档结构微调建议

建议在“2.5 状态哈希”之后，增加一个 《浮点污染隔离墙 (Float Contamination Firewall)》 章节：

物理隔离：战斗逻辑 System（LogicSet）的代码中，禁止出现 f32 / f64 类型。如果必须使用（如计算百分比），必须在函数签名上标注 #[deny(float_operations)]（可通过 Clippy 或自定义 Lint 实现）。
表现层降级：当 ViewModelSet 读取逻辑层数据用于 UI 显示（如血条百分比、伤害预估）时，允许将整数转换为浮点数进行渲染，但严禁将计算结果回写到逻辑层 Component 中。
