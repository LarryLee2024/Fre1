# 不要同时存在三套模式

如果长期保留：

```text
有的 mod.rs
有的 api.rs
有的 integration.rs
```

五万行还好。

五十万行以后：

```text
AI不知道该找哪里

开发者不知道该从哪里进入

代码审查标准不统一

领域边界逐渐失控
```

这是最大的风险。

---

# 我的推荐

统一成：

```text
mod.rs
integration/
```

删除：

```text
api.rs
```

---

原因是：

## mod.rs

职责：

```text
模块组织
模块导出
```

例如：

```rust
pub mod model;
pub mod service;
pub mod integration;

pub use model::*;
```

---

永远不要写业务逻辑。

---

## integration/

职责：

```text
跨领域入口
ACL
Facade
Gateway
```

例如：

```text
character
ability
effect
battle
tactical
```

都可以有：

```text
integration/
```

---

例如：

```text
tactical/

integration/
├── mod.rs
├── movement.rs
├── terrain.rs
├── targeting.rs
└── vision.rs
```

---

Ability：

```text
ability/

integration/
├── mod.rs
├── execution.rs
├── targeting.rs
└── cooldown.rs
```

---

Character：

```text
character/

integration/
├── mod.rs
├── attribute.rs
├── progression.rs
└── equipment.rs
```

---

# 为什么不要 api.rs

很多项目喜欢：

```text
api.rs
```

但问题是：

API是什么意思？

可能表示：

```text
公开接口
Facade
Service
Application Service
SDK
RPC
HTTP
```

含义太模糊。

---

对于你的项目：

```text
integration
```

含义极其明确：

```text
其它领域从这里进来
```

---

AI也容易理解。

对于你当前正在讨论的 Tactical。

integration/

integration.rs

20+

# 我会写进架构宪法

```text
规则：

mod.rs
    仅负责模块组织与导出。

plugin.rs
    仅负责 Bevy 注册。

integration/
    领域唯一对外入口（ACL / Facade）。

禁止新增 api.rs。

跨领域访问只能通过 integration/。
禁止直接访问其他领域的内部 model/service/component。
```

对于你这种长期维护的 SRPG，我认为这是比：

```text
mod.rs
api.rs
integration.rs
```

混用更稳定的方案。

一句话总结：

```text
mod.rs = 模块组织者

plugin.rs = Bevy注册者

integration/ = 领域门面（唯一对外入口）

api.rs = 删除
```

这是我认为最适合你未来 50万~100万行代码项目的统一规范。
