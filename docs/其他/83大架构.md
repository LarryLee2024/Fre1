我会直接站在**Bevy 0.18 + 长期维护SRPG + 未来几十万行代码 + 单人开发 + AI协作开发**的角度来评审。
主要原则：
- Shared继续瘦身
- Core继续强化DDD边界
- Infrastructure继续按技术能力分层
- Content成为唯一内容事实源
- Modding降级为跨层能力
- Tools独立存在
- App保持最薄

## src/源码架构
src/
	├── main.rs                   # 程序入口，根据 feature 启动 game/editor/headless
	├── lib.rs                    # 库根，导出各层公共接口
	│
	│                           # 经典 DDD / clean architecture
	├── shared/                   # Layer 1：底层原子层。数学库、集合、序列化、日志等，零业务语义
	├── core/                     # Layer 2：领域规则层。实体、值对象、领域服务、聚合根、领域事件
	├── infra/                    # Layer 3：技术实现层。渲染适配器、持久化、网络、音频等"脏活"
	│                           # 横切能力，是 runtime / role
	├── app/                      # 横切1：启动装配层。Composition Root，唯一知道所有层存在的模块
	├── content/                  # 横切2：内容桥接层。数据驱动的核心，DSL + 配置 + gameplay 粘合
	├── tools/                    # 横切3：开发工具层。编辑器、调试器、热重载（feature-gated）
	└── modding/                  # 横切4：Mod扩展层。跨层聚合，暴露稳定的 Modding API

- Domain 标准内部结构（强制执行）
```bash
domain_name/
	├── plugin.rs          # 唯一对外入口，注册组件、系统、事件
	├── components.rs      # 本系统专属 ECS 组件
	├── systems/           # 本系统业务系统（按子模块拆分）
	│   ├── mod.rs
	│   ├── xxx_system.rs
	│   └── yyy_system.rs
	├── events.rs          # 本系统对外发布的领域事件
	├── error.rs           # 本系统专属错误枚举
	├── rules/             # 纯业务规则（优先纯函数，无 ECS 依赖）
	│   ├── formulas.rs    # 业务计算公式
	│   └── rules.rs       # 玩法规则判定
	└── integration.rs     # 集成层：唯一调用 Layers 能力的入口，编排底层机制实现业务
```
