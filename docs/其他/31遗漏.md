## 一、Bevy ECS 代码层规范（规则文档最容易忽略的落地约束）
你已经定义了"业务规则是什么"，但缺少"Bevy代码怎么写才不会乱"的底层编码契约，这是架构从文档落到代码的关键断层。

### 1. 组件设计规范 `component_design_rules.md`
- 核心缺失：没有明确「标记组件 / 数据组件 / 状态组件」的划分标准与命名约定
- 关键内容：
  - 标记组件（Marker）：零数据，仅用于查询过滤（如 `IsAlly`、`IsDead`），严格禁止携带字段
  - 数据组件：纯数据容器，禁止包含任何业务方法（如 `Health`、`Position`）
  - 状态组件：生命周期有限的临时状态（如 `CastingSkill`、`Moving`），必须随状态结束自动清理，禁止残留
  - 强制规则：单组件职责唯一；所有可序列化组件必须标注版本号；禁止跨领域的"上帝组件"
- 价值：Bevy项目后期90%的混乱，都始于组件越写越大、职责越混，最终变成不可维护的大泥球。

### 2. 系统编写铁律 `system_design_rules.md`
- 核心缺失：没有约束System的粒度、参数边界与依赖方式
- 关键内容：
  - 单系统参数上限（如不超过8个Query/Resource），超过必须拆分
  - 禁止系统间直接函数调用，所有跨系统通信必须走Event或组件状态
  - 读操作与写操作系统分离原则，最大化Bevy调度器的并行空间
  - 命名强制规范：「阶段 + 行为 + 对象」（如 `battle_apply_buff_damage_system`）
- 价值：Bevy的并行调度能力完全依赖系统解耦，无规范的系统会快速退化成面条代码，性能和可维护性双输。

### 3. 插件边界与依赖契约 `plugin_contract_rules.md`
- 核心缺失：你已有 `plugin-design.md` 讲设计，但缺少插件间的依赖规则与通信边界
- 关键内容：
  - 插件必须显式声明依赖的其他插件，禁止隐式依赖
  - 插件只能通过「公共Event + 公共Resource + 公共组件」对外暴露能力，禁止调用其他插件的私有系统
  - 分层禁令：领域插件不能依赖UI插件；基础设施插件不能反向依赖领域插件
- 价值：插件是Bevy的模块化单元，插件边界破了，整个分层架构就形同虚设。

## 二、SRPG 核心机制的架构补全
你覆盖了战斗、技能、Buff等主链路规则，但以下核心机制的**架构抽象设计**存在缺失。

### 1. 命令总线与操作校验 `command_bus_design.md`
- 核心缺失：有 `input_rules.md` 讲输入，但缺少「输入 → 命令 → 校验 → 执行」的中间层抽象
- 关键内容：
  - 所有玩家操作/AI操作统一封装为 `Command` 对象（如 `CastSkillCommand`、`MoveCommand`），而非直接修改状态
  - 命令校验层：执行前统一检查权限、消耗、目标合法性，全程只读、不修改任何状态
  - 命令执行层：仅做状态变更，不重复做校验
  - 命令队列：支持撤销、回放、批量原子执行
- 价值：这是SRPG实现「操作可回滚、可回放、可同步」的核心抽象，也是输入与领域逻辑的解耦关键。

### 2. 确定性执行规范 `determinism_rules.md`
- 核心缺失：有 `replay_rules.md` 讲回放格式，但没有保证"同样输入必定得到同样结果"的底层规则
- 关键内容：
  - 全局随机数种子管理：所有战斗随机必须走统一的确定性随机源，禁止直接使用系统随机
  - 数值精度约束：核心战斗数值禁止浮点数，统一用定点数或整数；若必须使用，明确精度与舍入规则
  - 迭代顺序约束：所有ECS查询的迭代顺序必须显式排序，禁止依赖Bevy默认的无序迭代
- 价值：没有确定性，回放系统就是摆设；未来做联机、Bug复现、自动化测试都会彻底失控。

### 3. 战斗状态机设计 `battle_fsm_design.md`
- 核心缺失：有 `turn_rules.md` 讲回合规则，但没有战斗全流程的状态流转契约
- 关键内容：
  - 战斗阶段枚举：初始化 → 回合开始 → 行动阶段 → 结算阶段 → 回合结束 → 胜负判定
  - 每个状态的进入/退出钩子，以及允许注册的扩展点
  - 状态跳转的合法路径，禁止非法跳转（如行动中直接跳到结算）
  - 与Bevy State/Schedule的一一映射关系
- 价值：战斗是典型的有限状态机，没有统一规范，后续加剧情演出、特殊机制会到处打补丁，逻辑越来越乱。

### 4. 寻路与范围计算 `pathfinding_design.md`
- 核心缺失：有 `map_terrain_rules.md` 讲地形规则，但没有寻路基础设施的分层设计
- 关键内容：
  - 寻路算法抽象层（A* 等），与具体地形数据解耦，支持未来替换算法
  - 移动范围、攻击范围、技能范围的统一计算接口规范
  - 地形消耗、单位阻挡、遮挡的统一处理逻辑
  - 计算结果缓存策略：哪些范围可以复用、什么时候失效
- 价值：这是战棋游戏的核心性能瓶颈之一，也是最容易重复造轮子、各写各的模块。

## 二、五层测试详细定义
### 1. 领域单元测试层（纯逻辑，零运行时依赖）
**存放位置**：`core/[领域模块]/tests/`
**核心定位**：验证单条游戏规则的正确性，是测试金字塔的底座，占比 70% 以上。

#### 测试范围
- 纯数值公式：伤害计算、属性加成、堆叠规则、消耗计算
- 单领域逻辑：技能命中判定、Buff 生效/失效规则、回合推进条件
- 边界与异常：非法输入处理、错误类型返回、极值下的规则表现

#### 强制约束
- ❌ 绝对禁止引入 Bevy 任何依赖，不使用 ECS、不加载资源
- ❌ 不依赖任何 `content` 配置数据，测试用例硬编码入参
- ✅ 全部为纯函数测试，输入输出一一对应，毫秒级执行

```rust
}
```

### 2. 领域集成测试层（跨模块规则联动）
**存放位置**：`core/[聚合根模块]/tests/`（如 `core/battle/tests/`）
**核心定位**：验证多个领域模块组合后的规则一致性，不涉及引擎运行时。

#### 测试范围
- 跨领域链路：技能释放 → 伤害结算 → Buff 挂载 → 属性变更 全链路
- 状态流转：回合开始 → 行动 → 结算 → 回合结束的完整状态机
- 聚合不变量：战斗结束后所有单位状态合法、Buff 数量不超限等

#### 特点
- 仍然是纯 Rust 代码，不启动 Bevy
- 使用领域对象直接组合，模拟完整战斗上下文
- 重点验证模块间的契约是否符合 `layer-contracts.md` 约定

---

### 3. 系统集成测试层（Bevy ECS 运行时验证）
**存放位置**：`tests/integration/` 或对应插件模块内
**核心定位**：验证架构在 Bevy 引擎中的落地正确性，确认 ECS 系统、事件、组件的交互符合设计预期。

#### 测试范围
- 插件装配：注册指定 Plugin 后，必要的 Resource、System 是否正常初始化
- 系统调度：系统执行顺序是否符合 `schedules` 设计，是否存在顺序依赖bug
- 事件流转：发送领域事件后，对应系统是否正确触发、组件是否正确变更
- ECS 通信：跨模块通过事件/组件交互是否符合 `ecs_communication_rules.md`

#### 技术要点
- 使用 Bevy 原生测试能力，运行 `headless` 无头模式
- 通过 `World::new()` 手动构造最小测试环境，只加载被测插件
- 重点测「架构落地」，不测「业务规则」——业务规则已经在领域层测完

---

### 4. 端到端回归测试层（全链路黑盒）
**存放位置**：项目根目录 `tests/`
**核心定位**：验证完整游戏的可用性与稳定性，作为 CI 门禁的最后一道防线。

#### 测试分类与范围
| 分类 | 测试内容 | 执行频率 |
|------|---------|----------|
| 主流程测试 | 进入战斗 → 执行操作 → 推进回合 → 胜负判定 完整流程 | 每日构建 |
| 存档兼容测试 | 各版本存档能否正常加载、数据不丢失 | 发布前 |
| MOD 兼容测试 | 标准 MOD 能否正常加载、不破坏基础游戏逻辑 | 发布前 |
| 错误路径测试 | 异常输入、非法配置下游戏是否可控报错，不崩溃 | 每周回归 |

#### 强制约束
- 以黑盒方式调用游戏公共接口，不访问任何内部私有结构
- 用例只覆盖核心主干，不追求覆盖细节，细节由下层测试保障
- 执行耗时控制在分钟级，避免成为 CI 瓶颈

---

### 5. Testbeds 测试沙盒层（开发调试核心资产）
**存放位置**：项目根目录 `testbeds/`
**核心定位**：不是自动化测试代码，而是可运行、可视化、可交互的调试沙盒，是 SRPG 项目长期开发的效率杠杆。它既是调试工具，也是自动化测试的用例生成器。

#### 核心沙盒与用途
1. **battle_simulator 战斗模拟器**
    - 可加载任意地图、任意单位配置，快速启动一场完整战斗
    - 支持单步执行、回退、状态快照，用于调试战斗逻辑
    - 录制的战斗流程可直接导出为回放文件，自动转为回归测试用例

2. **skill_playground 技能调试台**
    - 单独加载技能系统，实时调整技能参数，立即看到结算结果
    - 用于技能设计、数值调试，不需要启动完整游戏
    - 支持批量跑技能用例，快速验证数值平衡性

3. **ai_debug_arena AI 调试场**
    - 可视化展示 AI 决策树、权重计算、最终选择
    - 固定场景下反复验证 AI 行为一致性，排查决策异常
    - 对应你的 `ai_rules.md`，是 AI 规则落地的核心调试工具

4. **balance_workbench 数值工作台**
    - 批量执行指定配置的战斗，统计伤害分布、胜率、回合时长
    - 用于平衡性验证，支持多版本配置对比
    - 对接 `content` 数据，配置修改后一键跑基准测试

5. **replay_validator 回放验证器**
    - 批量重放历史回放文件，校验同一输入是否得到完全相同的结果
    - 是确定性保障的核心工具，每次引擎升级、架构重构后必跑
    - 自动对比状态哈希，输出不一致的帧位置与差异点

## 三、公共测试支撑体系
**存放位置**：`src/shared/testing/`
这是整个测试体系唯一的公共依赖层，所有测试都可以复用，禁止在各模块内重复造测试轮子。

### 1. 测试夹具库（Fixture）
- 标准测试数据构造器：`create_test_character()`、`create_test_skill()`
- 预设战斗场景：`standard_1v1_battle()`、`full_party_battle()`
- 统一的构造约定，确保所有测试使用一致的基准数据

### 2. 确定性基础设施
- **可控随机源**：测试时注入固定种子的随机数生成器，完全替代全局随机
- **模拟时间轴**：脱离真实帧时间，手动推进游戏 tick，保证时序可复现
- **迭代排序工具**：统一ECS查询迭代顺序，消除 Bevy 默认无序迭代带来的不确定性
- **状态哈希工具**：对战斗全局状态生成唯一哈希，快速校验一致性

### 3. 自定义断言扩展
- 数值容差断言：处理浮点计算的精度问题
- 状态不变量断言：一键校验战斗全局规则是否被打破
- 错误类型断言：验证指定场景返回预期的错误类型

---

## 四、内容数据测试层
**存放位置**：`content/tests/`
这是数据驱动项目最容易遗漏的一层——代码没问题，但配置写错一样会炸游戏。

### 测试范围
- **引用完整性**：技能引用的 Buff ID、特效 ID 是否真实存在
- **数值合法性**：伤害是否为正、冷却时间是否越界、属性加成比例是否超限
- **格式合规性**：所有配置文件是否符合 schema、必填字段是否缺失
- **规则一致性**：配置数值是否符合领域规则约束（如堆叠数量上限）

### 价值
- 把内容配置的错误拦截在提交阶段，避免运行时才发现
- 支持 MOD 内容校验，第三方 MOD 提交时自动检查合规性
- 是大规模扩充内容时，保障质量不滑坡的核心闸门

## 五、落地优先级建议
结合你当前「主链路打通、进入内容扩展期」的阶段，按价值从高到低落地：

1. **第一优先级**：搭建 `shared/testing` 基础工具 + 核心领域单元测试
   先把公式、技能、Buff的核心规则用单元测试钉死，这是后续所有扩展的基础。

2. **第二优先级**：搭建 `battle_simulator` 沙盒 + 回放确定性验证
   这是SRPG开发的效率倍增器，后续调试、测数值、复现bug都靠它，同时把确定性底线守住。

3. **第三优先级**：补充内容数据校验 + 核心E2E测试
   开始大量加内容前，把数据校验闸门加上；CI接入核心流程回归测试。

4. **第四优先级**：完善其余沙盒工具 + 全量回归用例
   随着项目扩张逐步补齐，不用初期一次性做完。

## 一句话总结
- **领域单元测试** 钉死规则正确性
- **系统集成测试** 验证架构落地合理性
- **Testbeds 沙盒** 支撑日常开发调试
- **内容数据测试** 守住配置质量底线
- **E2E 回归测试** 做最终稳定性门禁

这套体系完全匹配你的DDD分层与长期SRPG规划，既能支撑未来上千技能、上百关卡的内容扩张，也能保障MOD接入、引擎升级时的架构稳定性。
