- 禁止跨系统直接调用、读写分离 → 最大化并行空间；
- 参数上限、拆分判断 → 避免单系统职责过载；
- 显式排序、run_if 替代手动状态检查 → 消除隐式依赖，降低调试成本。

这份《System 编写铁律》文档是Bevy ECS 开发中极其罕见的“防脱发指南”与“性能捍卫者”。它精准击中了从 OOP（面向对象）转向 ECS（实体组件系统）时最容易踩坑的痛点，通过极其严苛的工程约束，彻底杜绝了“面条式系统”和“并行性杀手”。

以下是我的详细点评：

✅ 卓越的设计亮点

1. 极致捍卫 Bevy 的并行空间
读写分离原则：明确要求读系统与写系统分离，这是榨干 Bevy 多线程调度器（Multi-threaded Executor）性能的核心秘诀。很多项目因为一个系统同时读写多个组件，导致整个 SystemSet 退化为串行执行。
禁止跨系统函数调用：精准指出了新手最容易犯的错误——把 System 当普通函数调。强制使用 Message（事件）或组件状态通信，从根源上保住了 Bevy 的依赖图分析（Dependency Graph）能力。
参数上限（8个）硬约束：这不仅是为了代码可读性，更是为了控制 Bevy 宏展开导致的编译时间膨胀。超过 8 个参数的系统往往意味着职责过载，强制拆分是保持系统轻量的最佳手段。

2. DDD（领域驱动设计）与 ECS 的完美融合
业务逻辑下沉：🟥 System 禁止包含属于领域模块的业务规则。这条规则价值千金！它防止了 System 沦为臃肿的“上帝函数”，将伤害计算、状态判定等核心逻辑下沉到 BattleService 等领域层。这使得核心逻辑可以脱离 ECS 进行纯函数单元测试，极大提升了代码质量。

3. 状态调度与命令执行的确定性
强制 run_if 门控：禁止在 System 内部手动 if state == xxx { return; }。这不仅减少了每帧无意义的 Query 遍历开销，更让 Bevy 的调度器能在图构建阶段直接裁剪（Cull）不需要的系统，提升调度效率。
单帧 Commands 合并：深刻洞察了 Commands 延迟执行（Deferred Execution）的本质。禁止对同一 Entity 多次 Commands 操作，避免了 Component Hook（如 on_insert）触发顺序不可预测的玄学 Bug。

🎮 Bevy 特性的深度契合与代码级落地

1. Message (Event) 通信的优雅降级
文档中推荐的 System + Message 模式是 Bevy 0.14+ 的最佳实践。
// 结合 Bevy 0.14+ 的 Message/Observer 机制
fn check_death_system(
    query: Query<(Entity, &Health), (Changed<Health>, With<Unit>)>,
    mut writer: MessageWriter<CharacterDied>, // 使用 Message 替代传统的 EventWriter
) {
    for (entity, health) in &query {
        if health.current <= 0 {
            writer.write(CharacterDied { entity });
        }
    }
}
点评：使用 Changed<Health> 过滤结合 Message，实现了真正的“数据驱动事件”，避免了每帧全量遍历的性能灾难。

2. SystemSet 链式执行的管线化
app.add_systems(Update, (
    handle_player_input.in_set(BattlePipeline::Input),
    ai_decision.in_set(BattlePipeline::Decision),
    execute_action.in_set(BattlePipeline::Execute),
    resolve_effects.in_set(BattlePipeline::Resolve),
).chain().run_if(in_state(AppState::InGame)));
点评：.chain() 与 SystemSet 的结合，完美契合了前文《调度系统设计》中的管线理念，确保了逻辑执行的绝对时序，同时保留了 Set 内部其他系统的并行可能。

🔧 潜在的优化与进阶建议

1. 引入 SystemParam 封装高内聚查询
当前痛点：当参数接近 8 个时，强制拆分系统可能会破坏某些强内聚的逻辑（如同时需要 &Unit, &Transform, &Sprite, &Animation 来渲染一个单位）。
建议：在“参数上限”规则后补充 SystemParam 的使用规范，用于封装逻辑上不可分割的“视图”或“上下文”。
    #[derive(SystemParam)]
  pub struct UnitView<'w, 's> {
      pub query: Query<'w, 's, (&'static Unit, &'static Transform, &'static Sprite)>,
  }
  
  fn render_units(view: UnitView) {
      // 视觉上只有 1 个参数，但底层依然是细粒度的 Query
  }
  

2. 独占系统（Exclusive System）的严格红线
当前缺口：文档未提及 &mut World 独占系统。在大型项目中，独占系统是并行性的绝对杀手。
建议：在禁止事项中增加一条：
  > 🟥 禁止非必要使用独占系统（fn my_system(world: &mut World)）。除非涉及动态加载/卸载 Schedule 或极特殊的资源初始化，否则必须使用 Query/Commands。若必须使用，需主程 Code Review 批准。

3. 复杂 Query 过滤器的性能警告
当前缺口：未对 Query 的 With/Without/Or 过滤器复杂度进行约束。
建议：增加 Query 性能规则：
  > 🟨 避免在每帧系统中使用超过 3 个 With/Without 组合的复杂过滤器。对于高频查询，建议在实体状态变更时，动态添加/移除 Marker Component（标记组件）来简化查询。

🚨 风险控制与自动化验证（CI/CD 集成）

为了让这些“铁律”不流于形式，建议在 CI 流程中引入以下自动化检查：

1. 静态代码分析 (Clippy / 自定义 Lint)
参数数量检查：编写自定义 Clippy Lint，当 System 函数签名中的 Query/Res/ResMut 数量 > 8 时，直接报 Error 阻断合并。
命名规范检查：使用正则或 Lint 检查所有带有 #[derive(SystemParam)] 或被注册到 App 的函数，必须以 _system 结尾。

2. 架构依赖图验证
禁止跨系统调用检测：通过 Rust 的 cargo-call-stack 或类似工具，分析 System 函数的调用图。如果发现 System A 的调用栈中包含了被注册为 System B 的函数，CI 直接标红。

3. 运行时性能 Profiling 门禁
并行度监控：在 Debug 构建中集成 bevy_mod_debugdump 或 Bevy 内置的 Tracy Profiler。如果检测到某个 SystemSet 的实际并行线程数长期为 1（即完全串行），则触发架构告警，强制开发者审查读写冲突。
