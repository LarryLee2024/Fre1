# 存档持久化领域

Version: 1.0
Status: Proposed

存档持久化领域管理游戏状态的保存、加载、格式版本化和向后兼容。只保存 Instance 数据，Definition 通过 Strong ID 引用从 Registry 恢复。

核心原则：
- 只保存 Instance，禁止保存 Definition（Definition 不可变，从配置加载）
- 格式版本只增不减，旧版本必须可通过迁移管线加载
- 存档必须在状态一致点生成（TurnEnd 阶段后）
- 存档数据使用人类可读格式（开发阶段 JSON），预留二进制选项

---

# 术语定义

## 存档（Save Data）

游戏运行状态的持久化快照。包含 Instance 数据、格式版本号、时间戳和元信息。加载时从 Registry 恢复 Definition 引用。

不是配置文件。不是日志。不是录像。不是回放文件。

关键属性：
- 包含版本号（SemVer）、时间戳、Instance 数据、元信息
- 通过 Definition ID 引用不可变配置
- 可被序列化为 JSON 或二进制格式
- 每个存档是自包含的完整游戏状态快照

---

## 存档格式（Save Format）

存档数据的序列化格式。定义存档文件的结构和字段布局。

不是文件扩展名。不是数据库 Schema。不是内存布局。

关键属性：
- 开发阶段使用 JSON（人类可读、可调试）
- 生产阶段可通过 feature flag 切换到二进制格式
- 格式变更必须递增版本号
- 不同格式之间通过迁移函数转换

---

## 格式版本（Format Version）

存档结构的版本号，使用 SemVer 格式。标识存档结构的兼容性等级。

不是游戏版本。不是存档次数。不是系统版本。

关键属性：
- 使用 SemVer（major.minor.patch）格式
- major 变更 = 不兼容变更（需要迁移函数）
- minor 变更 = 新增可选字段（向后兼容）
- patch 变更 = 字段修复（向后兼容）
- 只增不减

---

## 格式迁移（Format Migration）

旧格式存档到新格式的转换过程。通过迁移函数链将旧版本存档逐步升级到最新版本。

不是存档加载。不是数据修补。不是热修复。不是存档覆盖。

关键属性：
- 迁移函数是纯函数（输入旧数据，输出新数据）
- 迁移不修改原始数据（不可变转换）
- 迁移失败时回滚到原始数据
- 每个版本跳跃有独立的迁移函数（V1→V2, V2→V3...）
- 迁移链支持任意旧版本到最新版本

---

## 存档快照（Save Snapshot）

存档生成时游戏状态的完整一致快照。保证状态一致性——不在行动中途存档。

不是增量保存。不是实时同步。不是增量备份。不是数据库事务。

关键属性：
- 快照在 TurnEnd 阶段后生成
- 快照包含当前回合所有实体状态的完整一致视图
- 快照不包含 Definition 数据（只记录引用 ID）
- 快照包含随机种子（保证回放确定性）

---

## 存档槽位（Save Slot）

一个独立的存档文件位。标识一个存档位置，与具体文件路径解耦。

不是文件路径。不是存档索引。不是存档内容。

关键属性：
- 槽位是逻辑标识（如 Slot0、Slot1、AutoSave）
- 槽位到文件路径的映射由配置决定
- 每个槽位独立管理（互不覆盖）
- 槽位数量有限制（由配置定义）

---

## 自动存档（Auto Save）

特定时机自动触发的存档。在关键节点（如 TurnEnd、关卡完成）自动保存。

不是存档类型。不是快照类型。不是手动存档。

关键属性：
- 自动存档使用专用槽位（AutoSave Slot）
- 自动存档覆盖上次自动存档（不累积）
- 自动存档时机由游戏规则定义
- 自动存档失败不影响游戏进行

---

## 最低支持版本（Minimum Supported Version）

可加载的最旧存档版本号。低于此版本的存档拒绝加载。

不是当前版本。不是兼容性声明。

关键属性：
- 支持最近 3 个 MAJOR 版本的存档
- 低于最低支持版本的存档：拒绝加载 + ERROR 日志 + 提示用户
- 版本号低于最低支持时不尝试加载（避免数据损坏）
- 来源：`docs/architecture/save_migration_rules.md`

---

# 领域边界

## 本领域负责

- 存档的序列化和反序列化
- 存档格式版本管理（SemVer）
- 存档格式迁移（旧版本 → 新版本）
- 存档完整性校验
- 存档槽位管理
- 自动存档触发时机
- 存档文件的读写操作
- Instance 数据的保存和恢复
- Definition 引用的恢复（从 Registry）

## 本领域不负责

- 哪些数据值得保存（由 Core 层业务规则决定）
- Definition 的加载和管理（由 Content 层负责）
- Registry 的维护（由 Core 层负责）
- 游戏状态的业务逻辑（由 Core 层负责）
- 回放文件的生成（由 Replay 领域负责）
- UI 层存档界面（由 UI 层负责）
- 云存档同步（由 CloudSave 领域负责）

## 跨领域通信方式

| 通知内容 | 通信方式 | 目标领域 |
|----------|----------|----------|
| 存档触发信号 | Message | Core 层（TurnEnd 阶段） |
| 存档完成通知 | Message | UI 层（更新存档界面） |
| 格式迁移失败 | Message | UI 层（显示错误信息） |
| Definition 引用恢复 | Resource 读取 | Core 层 Registry |
| 自动存档触发 | Message | Core 层（TurnEnd 阶段） |

# 生命周期

## 状态列表

| 状态 | 含义 | 可转换到 |
|------|------|----------|
| Idle | 无存档操作 | Saving, Loading |
| Saving | 正在保存存档 | Idle, SaveError |
| SaveError | 保存失败 | Idle, Saving |
| Loading | 正在加载存档 | Idle, LoadError |
| LoadError | 加载失败 | Idle, Loading |
| Migrating | 正在迁移格式 | Idle, MigrationError |
| MigrationError | 迁移失败 | Idle |

## 状态转换图

```
Idle → Saving → Idle
  ↓       ↓
  ↓    SaveError → Idle
  ↓
Loading → Idle
  ↓
LoadError → Idle
  ↓
Migrating → Idle
  ↓
MigrationError → Idle
```

## 转换条件

| 从 | 到 | 条件 |
|----|-----|------|
| Idle | Saving | 接收到存档触发信号（手动/自动） |
| Saving | Idle | 序列化和写入完成 |
| Saving | SaveError | 序列化失败或写入失败 |
| SaveError | Idle | 用户确认错误或自动重试 |
| Idle | Loading | 接收到加载请求 |
| Loading | Idle | 文件读取、版本检查、迁移（如需要）、反序列化完成 |
| Loading | LoadError | 文件读取失败或反序列化失败 |
| LoadError | Idle | 用户确认错误或自动重试 |
| Loading | Migrating | 存档版本低于当前版本 |
| Migrating | Idle | 迁移函数执行成功 |
| Migrating | MigrationError | 迁移函数执行失败 |

# 不变量

## 不变量1：存档必须在 TurnEnd 阶段后生成

任意时刻：

存档快照必须在 TurnEnd 阶段完成后生成。禁止在行动中途、战斗阶段中间、动画播放中生成存档。

违反表现：

存档包含不完整的行动状态（如伤害已计算但 Buff 未结算）。加载后游戏状态不一致。

---

## 不变量2：格式版本只增不减

任意时刻：

存档格式版本号只能递增，禁止递减或保持不变。每次格式变更必须递增版本号。

违反表现：

新版本代码尝试加载旧版本格式的存档时，无法确定迁移路径。数据丢失或解析错误。

---

## 不变量3：旧格式存档必须能通过迁移管线加载

任意时刻：

对于所有历史版本的存档格式，必须提供完整的迁移函数链，确保可以从任意旧版本迁移到最新版本。

违反表现：

玩家无法加载旧版本存档。游戏更新后存档丢失。

---

## 不变量4：存档不得包含 Definition 数据

任意时刻：

存档中只包含 Instance 数据和 Definition 的引用 ID。禁止将 Definition 配置数据直接嵌入存档。

违反表现：

存档文件膨胀。热重载后 Definition 更新，但存档中的 Definition 未更新，导致配置不一致。

---

## 不变量5：存档必须包含关键元信息

任意时刻：

每个存档必须包含以下字段：格式版本号（SemVer）、时间戳、当前回合号、GameOverState、随机种子。缺失任何字段的存档视为无效。

违反表现：

无法确定存档的版本号（无法迁移）。无法确定存档的时间点（无法恢复回合）。无法保证回放确定性（缺少随机种子）。

---

## 不变量6：迁移函数必须是纯函数

任意时刻：

迁移函数不依赖外部状态（如 Registry、Resource、全局变量）。输入旧数据，输出新数据，无副作用。

违反表现：

迁移函数依赖 Registry 但 Registry 未加载，导致迁移失败；迁移函数修改全局状态，导致不可预测行为。

---

## 不变量7：迁移前必须备份原始文件

任意时刻：

迁移前创建临时备份文件，迁移成功后删除备份，迁移失败后保留原始文件。禁止迁移失败修改原始文件。

违反表现：

迁移失败时原始文件被修改或删除，玩家存档永久丢失。

---

## 不变量8：每个迁移函数必须校验输入输出

任意时刻：

每个迁移函数必须验证输入数据合法性（版本号、字段完整性）和输出数据合法性。禁止不验证直接返回。

违反表现：

非法输入导致迁移函数 panic 或输出损坏数据，加载后游戏状态错乱。

---

# 业务规则

## 规则1：存档保存时机

允许：
- 在 TurnEnd 阶段完成后保存存档
- 在 MainMenu 阶段保存存档
- 在关卡间切换时保存存档

禁止：
- 在战斗阶段中间保存存档
- 在动画播放中保存存档
- 在回合选择阶段保存存档
- 在 BattleEnd 之前保存存档

必须：
- 自动存档在 TurnEnd 阶段后触发
- 手动存档只在允许的阶段执行

---

## 规则2：存档格式版本管理

允许：
- minor 版本新增可选字段（向后兼容）
- patch 版本修复字段错误（向后兼容）
- major 版本进行不兼容变更（必须提供迁移函数）

禁止：
- 删除已有字段（必须保留并标记为 deprecated）
- 修改已有字段的类型（必须新增字段替代）
- 递减版本号
- 跨 major 版本不提供迁移函数

必须：
- 每次格式变更递增版本号
- 新增字段时标注默认值（保证旧数据兼容）
- major 版本变更时提供完整的迁移函数链

---

## 规则3：存档数据完整性

允许：
- 使用 JSON 格式存储（开发阶段）
- 使用二进制格式存储（生产阶段）
- 在存档末尾附加校验和

禁止：
- 不写校验和就完成存档
- 存档文件截断后继续使用
- 忽略校验和验证结果

必须：
- 保存后立即校验存档完整性
- 加载时首先验证校验和
- 校验失败时返回 LoadError，不返回损坏数据

---

## 规则4：Definition 引用恢复

允许：
- 存档中记录 Definition 的 Strong ID（UnitId、SkillId 等）
- 加载时从 Registry 根据 ID 恢复 Definition 引用
- 加载时验证所有 Definition ID 在 Registry 中存在

禁止：
- 将 Definition 数据嵌入存档
- 加载时创建新的 Definition
- 忽略缺失的 Definition ID

必须：
- 加载后所有 Definition 引用必须有效
- 缺失 Definition ID 时返回 LoadError
- 加载完成后验证 Instance 与 Definition 的一致性

---

## 规则5：自动存档管理

允许：
- 使用专用槽位（AutoSave）存储自动存档
- 覆盖上次自动存档（单槽位轮转）
- 在 TurnEnd 阶段后触发自动存档

禁止：
- 自动存档累积多个文件
- 自动存档失败时中断游戏
- 手动存档和自动存档使用同一槽位

必须：
- 自动存档使用独立的槽位标识
- 自动存档失败时记录日志但不阻塞游戏
- 自动存档遵循与手动存档相同的格式和完整性规则

---

## 规则6：最低支持版本策略

允许：
- 支持最近 3 个 MAJOR 版本的存档
- 拒绝低于最低支持版本的存档加载

禁止：
- 加载低于最低支持版本的存档（必须拒绝）
- 支持超过 3 个 MAJOR 版本的旧存档

必须：
- 低于最低支持版本时：拒绝加载 + ERROR 日志 + 提示用户更新游戏或重新开始
- 不尝试加载（避免数据损坏）

---

## 规则7：迁移函数约束

允许：
- 迁移函数是纯函数（不依赖外部状态）
- 迁移函数验证输入输出数据合法性
- 迁移函数有对应的单元测试

禁止：
- 迁移函数依赖外部状态（Registry、Resource）
- 迁移函数不验证输入输出
- 迁移函数修改输入数据（必须不可变）

必须：
- 每个迁移函数处理一个版本跳跃（V1→V2）
- 迁移函数验证输入版本号正确
- 迁移函数验证输出数据合法性
- 迁移函数记录 WARN 日志（表示正在执行迁移）

---

## 规则8：迁移执行时机

允许：
- 迁移在游戏启动时或加载存档时执行
- 迁移在反序列化之前执行（数据转换）

禁止：
- 迁移在战斗中执行
- 迁移阻塞主线程超过 5 秒

必须：
- 迁移仅在加载时执行
- 迁移不阻塞主线程超过 5 秒
- 迁移失败时保留原始文件

---

## 规则9：备份策略

允许：
- 迁移前创建临时备份文件
- 迁移成功后删除备份
- 迁移失败后保留备份供用户恢复

禁止：
- 迁移失败时修改或删除原始文件
- 迁移前不创建备份

必须：
- 迁移前备份原始文件
- 迁移成功后删除备份
- 迁移失败后保留原始文件 + 备份文件

---

# 流程管线

## 管线1：存档保存管线

```
状态快照 → 序列化 → 写入文件 → 校验完整性
```

### Step1：状态快照

输入：当前 World 状态（TurnEnd 阶段后）
处理：收集所有 Instance 数据，记录 Definition 引用 ID，记录回合号、GameOverState、随机种子
输出：SaveData 结构（内存中）
禁止：快照过程中修改任何 ECS 状态、快照过程中触发新事件

### Step2：序列化

输入：SaveData 结构
处理：将 SaveData 序列化为目标格式（JSON/二进制），附加版本号和校验和
输出：字节流
禁止：序列化过程中修改 SaveData、跳过版本号写入

### Step3：写入文件

输入：序列化后的字节流
处理：写入目标槽位的文件路径，使用临时文件+原子替换策略
输出：文件写入完成确认
禁止：直接覆盖旧文件（必须先写临时文件再替换）、写入完成后不清理临时文件

### Step4：校验完整性

输入：已写入的文件
处理：重新读取文件，验证校验和，验证反序列化结果与原始 SaveData 一致
输出：校验通过/失败
禁止：跳过校验、校验失败后继续使用该存档

---

## 管线2：存档加载管线

```
读取文件 → 版本检查 → 格式迁移（如需要）→ 反序列化 → 恢复游戏状态
```

### Step1：读取文件

输入：目标槽位的文件路径
处理：读取文件全部内容，验证文件存在和可读
输出：字节流
禁止：文件不存在时返回空数据、忽略文件读取错误

### Step2：版本检查

输入：字节流
处理：解析版本号字段，与当前代码版本比较
输出：版本匹配/需要迁移
禁止：跳过版本号解析、版本号无法解析时继续加载

### Step3：格式迁移（如需要）

输入：旧版本存档数据
处理：按迁移函数链逐步升级（V1→V2→V3...），直到当前版本
输出：最新版本存档数据
禁止：跳过中间版本（必须逐版本迁移）、迁移失败后返回部分数据

### Step4：反序列化

输入：最新版本存档字节流
处理：反序列化为 SaveData 结构，验证字段完整性
输出：SaveData 结构
禁止：反序列化失败后返回空数据、跳过字段完整性验证

### Step5：恢复游戏状态

输入：SaveData 结构
处理：从 Registry 恢复 Definition 引用，重建 ECS 状态，恢复回合号和随机种子
输出：完整的游戏状态
禁止：Definition ID 缺失时静默忽略、恢复过程中修改 Definition 数据

---

## 管线3：格式迁移管线

```
旧版本存档 → 版本检测 → 迁移函数链 → 最新版本存档
```

### Step1：版本检测

输入：旧版本存档数据
处理：读取存档中的版本号，确定需要执行的迁移函数列表
输出：迁移函数列表
禁止：版本号无法识别时继续迁移

### Step2：迁移函数链

输入：旧版本存档数据 + 迁移函数列表
处理：按顺序执行每个迁移函数，每个函数将数据从版本 N 升级到版本 N+1
输出：最新版本存档数据
禁止：跳过任何迁移步骤、迁移函数修改输入数据（必须不可变）

---

# 数据结构

## SaveData（存档数据）

职责：存档文件的顶层结构，包含所有存档信息

结构：
- version：SemVer — 存档格式版本号
- timestamp：u64 — 保存时间戳（Unix 时间戳）
- turn_number：u32 — 当前回合号
- game_over_state：GameState — 当前游戏状态
- random_seed：u64 — 随机种子（保证回放确定性）
- instances：InstanceData — 所有运行时 Instance 数据
- metadata：SaveMetadata — 存档元信息

要求：
- 版本号必须与当前代码版本兼容
- 所有字段必须完整（不允许省略）
- instances 中只包含 Instance 数据，禁止包含 Definition

---

## InstanceData（Instance 数据集）

职责：所有需要持久化的运行时 Instance 数据集合

结构：
- units：Vec<UnitInstance> — 所有单位的运行时状态
- buffs：Vec<BuffInstance> — 所有激活的 Buff 实例
- inventory：InventoryInstance — 背包状态
- map_state：MapInstance — 地图状态（已探索区域、单位位置等）
- turn_state：TurnInstance — 回合状态机当前状态

要求：
- 每个 Instance 通过 Strong ID 引用对应的 Definition
- 只包含运行时可变数据
- 不包含任何 Definition 配置数据

---

## SaveMetadata（存档元信息）

职责：存档的辅助信息，不包含游戏状态

结构：
- save_name：String — 存档显示名称
- slot：SaveSlot — 存档槽位标识
- is_auto_save：bool — 是否为自动存档
- game_version：String — 生成存档的游戏版本
- play_time：u64 — 累计游戏时间（秒）

要求：
- 元信息不影响游戏状态恢复
- 元信息用于 UI 显示和管理

---

## SemVer（语义化版本号）

职责：存档格式的版本标识

结构：
- major：u32 — 主版本号（不兼容变更）
- minor：u32 — 次版本号（向后兼容的新增）
- patch：u32 — 补丁版本号（向后兼容的修复）

要求：
- major 变更必须提供迁移函数
- minor 变更新增字段必须有默认值
- 只增不减

# 禁止事项

禁止：在战斗阶段中间保存存档

原因：战斗阶段中间状态不完整（伤害已计算但 Buff 未结算、行动已发起但动画未完成），存档会导致状态不一致。

违反后果：加载后游戏状态错乱，单位属性不一致，战斗结果不可预测。

---

禁止：将 Definition 数据嵌入存档

原因：Definition 是不可变配置，从 RON 文件加载。嵌入存档会导致配置更新后存档中的 Definition 过时。

违反后果：热重载后 Definition 更新，但存档中的 Definition 未更新，配置不一致。存档文件膨胀。

---

禁止：跨 major 版本不提供迁移函数

原因：major 版本变更意味着不兼容的格式修改，没有迁移函数无法将旧版本存档转换为新版本。

违反后果：游戏更新后玩家无法加载旧版本存档，导致存档丢失。

---

禁止：跳过存档完整性校验

原因：校验和是检测存档损坏的唯一手段。跳过校验可能加载损坏的数据。

违反后果：加载损坏的存档导致游戏状态错乱、崩溃或数据丢失。

---

禁止：直接覆盖旧存档文件

原因：直接覆盖在写入中断时会同时损坏新旧文件，导致存档永久丢失。

违反后果：写入中断时存档损坏，无法恢复。

---

禁止：加载时忽略缺失的 Definition ID

原因：缺失 Definition ID 意味着存档引用的配置在当前 Registry 中不存在。静默忽略会导致运行时错误。

违反后果：加载后游戏状态不完整，运行时 panic 或数据错乱。

---

禁止：存档格式无版本号

原因：没有版本号无法判断存档格式，无法确定迁移路径。

违反后果：旧版本存档无法加载，加载时数据解析错误。

---

禁止：删除旧版本的迁移函数

原因：迁移函数是永久资产，删除后无法从旧版本迁移到新版本。

违反后果：旧版本存档无法加载，玩家存档丢失。

---

禁止：MAJOR 变更不提供迁移路径

原因：MAJOR 变更意味着不兼容的格式修改，没有迁移函数无法将旧版本存档转换为新版本。

违反后果：游戏更新后玩家无法加载旧版本存档，导致存档丢失。

---

禁止：迁移失败修改原始文件

原因：迁移失败时必须保持原始文件不变，否则玩家存档永久丢失。

违反后果：迁移失败后原始文件被修改或删除，玩家无法恢复存档。

---

# AI 修改规则

## 如果新增存档字段

允许：
- 在 SaveData 中新增可选字段
- 为新字段提供默认值
- 在 minor 版本中新增字段

禁止：
- 删除已有字段
- 修改已有字段的类型
- 新增字段不提供默认值

优先检查：
- 新字段是否有默认值（保证旧数据兼容）
- 版本号是否已递增
- 反序列化时新字段缺失是否能正确处理默认值

---

## 如果修改格式迁移函数

允许：
- 新增迁移函数（V2→V3）
- 修复现有迁移函数的 Bug
- 优化迁移函数性能

禁止：
- 修改已有迁移函数的输出格式
- 跳过中间版本直接迁移（必须逐版本）
- 迁移函数修改输入数据（必须不可变）

优先检查：
- 迁移函数是否为纯函数（不修改输入）
- 迁移函数链是否覆盖所有历史版本
- 迁移后数据是否通过完整性校验

---

## 如果修改存档保存逻辑

允许：
- 修改保存时机（在允许的阶段内）
- 新增自动存档触发点
- 优化保存性能

禁止：
- 在战斗阶段中间添加保存点
- 保存时修改 ECS 状态
- 跳过完整性校验

优先检查：
- 保存时机是否在 TurnEnd 阶段后
- 保存后是否立即校验完整性
- 自动存档失败是否不阻塞游戏

---

## 如果修改存档加载逻辑

允许：
- 优化加载性能
- 新增加载错误处理
- 改善 Definition 引用恢复逻辑

禁止：
- 加载时修改 Definition 数据
- 忽略缺失的 Definition ID
- 跳过版本检查

优先检查：
- 加载后 Definition 引用是否全部有效
- 版本检查是否正确识别需要迁移的存档
- 加载失败时是否返回正确的错误类型

---

## 如果测试失败

排查顺序：
1. 检查存档格式版本号是否正确递增
2. 检查迁移函数链是否完整覆盖所有历史版本
3. 检查 Definition 引用恢复是否成功（Registry 中是否存在对应 ID）
4. 检查存档完整性校验是否通过
5. 检查 Instance 数据是否包含 Definition 数据（违反不变量4）

---

## 如果新增存档迁移函数

允许：
- 新增迁移函数（V2→V3）
- 修复现有迁移函数的 Bug
- 优化迁移函数性能

禁止：
- 修改已有迁移函数的输出格式
- 跳过中间版本直接迁移（必须逐版本）
- 迁移函数修改输入数据（必须不可变）

优先检查：
- 迁移函数是否为纯函数（不依赖外部状态）
- 迁移函数是否验证输入输出数据合法性
- 迁移函数链是否覆盖所有历史版本
- 迁移后数据是否通过完整性校验
- 是否有对应的单元测试

---

## 如果修改最低支持版本

允许：
- 调整支持的 MAJOR 版本数量（当前：3 个）
- 更新最低支持版本号

禁止：
- 不提供迁移路径就提高最低支持版本
- 支持超过 3 个 MAJOR 版本的旧存档

优先检查：
- 迁移函数链是否覆盖新的最低支持版本到当前版本
- 低于最低支持版本时是否拒绝加载 + ERROR 日志 + 提示用户
- 是否有测试覆盖版本拒绝场景
