# 战斗回放领域

Version: 1.0
Status: Proposed

战斗回放领域管理战斗的确定性重放，基于审计事件流重新执行战斗过程。回放 = 从初始状态 + 重放审计事件，不是逐帧录制。

核心原则：
- 确定性优先：相同初始条件 + 相同事件流 → 相同结果
- 事件驱动：只记录决策结果，不记录逐帧状态
- 与审计模块天然集成：审计事件是唯一事实源
- 回放期间不接收任何用户输入

---

# 术语定义

## 战斗回放（Battle Replay）

从初始状态 + 审计事件流重新执行战斗的过程。通过重放每个决策点的审计事件，从初始条件开始逐步还原完整的战斗过程。

不是逐帧录像。不是视频录制。不是存档加载。不是实时模拟。

关键属性：
- 回放是确定性的：同一回放文件多次回放结果完全相同
- 回放从初始条件开始，按事件序列逐步执行
- 回放期间不接收用户输入
- 回放文件远小于逐帧录制

---

## 审计事件（Audit Event）

战斗中发生的每一个决策结果记录。记录状态转换的结果，不记录中间过程。

不是日志。不是系统诊断。不是调试输出。不是存档数据。

关键属性：
- 每个事件记录一个决策的结果（如伤害值、Buff 应用、位置移动）
- 事件在白名单中注册（未注册的事件不被记录）
- 事件包含上下文信息（回合号、阶段、来源实体）
- 事件支持序列化（用于存储和传输）
- 参见 `shared_layer_rules.md#Audit`

---

## 回放文件（Replay File）

存储初始条件 + 审计事件流的文件。包含回放所需的全部信息。

不是存档。不是视频文件。不是日志文件。不是配置文件。

关键属性：
- 包含初始条件（回合0的完整状态）
- 包含审计事件序列（按时间顺序排列）
- 包含元数据（战斗ID、时长、胜负结果）
- 格式与存档格式独立（不复用 SaveData 格式）

---

## 初始条件（Initial Conditions）

回放开始时还原的战斗状态。包括回合0的单位属性、随机种子、地图配置等。

不是存档。不是完整游戏状态。不是 Definition 数据。不是实时状态。

关键属性：
- 初始条件在战斗初始化时记录
- 只包含战斗相关的状态（不包含 MainMenu 等非战斗状态）
- 包含随机种子（保证回放确定性）
- 包含单位初始属性、地图配置、关卡配置引用
- 初始条件是只读的，回放期间不修改

---

## 确定性（Determinism）

相同初始条件 + 相同事件流 → 相同结果的保证。回放系统的核心约束。

不是固定结果。不是无随机。不是无变化。不是幂等性。

关键属性：
- 确定性依赖随机种子（相同种子 → 相同随机序列）
- 确定性依赖初始条件（相同初始状态 → 相同起点）
- 确定性依赖事件序列（相同事件 → 相同决策路径）
- 任何非确定性因素都会破坏回放

---

## 随机种子（Random Seed）

保证回放确定性的随机数种子。回放时使用相同种子重放随机序列。

不是随机数。不是存档部分。不是配置数据。不是用户输入。

关键属性：
- 随机种子在战斗初始化时生成
- 回放时使用相同种子初始化随机数生成器
- 随机种子存储在初始条件中
- 每次回放使用相同种子

---

## 回放速度（Replay Speed）

回放时的播放速度控制。支持 1x、2x、4x 和暂停。

不是时间缩放。不是帧率控制。不是逻辑加速。不是动画速度。

关键属性：
- 回放速度只影响事件之间的间隔时间
- 回放速度不影响游戏逻辑执行
- 回放速度可随时调整（暂停/恢复）
- 速度值为正整数倍数（1x、2x、4x）

---

# 领域边界

## 本领域负责

- 回放文件的生成（录制）
- 回放文件的加载和播放
- 回放速度控制（1x/2x/4x/暂停）
- 初始条件的记录和还原
- 审计事件的重放执行
- 回放确定性保证

## 本领域不负责

- 审计事件的收集（由 Audit 模块负责）
- 存档的生成和加载（由 Persistence 领域负责）
- 战斗逻辑的执行（由 Core 层 Battle 领域负责）
- 非战斗状态的回放（只回放战斗）
- 用户输入处理（回放期间禁止输入）
- UI 界面（由 UI 层负责）

## 跨领域通信方式

| 通知内容 | 通信方式 | 目标领域 |
|----------|----------|----------|
| 初始条件记录 | Resource 读取 | Core 层（Battle 领域） |
| 审计事件流 | Resource 读取 | Audit 模块 |
| 回放开始/结束 | Message | UI 层 |
| 回放进度更新 | Message | UI 层 |
| 回放错误 | Message | UI 层 |

# 生命周期

## 状态列表

| 状态 | 含义 | 可转换到 |
|------|------|----------|
| Idle | 无回放 | Recording, Playing |
| Recording | 正在录制战斗事件 | Idle, RecordError |
| RecordError | 录制失败 | Idle |
| Playing | 正在播放回放 | Idle, Paused, PlayError |
| Paused | 回放暂停 | Playing, Idle |
| PlayError | 播放失败 | Idle |

## 状态转换图

```
Idle → Recording → Idle
  ↓       ↓
  ↓    RecordError → Idle
  ↓
Playing → Paused → Playing
  ↓          ↓
PlayError   Idle
  ↓
Idle
```

## 转换条件

| 从 | 到 | 条件 |
|----|-----|------|
| Idle | Recording | 战斗开始，初始化初始条件 |
| Recording | Idle | 战斗结束，生成回放文件 |
| Recording | RecordError | 录制过程中发生错误 |
| RecordError | Idle | 用户确认错误或自动重试 |
| Idle | Playing | 加载回放文件并开始播放 |
| Playing | Paused | 用户触发暂停 |
| Paused | Playing | 用户触发恢复 |
| Playing | Idle | 回放结束或用户停止 |
| Paused | Idle | 用户停止回放 |
| Playing | PlayError | 播放过程中发生错误 |
| PlayError | Idle | 用户确认错误或自动重试 |

# 不变量

## 不变量1：回放必须是确定性的

任意时刻：

同一回放文件，无论回放多少次，结果完全相同。回放期间的所有随机数必须由随机种子控制。

违反表现：

两次回放结果不同（伤害值、暴击、Buff 触发等不一致）。回放结果与原始战斗结果不一致。

---

## 不变量2：回放只记录决策结果

任意时刻：

回放文件中只记录每个决策的结果（伤害值、Buff 应用、位置移动），不记录逐帧状态（不记录每帧的动画进度、粒子效果等）。

违反表现：

回放文件膨胀。回放变成逐帧播放而非事件重放。回放速度控制失效。

---

## 不变量3：回放文件不包含随机数

任意时刻：

回放文件中只包含随机种子，不包含随机数序列。回放时使用相同种子重新生成随机数序列。

违反表现：

回放文件膨胀。不同随机数生成器实现可能导致结果不一致。

---

## 不变量4：回放期间不接收用户输入

任意时刻：

回放播放期间，系统不处理任何用户输入（键盘、鼠标、手柄）。回放完全由事件序列驱动。

违反表现：

用户输入干扰回放执行，导致回放结果不一致。回放变成交互式模拟而非确定性重放。

---

## 不变量5：回放只复盘战斗状态

任意时刻：

回放只处理战斗相关的状态（单位属性、地图、回合、Buff 等）。不涉及 MainMenu、LevelSelect 等非战斗状态。

违反表现：

回放尝试恢复非战斗状态，导致状态不一致或崩溃。

---

# 业务规则

## 规则1：回放录制时机

允许：
- 在战斗开始时初始化初始条件
- 在战斗进行中记录审计事件
- 在战斗结束时生成回放文件

禁止：
- 在战斗开始前录制（初始条件不完整）
- 在战斗结束后录制（事件流不完整）
- 录制过程中修改游戏逻辑

必须：
- 初始条件在战斗初始化时完整记录
- 所有审计事件按时间顺序记录
- 战斗结束时立即生成回放文件

---

## 规则2：回放确定性保证

允许：
- 使用随机种子控制随机数序列
- 使用确定性算法执行游戏逻辑
- 使用固定时间步长模拟

禁止：
- 回放期间使用系统时间作为随机源
- 回放期间处理用户输入
- 回放期间修改游戏规则

必须：
- 所有随机数由随机种子控制
- 回放使用与原始战斗相同的初始条件
- 回放使用与原始战斗相同的事件序列

---

## 规则3：回放文件格式

允许：
- 使用 JSON 格式存储（开发阶段）
- 使用二进制格式存储（生产阶段）
- 在回放文件中包含元数据

禁止：
- 回放文件格式与存档格式混淆
- 回放文件包含 Definition 数据
- 回放文件包含逐帧状态

必须：
- 回放文件包含初始条件
- 回放文件包含审计事件序列
- 回放文件包含元数据（战斗ID、时长、胜负结果）

---

## 规则4：回放速度控制

允许：
- 调整回放速度（1x、2x、4x）
- 暂停和恢复回放
- 回放速度可随时调整

禁止：
- 修改游戏逻辑的执行速度
- 修改渲染帧率
- 调整非整数倍速度（如 1.5x）

必须：
- 速度变化不影响游戏逻辑
- 暂停时冻结所有事件执行
- 恢复时从暂停点继续

---

## 规则5：回放错误处理

允许：
- 回放文件损坏时返回错误
- 事件序列不完整时提前终止
- 初始条件与当前版本不兼容时返回错误

禁止：
- 忽略回放错误继续播放
- 修改回放文件修复错误
- 回放错误时回退到交互模式

必须：
- 回放错误时停止播放并通知用户
- 回放错误不影响原始战斗数据
- 回放错误记录到日志

---

# 流程管线

## 管线1：回放录制管线

```
战斗初始化记录初始条件 → 每个决策点记录审计事件 → 战斗结束写入回放文件
```

### Step1：记录初始条件

输入：战斗初始化状态（回合0）
处理：记录所有单位初始属性、地图配置、关卡配置引用、随机种子
输出：InitialConditions 结构
禁止：记录不完整的初始条件、记录 Definition 数据

### Step2：记录审计事件

输入：战斗进行中的 DomainEvent
处理：将事件追加到事件序列，验证事件在白名单中
输出：事件序列（内存中）
禁止：记录未在白名单中的事件、记录逐帧状态

### Step3：写入回放文件

输入：InitialConditions + 事件序列 + 元数据
处理：序列化为回放文件格式，写入磁盘
输出：回放文件路径
禁止：跳过元数据写入、写入 Definition 数据

---

## 管线2：回放执行管线

```
加载回放文件 → 还原初始条件 → 逐步消耗审计事件 → 每个事件触发对应游戏逻辑 → 回放结束
```

### Step1：加载回放文件

输入：回放文件路径
处理：读取文件，反序列化为 ReplayData 结构，验证完整性
输出：ReplayData 结构
禁止：加载损坏的文件、跳过完整性验证

### Step2：还原初始条件

输入：InitialConditions
处理：根据初始条件重建回合0的完整战斗状态（单位属性、地图、随机种子）
输出：回合0的战斗状态
禁止：使用当前游戏状态替代初始条件、跳过随机种子初始化

### Step3：逐步消耗审计事件

输入：审计事件序列
处理：按时间顺序逐个消耗事件，每个事件触发对应的游戏逻辑
输出：事件执行完成确认
禁止：跳过事件、乱序执行事件、执行未在事件序列中的逻辑

### Step4：回放结束

输入：所有事件执行完成
处理：生成回放结束通知，清理回放状态
输出：回放结束确认
禁止：回放结束后保留游戏状态、回放结束后继续执行事件

# 数据结构

## ReplayData（回放数据）

职责：回放文件的顶层结构，包含回放所需的全部信息

结构：
- initial_conditions：InitialConditions — 回放开始时的战斗状态
- events：Vec<AuditEvent> — 审计事件序列（按时间顺序）
- metadata：ReplayMetadata — 回放元数据

要求：
- initial_conditions 必须完整（回合0的完整状态）
- events 必须按时间顺序排列
- metadata 包含战斗ID、时长、胜负结果

---

## InitialConditions（初始条件）

职责：回放开始时还原的战斗状态

结构：
- turn_number：u32 — 回合号（固定为0）
- units：Vec<UnitSnapshot> — 所有单位的初始属性
- map_config：String — 地图配置引用（Strong ID）
- level_config：String — 关卡配置引用（Strong ID）
- random_seed：u64 — 随机种子
- buff_states：Vec<BuffSnapshot> — 初始 Buff 状态

要求：
- 只包含战斗相关的状态
- 不包含 Definition 数据（只记录引用 ID）
- random_seed 用于保证回放确定性

---

## UnitSnapshot（单位快照）

职责：单位在初始条件中的状态

结构：
- unit_id：String — 单位 Strong ID
- template_id：String — 单位模板 Strong ID（用于从 Registry 恢复 Definition）
- position：Position — 初始位置
- attributes：AttributeSet — 初始属性值
- tags：GameplayTags — 初始标签

要求：
- unit_id 必须在当前 Registry 中存在
- template_id 用于恢复 Definition 引用
- attributes 只包含运行时可变属性

---

## ReplayMetadata（回放元数据）

职责：回放文件的辅助信息

结构：
- battle_id：String — 战斗唯一标识
- duration：u64 — 战斗时长（秒）
- result：BattleResult — 战斗结果（胜负）
- total_turns：u32 — 总回合数
- created_at：u64 — 回放生成时间戳

要求：
- 元数据不影响回放执行
- 元数据用于 UI 显示和管理

---

## ReplaySpeed（回放速度）

职责：回放播放速度控制

结构：
- multiplier：u32 — 速度倍数（1、2、4）

要求：
- 只支持正整数倍数
- 暂停时 multiplier 为 0
- 速度变化不影响游戏逻辑

---

# 禁止事项

禁止：逐帧录制输入

原因：逐帧录制丧失确定性保证。输入依赖用户操作，不同用户操作导致不同结果，回放无法还原。

违反后果：回放结果与原始战斗不一致。回放文件膨胀。回放速度控制失效。

---

禁止：回放过程中允许随机数不使用种子

原因：不使用种子的随机数是不确定性的。每次回放产生不同的随机序列，破坏回放确定性。

违反后果：两次回放结果不同。回放无法复现原始战斗结果。

---

禁止：回放过程修改游戏规则

原因：回放的目的是复现原始战斗过程。修改游戏规则会导致回放结果与原始战斗不一致。

违反后果：回放结果与原始战斗不一致。回放失去调试和验证价值。

---

禁止：回放期间处理用户输入

原因：用户输入会干扰回放执行。回放应该完全由事件序列驱动。

违反后果：回放变成交互式模拟。回放结果不确定。回放失去确定性保证。

---

禁止：回放文件包含逐帧状态

原因：逐帧状态导致回放文件膨胀。回放应该基于事件重放，不是逐帧播放。

违反后果：回放文件大小远超事件重放方案。回放速度控制变为帧率控制，与游戏逻辑耦合。

---

禁止：回放只复盘非战斗状态

原因：回放只处理战斗相关的状态。非战斗状态（MainMenu、LevelSelect 等）不在回放范围内。

违反后果：回放尝试恢复非战斗状态，导致状态不一致或崩溃。

---

禁止：回放错误时回退到交互模式

原因：回放错误应该停止播放并通知用户。回退到交互模式会导致状态不一致。

违反后果：回放错误后游戏状态不一致。用户操作可能基于错误状态。

---

# AI 修改规则

## 如果新增审计事件类型

允许：
- 在 Audit 模块的白名单中注册新事件类型
- 在回放录制管线中记录新事件
- 在回放执行管线中处理新事件

禁止：
- 未在白名单中注册就记录事件
- 事件中包含逐帧状态
- 事件中包含随机数（只包含决策结果）

优先检查：
- 新事件是否在 Audit 白名单中注册
- 新事件是否包含随机数（应该只包含结果）
- 新事件的序列化格式是否与现有事件一致

---

## 如果修改回放文件格式

允许：
- 新增可选字段
- 修改序列化格式
- 优化文件大小

禁止：
- 删除已有字段
- 修改已有字段的类型
- 格式变更不递增版本号

优先检查：
- 新字段是否有默认值（保证旧文件兼容）
- 版本号是否已递增
- 反序列化时新字段缺失是否能正确处理

---

## 如果修改回放执行逻辑

允许：
- 优化回放性能
- 新增回放速度选项
- 改善回放错误处理

禁止：
- 回放期间修改游戏规则
- 回放期间处理用户输入
- 跳过事件序列中的事件

优先检查：
- 回放执行是否与原始战斗逻辑一致
- 回放速度变化是否不影响游戏逻辑
- 回放错误是否正确停止并通知用户

---

## 如果修改初始条件记录

允许：
- 新增初始条件字段
- 优化初始条件大小
- 改善初始条件还原逻辑

禁止：
- 记录 Definition 数据
- 记录不完整的初始条件
- 跳过随机种子记录

优先检查：
- 初始条件是否包含所有战斗相关状态
- 初始条件是否不包含 Definition 数据
- 随机种子是否正确记录和恢复

排查顺序：
1. 检查随机种子是否在初始条件中正确记录和恢复
2. 检查审计事件是否在白名单中注册
3. 检查事件序列是否按时间顺序排列
4. 检查回放执行是否与原始战斗逻辑一致
5. 检查回放文件格式版本号是否正确
