# 测试体系领域

Version: 1.1
Status: Proposed

测试体系领域管理多层测试系统的规则、组织和执行，确保游戏规则、架构落地和内容完整性的分层验证。

核心原则：
- 分层对齐：测试层级与架构层级一一对应，每层只验证自身职责
- 权责封闭：领域规则测试绝不依赖 Bevy 运行时，系统测试绝不硬编码业务数值
- 确定性优先：所有可自动化测试必须可复现，随机、时间、迭代顺序全部可控

## 宪法合规矩子

| 条款 | 级别 | 落地规则 |
|------|------|----------|
| 18.1.1 测试优先 | 🟥 | 所有功能必须优先编写测试，其次才是手工验证 |
| 18.1.2 Bug 修复流程 | 🟥 | 发现 Bug 后必须先编写重现测试，再修复 Bug |
| 18.2.1 三层测试体系 | 🟥 | 单元（纯函数）→ 集成（Feature）→ 回放（Battle Replay） |
| 18.2.2 SRPG 测试优先 | 🟩 | 复杂 SRPG 逻辑优先使用 Battle Replay 测试，而非 BDD |
| 18.3.1 Builder 模式 | 🟥 | 核心测试必须使用 Builder 模式构造测试数据 |
| 18.3.2 Golden Test | 🟥 | 战斗回放等稳定输出必须使用金文件对比测试 |
| 18.4.1 战斗完全可重现 | 🟥 | 相同初始条件 + 相同输入 + 相同 RNG 种子 → 完全一致结果 |
| 18.4.2 禁止系统时间 | 🟥 | 核心战斗逻辑禁止直接读取系统时间 |

# 术语定义

## 领域单元测试（Domain Unit Test）

纯 Rust、零 Bevy 依赖的单条游戏规则验证，是测试金字塔的底座。位于 `core/[领域模块]/tests/`。

不是集成测试。不是系统测试。

关键属性：
- 绝对禁止引入 Bevy 任何依赖，不使用 ECS、不加载资源
- 不依赖任何 content 配置数据，测试用例硬编码入参
- 全部为纯函数测试，输入输出一一对应，毫秒级执行
- 每条领域规则至少对应一组单元测试
- 覆盖纯数值公式、单领域逻辑、边界与异常

---

## 领域集成测试（Domain Integration Test）

跨模块规则链验证，多个领域模块组合后的规则一致性验证，不涉及引擎运行时。位于 `core/[聚合根模块]/tests/`。

不是 E2E 测试。不是单元测试。

关键属性：
- 仍然是纯 Rust 代码，不启动 Bevy
- 使用领域对象直接组合，模拟完整战斗上下文
- 验证跨领域链路（技能释放 → 伤害结算 → Buff 挂载 → 属性变更）
- 验证状态流转（回合开始 → 行动 → 结算 → 回合结束）
- 验证聚合不变量（战斗结束后所有单位状态合法）

---

## 系统集成测试（System Integration Test）

Bevy headless 模式下的架构落地验证，确认 ECS 系统、事件、组件的交互符合设计预期。位于 `tests/integration/`。

不是业务规则测试。不是性能测试。

关键属性：
- 使用 Bevy 原生测试能力，运行 headless 无头模式
- 通过 `World::new()` 手动构造最小测试环境，只加载被测插件
- 重点测「架构落地」，不测「业务规则」——业务规则已在领域层测完
- 覆盖插件装配、系统调度、事件流转、ECS 通信

---

## 端到端测试（E2E Test）

完整游戏流程的黑盒验证，作为 CI 门禁的最后一道防线。位于项目根目录 `tests/`。

不是底层测试。不是白盒测试。

关键属性：
- 以黑盒方式调用游戏公共接口，不访问任何内部私有结构
- 用例只覆盖核心主干，不追求覆盖细节
- 执行耗时控制在分钟级，避免成为 CI 瓶颈
- 覆盖主流程测试、存档兼容测试、MOD 兼容测试、错误路径测试
- 基于回放的回归测试：使用录制的命令流进行回归验证，同一输入必须得到完全相同的结果

> **优化来源**: docs/architecture/testing_architecture.md

---

## 测试沙盒（Testbed）

可交互、可视化、可调试的开发工具，是 SRPG 项目长期开发的效率杠杆。位于项目根目录 `testbeds/`。

不是自动化测试。不是生产代码。

关键属性：
- 每个 Testbed 必须可以独立运行
- 每个 Testbed 必须有 README 说明使用方法
- Testbeds 代码可以引用内部实现（开发工具权限更宽）
- Testbeds 代码永不进入 Release 构建
- 包含 battle_simulator、skill_playground、ai_debug_arena、balance_workbench、replay_validator
- Testbed 录制的流程可直接导出为回归测试用例，是可复用的调试资产
- replay_validator 批量重放历史回放文件，校验同一输入是否得到完全相同的结果，是确定性保障的核心工具

> **优化来源**: docs/architecture/testing_architecture.md

---

## 测试夹具（Test Fixture）

可复用的测试数据构造器，位于 `shared/testing/fixtures.rs`。

不是测试用例。不是生产数据。

关键属性：
- 只提供辅助函数，不包含测试逻辑
- 提供标准测试角色创建、标准战斗场景构建
- 支持参数化构造不同规格的测试数据
- 被领域单元测试和领域集成测试共同使用
- 禁止在生产代码中引用

---

## 内容数据测试（Content Data Test）

RON 配置文件的合法性校验，验证游戏内容数据的引用完整性、数值合法性和格式合规性。位于 `content/tests/`。

不是游戏逻辑测试。不是代码测试。

关键属性：
- 覆盖引用完整性（技能引用的 Buff ID 是否存在）
- 覆盖数值合法性（伤害是否为正、冷却时间是否越界）
- 覆盖格式合规性（配置文件是否符合 schema）
- 覆盖规则一致性（配置数值是否符合领域规则约束）

---

# 领域边界

## 本领域负责

- 测试分层策略（领域单元 → 领域集成 → 系统集成 → E2E，金字塔比例管控）
- 测试编写规范（命名、组织、质量标准）
- 测试公共工具库（`shared/testing/`：fixtures、deterministic、assertions）
- Testbed 沙盒定义（battle_simulator、skill_playground 等五大沙盒）
- CI 门禁规则（哪些测试失败阻断合并）
- 内容数据测试（RON 配置校验）

## 本领域不负责

- 具体游戏规则的实现（由各业务领域负责）
- 测试用例的具体编写（由 @test-guardian 和 @feature-developer 负责）
- 代码审查中的测试覆盖率检查（由 @code-reviewer 负责）
- Bevy 引擎的测试基础设施（由 Infrastructure 层负责）
- 测试数据的具体生成（由各业务领域的 fixture 负责）

## 跨领域通信方式

| 通知内容 | 通信方式 | 目标领域 |
|----------|----------|----------|
| 测试失败报告 | CI 门禁 | 所有领域 |
| 测试夹具 | 函数调用 | 各业务领域的 tests/ 目录 |
| 确定性 RNG | 函数调用 | 需要确定性测试的领域 |
| Testbed 调试数据 | Resource 读取 | 各业务领域（调试阶段） |

# 生命周期

本领域无游戏状态机，测试体系本身按金字塔层级组织，各层之间无运行时状态转换。

测试执行按 CI 流水线顺序：

```
编译检查 → Lint 检查 → 单元测试 → 领域集成 → 系统集成 → 内容数据 → E2E → 性能基准
```

任何一层失败即阻断后续执行。

# 不变量

## 不变量1：领域单元测试零 Bevy 依赖

任意时刻：

领域单元测试的 `use` 语句只允许标准库和被测业务模块的纯 Rust 代码。禁止引入 `bevy::prelude::*`、`bevy::app::*` 等 Bevy 依赖。

违反表现：

领域单元测试文件中出现 `use bevy::prelude::*` 或 `App::new()` 等 Bevy 类型。

---

## 不变量2：领域单元测试不依赖内容配置数据

任意时刻：

领域单元测试的入参必须硬编码或通过测试夹具构造，禁止从 `assets/` 目录加载 RON 配置文件。

违反表现：

领域单元测试中出现 `load_asset("assets/skills/fireball.ron")` 等配置加载调用。

---

## 不变量3：测试独立

任意时刻：

任意两个测试函数之间不共享可变状态。每个测试函数的执行结果不依赖其他测试的执行顺序或执行结果。

违反表现：

测试 A 写入全局 Resource，测试 B 读取该 Resource 并依赖其值。测试 A 和 B 并行执行时结果不一致。

---

## 不变量4：测试不依赖特定 RNG 输出

任意时刻：

所有可自动化测试必须使用确定性 RNG（固定种子），禁止依赖系统随机数或未固定种子的随机序列。

违反表现：

测试在不同机器上执行时，因随机数序列不同导致通过/失败不一致。

---

## 不变量5：每个测试函数只验证一种行为

任意时刻：

每个 `#[test]` 函数只验证一个明确的行为或规则。一个测试函数中禁止包含多个不相关的断言组。

违反表现：

一个测试函数同时验证伤害计算和 Buff 施加，伤害测试通过但 Buff 测试失败时，无法快速定位问题。

---

## 不变量6：Testbed 永不进入 Release 构建

任意时刻：

Testbeds 代码通过 feature gate 或条件编译隔离，Release 构建中不包含任何 Testbed 代码。

违反表现：

Release 构建的二进制文件中包含 `testbeds/battle_simulator/main.rs` 的代码，增加二进制体积。

---

# 业务规则

## 规则0：测试优先原则 [宪法 18.1.1 🟥]

🟥 **所有功能必须优先编写测试，其次才是手工验证。**

必须：
- 🟥 新增功能必须先编写测试用例，再编写实现代码
- 🟥 Bug 修复必须先编写重现测试，再修复 Bug（18.1.2）
- 🟥 修复的 Bug 必须沉淀为永久测试资产（18.4.5）
- 🟩 测试用例应覆盖正常路径和边界条件
- 🟩 测试失败信息应足够定位问题根因

禁止：
- 🟥 新增核心业务系统未附带任何测试用例
- 🟥 Bug 修复后不保留重现测试
- 🟥 为通过测试修改业务逻辑

---

## 规则0.1：Golden Test 规范 [宪法 18.3.2 🟥]

🟥 **战斗回放、配置导出等稳定输出必须使用金文件对比测试。**

必须：
- 🟥 Battle Replay 的输出必须与金文件对比，版本升级后输出变化必须显式确认
- 🟥 配置导出、数值公式输出等稳定结果使用金文件测试
- 🟩 金文件测试纳入 CI 回归测试套件
- 🟩 金文件变化需代码审查确认

禁止：
- 🟥 金文件测试被标记为 ignore
- 🟥 修改核心规则后不更新金文件

允许：
- 金文件变化时显式确认并更新

---

## 规则0.2：Bug→回放→测试规范 [宪法 18.4.3 🟥]

🟥 **所有战斗相关 Bug 必须通过 Battle Replay 重现并转化为永久测试用例。**

必须：
- 🟥 发现战斗 Bug 后，必须先录制回放文件（初始条件 + 命令流）
- 🟥 回放文件作为永久测试资产沉淀
- 🟥 同一回放文件在不同版本执行结果必须一致（Golden Test）
- 🟩 回放验证测试纳入 CI 回归测试套件
- 🟩 Bug 描述和修复说明附在测试用例注释中

禁止：
- 🟥 修复战斗 Bug 但不录制对应的回放测试用例
- 🟥 修复 Bug 后删除重现测试
- 🟥 回放测试用例被标记为 ignore 超过 1 个月

---

## 规则0.3：确定性测试环境 [宪法 18.4.1 🟥]

🟥 **相同初始条件 + 相同输入序列 + 相同 RNG 种子，必须得到完全一致的结果。**

必须：
- 🟥 所有可自动化测试使用确定性 RNG（固定种子）
- 🟥 核心战斗逻辑禁止直接读取系统时间（18.4.2）
- 🟥 使用统一的 GameTime 服务替代系统时间
- 🟩 测试数据通过 Builder 模式构造（18.3.1）
- 🟩 测试可并行执行，不依赖外部状态

禁止：
- 🟥 测试中使用 `rand::random()` 等非确定性随机源
- 🟥 测试中使用 `std::time::Instant` 等系统时间
- 🟥 测试之间共享可变状态

---

## 规则1：测试金字塔比例 [宪法 18.2.1 🟥]

🟥 **五层测试体系**（扩展宪法 18.2.1 三层模型）：领域单元测试（纯函数）→ 领域集成测试（跨模块规则链）→ 系统集成测试（ECS 架构落地）→ E2E 回归测试（黑盒主流程）→ Testbed 沙盒（交互式调试）

必须：
- 🟥 领域单元测试占比 70%，毫秒级执行
- 🟥 领域集成测试占比 15%，百毫秒级执行
- 🟥 系统集成测试占比 10%，秒级执行
- 🟥 E2E 回归测试占比 5%，分钟级执行
- 🟩 Testbed 沙盒为可选开发调试工具，不纳入 CI 自动化
- 🟩 SRPG 复杂逻辑优先使用 Battle Replay 测试（18.2.2）

禁止：
- 🟥 系统集成测试中硬编码业务数值（如具体伤害值）
- 🟥 E2E 测试覆盖底层实现细节
- 🟥 用 E2E 测试替代单元测试的规则验证

---

## 规则2：每条领域规则对应单元测试

必须：
- 每条领域规则文档（`docs/domain/*_rules.md`）至少对应一组单元测试
- 单元测试文件名描述测试范围（如 `damage_calculation.rs`）
- 测试函数名描述预期行为（如 `physical_damage_respects_armor_reduction`）

禁止：
- 领域规则存在但没有对应单元测试
- 单元测试只覆盖正常路径，不覆盖边界条件

---

## 规则3：系统集成测试只测架构落地

必须：
- 使用 Bevy headless 模式运行
- 只验证插件装配、系统调度、事件流转
- 通过 `World::new()` 构造最小测试环境

禁止：
- 在系统集成测试中硬编码业务规则数值
- 在系统集成测试中验证伤害计算、Buff 生效等业务逻辑
- 系统集成测试依赖领域层的测试夹具

---

## 规则4：E2E 测试黑盒方式

必须：
- 只使用游戏公共 API
- 只覆盖核心主干流程
- 执行耗时控制在分钟级

禁止：
- E2E 测试访问内部私有结构
- E2E 测试依赖特定执行顺序
- E2E 测试中直接操作 ECS World

---

## 规则5：CI 门禁 [宪法 20.7 🟥]

🟥 **cargo test 全部测试通过才能合并入主分支。**

必须：
- 🟥 领域单元测试失败阻断合并
- 🟥 领域集成测试失败阻断合并
- 🟥 系统集成测试失败阻断合并
- 🟥 E2E 测试失败阻断合并
- 🟩 配置数据校验全部通过才能合并（12.2.2）

允许：
- 性能基准回归 >10% 标记警告，不阻断合并
- Testbed 手动运行，不纳入 CI 自动化

禁止：
- 🟥 跳过失败的测试直接合并
- 🟥 标记测试为 ignore 来绕过 CI 失败

---

## 规则6：shared/testing 提供公共工具

必须：
- fixture builder 提供标准测试角色和战斗场景构造
- 确定性 RNG 提供固定种子随机数生成
- 自定义断言提供数值容差、状态不变量、错误类型校验
- 所有公共工具只在测试配置下编译

禁止：
- 在生产代码中引用 shared/testing 模块
- 在 shared/testing 中包含业务断言逻辑
- 测试夹具中硬编码业务规则判断

---

## 规则7：测试资产作为共享调试资产

必须：
- 测试夹具（fixtures）同时作为开发调试资产，被全项目复用
- Testbed 沙盒录制的流程可直接导出为回归测试用例
- 测试失败时自动保存状态快照，便于复现和调试
- 快照文件使用 RON 格式，包含帧号、时间戳、RNG 种子

禁止：
- 测试资产仅限测试使用，不允许开发调试复用
- 测试失败时丢弃调试信息

允许：
- 开发者使用 Testbed 沙盒交互式调试
- 将 Testbed 录制的流程转换为自动化回归测试

> **优化来源**: docs/architecture/testing_architecture.md

---

## 规则8：测试维护生命周期

测试有明确的生命周期阶段：新建 → 活跃 → 过时 → 废弃 → 删除。

### 测试废弃协议（Test Deprecation Protocol）

当业务规则变更导致测试不再适用时，禁止直接删除测试，必须遵循以下流程：

1. 标记 `#[deprecated]` 并注明废弃原因和版本
2. 标记 `#[ignore]` 使其在 CI 中跳过
3. 保留 1 个版本周期后方可删除
4. 废弃测试仍保留注释说明迁移去向

### 测试审查检查清单

每月审查一次测试健康度：

| 检查项 | 方法 | 处理方式 |
|--------|------|----------|
| 测试是否仍覆盖有效规则 | 对照领域规则文档 | 无对应规则 → 标记废弃 |
| 测试是否在 CI 中运行 | 检查 `#[ignore]` 标记 | 被忽略 > 3 个月 → 标记废弃 |
| 测试是否与其他测试重复 | 代码审查 | 重复 → 合并或删除 |
| 测试是否硬编码过时数值 | 对照 RON 配置 | 数值过时 → 更新 fixtures |
| 测试执行时间是否异常 | CI 报告分析 | > 10s → 优化或拆分 |

约束：
- 🟥 禁止直接删除仍有参考价值的测试
- 🟥 禁止通过删除测试来消除 CI 失败
- 🟩 废弃测试必须注明原因和迁移去向
- 🟩 测试审查每月执行一次

> **优化来源**: docs/architecture/testing_architecture.md

# 流程管线

## CI 测试执行管线

```
编译检查 → Lint 检查 → 单元测试 → 领域集成 → 系统集成 → 内容数据 → E2E → 性能基准
```

### 编译检查

输入：全部源代码
处理：`cargo check` 确保代码可编译
输出：编译通过/失败
禁止：编译失败时继续执行后续步骤

### Lint 检查

输入：可编译的源代码
处理：`cargo clippy` 检查代码质量
输出：lint 警告/错误列表
禁止：忽略 clippy 错误继续执行

### 单元测试

输入：领域模块代码
处理：`cargo test --lib` 执行所有领域单元测试
输出：测试通过/失败报告
禁止：单元测试失败时跳过继续

### 领域集成

输入：跨模块组合代码
处理：`cargo test --test integration` 执行领域集成测试
输出：测试通过/失败报告
禁止：集成测试失败时跳过继续

### 系统集成

输入：插件和 ECS 系统代码
处理：`cargo test --test system` 执行系统集成测试
输出：测试通过/失败报告
禁止：系统测试失败时跳过继续

### 内容数据

输入：RON 配置文件
处理：`cargo test --test content` 执行内容数据校验
输出：配置校验通过/失败报告
禁止：配置校验失败时跳过继续

### E2E

输入：完整游戏公共 API
处理：`cargo test --test e2e` 执行端到端回归测试
输出：测试通过/失败报告
禁止：E2E 失败时跳过继续

### 性能基准

输入：性能基准测试代码
处理：`cargo bench` 执行性能基准
输出：性能回归报告（>10% 标记警告）
禁止：性能基准作为阻断门禁

## 测试编写管线

```
识别被测规则 → 选择测试层级 → 构造夹具 → 编写断言 → 独立性验证
```

### 识别被测规则

输入：被测行为的领域规则文档
处理：确认被测规则的精确语义和边界条件
输出：明确的被测行为描述
禁止：在未阅读领域规则文档的情况下编写测试

### 选择测试层级

输入：被测行为描述
处理：根据是否涉及 Bevy 运行时选择层级
输出：目标测试层级（单元/集成/系统/E2E）
禁止：在系统集成测试层级验证业务规则

### 构造夹具

输入：测试所需的数据结构
处理：使用 shared/testing 的 fixture builder 构造测试数据
输出：可复用的测试数据
禁止：在测试中硬编码业务数值（应使用 fixtures）

### 编写断言

输入：预期行为描述
处理：编写精确的断言，覆盖正常路径和边界条件
输出：完整的测试函数
禁止：一个测试函数验证多个不相关行为

### 独立性验证

输入：编写完成的测试
处理：验证测试可并行执行、不依赖外部状态
输出：通过独立性验证的测试
禁止：共享可变状态的测试通过审查

# 数据结构

## TestGame（E2E 测试门面）

职责：封装游戏公共 API，提供 E2E 测试的黑盒操作接口

结构：
- state — 当前 AppState 状态
- world — Bevy World 实例

要求：
- 只暴露公共 API（start_stage、select_unit、move_unit、attack、end_turn）
- 不暴露内部 ECS 组件、Resource 的直接访问
- 内部状态对外不可见

---

## DeterministicRng（确定性随机数）

职责：提供固定种子的伪随机数生成，确保测试可复现

结构：
- seed — 随机种子
- state — 当前状态

要求：
- 使用固定算法（如 LCG），保证跨平台一致性
- 提供 next_u32 方法
- 禁止使用系统随机数

---

## BattleContext（战斗测试上下文）

职责：封装多单位战斗场景的测试数据

结构：
- party — 玩家单位列表
- enemies — 敌方单位列表

要求：
- 由 fixture builder 构造
- 支持不同规模的战斗场景（1v1、团队战）
- 单位属性可参数化

# 禁止事项

禁止：领域层测试引入 Bevy 依赖

原因：领域单元测试的核心价值是毫秒级执行和纯函数验证。引入 Bevy 会大幅降低执行速度，且违反分层对齐原则。

违反后果：单元测试执行时间从毫秒级退化到秒级，测试金字塔比例失衡，CI 执行时间膨胀。

---

禁止：测试之间共享可变状态

原因：共享可变状态导致测试执行顺序敏感，无法并行执行，且失败时难以定位根因。

违反后果：测试在并行执行时随机失败（flaky tests），CI 结果不可靠。

---

禁止：为通过测试修改业务逻辑

原因：测试的目的是验证业务逻辑的正确性，如果业务逻辑为适配测试而修改，测试失去验证价值。

违反后果：业务逻辑被测试倒退，规则语义被篡改，真实 Bug 被掩盖。

---

禁止：为通过测试修改测试适配错误逻辑

原因：测试失败时应修复业务逻辑，不是修改测试来让错误的行为通过验证。

违反后果：测试变成错误逻辑的背书，测试通过但游戏行为不正确。

---

禁止：删除测试来消除失败

原因：删除失败的测试会降低测试覆盖率，隐藏已发现的问题。

违反后果：已发现的 Bug 在后续版本中复发，测试覆盖率持续下降。

---

禁止：测试函数验证多个不相关的行为

原因：一个测试函数验证多个行为时，失败时无法快速定位是哪个行为出了问题。

违反后果：调试时间增加，一个测试失败可能掩盖多个独立问题。

---

禁止：E2E 测试访问内部私有结构

原因：E2E 测试的目的是验证公共 API 的可用性，访问内部结构会破坏黑盒原则。

违反后果：E2E 测试与内部实现耦合，重构时大量 E2E 测试需要同步修改。

---

禁止：测试中硬编码业务数值而不用 fixtures

原因：硬编码的业务数值分散在各测试文件中，业务规则变更时需要逐个修改，维护成本高。

违反后果：业务规则变更时大量测试因数值过期而失败，修改成本高且容易遗漏。

---

禁止：Testbed 代码进入 Release 构建

原因：Testbed 是开发调试工具，包含大量调试逻辑和可视化代码，不应出现在发布版本中。

违反后果：Release 二进制体积膨胀，暴露内部调试接口，潜在的安全风险。

---

禁止：跳过失败的测试直接合并

原因：CI 门禁的意义在于保证合并前代码质量。跳过失败测试等于放弃质量保障。

违反后果：已知问题被带入主分支，多人协作时问题扩散，修复成本指数增长。

# AI 修改规则

## 如果新增测试层级或测试工具

允许：
- 在 `shared/testing/` 中新增辅助函数
- 在 `testbeds/` 中新增沙盒工具
- 调整测试金字塔比例（需架构评审）

禁止：
- 在 shared/testing 中包含业务逻辑
- 在生产代码中引用 shared/testing
- 创建新的顶级测试目录（应放在现有目录结构中）

优先检查：
- 新工具是否被两个以上测试文件使用
- 新工具是否零业务依赖（只用标准库和共享层类型）
- Testbed 代码是否通过 feature gate 隔离

---

## 如果修改测试编写规范

允许：
- 增加测试命名约束
- 增加断言质量要求
- 调整 CI 门禁规则

禁止：
- 允许领域层测试引入 Bevy 依赖
- 放宽测试独立性要求
- 允许删除测试来消除失败

优先检查：
- 新规范是否符合测试金字塔比例
- 新规范是否增加不必要的测试负担
- 新规范是否与现有测试兼容

---

## 如果新增 Testbed 沙盒

允许：
- 在 `testbeds/` 目录下创建新沙盒
- 引用内部实现（开发工具权限更宽）
- 提供 README 说明使用方法

禁止：
- Testbed 代码进入 Release 构建
- Testbed 之间共享可变状态
- Testbed 替代自动化测试

优先检查：
- 沙盒是否能独立运行
- 是否有明确的使用文档
- 是否通过 feature gate 与 Release 构建隔离

---

## 如果测试失败

排查顺序：
1. 检查测试是否引入了 Bevy 依赖（违反不变量1）
2. 检查测试是否依赖内容配置数据（违反不变量2）
3. 检查测试之间是否共享可变状态（违反不变量3）
4. 检查是否依赖特定 RNG 输出（违反不变量4）
5. 检查一个测试函数是否验证了多个行为（违反不变量5）
6. 检查测试断言是否精确匹配预期行为
7. 检查被测代码是否违反领域规则文档中的约束
