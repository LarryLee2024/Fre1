# 资产组织 领域规则

Version: 1.0
Status: Draft
Applies To: 资产组织与管理

> **优化来源**: docs/architecture/asset-organization.md

资产组织领域管理美术资产的物理组织、命名规范、外包协作流程和内容包组织，确保美术团队可独立工作，不依赖程序团队即可提交资源。

核心原则：
- 🟩 **1.1.3 Rule/Content 强制分离**：代码是规则，配置是内容（宪法条款 1.1.3）
- 🟩 **12.4.1 平衡参数全配置化**：所有数值平衡参数必须放入配置文件（宪法条款 12.4.1）
- 三树分离：美术资源、游戏配置、代码逻辑物理隔离
- Content Packs：资源按功能单元组织，而非文件类型
- 外包友好：美术团队只修改美术目录，不触碰代码和配置
- 自动化校验：所有资源提交必须通过自动校验

---

# 1. 统一术语

## 三树分离（Three-Tree Separation）

美术资源、游戏配置、代码逻辑三大类资产在物理目录上彻底隔离。

不是混合存放。不是按文件类型分散到各处。

关键属性：
- assets/：二进制美术资源（精灵图、音频、字体等）
- content/：游戏配置数据（RON 文件）
- src/：游戏逻辑代码
- 三大树各自独立演进，互不干扰
- 美术团队只操作 assets/ 下的子目录

---

## Content Packs（内容包）

资源按**功能单元**组织，一个功能（技能、Buff、物品）的所有相关资源物理上相邻。

不是按文件类型分散存放。不是按角色类型堆叠。

关键属性：
- 每个功能单元（技能、Buff、物品）拥有独立的资源目录
- 功能单元内的图标、配置、特效文件物理相邻
- 运行时共享资源存放在 _shared/ 目录
- 编辑器模板存放在 _templates/ 目录
- 实验性资源存放在 _wip/ 目录
- 废弃资源存放在 _deprecated/ 目录

---

## 资源命名空间（Asset Namespace）

资源路径直接映射为唯一的命名空间标识，格式为 namespace + category + name。

不是随意路径。不是硬编码前缀。

关键属性：
- 路径结构：content_packs/{namespace}/{category}/{name}
- 例：content_packs/base/skills/fireball/icon.png → namespace: "base", category: "skills", name: "fireball/icon"
- 代码中禁止硬编码路径前缀，必须通过资源解析器访问

---

## 外包美术工作流（Outsourcing Art Workflow）

美术团队在独立分支上工作，只修改美术相关目录，通过自动化校验后合并到主分支。

不是自由提交。不是人工逐文件审查。

关键属性：
- 每个美术人员有独立的工作分支
- 只能修改 assets/ 下的美术子目录
- 提交前必须通过自动化格式校验
- 校验失败时提供明确的修正建议

---

## 资源校验（Asset Validation）

自动化脚本对所有资源提交执行格式、命名、尺寸、引用完整性的校验。

不是人工审查。不是可选流程。

关键属性：
- 格式合规性校验（PNG、OGG、WAV 等标准格式）
- 图片尺寸校验（2 的幂次方、最大 4096x4096）
- 文件大小校验（单文件不超过 5MB）
- 命名规范校验（snake_case、单数名词、无空格/特殊字符）
- 目录结构校验（按角色/类型组织）
- Alpha 通道完整性校验（Sprite 必须使用 Alpha 通道）
- 资源引用完整性校验（所有引用必须指向存在的资源）

---

## 角色资源目录（Character Resource Directory）

每个角色（Character）的所有相关资源在一个独立目录中组织，包含精灵图、动画数据、头像、缩略图和专属特效。

不是散落在多个目录。不是按文件类型分散。

关键属性：
- 每个角色一个独立目录
- 精灵图按动作命名（idle、move、attack、skill、hurt、dead）
- 动画数据与精灵图配套
- 头像包含常规和受伤两种状态
- 专属特效与角色物理相邻

---

## 战斗地图资源（Battle Map Resource）

战斗地图的美术资源和游戏数据分离存放，美术资源在 assets/ 下，游戏配置在 content/ 下。

不是美术和配置混放。不是全部堆在一处。

关键属性：
- 美术资源（地形瓦片图集、物件图集、背景、前景、小地图）存放在 assets/ 下
- 游戏数据（关卡配置、敌人配置、胜负条件）存放在 content/ 下
- 禁止把敌人配置放在美术目录中

---

## 音频资源规范（Audio Resource Specification）

音频文件按类型使用标准格式，命名遵循统一的类别前缀规则。

不是任意格式。不是随意命名。

关键属性：
- BGM 和语音使用 OGG 格式（支持循环）
- 短音效使用 WAV 格式
- 环境音使用 OGG 格式
- 命名格式：sfx_{category}_{description}_{variant}.wav

---

## 本地化资源（Localization Resources）

多语言文本资源按语言分目录存储，每个语言包含按领域拆分的配置文件。

不是单一文件包含所有语言。不是硬编码文本。

关键属性：
- 按语言分目录（en/、zh_cn/、zh_tw/、ja/、ko/）
- 每个语言下按领域拆分文件（ui、skills、buffs、items、dialogue、quest、tutorial）
- 使用键值对格式存储翻译

---

## UI 主题资源（UI Theme Resources）

UI 主题（颜色、尺寸、样式）通过独立的配置文件定义，支持多主题切换。

不是硬编码样式。不是单一主题。

关键属性：
- 默认主题必须存在
- 支持暗色、经典等扩展主题
- 每个主题包含颜色、尺寸、样式三类配置

---

# 2. 领域边界

## 本领域负责

- 三树分离的目录组织规范
- Content Packs 的功能单元组织方式
- 资源命名规范（snake_case、单数名词、无特殊字符）
- 角色资源目录结构标准
- 战斗地图资源组织规范
- 音频资源格式和命名规范
- 本地化资源的多语言目录组织
- UI 主题资源的目录结构
- 外包美术工作流（分支策略、提交规范）
- 资源自动化校验（格式、命名、尺寸、引用完整性）
- 资源命名空间协议（路径到命名空间的映射）
- 新资源添加检查清单

## 本领域不负责

- 资源的运行时加载和生命周期管理（由资源生命周期管理领域负责）
- RON 配置文件的内容格式定义（由内容系统领域负责）
- 游戏逻辑代码的组织（由架构领域负责）
- 美术资源的具体制作标准（由美术规范领域负责）
- 资源管线的脚本实现（由基础设施领域负责）
- MOD 资源的组织方式（由 MOD 系统领域负责）

## 跨领域通信方式

| 通知内容 | 通信方式 | 目标领域 |
|----------|----------|----------|
| 资源目录变更通知 | Message | 资源生命周期管理领域 |
| 资源校验结果 | Message | 内容系统领域 |
| 新资源添加通知 | Message | 资源生命周期管理领域 |
| 命名空间变更通知 | Message | 内容系统领域 |

# 3. 生命周期

## 资源状态列表

| 状态 | 合义 | 可转换到 |
|------|------|----------|
| WIP | 实验性资源，在 _wip/ 中测试 | Validating, Deprecated |
| Validating | 正在校验格式和引用 | Valid, Invalid |
| Valid | 校验通过，可提交 | Active |
| Invalid | 校验失败，需修正 | Validating |
| Active | 正式资源，参与构建 | Deprecated |
| Deprecated | 已废弃，保留用于回溯 | Removed |

## 状态转换图

```
WIP → Validating → Valid → Active → Deprecated → Removed
           ↓
        Invalid → Validating（修正后重试）
```

## 转换条件

| 从 | 到 | 条件 |
|----|-----|------|
| WIP | Validating | 美术人员将资源移入正式目录 |
| WIP | Deprecated | 实验性资源被确认废弃 |
| Validating | Valid | 自动化校验全部通过 |
| Validating | Invalid | 自动化校验存在失败项 |
| Invalid | Validating | 修正后重新提交校验 |
| Valid | Active | 代码审查通过并合并 |
| Active | Deprecated | 资源被新版本替代 |

# 4. 不变量（Invariants）

## 不变量1：三树物理隔离

任意时刻：

美术资源、游戏配置、代码逻辑必须分别存放在 assets/、content/、src/ 三个独立的顶级目录中。

违反表现：

配置数据（RON 文件）出现在 assets/ 目录中。美术资源出现在 content/ 目录中。代码出现在 assets/ 或 content/ 目录中。

---

## 不变量2：资源引用完整性

任意时刻：

所有内容包中的 RON 配置文件引用的资源路径必须在文件系统中存在。

违反表现：

技能配置引用的图标路径 assets/art/skills/fireball/icon.png 在文件系统中不存在。

---

## 不变量3：命名规范一致性

任意时刻：

所有资源文件名必须使用 snake_case 格式，目录名使用单数名词，禁止空格和特殊字符。

违反表现：

文件名包含大写字母（如 FireBall.png）。目录名使用复数（如 skills/）。文件名包含空格或特殊字符。

---

## 不变量4：美术团队目录隔离

任意时刻：

美术团队只能修改 assets/ 下的美术子目录（art/、audio/、ui/、fonts/），禁止修改 src/、content/、assets/definitions/、assets/rules/。

违反表现：

美术人员的提交包含 src/ 或 content/ 目录下的文件变更。

---

## 不变量5：伴随文件同名同目录

任意时刻：

资源的元数据伴随文件（如配置文件）必须与资源文件同名同目录存放。

违反表现：

技能图标的伴随配置文件存放在 configs/ 目录中，而非与图标文件在同一目录。

---

## 不变量6：_wip/ 资源不参与构建

任意时刻：

_wip/ 目录下的实验性资源不会被资源加载器加载，不参与游戏构建。

违反表现：

_wip/ 目录下的测试资源被加载到游戏中，导致意外行为。

---

## 不变量7：_deprecated/ 资源不参与构建

任意时刻：

_deprecated/ 目录下的废弃资源保留用于 Git 历史回溯，但不参与游戏构建。

违反表现：

已废弃的旧版资源被加载到游戏中，与新版资源产生冲突。

---

## 不变量8：图片尺寸为2的幂次方

任意时刻：

所有 Sprite 类图片的宽和高必须为 2 的幂次方（如 32、64、128、256、512、1024、2048、4096）。

违反表现：

角色精灵图的尺寸为 300x300，非 2 的幂次方。

---

# 5. 业务规则

## 规则1：Content Packs 功能单元组织

必须：
- 每个技能、Buff、物品拥有独立的功能单元目录
- 功能单元内的图标、配置、特效文件物理相邻
- 运行时共享资源存放在 _shared/ 目录
- 编辑器模板存放在 _templates/ 目录

禁止：
- 按文件类型分散存放同一功能的所有资源
- 将功能单元的资源拆分到多个不相关的目录
- 在 _templates/ 中存放运行时资源

允许：
- _shared/ 中的资源被多个功能单元引用
- _wip/ 中存放实验性资源进行测试
- _deprecated/ 中保留旧版资源用于回溯

---

## 角色资源规范

必须：
- 每个角色一个独立目录（assets/art/characters/{character_name}/）
- 精灵图按动作命名（idle.png、move.png、attack.png、skill.png、hurt.png、dead.png）
- 动画数据与精灵图配套（idle.anim、attack.anim）
- 头像包含常规和受伤两种状态（normal.png、damaged.png）

禁止：
- 角色资源散落在多个不相关的目录
- 精灵图命名使用非标准动作名
- 头像缺少常规或受伤状态

允许：
- 角色专属特效与角色物理相邻
- 缩略图（avatar/small.png）用于 UI 显示

---

## 战斗地图资源规范

必须：
- 地图美术资源存放在 assets/art/maps/battle_maps/{map_name}/
- 地图数据（关卡配置、敌人配置）存放在 content/stages/
- 每个地图包含地形瓦片图集、物件图集、背景、前景、小地图

禁止：
- 把敌人配置放在美术目录中
- 地图美术资源和游戏数据混放
- 地图缺少小地图缩略图

允许：
- 地图资源目录中包含多个图集文件
- 前景装饰为可选资源

---

## 音频资源规范

必须：
- BGM 使用 OGG 格式（支持循环）
- 短音效使用 WAV 格式
- 语音使用 OGG 格式
- 环境音使用 OGG 格式
- 音频命名遵循 sfx_{category}_{description}_{variant}.wav 格式

禁止：
- BGM 使用 WAV 格式（文件过大，不支持循环）
- 音频文件名包含空格或特殊字符
- 音频文件名使用非标准命名格式

允许：
- 音频文件包含变体编号（01、02 等）
- 同一类别下有多个音频变体

---

## 命名规范规则

必须：
- 所有文件名使用 snake_case（如 fire_ball.png）
- 目录名使用单数名词（如 skill/ 而非 skills/）
- 文件名无空格（如 hero_portrait.png）
- 文件名无特殊字符（如 ice_sword.png）
- 伴随文件与资源文件同名同目录（如 fireball/icon.png + fireball/config.ron）

禁止：
- 文件名使用大写字母（如 FireBall.png）
- 文件名使用连字符（如 fire-ball.png）
- 目录名使用复数（如 skills/）
- 伴随文件物理分离（如 skills/fireball/icon.png + configs/skills/fireball.ron）

允许：
- 文件名包含下划线分隔
- 伴随文件使用相同的基础名称

---

## 外包美术分支策略

必须：
- 每个美术人员有独立的工作分支（art/{artist_name}/）
- 美术分支从 main 分支创建
- 提交前必须通过自动化校验
- 校验失败时提供明确的修正建议

禁止：
- 美术人员直接在 main 分支上提交
- 跳过自动化校验直接合并
- 美术分支包含 src/ 或 content/ 的变更

允许：
- 美术分支包含多个资源文件的批量提交
- 校验失败后修正并重新提交

---

## 资源校验规则

必须：
- 所有资源提交前通过自动化校验
- 校验内容包括：格式合规、图片尺寸（2 的幂次方）、文件大小（≤5MB）、命名规范、目录结构、Alpha 通道、色彩空间（sRGB）
- 校验失败时提供文件路径、违规原因、修正建议
- 资源引用完整性校验：所有 RON 配置引用的资源必须存在，所有资源必须至少被一个 RON 配置引用（排除 _shared）

禁止：
- 跳过自动化校验直接合并
- 校验失败时忽略错误继续提交
- 提交临时文件、PSD 等非标准格式文件

允许：
- 校验失败后修正并重新提交
- _shared/ 目录中的资源可以不被任何 RON 配置引用

---

## 新资源添加检查清单

必须：
- 确认资源放入正确的功能单元目录
- 确认命名符合 snake_case + 单数规则
- 确认伴随的配置文件存在
- 确认在 _wip/ 中测试过再移入正式目录
- 确认通过了自动化校验
- 确认文件大小在限制范围内（≤5MB）
- 确认图片尺寸为 2 的幂次方
- 确认 Sprite 类资源使用了 Alpha 通道

禁止：
- 不符合命名规范的资源直接提交
- 未经 _wip/ 测试的新资源直接移入正式目录
- 文件大小超过 5MB 的资源提交

允许：
- 在 _wip/ 中进行多轮测试
- 校验失败后修正并重新提交

---

# 6. 流程管线

## 资源提交校验管线（Asset Submission Validation Pipeline）

```
美术提交 PR → 自动化校验 → 格式检查 → 代码审查 → 合并到 main
```

### Step1：美术提交 PR

输入：美术资源文件变更
处理：美术人员在独立分支上提交资源变更
输出：PR 请求
禁止：美术人员直接在 main 分支上提交

### Step2：自动化校验

输入：PR 中的资源文件
处理：执行格式、命名、尺寸、引用完整性校验
输出：校验报告（通过/失败）
禁止：跳过自动化校验

### Step3：格式检查

输入：校验报告
处理：检查校验是否全部通过
输出：通过 → 代码审查；失败 → 通知美术修正
禁止：校验失败时继续流程

### Step4：代码审查

输入：校验通过的 PR
处理：人工审查资源质量和目录组织
输出：审查通过 → 合并；不通过 → 通知美术修正
禁止：跳过代码审查直接合并

### Step5：合并到 main

输入：审查通过的 PR
处理：合并到 main 分支
输出：资源正式纳入项目
禁止：合并包含 src/ 或 content/ 变更的 PR

---

## 资源引用完整性校验管线（Asset Reference Integrity Pipeline）

```
扫描 RON 配置 → 提取引用路径 → 验证文件存在 → 反向检查孤立资源
```

### Step1：扫描 RON 配置

输入：content_packs/ 目录下的所有 RON 配置文件
处理：读取并解析所有 RON 配置
输出：配置文件列表
禁止：跳过任何 RON 配置文件

### Step2：提取引用路径

输入：RON 配置文件内容
处理：提取所有引用的资源路径
输出：资源路径列表
禁止：忽略可选引用字段

### Step3：验证文件存在

输入：资源路径列表
处理：逐个验证文件是否在文件系统中存在
输出：存在/缺失状态
禁止：跳过验证直接标记为 OK

### Step4：反向检查孤立资源

输入：所有资源文件列表
处理：检查每个资源文件是否至少被一个 RON 配置引用（排除 _shared）
输出：孤立资源列表
禁止：忽略孤立资源

---

## 新资源添加管线（New Asset Addition Pipeline）

```
创建 _wip/ 资源 → 测试验证 → 移入正式目录 → 自动化校验 → 合并
```

### Step1：创建 _wip/ 资源

输入：新资源文件
处理：在 _wip/ 目录中创建实验性资源
输出：_wip/ 中的资源文件
禁止：直接在正式目录中创建新资源

### Step2：测试验证

输入：_wip/ 中的资源
处理：在游戏中加载并测试资源效果
输出：测试通过/失败
禁止：未测试直接移入正式目录

### Step3：移入正式目录

输入：测试通过的资源
处理：将资源从 _wip/ 移入对应的功能单元目录
输出：正式目录中的资源文件
禁止：跳过测试直接移入

### Step4：自动化校验

输入：正式目录中的资源
处理：执行完整的自动化校验
输出：校验报告
禁止：跳过校验直接合并

### Step5：合并

输入：校验通过的资源
处理：提交 PR 并合并到 main
输出：资源正式纳入项目
禁止：校验失败时合并

# 7. 数据结构

## Content Pack（内容包）

职责：按功能单元组织相关资源，提供资源的逻辑分组

结构：
- 功能单元目录：每个技能、Buff、物品独立存放
- _shared/：运行时共享资源
- _templates/：编辑器模板
- _wip/：实验性资源
- _deprecated/：废弃资源

要求：
- 每个功能单元目录包含图标、配置、特效等所有相关资源
- _shared/ 中的资源可被多个功能单元引用
- _templates/ 中的资源不参与构建
- _wip/ 中的资源不被加载
- _deprecated/ 中的资源不参与构建

---

## 角色资源目录结构（Character Resource Directory Structure）

职责：定义每个角色的标准资源组织方式

结构：
- sprite/：精灵图（按动作命名）
- animation/：动画数据（与精灵图配套）
- portrait/：头像（常规和受伤）
- avatar/：缩略图
- vfx/：专属特效

要求：
- 每个角色一个独立目录
- 精灵图包含所有必要动作（idle、move、attack、skill、hurt、dead）
- 动画数据与精灵图配套
- 头像包含常规和受伤两种状态

---

## 战斗地图资源目录结构（Battle Map Resource Directory Structure）

职责：定义每个战斗地图的标准资源组织方式

结构：
- terrain.png：地形瓦片图集
- objects.png：地图物件图集
- background.png：背景图
- foreground.png：前景装饰
- minimap.png：小地图缩略图

要求：
- 每个地图一个独立目录
- 包含所有必要的图集文件
- 小地图缩略图为必选资源

---

## 音频资源命名结构（Audio Resource Naming Structure）

职责：定义音频文件的标准命名格式

结构：
- 格式：sfx_{category}_{description}_{variant}.wav
- 类别：battle、skill、ui、item、buff 等
- 描述：具体动作或效果
- 变体：01、02 等编号

要求：
- 所有音频文件遵循统一命名格式
- 变体编号用于区分同一动作的不同音频
- 文件名无空格和特殊字符

---

## 资源校验结果结构（Asset Validation Result Structure）

职责：记录资源校验的结果和详细信息

结构：
- 文件路径：被校验的资源文件路径
- 校验状态：通过/失败
- 违规原因：具体的违规描述
- 修正建议：如何修正违规

要求：
- 校验结果必须包含文件路径和违规原因
- 修正建议必须具体可操作
- 校验失败时必须提供完整的错误信息

---

# 8. 禁止事项

禁止：把游戏配置数据放在 assets/ 目录

原因：违反三树分离原则，配置数据应存放在 content/ 目录。

违反后果：配置数据和美术资源混放，难以维护和扩展。

---

禁止：把 Rust 代码放在 assets/ 目录

原因：违反三树分离原则，代码应存放在 src/ 目录。

违反后果：代码和美术资源混放，版本控制混乱。

---

禁止：把开发脚本放在 assets/ 目录

原因：违反三树分离原则，脚本应存放在 scripts/ 目录。

违反后果：脚本和美术资源混放，维护困难。

---

禁止：美术团队修改 src/ 或 content/ 目录

原因：美术团队只负责美术资源，代码和配置的修改应由程序团队负责。

违反后果：美术人员的修改可能破坏游戏逻辑或配置完整性。

---

禁止：使用非标准格式（如 PSD、AI 放入 assets）

原因：非标准格式无法被游戏引擎直接加载，需要转换为标准格式。

违反后果：游戏引擎无法加载资源，构建失败。

---

禁止：在代码中硬编码 content_packs/ 前缀

原因：硬编码路径前缀会导致资源组织变更时需要修改代码。

违反后果：每次资源路径变更都需要修改代码，无法支持扩展和 MOD。

---

禁止：跳过 _wip/ 直接在正式目录创建新资源

原因：新资源未经测试直接进入正式目录，可能导致构建失败或游戏异常。

违反后果：未经验证的资源进入游戏，影响游戏稳定性。

---

禁止：资源命名使用大写字母、空格或特殊字符

原因：非标准命名会导致跨平台兼容性问题和自动化脚本解析错误。

违反后果：CI/CD 流程失败，资源无法正确加载。

---

禁止：提交临时文件、PSD 等非标准格式文件

原因：非标准格式文件会增加仓库体积，且无法被游戏引擎使用。

违反后果：仓库体积膨胀，构建时间增加。

---

禁止：_wip/ 目录中的资源被资源加载器加载

原因：实验性资源未经验证，加载到游戏中可能导致意外行为。

违反后果：测试资源出现在游戏中，影响游戏体验。

---

禁止：_deprecated/ 目录中的资源参与构建

原因：废弃资源已被新版本替代，参与构建会导致资源冲突。

违反后果：新旧资源同时存在，产生不可预期的行为。

---

禁止：资源引用完整性校验失败时继续提交

原因：孤立资源或断链配置会导致运行时错误。

违反后果：游戏加载时找不到资源，产生运行时错误。

---

禁止：代码中硬编码魔法数字

原因：违反宪法条款 12.4.1，所有平衡参数必须放入配置文件。

违反后果：每次数值调整都需要程序员介入，策划无法独立调优。

---

# 9. AI 修改规则

## 如果新增资源类型

允许：
- 在对应的 Content Pack 中创建新的功能单元目录
- 遵循现有的命名规范和目录结构
- 在 _wip/ 中测试后再移入正式目录

禁止：
- 不遵循现有的命名规范
- 跳过 _wip/ 测试直接移入正式目录
- 不进行资源引用完整性校验

优先检查：
- 新资源是否放入正确的功能单元目录
- 命名是否符合 snake_case + 单数规则
- 伴随的配置文件是否存在
- 是否通过了自动化校验

---

## 如果修改命名规范

允许：
- 新增命名规则（需在本文档中记录）
- 调整命名规则的优先级

禁止：
- 删除已有命名规则（破坏兼容性）
- 修改已有资源的命名（需要全局重命名）
- 引入与现有规则冲突的新规则

优先检查：
- 新规则是否与现有规则一致
- 是否需要更新自动化校验脚本
- 是否需要重命名已有资源

---

## 如果调整目录结构

允许：
- 新增顶级目录（如新的资源类型）
- 在现有目录中新增子目录

禁止：
- 删除已有目录（破坏兼容性）
- 移动已有资源到新位置（需要全局更新引用）
- 合并多个目录（破坏功能单元组织）

优先检查：
- 新目录是否与三树分离原则一致
- 是否需要更新资源校验脚本
- 是否需要更新资源引用完整性校验

---

## 如果修改外包工作流

允许：
- 调整美术团队的分支策略
- 新增自动化校验规则
- 优化校验失败的反馈机制

禁止：
- 移除自动化校验（降低质量）
- 允许美术团队修改 src/ 或 content/ 目录
- 跳过代码审查直接合并

优先检查：
- 新工作流是否保持美术团队的独立性
- 自动化校验是否覆盖所有必要的检查项
- 校验失败反馈是否清晰可操作

---

## 如果测试失败

排查顺序：
1. 检查资源是否放入正确的目录（三树分离）
2. 检查命名是否符合规范（snake_case、单数、无特殊字符）
3. 检查资源引用完整性（所有引用必须存在）
4. 检查图片尺寸是否为 2 的幂次方
5. 检查文件大小是否在限制范围内（≤5MB）
6. 检查 _wip/ 和 _deprecated/ 目录中的资源是否被意外加载
