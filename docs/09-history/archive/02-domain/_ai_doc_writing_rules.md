# AI 领域文档撰写规则

Version: 1.0

本文件规定 AI 撰写领域文档时必须遵循的格式、顺序和约束。

目标：让 AI 遇到任何新领域，都能产出自身最易理解、最具约束力的领域文档。

---

# 核心原则

1. **文档的第一受众是 AI，不是人**
2. **约束力 > 可读性 > 优雅性**
3. **规则不会变，代码会变** — 写规则，不写代码
4. **禁止 > 必须 > 允许** — 禁止事项是最高优先级内容
5. **概念先于流程** — 先定义是什么，再定义怎么做

---

# 文档结构（强制顺序）

每个领域文档必须按以下顺序组织，不得跳过章节，不得调换顺序：

```
1. Domain Overview     — 领域概述
2. Glossary            — 术语定义
3. Domain Boundary     — 领域边界
4. Lifecycle           — 生命周期 / 状态机
5. Invariants          — 不变量
6. Rules               — 业务规则
7. Pipelines           — 流程管线
8. Data Structures     — 数据结构
9. Forbidden           — 禁止事项
10. AI Decision Rules  — AI 修改规则
```

---

# 各章节撰写规范

## 1. Domain Overview — 领域概述

**篇幅**：3~5 行

**内容**：这个领域管什么，一句话定义。

**格式**：

```
# XXX 领域

XXX 领域管理 [一句话职责]。

核心原则：
- 原则1
- 原则2
- 原则3
```

**禁止**：超过 5 行概述、写背景介绍、写设计哲学。

---

## 2. Glossary — 术语定义

**这是最重要的章节。**

AI 最常犯的错误是概念混淆。术语定义是第一道防线。

**格式**：

```
# 术语定义

## 术语A

[一句话定义]。

不是 [容易混淆的概念]。

关键属性：
- 属性1：说明
- 属性2：说明

---

## 术语B

[一句话定义]。

不是 [容易混淆的概念]。

关键属性：
- 属性1：说明
```

**规则**：

1. 每个术语必须包含 **"不是"** 部分 — 明确与易混淆概念的区别
2. 术语之间有依赖关系时，先定义被依赖的术语
3. 同一概念只能有一个术语名，禁止别名
4. 术语名在整个项目内唯一，跨文档不重复定义
5. 如果术语在其他领域文档已定义，写 `参见 xxx_rules.md#术语A`，不重复定义

**评判标准**：如果一个术语的 "不是" 部分为空，说明定义不够精确，必须补充。

---

## 3. Domain Boundary — 领域边界

**格式**：

```
# 领域边界

## 本领域负责

- 职责1
- 职责2

## 本领域不负责

- 职责X（由 XXX 领域负责）
- 职责Y（由 YYY 领域负责）

## 跨领域通信方式

| 通知内容 | 通信方式 | 目标领域 |
|----------|----------|----------|
| XXX事件 | Message | YYY领域 |
| ZZZ状态 | Observer | WWW领域 |
```

**规则**：

1. "不负责" 列表必须和 "负责" 列表一样详细
2. 每项 "不负责" 必须指明由谁负责
3. 跨领域通信只允许 Message / Observer / Command，禁止直接调用

---

## 4. Lifecycle — 生命周期 / 状态机

**格式**：

```
# 生命周期

## 状态列表

| 状态 | 含义 | 可转换到 |
|------|------|----------|
| StateA | 说明 | StateB, StateC |
| StateB | 说明 | StateC, StateD |

## 状态转换图

StateA → StateB → StateC → StateD
                  ↘ StateE

## 转换条件

| 从 | 到 | 条件 |
|----|-----|------|
| StateA | StateB | [触发条件] |
| StateB | StateC | [触发条件] |
```

**规则**：

1. 每个状态必须列出所有合法的转换目标
2. 不在转换表中的转换 = 禁止转换
3. 必须标注初始状态和终态
4. 如果领域没有状态机（纯计算），写 "本领域无状态机，为纯函数式计算"

---

## 5. Invariants — 不变量

**这是防 AI 绕过规则最有效的章节。**

**格式**：

```
# 不变量

## 不变量1：[名称]

任意时刻：

[条件A] 必须成立。

违反表现：

[描述违反时会出现什么异常]

---

## 不变量2：[名称]

[条件B] 必须成立。
```

**规则**：

1. 不变量必须是 **可验证的** — 能写断言
2. 不变量必须标注检查时机（"任意时刻" / "阶段X结束后" / "操作Y完成后"）
3. 每个不变量必须描述违反时的表现
4. 不变量数量控制在 3~10 条，太多说明领域拆分不够

---

## 6. Rules — 业务规则

**格式**：

```
# 业务规则

## 规则1：[名称]

允许：
- 行为A
- 行为B

禁止：
- 行为C

必须：
- 行为D

---

## 规则2：[名称]

允许：
- ...

禁止：
- ...

必须：
- ...
```

**规则**：

1. 每条规则必须包含 **允许 / 禁止 / 必须** 中的至少两种
2. "禁止" 优先级最高，写在最前面
3. 规则之间不能矛盾 — 如有冲突，标注优先级
4. 规则粒度：一条规则一个主题，不要把多条规则混在一起
5. 规则编号稳定，新增规则追加，不插入中间

---

## 7. Pipelines — 流程管线

**格式**：

```
# 流程管线

## 管线名称

Step1 → Step2 → Step3 → Step4

### Step1：[名称]

输入：[什么]
处理：[做什么]
输出：[产生什么]
禁止：[不能做什么]

### Step2：[名称]

输入：[什么]
处理：[做什么]
输出：[产生什么]
禁止：[不能做什么]
```

**规则**：

1. 每个步骤必须标注 **禁止** — 明确该步骤不能越权做什么
2. 步骤之间是严格顺序，禁止跳步
3. 管线必须有明确的输入和输出
4. 如果管线有分支，用条件标注

---

## 8. Data Structures — 数据结构

**格式**：

```
# 数据结构

## 结构名

职责：[一句话]

结构：
- 字段1：类型 — 说明
- 字段2：类型 — 说明

要求：
- [约束1]
- [约束2]
```

**规则**：

1. **禁止写代码** — 只写职责、字段说明、约束
2. **禁止写实现细节** — 不写 derive 宏、不写生命周期标注
3. 字段说明用自然语言，不用类型系统语言
4. 区分 Definition 和 Instance — 如果领域有此分离，必须标注

---

## 9. Forbidden — 禁止事项

**这是全文档约束力最强的章节。**

**格式**：

```
# 禁止事项

禁止：[描述]

原因：[为什么]

违反后果：[会出什么问题]

---

禁止：[描述]

原因：[为什么]

违反后果：[会出什么问题]
```

**规则**：

1. 每条禁止事项必须写明 **原因** 和 **违反后果**
2. 禁止事项不能和 Rules 中的 "禁止" 重复 — 此处放跨规则的全局禁止
3. 禁止事项按严重程度排序：最严重的在前
4. 禁止事项数量控制在 5~15 条

---

## 10. AI Decision Rules — AI 修改规则

**这是 AI 项目特有的章节，直接决定后续 AI 修改代码的质量。**

**格式**：

```
# AI 修改规则

## 如果新增 [变更类型A]

允许：
- [可以做什么]

禁止：
- [不能做什么]

优先检查：
- [先看什么]

---

## 如果修改 [变更类型B]

允许：
- [可以做什么]

禁止：
- [不能做什么]

优先检查：
- [先看什么]

---

## 如果测试失败

排查顺序：
1. [第一步检查什么]
2. [第二步检查什么]
3. [第三步检查什么]
```

**规则**：

1. 覆盖该领域最常见的 3~5 种变更场景
2. 每种场景必须有 "优先检查" — 指导 AI 的排查路径
3. "测试失败" 场景必须包含，且排查顺序从规则层到实现层
4. 禁止写 "根据情况判断" 这种模糊指导

---

# 全局撰写规则

## 语言

1. 用短句，不用长句
2. 用列表，不用段落
3. 用肯定/否定句，不用疑问句
4. 用具体描述，不用抽象描述

## 精确性

1. "必须" = 没有例外
2. "允许" = 可以做也可以不做
3. "禁止" = 绝对不能做
4. "应该" = 不用这个词，改为 "必须" 或 "允许"
5. "建议" = 不用这个词，改为 "允许" 或删除

## 代码

1. 禁止在文档中写可执行代码
2. 数据结构只写职责和约束，不写 derive / impl
3. 代码片段仅在必须说明算法逻辑时使用，且用伪代码

## 一致性

1. 术语名在全文中必须完全一致，禁止缩写、别名
2. 同一概念在不同文档中的定义必须一致
3. 规则编号稳定，新增追加，不插入中间

## 冲突解决

1. 本文档内规则冲突时，Forbidden > Invariants > Rules > Pipelines
2. 跨文档规则冲突时，domain_rules.md > 具体领域文档
3. 规则与代码冲突时，规则优先，代码必须修改

## 版本化

1. 文档头部必须标注 Version
2. 规则变更时递增 Version
3. 废弃规则标注 `[DEPRECATED since vX.X]`，保留 1 个版本后删除
4. 新增规则标注 `[NEW since vX.X]`，保留 1 个版本后去除标记

---

# 质量检查清单

文档完成后，AI 必须逐项检查：

- [ ] 每个术语都有 "不是" 部分
- [ ] 领域边界 "不负责" 和 "负责" 一样详细
- [ ] 生命周期覆盖所有合法转换
- [ ] 不变量可验证（能写断言）
- [ ] 每条规则至少有 "禁止" 或 "必须"
- [ ] 管线每个步骤都有 "禁止"
- [ ] 数据结构无代码、无实现细节
- [ ] 禁止事项都有原因和后果
- [ ] AI Decision Rules 覆盖 3+ 种变更场景
- [ ] 无 "应该" "建议" 等模糊词
- [ ] 术语全文一致无别名
- [ ] Version 已标注
