对于你这种目标 **50万行代码、长期维护、AI大量参与开发、Bevy UI复杂度较高的SRPG项目**，我不建议走两个极端：

❌ 完全Figma驱动
❌ 完全AI直接手写UI

最佳实践其实是：

> **ASCII Wireframe（线框图） + Flexbox规范文档 + 少量Figma关键页面**
>
> 三者结合。

# 为什么AI写UI总是乱？

因为AI缺少：

1. 页面信息架构（Information Architecture）
2. 视觉层级（Visual Hierarchy）
3. 布局约束（Layout Constraints）
4. 组件库规范（Design System）

例如你说：

```rust
spawn_inventory_screen()
```

AI并不知道：

* 左边是不是角色栏
* 右边是不是物品栏
* 底部是不是说明栏
* 哪些区域固定宽度
* 哪些区域flex-grow

所以AI只能猜。

猜出来的UI一定乱。

# 如果是我设计你的项目

我会规定一个 ADR：

```text
ADR-UI-001

UI开发流程

1. Screen必须先有ASCII Wireframe

2. Screen必须定义Flexbox结构

3. Screen不得直接手写布局

4. Screen只能组合Widget

5. Widget负责具体UI实现

6. 核心Screen可选Figma设计稿

7. AI生成UI前必须阅读对应Screen文档
```

对于你这种长期维护的 Bevy SRPG，这套流程的收益远高于投入。

**结论：**

* 小型项目：AI直接写即可。
* 中型项目：ASCII Wireframe + Flexbox描述。
* 你这种50万行级别项目：**ASCII Wireframe（必需） + Design System（必需） + Widget组件库（必需） + 少量核心页面Figma（可选）**。

如果只能选一个最划算的投入，我会优先做 **ASCII Wireframe + Flexbox规范文档**，而不是先做Figma。这样对AI生成Bevy UI的提升最大。

这是我认为最适合你这种 **Bevy + DDD + Widget化 + AI辅助开发 + 50万行级别项目** 的版本。

不要把它理解成 UI 文档。

应该理解成：

> **AI 可消费（AI-Consumable）的 UI 架构规范**

目标不是给人看。

目标是：

```text
让 Claude
让 ChatGPT
让 Gemini
让 OpenCode

看完后

100%知道怎么生成UI
```

# UI Screen Specification v1.0

## 目标

本规范用于描述 Screen 的结构。

目标：

* AI 能直接生成 Bevy UI
* 人类能快速阅读
* Layout 与 Widget 解耦
* Layout 可长期维护
* Layout 可独立演进

禁止：

* 在 Screen 文档描述视觉样式
* 在 Screen 文档描述颜色
* 在 Screen 文档描述字体
* 在 Screen 文档描述动画

这些属于 Design System。

Screen 文档只描述：

* 信息架构
* Widget结构
* Flexbox布局
* 交互区域

---

# 1. 基础原则

Screen = Layout

Widget = Implementation

Screen 不关心 Widget 内部实现。

例如：

InventoryScreen

包含：

* CharacterPanel
* InventoryGrid
* DescriptionPanel

但不描述：

* CharacterPanel 内部如何实现
* InventoryGrid 内部如何实现

---

# 2. Screen Header

每个 Screen 必须先定义：

Screen Name

Purpose

Navigation

Example

Screen: Inventory

Purpose:
管理角色装备与道具

Navigation:
MainMenu -> Character -> Inventory

---

# 3. ASCII Wireframe

必须存在。

用于表达信息架构。

Example

+------------------------------------------------------+
| Header                                               |
+------------------------------------------------------+
| CharacterPanel | InventoryGrid                       |
|                |                                     |
|                |                                     |
+------------------------------------------------------+
| DescriptionPanel                                     |
+------------------------------------------------------+

要求：

* 必须标注所有区域名称
* 不允许出现模糊区域
* 不允许出现 Anonymous Panel
* 不允许出现 Empty Box

错误示例：

+----------------+
|                |
+----------------+

正确示例：

+----------------+
| EquipmentPanel |
+----------------+

---

# 4. Widget Tree

必须定义。

格式：

Root
├── HeaderBar
├── Body
│   ├── CharacterPanel
│   └── InventoryGrid
└── DescriptionPanel

要求：

必须完整。

禁止隐藏节点。

禁止省略重要容器。

---

# 5. Flexbox Layout

必须定义。

格式：

Root:
direction: Column

HeaderBar:
height: 64

Body:
flex_grow: 1
direction: Row

CharacterPanel:
width: 320

InventoryGrid:
flex_grow: 1

DescriptionPanel:
height: 180

要求：

必须明确：

* width
* height
* flex_grow

禁止：

不写尺寸。

错误：

CharacterPanel

正确：

CharacterPanel:
width: 320

---

# 6. Responsive Rules

必须定义。

Example

<1280

CharacterPanel:
width: 240

> =1280

CharacterPanel:
width: 320

> =1920

CharacterPanel:
width: 400

如果项目固定分辨率：

明确写：

Responsive:
None

---

# 7. Region Responsibility

每个区域必须定义职责。

CharacterPanel

Responsibilities:

* 当前角色头像
* 当前角色属性
* 当前角色装备

InventoryGrid

Responsibilities:

* 展示背包物品
* 支持分页
* 支持筛选

DescriptionPanel

Responsibilities:

* 展示当前选中物品说明

禁止：

Responsibilities:
显示内容

过于模糊。

---

# 8. Widget Contract

引用 Widget 时必须声明：

CharacterPanel

Inputs:

* CharacterId

Outputs:

* CharacterSelected

InventoryGrid

Inputs:

* InventoryData

Outputs:

* ItemHovered
* ItemSelected

目的：

让 AI 知道数据流。

---

# 9. State Mapping

定义状态。

InventoryScreen

States:

Loading

Empty

Normal

Error

每个状态如何布局必须明确。

Example

Loading:
显示 LoadingWidget

Empty:
显示 EmptyInventoryWidget

Normal:
显示正常界面

Error:
显示 ErrorWidget

---

# 10. Focus Navigation

键盘/手柄项目必须写。

Example

CharacterPanel
↓
InventoryGrid
↓
DescriptionPanel

用于：

* 键盘导航
* 手柄导航
* SteamDeck

---

# 11. Interaction Zones

定义交互。

InventoryGrid

Supports:

* Click
* Hover
* Drag
* Drop

DescriptionPanel

Supports:

* None

---

# 12. Overlay Definition

定义覆盖层。

Example

InventoryScreen

Overlays:

* TooltipLayer
* ModalLayer
* NotificationLayer

层级：

TooltipLayer
↑
ModalLayer
↑
NotificationLayer

---

# 13. Screen Lifecycle

定义生命周期。

OnEnter

* LoadInventory

OnReady

* SelectCurrentCharacter

OnExit

* ClearSelection

---

# 14. Data Ownership

必须明确。

InventoryScreen

Owns:

* SelectedItem

Uses:

* InventoryRepository

禁止：

多个 Widget 同时拥有状态。

---

# 15. AI Generation Rules

AI生成UI时必须遵守：

1.

严格按照 Widget Tree 生成。

2.

严格按照 Flexbox Layout 生成。

3.

禁止新增区域。

4.

禁止删除区域。

5.

禁止修改 Widget 名称。

6.

禁止推测业务逻辑。

7.

禁止推测视觉设计。

8.

只负责布局实现。

9.

Widget 内部逻辑由 Widget 自身负责。

10.

若文档与代码冲突：
以 Screen 文档为准。

---

# 16. Definition of Done

一个 Screen 文档完成必须满足：

[ ] Header

[ ] ASCII Wireframe

[ ] Widget Tree

[ ] Flexbox Layout

[ ] Responsibilities

[ ] Widget Contract

[ ] State Mapping

[ ] Focus Navigation

[ ] Interaction Zones

[ ] Overlay Definition

[ ] Lifecycle

[ ] Data Ownership

否则视为不完整。

如果只允许我再加一个规则，而且是**对 AI 生成质量提升最大的规则**，那就是：

```text
ADR-UI-002

任何 Screen 文档必须同时包含：

1. ASCII Wireframe
2. Widget Tree
3. Flexbox Layout

三者缺一不可。

ASCII 负责空间结构
Widget Tree 负责实体结构
Flexbox 负责实现约束

仅有其中任何一个，
AI生成质量都会明显下降。
```

很多团队只画线框图，AI不知道父子关系；只写 Widget Tree，AI不知道空间布局；只写 Flexbox，AI不知道业务结构。对于大型 Bevy 项目，这三个部分组合起来才是最稳定、最适合 AI 的 UI 规范。
有，而且下面这些反而是很多大型项目后期踩坑后才补上的。前面那份已经覆盖了 80%，下面是剩余最重要的 20%。

---

# 17. Widget ID（极其重要）

不要只写：

```text
CharacterPanel
InventoryGrid
```

要写稳定 ID：

```yaml
widgets:
  - id: character_panel
    widget: CharacterPanel

  - id: inventory_grid
    widget: InventoryGrid
```

原因：

后期会出现：

```text
InventoryScreen

├── LeftInventoryGrid
├── RightInventoryGrid
```

AI会混乱。

应该：

```yaml
left_inventory_grid
right_inventory_grid
```

永远使用稳定ID。

---

# 18. Layout Intent（非常重要）

很多布局尺寸无法表达设计意图。

例如：

```yaml
CharacterPanel:
  width: 320
```

AI不知道为什么320。

应该：

```yaml
CharacterPanel:
  width: 320

intent:
  固定角色信息区域
  不允许压缩
```

例如：

```yaml
InventoryGrid:
  flex_grow: 1

intent:
  占据剩余空间
```

AI理解会提高很多。

---

# 19. Layout Priority

定义压缩优先级。

例如：

```yaml
priority:

  inventory_grid:
    shrink: high

  character_panel:
    shrink: none

  action_bar:
    shrink: none
```

否则AI经常把重要区域压没。

---

# 20. Scroll Policy

必须写。

很多AI最容易遗漏。

例如：

```yaml
CharacterPanel:
  scroll: none

InventoryGrid:
  scroll: vertical

QuestList:
  scroll: vertical

Description:
  scroll: none
```

否则后期内容超长时全崩。

---

# 21. Overflow Policy

例如：

```yaml
text:
  overflow: ellipsis
```

或者：

```yaml
overflow: clip
```

或者：

```yaml
overflow: scroll
```

大型项目迟早遇到：

```text
德语
俄语
日语
中文
```

文本爆炸。

提前定义。

---

# 22. Empty State（非常重要）

不要只写：

```yaml
InventoryGrid
```

必须写：

```yaml
empty_state:
  EmptyInventoryWidget
```

例如：

```text
没有任务
没有好友
没有装备
没有存档
```

这些状态必须提前设计。

---

# 23. Error State

同理。

```yaml
error_state:
  ErrorWidget
```

不要等后面补。

---

# 24. Loading State

```yaml
loading_state:
  LoadingWidget
```

AI经常忘。

---

# 25. Selection Model

SRPG特别重要。

例如：

```yaml
selection:

  single
```

或者：

```yaml
selection:

  multi
```

或者：

```yaml
selection:

  none
```

例如：

```text
InventoryGrid
CharacterList
SkillList
```

都要定义。

---

# 26. Ownership Boundary（DDD项目非常重要）

Screen文档中明确：

```yaml
InventoryScreen

owns:

  selected_item

does_not_own:

  inventory_data
```

这样AI不会把Repository塞进Widget。

---

# 27. Event Contract（非常重要）

不要写：

```text
点击物品
```

要写：

```yaml
events:

  ItemSelected

  ItemHovered

  ItemDoubleClicked
```

AI看到后直接生成事件流。

---

# 28. Widget Reuse Policy

规定：

```yaml
reuse_only:
  true
```

意思：

```text
禁止创建新Widget
```

AI必须使用现有：

```text
PanelWidget
ListWidget
GridWidget
TooltipWidget
```

大型项目收益巨大。

---

# 29. Forbidden Rules

很多团队不写。

实际上非常有用。

例如：

```yaml
forbidden:

  - 不允许嵌套ScrollView

  - 不允许超过4层容器

  - 不允许业务逻辑进入Screen

  - 不允许直接访问Repository
```

AI违规率会下降很多。

---

# 30. Accessibility（未来收益巨大）

虽然你可能暂时不用。

建议提前预留：

```yaml
accessibility:

  focusable: true

  keyboard_navigation: true

  gamepad_navigation: true
```

否则后面Steam Deck适配会很痛苦。

---

# 31. Z-Layer

很多UI文档没有。

实际上Bevy特别需要。

```yaml
z_layers:

  background: 0

  content: 100

  tooltip: 200

  modal: 300

  notification: 400
```

AI直接知道层级。

---

# 32. Animation Ownership

提前定义：

```yaml
animation:

  owner: widget
```

或者：

```yaml
animation:

  owner: screen
```

否则后期：

```text
Tooltip动画
Window动画
Panel动画
```

全乱。

---

# 33. Widget Budget（大型项目神器）

限制复杂度。

例如：

```yaml
budget:

  max_widget_depth: 6

  max_children_per_container: 20
```

避免：

```text
Node
 └── Node
     └── Node
         └── Node
             └── Node
                 └── Node
```

AI特别喜欢生成这种树。

---

# 34. Screen Metrics（我最推荐新增的一条）

每个Screen最后必须统计：

```yaml
metrics:

  widget_count: 12

  container_count: 4

  interactive_count: 6

  overlay_count: 2
```

这样AI修改后：

```text
原来12个Widget

现在变成31个Widget
```

立刻知道失控了。

---

如果是我给 **50万行 Bevy 项目** 制定最终版规范，我会把前面那份作为主规范，然后额外强制增加这 **7条最高价值规则**：

```text
Widget ID
Layout Intent
Scroll Policy
Event Contract
Ownership Boundary
Forbidden Rules
Screen Metrics
```

这7项是大多数UI规范里没有，但对 AI 生成、DDD架构、长期维护价值最高的部分。它们解决的不是“怎么画UI”，而是“怎么防止10万行以后UI体系失控”。
