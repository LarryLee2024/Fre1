# ① Infra ↔ Content 双向依赖

这是最致命的问题。

如果你现在文档里真写的是：

```text
Infra → Content
Content → Infra
```

那确实有硬伤。

---

因为：

```text
Content
```

本质是：

```text
业务内容
```

---

而：

```text
Infra
```

本质是：

```text
技术能力
```

---

正确方向：

```text
Content
    ↓
Infra
```

允许。

---

例如：

```rust
SkillLoader
    ↓
AssetReader
```

---

反过来：

```text
Infra
    ↓
Content
```

绝对禁止。

---

否则：

```rust
asset_loader.rs

match content_type {
    Skill => ...
    Buff => ...
}
```

开始出现。

---

几年后：

```text
Infrastructure知道所有业务
```

边界彻底消失。

---

这个批评100%正确。

---

# ② ECS归属问题

这位直接打中了Bevy架构最大的难点。

他说：

> Entity Component 到底属于哪层？

这个问题我之前其实刻意没有展开。

因为这是：

```text
Bevy版DDD
```

最难的问题。

---

我现在给你最终答案：

---

# Entity

属于：

```text
Infrastructure
```

不是Core。

---

因为：

```rust
Entity
```

是Bevy实现细节。

---

删掉Bevy：

```rust
Entity(123)
```

没有意义。

---

# Component

分两类

---

## 类型1

业务Component

属于Core

例如：

```rust
Health
Mana
Position
Faction
Level
```

---

这些其实是：

```text
领域数据
```

---

即使换引擎：

```rust
Health
```

仍存在。

---

所以：

```rust
#[derive(Component)]
pub struct Health
```

允许在Core。

---

注意：

这里只依赖：

```toml
bevy_ecs
```

不是：

```toml
bevy_render
bevy_asset
bevy_audio
```

---

## 类型2

技术Component

属于Infrastructure

例如：

```rust
Sprite
AudioPlayer
UiNode
AnimationPlayer
```

---

这些显然是：

```text
技术实现
```

---

应该放：

```text
Infrastructure
```

---

这位提出的方案：

```text
业务Component → Core

技术Component → Infra
```

我认为是正确答案。

---
