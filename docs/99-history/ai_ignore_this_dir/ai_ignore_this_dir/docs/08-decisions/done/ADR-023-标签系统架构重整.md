# ADR-023: 标签系统架构重整

## 状态

Proposed

## 背景

标签系统（GameplayTag）是 SRPG 中跨所有业务领域的核心分类机制，被技能、Buff、装备、物品、角色、地图、修饰规则、AI 决策等多个子系统依赖。当前系统存在三重数据源重复、分类体系不完整、RON 覆盖缺失、国际化违规等架构问题，需要一次系统性的架构重整。

### 当前问题分析

**问题 1：三重数据源重复（P0）**

标签元数据分散在三个地方，任一不同步即导致不一致：

| 数据源 | 文件 | 维护方式 | 覆盖度 |
|--------|------|----------|--------|
| GameplayTag 常量 | `src/core/tag.rs` | 手工 | 30+ 标签 |
| register_defaults() | `src/core/tag_def.rs` | 手工（重复 RON） | 与 RON 不完全一致 |
| RON 定义 | `content/definitions/tags.ron` | 手工 | 仅 16 个标签 |

**问题 2：RON 覆盖不完整**

`content/definitions/tags.ron` 仅定义了 16 个标签，以下 15+ 标签仅存在于代码中但无 RON 元数据：
- Movement：FLYING, MOUNTED, SWIMMING
- ItemType：CONSUMABLE, AMMO, MATERIAL, CURRENCY, QUEST_ITEM
- ConsumableSub：HEALING, POTION, SCROLL, FOOD
- EquipmentAttribute：HEAVY_ARMOR, LIGHT_ARMOR, SHIELD, TWO_HANDED, MARTIAL, SIMPLE
- WeaponType：SWORD, AXE, BOW, STAFF

**问题 3：TagCategory 覆盖不完整**

`TagCategory` 枚举仅 6 个变体，缺少 Movement、ItemType、WeaponType、EquipmentAttribute 等实际已使用的分类维度。

**问题 4：label() 硬编码中文（P1）**

`GameplayTag::label()` 返回硬编码中文字符串，违反了 ADR-017（国际化架构）和 ADR-018（国际化迁移方案）的"Key 驱动内容"原则。`TagRegistry::display_name()` 虽然已提供从 RON 获取显示名的能力，但代码中仍有 30+ 处硬编码。

**问题 5：active_tags() 手工维护**

`GameplayTags::active_tags()` 维护一个包含所有标签的静态列表，新增标签时极易遗忘。

**问题 6：u64 位掩码耗尽风险**

当前 30/64 bits 已占用。若项目持续扩展（新元素类型、伤害类型、状态、物品分类等），位掩码空间将紧张。需要建立告警机制和扩展预案。

**问题 7：标签来源管理缺乏统一入口**

`PersistentTags` 存储 `from_traits` 和 `from_equipment` 两层来源，但最终将标签合并到 `GameplayTags` 的过程在 `spawn.rs` 中通过手动位运算完成，没有统一的 `rebuild_tags()` 函数（类比装备系统的 `rebuild_trait_effects()`）。

## 决策

### 1. RON 为元数据唯一事实源

`content/definitions/tags.ron` 是标签元数据（display_name、description、category）的唯一事实源。`tag_def.rs` 中的 `register_defaults()` **必须删除**。

`TagDefPlugin` 的行为改为：
- RON 文件存在 → 从 RON 加载，覆盖所有 `TagDefinition`
- RON 文件缺失或格式错误 → panic（启动失败，因为标签系统是核心基础设施）

迁移后，新增标签的元数据只需编辑一个文件。

### 2. TagCategory 扩展

`TagCategory` 从 6 个变体扩展为 10 个，覆盖所有现有标签：

```rust
pub enum TagCategory {
    Element,           // 元素：FIRE, ICE, POISON
    Status,            // 状态：STUN, BURN, REGEN
    Weapon,            // 攻击类型：MELEE, RANGED
    WeaponType,        // 武器类型：SWORD, AXE, BOW, STAFF
    Class,             // 职业：WARRIOR, ARCHER, MAGE
    Movement,          // 移动类型：FLYING, MOUNTED, SWIMMING
    SkillType,         // 技能类型：SKILL_ACTIVE, SKILL_PASSIVE
    BuffType,          // Buff 类型：BUFF, DEBUFF
    ItemType,          // 物品类型：CONSUMABLE, AMMO, MATERIAL, CURRENCY, QUEST_ITEM
    EquipmentAttribute,// 装备属性：HEAVY_ARMOR, LIGHT_ARMOR, SHIELD, TWO_HANDED, MARTIAL, SIMPLE
}
```

### 3. TagName 枚举 — 保留在代码中

`TagName` 枚举（`src/core/tag.rs`）**保留**在 Rust 代码中。理由：
- RON 配置文件的标签引用使用 `TagName` 枚举值（类型安全），避免运行时字符串解析
- Rust 枚举提供编译时穷举检查，防止拼写错误
- 枚举变体到 `GameplayTag` 位掩码的转换（`to_tag()`）是零开销

**新增标签仍需修改代码**（添加 TagName 枚举变体），这符合"新增范畴需要代码修改"的原则。但元数据（显示名、描述、分类）只改 RON。

### 4. 标签添加流程规范化

添加一个标签需要以下改动：

```
1. src/core/tag.rs
   - 添加 TagName 枚举变体
   - 添加 GameplayTag 常量（位掩码）
   - 添加 label() 和 to_tag() 分支
   - 添加 active_tags() 静态列表（当前方式）

2. content/definitions/tags.ron
   - 添加 (tag: ..., display_name: ..., description: ..., category: ...)

可选（按需）：
3. content/modifiers/*.ron
   - 添加 ModifierRule 引用新标签
```

### 5. 删除 register_defaults()

`TagRegistry::register_defaults()` 是 RON 内容在 Rust 代码中的重复，**必须删除**。

替代方案：`TagDefPlugin` 中，RON 加载失败时从 `TagName` 枚举自动生成默认元数据（显示名回退到 `GameplayTag::label()`，分类使用统一的默认值映射）。

```rust
// TagDefPlugin 加载逻辑（伪代码）
fn build(&self, app: &mut App) {
    let mut registry = match TagRegistry::load_from_file("content/definitions/tags.ron") {
        Ok(r) => r,
        Err(_) => {
            // RON 缺失时从 TagName 枚举生成默认元数据
            TagRegistry::from_enum_defaults()
        }
    };
    app.insert_resource(registry);
}
```

### 6. label() 废弃与 TagRegistry 统一

`GameplayTag::label()` 标记为 `#[deprecated]`，最终移除。替代方案：
- 所有 UI 显示优先使用 `TagRegistry::display_name(tag)`（从 RON 读取）
- 回退逻辑：RON 中未定义时使用 `tag.label()`（保持向后兼容）

### 7. TagCategory 全分类验证

`TagDefPlugin` 启动时增加校验：所有 `TagName` 枚举变体在 `TagRegistry` 中都有对应 `TagDefinition`。

```rust
pub fn validate_tag_coverage(registry: &TagRegistry) -> Vec<GameplayTag> {
    // 返回 RON 中缺失定义的 GameplayTag 列表
    TagName::ALL.iter()
        .filter(|name| !registry.definitions.contains_key(&name.to_tag()))
        .map(|name| name.to_tag())
        .collect()
}
```

### 8. u64 位掩码耗尽告警

`TagDefPlugin` 启动时增加位掩码使用率检查：

```rust
// 位掩码使用率 ≥ 80% 时输出 WARN 日志
if GameplayTag::used_bits() >= 51 {  // 64 * 0.8 = 51.2
    bevy::log::warn!(..., "GameplayTag 位掩码使用率已超过 80%，请准备迁移到分层标签系统");
}
```

扩展预案：
- 短期（当前到 51 bits）：维持 u64 位掩码
- 中长期（接近 51 bits）：迁移到分层字符串 ID，利用 `TypeIdMap` 或 `HashMap<String, u64>` 实现运行时标签注册
- 该方案在架构中预留扩展点，不作提前实现

### 9. 标签来源管理统一化

引入 `rebuild_tags()` 函数（类比装备系统的 `rebuild_trait_effects()`）：

```rust
/// 从 PersistentTags 重建单位的 GameplayTags
/// 在 Trait 或装备变化后调用
pub fn rebuild_tags(
    persistent: &PersistentTags,
    tags: &mut GameplayTags,
) {
    tags.0 = persistent.from_traits.0 | persistent.from_equipment.0;
}
```

调用时机：
- 单位生成后（现有 spawn.rs 逻辑）
- Trait 变化后
- 装备穿脱后（equip/unequip 流程）

### 10. TagCategory 扩展的 RON 字段兼容

当前 RON 格式不存储 `category` 之外的额外字段。扩展后保持向下兼容：新增标签只需在 RON 中增加条目，旧 RON 文件自动补全默认值（`#[serde(default)]`）。

## Module Design

### 目标模块结构（迁移后）

```
src/core/tag.rs
├── GameplayTag(u64)          ← 位掩码常量
├── GameplayTags              ← 实体组件
├── TagName                   ← 枚举（RON 序列化用）
├── PersistentTags            ← 标签来源管理
├── rebuild_tags()            ← 统一标签重建
└── TagCategory               ← 标签分类

src/core/tag_def.rs
├── TagDefinition             ← 标签元数据
├── TagRegistry               ← 标签注册表
└── TagDefPlugin              ← 标签插件（RON 加载）

content/definitions/tags.ron  ← 标签元数据唯一事实源
```

### 领域文件更新

`docs/02-domain/tag/tag-rules.md` — **新增**

整合当前分散在多个领域文件中的标签规则，作为标签系统的统一领域参考。

## Communication Design

### 标签创建流程

```
需求：新增 FROST 元素
→ 1. TagName::Frost 添加到 tag.rs 枚举
→ 2. GameplayTag::FROST 常量添加（位掩码分配）
→ 3. content/definitions/tags.ron 添加 Frost 条目
→ 4. content/modifiers/*.ron 按需添加修饰规则
```

### 标签加载流程

```
tags.ron → TagDefPlugin (build) → TagRegistry
                                    ↓
                              校验：RON 覆盖所有 TagName 变体？
                                    ↓
                              校验：位掩码使用率 < 80%？
                                    ↓
                              insert_resource(TagRegistry)
```

### 标签查询流程

```
UI 显示 → TagRegistry::display_name(tag) → RON 元数据
ModifierRule → source_tag 匹配 → 位掩码 O(1)
技能条件 → GameplayTags::has(tag) → 位掩码 O(1)
```

## 边界定义

### 允许
- `core::tag` 被所有业务模块依赖（标签是跨模块共享的分类机制）
- `core::tag_def` 依赖 `core::registry_loader`（加载标签定义）
- `core::tag` 定义 `GameplayTag`、`GameplayTags`、`TagName`、`TagCategory`、`PersistentTags`
- `core::tag_def` 定义 `TagRegistry`（标签注册表 Resource）

### 禁止
- 🟥 禁止在 Rust 代码中硬编码标签显示文本（label() 废弃）
- 🟥 禁止 register_defaults() 模式（RON 是元数据唯一来源）
- 🟥 禁止绕过 TagRegistry 直接访问标签元数据
- 🟥 禁止运行时使用字符串查询标签（必须使用 TagName → GameplayTag）
- 🟥 禁止新增标签时不同步更新 RON 文件
- 🟥 禁止在 RON 中使用位掩码数值（必须使用 TagName 枚举）

## Forbidden（禁止事项）

- 🟥 **禁止 register_defaults()** — 理由：元数据重复，违反 Rule/Content 分离原则
- 🟥 **禁止 label() 用于 UI 显示** — 理由：违反 ADR-017 国际化架构，应使用 TagRegistry
- 🟥 **禁止新增标签不添加 RON 定义** — 理由：导致元数据缺失，违反数据驱动原则
- 🟥 **禁止 TagCategory 不匹配实际标签分类** — 理由：导致分类查询失效
- 🟥 **禁止跳过 rebuild_tags() 手动位运算** — 理由：导致标签来源管理混乱
- 🟥 **禁止在 RON 中使用位掩码数值** — 理由：可读性差，应使用 TagName 枚举

## Definition / Instance Design

- **Definition（不可变配置）**：`TagName` 枚举（代码定义）、`GameplayTag(u64)` 位掩码常量、`TagDefinition`（RON 元数据）、`TagRegistry`（全局注册表）
- **Instance（运行时状态）**：`GameplayTags`（实体组件）、`PersistentTags`（标签来源管理，含 from_traits + from_equipment）

## 后果

### 正面
- 消除三重数据源重复，标签元数据唯一来源
- TagCategory 覆盖所有标签分类，查询能力完整
- label() 废弃后全量走 TagRegistry，国际化合规
- u64 耗尽有预警和扩展预案
- 统一 rebuild_tags() 简化标签重建
- 添加标签流程明确，新增元数据仅改 RON

### 负面
- TagName 枚举仍需手工维护（无法纯数据驱动，Rust 类型系统约束）
- 现有 30+ 处 label() 调用需要迁移到 TagRegistry
- register_defaults() 删除后需要确保 RON 文件永远存在

## 替代方案

| 方案 | 优点 | 缺点 | 为何放弃 |
|------|------|------|----------|
| build.rs 从 RON 生成代码 | 完全消除手工 TagName | 构建复杂、文件依赖环 | 过度设计，TagName 枚举变更频率极低 |
| 运行时字符串标签 | 无限扩展 | 类型不安全、性能 O(n) | 违反宪法性能要求 |
| 分层 hash 标签 | 层级语义丰富 | 运行时开销、Need 大重构 | 当前复杂度不需要，预留为中长期方案 |
| 保持现状 | 无迁移成本 | 三重重复继续恶化 | 架构债务持续积累，不可接受 |
