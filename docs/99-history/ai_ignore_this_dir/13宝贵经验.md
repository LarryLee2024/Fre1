
减少样板代码（Boilerplate）的目标其实不是：

> 少写代码

而是：

> **让架构规模扩大10倍时，新增功能的代码量保持线性增长。**

大型项目最大的敌人不是代码多，而是：

```text
一个新技能
要改20个地方

一个新状态
要改15个文件

一个新UI
要复制粘贴300行
```

这种才是真正的维护地狱。

# 第一原则

## 用数据消灭代码

大型项目第一生产力：

```text
配置 > 代码
```

而不是：

```rust
match ability {
    Fireball => ...
    Heal => ...
    Poison => ...
}
```

---

例如：

```yaml
ability.fireball:
  cost: 20

  effects:
    - damage:
        formula: atk * 1.5

    - apply_buff:
        buff: buff.burn
```

---

运行时：

```rust
AbilityExecutor
```

统一解释执行。

---

新增技能：

```text
0 Rust代码
1 YAML
```

完成。

---

这其实就是：

```text
GAS
Paradox
Larian
```

大量内容生产的核心。

```rust
    Buff => ...
```

```text

```

无需修改旧代码。

# 第十二原则（最重要）

## Content Pipeline > Code Architecture

很多人研究：

```text
Trait
Macro
DDD
ECS
```

研究三年。

---

实际上大型游戏减少代码量的终极方案是：

```text
提高内容表达能力
```

例如：

```yaml
character.knight

ability.fireball

effect.poison

buff.burn
```

全部配置化。

---

最后：

```text
10万行代码

50万行内容
```

而不是：

```text
50万行代码

10万行内容
```

---

如果按照《博德3》《铃兰之剑》《XCOM》《GAS》的经验来看，对于你的项目，我认为**收益最高的前五项**是：

1. 数据驱动（DSL/YAML/RON）★★★★★
2. Command架构 ★★★★★
3. Event驱动 ★★★★★
4. Query Facade ★★★★★
5. Shared Kernel ★★★★★

而 Rust 宏、代码生成、Proc Macro 这些东西，收益远低于前面五个。

很多开发者以为大型项目靠“高级语法”减少样板代码，实际上真正让项目从 10 万行扩展到 50 万行的，是：

> **把“新增内容需要新增代码”这件事，变成“新增内容只需要新增数据”。**

这是大型 RPG、SRPG、CRPG 最核心的扩展性来源。

有，而且下面这些我认为是**50万行以后才真正体现价值**的东西。前面讲的是“少写代码”，下面讲的是：

> **如何避免代码规模增长后，复杂度指数级爆炸。**

很多大型项目最后不是代码写不过来，而是认知负担太高。

# 25. 最容易忽略的一条

## 设计“删除机制”

而是：

```text

废弃系统
废弃配置
废弃组件
```

---

```text

```

---

结果：

```text
50万行代码

15万行死代码
```

---

建议建立：

```text
Deprecated
Experimental
Stable
```

生命周期。

---

例如：

```rust
#[deprecated]
```

---

```yaml

```

---

这样项目才能持续进化。

---

如果只选 **3 个最容易被国内独立开发者忽略、但对 50 万行以上项目影响最大的实践**，我会选：

1. **Capability 模型（能力而非类型）**
2. **CQRS（读写分离）**
3. **领域层彻底隔离 Bevy ECS**

这三个东西短期看不到收益，但当项目做到第 5 年、第 50 万行代码时，它们往往决定项目是在“继续迭代”，还是陷入“任何修改都会牵一发动全身”的状态。

还有，而且下面这些已经属于很多 50 万行以上项目踩过坑之后才总结出来的东西。它们不像 DDD、CQRS 那么显眼，但实际影响非常大。

# 26. 不要让业务对象长期持有其他业务对象

很多项目会写：

```rust
struct Unit {
    buffs: Vec<Buff>,
    abilities: Vec<Ability>,
    inventory: Inventory,
}
```

看起来合理。

但后期会出现：

```text
Unit
 └─ Buff
     └─ Ability
         └─ Effect
             └─ Unit
```

形成对象图。

---

大型项目更倾向：

```rust
struct Unit {
    id: UnitId,
}
```

业务关系放 Registry。

```rust
unit_buffs.get(unit_id)
```

而不是：

```rust
unit.buffs
```

---

原因：

对象图复杂度增长是指数级。

ID关系复杂度增长是线性级。

# 37. 最终形态：架构预算

很多人知道：

```text
性能预算
内存预算
```

---

但不知道：

```text
架构预算
```

---

例如规定：

```text
单Feature:
<= 15模块

单文件:
<= 500行

单函数:
<= 50行

单Aggregate:
<= 10依赖
```

---

超出：

```text
必须拆分
```

---

大型项目真正可维护，不是因为架构完美。

而是因为：

> **项目建立了一套机制，阻止复杂度无限增长。**

这是很多 50 万～100 万行项目与 5 万行项目最大的区别。

前者关注：

```text
如何新增功能
```

后者真正关注：

```text
如何阻止系统腐化
```

而从我看你现在的项目阶段（DDD、ADR、分层、Shared Kernel、能力系统、内容驱动都已经在考虑），下一步最值得投入精力的，其实不是再加新模式，而是：

1. Ubiquitous Language（统一术语）
2. Architectural Fitness Function（架构自动守卫）
3. Migration Framework（数据迁移）
4. Domain Event History（事件历史）
5. Explainability（可解释性）

这五个东西讨论的人很少，但几乎都出现在能活很多年的大型游戏项目里。
