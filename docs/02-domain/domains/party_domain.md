## 1. 统一术语

| 术语 | 定义 | 职责边界 |
|------|------|----------|
| PartyRoster | 队伍成员名册，管理所有可加入队伍的角色和当前活跃成员 | 负责：成员列表的管理（加入/移除/排序），Party 的 LocalizationKey（name_key/desc_key）；不负责：成员的个人属性/状态 |
| PartyBuff | 队伍全局增益效果，特定条件激活的全队加成 | 负责：队伍级 Buff 的激活条件与效果；不负责：Buff 的单个目标施加 |
| BondState | 羁绊状态，角色间的特殊关系及对应的战斗加成 | 负责：羁绊的激活条件与等级；不负责：羁绊加成的数值计算 |
| Formation | 编队配置，战斗中队伍的阵型/站位策略 | 负责：阵型的定义与站位规则；不负责：阵型的效果计算 |
| ReserveMember | 预备队员，在战斗中不在场上但可替换的成员 | 负责：预备队员的管理；不负责：换人逻辑 |

### 队伍结构

```
Party（队伍）
 ├── 当前战斗成员（Active Members）：战斗中在场上的单位，通常 4 人
 │    ├── Member 1
 │    ├── Member 2
 │    ├── Member 3
 │    └── Member 4
 │
 ├── 预备队员（Reserve Members）：不在战斗中的队伍成员，最多 8 人
 │    ├── Reserve 1
 │    ├── Reserve 2
 │    └── ...
 │
 └── 队伍全局数据（Party Data）
      ├── PartyBuffs：[ ]
      ├── BondStates：[ ]
      └── 队伍资源（共享金钱/共享物品等，可配置）
```

### 羁绊系统

```
Bond（羁绊）
 ├── 触发条件：特定角色组合同时在场（如同阵营/同种族/特定配对）
 ├── 羁绊等级：通过共同战斗/剧情选择提升
 │    ├── 1 级：基础加成
 │    ├── 2 级：中级加成（更多战斗）
 │    └── 3 级：高级加成（战斗+剧情事件）
 ├── 效果范围：当前战斗的所有活跃成员
 └── 示例：
      ├── "兄弟连心"：A + B 同时在队 → 全队 +1 AC
      └── "精灵之友"：两个精灵同时在队 → 全队 +2 敏捷豁免
```

### 已对齐项目术语

- **Combat**：Party 在战斗开始/结束时管理队伍成员的战斗状态
- **CampRest**：CampRest 是换人和调整队伍配置的场所（长休时可换人）
- **Progression**：Progression 升级时可能需要和 Party 联动（新成员加入事件）

## 2. 队伍状态机

### 队伍配置状态

```
Exploration（探索模式——队伍全成员可上场）
   │  [进入战斗]
   ▼
Combat Formation（战斗编队——选定４人上场）
   │  [战斗中换人]
   ├──→ 消耗行动力/资源 → 替换场上成员
   │
   │  [战斗结束]
   ▼
Exploration（回到探索模式——可重新编队）
   │  [长休/营地]
   ▼
Camp（营地模式——可自由调整队伍配置）
```

## 3. 不变量（Invariants）

### 3.1 战斗人数上限
- **条件**：任何战斗开始时
- **不变量**：战斗中活跃成员不得超过 4 人
- **违反后果类型**：🔴 规则失败
- **违反后果**：超过上限导致战场拥挤/性能问题

### 3.2 队伍总人数上限
- **条件**：新成员加入时
- **不变量**：队伍总人数（活跃+预备）不得超过上限（默认 12 人）
- **违反后果类型**：🔴 规则失败
- **违反后果**：无限扩大队伍导致管理复杂

### 3.3 战斗换人限制
- **条件**：战斗中换人
- **不变量**：战斗中换人消耗 1 行动力，每回合最多 1 次
- **违反后果类型**：🔴 规则失败
- **违反后果**：无限制换人破坏战斗策略性

### 3.4 羁绊条件实时评估
- **条件**：队伍成员变化时
- **不变量**：羁绊激活状态必须在队伍成员变化时实时重新评估
- **违反后果类型**：🔴 程序错误
- **违反后果**：成员替换后旧羁绊还在生效/新羁绊未激活

### 3.5 羁绊唯一性
- **条件**：羁绊定义时
- **不变量**：同一角色不能同时激活多个相同类型的羁绊
- **违反后果类型**：🔴 规则失败
- **违反后果**：角色"左脚踏右船"，羁绊重叠导致加成混乱

## 4. 禁止事项（Forbidden）

- 🟥 禁止：战斗中直接修改队伍总人数 — 理由：队伍总人数改变应在非战斗状态进行
- 🟥 禁止：同一角色同时在活跃队伍和预备队伍中 — 理由：一个角色只能在一个状态
- 🟥 禁止：羁绊效果影响非队伍成员 — 理由：羁绊效果限制在队伍范围内
- 🟥 禁止：战斗中换人不消耗任何资源 — 理由：换人消耗 1 行动力是策略资源管理的一部分
- 🟥 禁止：PartyDef 中直接存储用户可见文本的自然语言 — 理由：必须使用 name_key/desc_key: LocalizationKey 引用。违反宪法 §22 Localization First。

## 5. 流程定义

### 5.1 队伍成员管理

- **输入**：加入/移除请求、角色实体
- **处理**：
  1. 加入：检查队伍总人数是否已达上限（不变量 3.2）
  2. 加入：将角色添加到队伍（活跃或预备，取决于当前状态）
  3. 移除：从队伍中移除角色
  4. 如果移除的是活跃成员且有预备成员，自动从预备补充
  5. 重新评估羁绊状态（不变量 3.4）
  6. 发布 MemberJoined / MemberRemoved 事件
- **输出**：成员变更事件
- **失败处理**：人数已达上限时加入失败 → 这是**规则失败**（预期业务分支，队伍名额已满需先移除成员）

### 5.2 战斗换人

- **输入**：场上活跃成员、要换上来的预备成员
- **处理**：
  1. 检查当前回合是否已有换人操作（不变量 3.3）
  2. 检查是否有足够的行动力
  3. 活跃成员离场（移除 CombatParticipant，保留位置）
  4. 预备成员上场（添加 CombatParticipant，出现在离场成员位置或预设位置）
  5. 消耗行动力
  6. 重新评估羁绊状态
  7. 发布 MemberSwapped 事件
- **输出**：MemberSwapped 事件
- **失败处理**：行动力不足/本回合已换人时换人失败 → 这是**规则失败**（预期业务分支，换人需消耗行动力且受每回合次数限制）

### 5.3 羁绊激活

- **输入**：队伍成员变化/羁绊等级提升
- **处理**：
  1. 遍历所有已定义的羁绊条件
  2. 检查当前活跃成员是否满足羁绊条件
  3. 满足条件且尚未激活 → 激活羁绊 → 应用羁绊的 Modifier（走队伍级 Modifier）
  4. 不再满足条件 → 解除羁绊 → 移除 Modifier
  5. 发布 BondActivated / BondDeactivated 事件
- **输出**：羁绊状态变化事件
- **失败处理**：羁绊条件检查失败不下激活 → 这是**规则失败**（预期业务分支，条件不满足时不激活羁绊）

## 6. 领域事件

| 事件名 | 触发时机 | 携带数据 | 订阅者 |
|--------|----------|----------|--------|
| MemberJoined | 新成员加入队伍时 | party_id, entity_id, role（active/reserve） | UI（更新队伍界面）、Party（重新评估羁绊）、日志（LogCode: PRY001） |
| MemberRemoved | 成员离开队伍时 | party_id, entity_id, reason | Party（重新评估羁绊）、Quest（检查任务条件）、日志（LogCode: PRY002） |
| MemberSwapped | 战斗中换人时 | party_id, outgoing_id, incoming_id | Combat（更新参与者）、Party（重新评估羁绊）、UI（换人动画）、日志（LogCode: PRY003） |
| BondActivated | 羁绊激活时 | party_id, bond_id, members[ ], effect_description | Modifier（应用羁绊加成）、UI（显示羁绊激活通知）、日志（LogCode: PRY004） |
| BondDeactivated | 羁绊解除时 | party_id, bond_id, reason（成员变化/等级不足） | Modifier（移除羁绊加成）、UI（显示羁绊解除通知）、日志（LogCode: PRY005） |

### 事件订阅关系图

```
MemberJoined / MemberRemoved
    │
    ├──→ Party：重新评估所有羁绊条件
    ├──→ UI：更新队伍面板
    ├──→ Quest：检查"特定成员在队"的任务条件
    └──→ 日志：记录队伍变更

MemberSwapped
    │
    ├──→ Combat：更新 CombatParticipant 列表
    ├──→ Party：重新评估羁绊
    ├──→ Cue：换人特效
    └──→ UI：更新战斗界面单位

BondActivated
    │
    ├──→ Modifier：注册羁绊的全局 Modifier
    ├──→ Aggregator：触发属性重算
    ├──→ UI：显示羁绊激活通知/特效
    └──→ Cue：羁绊特效
```

## 7. 与已有架构的对齐校验

- ✅ 架构边界：Party 域位于 `core/domains/party/`，components.rs 定义 PartyRoster/PartyBuff/BondState，systems/ 实现成员管理/换人/羁绊/队伍 Buff 系统，rules/ 定义羁绊/换人/人数限制规则
- ✅ 羁绊通过 Modifier 管线：BondActivated 的加成效果通过 Modifier 注册到全队实体
- ✅ 轻量领域：Party 核心是"成员管理+羁绊"，依赖链短
- ✅ 战斗换人需消耗行动力，管理 Action Economy
- ✅ LocalizationKey：本领域涉及的用户可见文本使用 LocalizationKey 而非硬编码文本（宪法 §22）
