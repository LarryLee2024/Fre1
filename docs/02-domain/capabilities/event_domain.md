## 1. 统一术语

| 术语 | 定义 | 职责边界 |
|------|------|----------|
| GameplayEvent | 系统间传递结构化数据的统一事件载体，是 Domain 间通信的唯一通道 | 负责：携带事件标签、上下文和数值载荷；不负责：事件的触发条件评估 |
| EventTag | 事件类型的标识标签，用于事件路由和订阅匹配 | 负责：事件分类与路由寻址；不负责：事件载荷数据的结构 |
| EventBus | 全局事件路由基础设施，负责接收事件并分发给所有订阅者 | 负责：事件的接收、过滤、分发；不负责：事件的业务含义 |
| EventSubscription | 事件订阅关系，定义谁对哪些事件感兴趣以及如何处理 | 负责：订阅关系的注册/注销管理；不负责：订阅者的具体处理逻辑 |
| EventPayload | 事件的载荷数据，封装触发该事件时的上下文信息 | 负责：传递事件相关的结构化数据；不负责：载荷数据的业务语义解释 |
| EventPriority | 事件分发优先级，决定同一事件的多个订阅者的接收顺序 | 负责：订阅者处理顺序的控制；不负责：事件的投递可靠性 |
| EventObserver | 观察者模式的订阅者，被动接收事件后执行响应逻辑 | 负责：响应特定事件；不负责：事件的产生与路由 |

### Event 与 Trigger 的职责分离

```
Event 领域：
  ┌─ 职责：系统间的结构化数据传递
  ├─ 示例：DamageDealt → 连锁闪电触发，UnitDied → 经验发放
  ├─ 消费者：全系统（Domain 间、Capabilities 间、Capabilities→Infra）
  ├─ 数据结构：重量级（携带 GameplayContext 完整载荷）
  └─ 典型场景：A 系统通知 B 系统"某件事已发生"

Trigger 领域（独立领域，见 trigger_domain.md）：
  ┌─ 职责：技能激活的条件检测
  ├─ 示例：OnDamaged → 激活反击技能，OnTurnStart → 激活恢复技能
  ├─ 消费者：Ability 系统
  ├─ 数据结构：轻量（仅条件表达式）
  └─ 典型场景：条件满足 → 激活一个可执行的能力
```

### 已对齐项目术语

- **GameplayContext**：事件的载荷载体，通过 EventPayload 传递 GameplayContextData（定义在 GameplayContext 领域）
- **Trigger**：技能的"激活条件"系统，与 Event 不同但互补。Event 是"已发生"的通知，Trigger 是"应发生"的条件
- **所有 Domain**：通过 EventBus 订阅/发布事件实现域间通信，禁止直接引用

## 2. 事件状态机

### 事件生命周期

```
Created（事件创建）
   │  [发布到 EventBus]
   ▼
Queued（已入队等待分发）
   │  [按优先级排序]
   ▼
Dispatching（分发中）
   │  [逐个投递到订阅者]
   ├──→ [订阅者处理成功]
   │        │
   │        ▼
   │    Delivered（已投递）
   │        │
   └──→ [订阅者处理异常]
            │
            ▼
        Failed（处理失败）
   │  [所有订阅者处理完毕]
   ▼
Consumed（事件消费完毕）
```

### 状态转换规则

| 转换 | 触发条件 | 动作 |
|------|---------|------|
| Created → Queued | 事件发布到 EventBus | 加入分发队列，按 EventPriority 排序 |
| Queued → Dispatching | 分发时机到达（帧同步点或立即） | 开始逐个投递 |
| Dispatching → Delivered | 单个订阅者处理成功 | 记录成功，继续投递下一订阅者 |
| Dispatching → Failed | 单个订阅者处理抛出异常 | 记录失败，继续投递（不影响其他订阅者） |
| 所有订阅者完成 → Consumed | 所有订阅者已处理完毕 | 从队列移除，事件生命周期结束 |
| 禁止 | Dispatching 状态下取消已投递的事件 | 事件分发是"发射后不管"，不支持撤回 |

## 3. 不变量（Invariants）

### 3.1 事件来源明确
- **条件**：任何 GameplayEvent 被创建时
- **不变量**：每个事件必须有明确的来源（触发该事件的 Domain/System 标识）
- **违反后果**：来源不明的事件被 EventBus 拒绝

### 3.2 事件分发不因单个订阅者失败而中断
- **条件**：事件分发过程中某个订阅者处理失败时
- **不变量**：单个订阅者的失败不影响事件继续分发给其他订阅者
- **违反后果**：一个订阅者的错误导致其他订阅者错过事件，产生"静默丢失"

### 3.3 事件顺序可预测
- **条件**：同一帧/同一批次的多个事件到达时
- **不变量**：
  - 同一 EventPriority 级别的事件按 FIFO（先入先出）顺序分发
  - 高优先级事件先于低优先级事件分发
- **违反后果**：事件分发顺序不可预测导致不确定的行为

### 3.4 事件不会导致循环触发
- **条件**：事件处理过程中产生新事件时
- **不变量**：同一事件类型在单次处理链中不得重复触发超过 N 次（防止无限循环）
- **违反后果**：事件循环风暴，系统挂起或栈溢出

### 3.5 订阅者可在运行时安全注册/注销
- **条件**：事件分发过程中订阅者注册或注销时
- **不变量**：当前正在分发的事件的订阅者集合不受注册/注销操作影响（操作延迟到下一事件生效）
- **违反后果**：并发修改订阅者集合导致分发遗漏或重复投递

## 4. 禁止事项（Forbidden）

- 🟥 禁止：Domain 间直接引用（`use`）代替 Event 通信 — 理由：架构铁律，Domain 间唯一通信通道是 Event
- 🟥 禁止：Event 携带可修改的可变引用 — 理由：事件数据应为不可变快照，接收方不应修改事件内容
- 🟥 禁止：将 Event 作为"返回值"使用（发出事件后等待另一系统的同步结果） — 理由：事件是单向通知，不是 RPC；需要结果反馈应使用 Command 模式
- 🟥 禁止：在单个事件分发周期内同步等待 — 理由：事件是异步机制，同步等待会导致死锁
- 🟥 禁止：用 Event 替代 Trigger 做技能激活条件判断 — 理由：职责分离——Event 做通知，Trigger 做条件检测

## 5. 流程定义

### 5.1 事件发布

- **输入**：EventTag、EventPayload（含 GameplayContext）、EventPriority
- **处理**：
  1. 校验 EventTag 是否已定义（事件类型白名单检查）
  2. 校验来源标识是否明确（不变量 3.1）
  3. 创建 GameplayEvent 实例
  4. 发布到 EventBus（进入分发队列）
- **输出**：GameplayEvent 实例
- **失败处理**：来源不明或 EventTag 未注册时发布失败

### 5.2 事件分发

- **输入**：EventBus 队列中的 GameplayEvent
- **处理**：
  1. 按 EventPriority 排序队列中的事件（高优先级先分发）
  2. 查找 EventTag 匹配的所有订阅者
  3. 逐个投递：
     a. 调用订阅者的处理逻辑
     b. 成功 → 记录 Delivered，继续下一个
     c. 失败 → 记录 Failed，继续下一个（不变量 3.2）
  4. 所有订阅者处理完毕后，标记事件为 Consumed
- **输出**：逐个投递结果
- **失败处理**：单个订阅者处理失败不影响其他订阅者；记录日志用于排查

### 5.3 订阅注册

- **输入**：订阅者标识、感兴趣的 EventTag 列表、处理优先级、处理回调
- **处理**：
  1. 检查订阅者标识是否唯一（避免重复注册）
  2. 注册订阅关系到 EventBus
  3. 如果当前有事件正在分发，注册延迟到当前事件分发完成后生效（不变量 3.5）
- **输出**：订阅确认
- **失败处理**：重复注册时覆盖旧注册并记录警告

### 5.4 订阅注销

- **输入**：订阅者标识、要取消订阅的 EventTag 列表
- **处理**：
  1. 从 EventBus 移除匹配的订阅关系
  2. 如果当前有事件正在分发，移除延迟到当前事件分发完成后生效（不变量 3.5）
- **输出**：注销确认
- **失败处理**：要注销的订阅不存在时忽略（幂等）

### 5.5 循环检测

- **输入**：当前正在处理的事件类型、事件链深度计数器
- **处理**：
  1. 每次发布事件时记录事件链深度
  2. 检测同一 EventTag 在链中的出现次数
  3. 如果同一事件类型在单次链中触发了 N 次（默认 N=5），触发循环保护
  4. 中断该链的事件分发，发布 CycleDetected 警告事件
- **输出**：CycleDetected 事件（仅 dev-tools 模式）
- **失败处理**：循环保护触发后，被中断的事件视为丢失，需通过其他方式恢复

## 6. 领域事件

> 注意：Event 领域自身不发布常规业务事件（避免循环），以下事件仅用于调试和监控。

| 事件名 | 触发时机 | 携带数据 | 订阅者 |
|--------|----------|----------|--------|
| EventPublished | 事件被发布到 EventBus 时（调试用，仅 dev-tools） | event_tag, source, priority, timestamp | 日志、调试工具 |
| EventDelivered | 事件成功投递到订阅者时（调试用，仅 dev-tools） | event_tag, subscriber_id, latency | 日志分析器、性能监控 |
| EventDeliveryFailed | 事件投递到订阅者失败时（用于监控告警） | event_tag, subscriber_id, error_message | 日志、告警系统 |
| EventCycleDetected | 检测到事件循环触发时（严重告警） | event_tag, cycle_depth, chain_trace | 架构检查工具、开发告警 |

### 事件分发架构图

```
[发布者 Domain]                  [EventBus]                     [订阅者 Domain]
      │                              │                              │
      │── GameplayEvent ──────────→  │                              │
      │                              │── (按 EventTag 匹配订阅者) → │
      │                              │                              │── 处理事件
      │                              │←── 处理结果 (成功/失败) ──── │
      │                              │── (继续下一订阅者) ────────→ │
      │                              │                              │
      │                              │── 全部处理完毕 ──→ 标记 Consumed
```

## 7. 与已有架构的对齐校验

- ✅ 架构边界：Event 能力领域位于 `core/capabilities/event/`，foundation/ 定义 gameplay_event.rs 和 event_type.rs，mechanism/ 定义 event_bus.rs、event_subscription.rs，符合 C1→C2 分层
- ✅ 术语一致：GameplayEvent、EventTag、EventBus、EventSubscription 与架构文档第六节完全一致
- ✅ 职责明确：Event 只做"通知与分发"，不做"条件检测"（Trigger 的职责）、不做"表现触发"（Cue 的职责）
- ✅ 与 Trigger 分离：Event 解决"系统间通信"，Trigger 解决"技能激活条件"，两者在架构文档中的区分得到严格遵守
- ✅ Domain 间通信：Event 是 Domain 间通信的唯一通道，符合架构文档第七节"Domain 间禁止直接引用"的铁律
