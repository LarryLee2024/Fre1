## 1. 统一术语

| 术语 | 定义 | 职责边界 |
|------|------|----------|
| SummonBond | 召唤者与召唤物之间的绑定关系，定义召唤物的归属和行为约束 | 负责：召唤者→召唤物的绑定管理；不负责：召唤物的个体属性 |
| SummonDuration | 召唤持续时间，定义召唤物在战场上的存在时长 | 负责：持续时间的类型（专注维持/固定回合/永久）与管理；不负责：到期后的自动消失处理 |
| SummonAI | 召唤物 AI 模式，定义召唤物在没有玩家指令时的自主行为 | 负责：AI 模式的分类与行为规则；不负责：AI 的具体决策逻辑 |
| SummonSlot | 召唤物占用格子数量，大型召唤物可能占用多个网格 | 负责：召唤物的战场占位规则；不负责：占位冲突的解决 |

### 召唤物分类

```
召唤物（按来源分类）
 ├── 法术召唤（Spell Summon）：通过法术召唤的生物
 │    示例：召唤动物/亡灵/元素生物/魔鬼
 │    特征：消耗法术位+专注，通常为固定持续时间
 │
 ├── 能力召唤（Ability Summon）：通过种族/职业能力召唤
 │    示例：德鲁伊的"自然伙伴"、游侠的"动物伙伴"
 │    特征：不消耗法术位，长休恢复
 │
 ├── 临时幻象（Illusion Summon）：非实体的幻象/投影
 │    示例：幻术系的幻象、假人
 │    特征：不可攻击，用于分散注意/侦察
 │
 └── 永久召唤（Permanent Summon）：永久的随从/宠物
       示例：魔宠、钢铁卫士
       特征：永久存在，死亡后可重新召唤
```

### 召唤物生命周期

```
Summoned（被召唤——创建 Entity，绑定召唤者）
   │  [初始化模板属性]
   ▼
Active（活跃——在战场上自主行动或听令）
   │  [每回合跟随召唤者行动顺序]
   │  [可接受召唤者指令/自动 AI]
   │
   ├──→ [持续时间到期] → 触发消失
   │
   ├──→ [召唤者死亡] → 级联消失
   │
   ├──→ [召唤者专注打断] → 专注类召唤自动消失
   │
   ├──→ [被击杀] → 死亡消失
   │
   └──→ [主动解除] → 驱散/解散
           │
           ▼
Expired（消失——Entity 被销毁/隐藏）
```

### 已对齐项目术语

- **Effect**：召唤持续时间通过 Effect(Duration) 管理；专注由 Spell 领域的 Concentration 管理
- **Combat**：召唤物是 CombatParticipant，参与先攻排序
- **Tactical**：召唤物占用网格位置
- **Ability**：召唤是一个特殊 Ability，激活时创建召唤物 Entity
- **Condition**：召唤条件检查（如"召唤者同时只能维持一个专注召唤"）

## 2. 召唤状态机

```
Pending（召唤中——施法/准备中）
   │  [施法完成/能力激活]
   ▼
Active（活跃中）
   │  [每回合：跟随召唤者行动顺序行动]
   │  [持续效果：Effect(Duration) 倒计时中]
   │
   ├──→ [Effect 到期] → 自动消失
   ├──→ [召唤者专注打断] → 专注类召唤消失
   ├──→ [召唤者死亡] → 级联消失
   ├──→ [被击杀] → 死亡
   └──→ [主动解散] → 驱散
           │
           ▼
Expired（已消失——Entity 清理）
```

## 3. 不变量（Invariants）

### 3.1 召唤者生死约束
- **条件**：召唤物活跃时
- **不变量**：召唤者死亡时，所有由该召唤者创建的召唤物必须立即消失
- **违反后果**：召唤者在召唤物之前死亡后，召唤物仍存在于战场

### 3.2 专注召唤唯一性
- **条件**：施法者维持专注召唤时
- **不变量**：一个施法者同时只能维持一个需要专注的召唤物（Spell 领域的专注规则）
- **违反后果**：多个专注召唤同时存在，违反专注唯一性

### 3.3 召唤物行动权
- **条件**：战斗中召唤物行动时
- **不变量**：召唤物拥有独立的行动回合（在召唤者的先攻位置附近行动），不由召唤者代行
- **违反后果**：召唤物不消耗额外行动资源但有独立行动权——这是召唤的价值

### 3.4 召唤物模板一致性
- **条件**：召唤物被创建时
- **不变量**：召唤物的属性/能力必须基于一个已注册的模板（召唤物 Def）
- **违反后果**：无模板的召唤物属性异常

### 3.5 占位不冲突
- **条件**：召唤物被召唤到战场时
- **不变量**：召唤物出生位置/占用的格子必须是可通行的且没有被其他单位占用
- **违反后果**：召唤物与其他单位重叠

---

## 4. 禁止事项（Forbidden）

- 🟥 禁止：召唤者死亡后召唤物仍在场上行动 — 理由：召唤者与召唤物是绑定关系，召唤者死亡→召唤物消失
- 🟥 禁止：一个施法者同时维持多个专注召唤 — 理由：专注唯一性
- 🟥 禁止：召唤物产生自己的召唤物（嵌套召唤） — 理由：嵌套召唤导致召唤链失去控制
- 🟥 禁止：召唤物使用需要消耗资源的能力（如法术位） — 理由：召唤物不是独立施法者，不拥有召唤者的资源

## 5. 流程定义

### 5.1 召唤物创建

- **输入**：召唤者、召唤物模板、召唤位置
- **处理**：
  1. 检查召唤位置是否合法（可通行且无单位——不变量 3.5）
  2. 检查召唤者是否已有活跃召唤物（检查专注/数量限制）
  3. 基于召唤物模板创建 Entity：
     a. 复制模板属性（Attribute、Tag、Ability）
     b. 设置 SummonBond（绑定召唤者）
     c. 设置 SummonDuration（通过 Effect(Duration) 管理）
     d. 设置 SummonAI（AI 模式）
  4. 如果为专注召唤，建立召唤者的专注连接
  5. 注册为 CombatParticipant（如在战斗中）
  6. 发布 SummonCreated 事件
- **输出**：SummonCreated 事件（召唤者、召唤物 Entity、模板、位置）
- **失败处理**：位置不合法/专注冲突/数量已达上限时创建失败

### 5.2 召唤物消失

- **输入**：消失原因（时间到期/专注打断/召唤者死亡/被击杀/主动解散）
- **处理**：
  1. 根据原因判定消失行为：
     a. 时间到期 → Effect(Duration) 到期触发
     b. 专注打断 → Spell 领域专注打断通知
     c. 召唤者死亡 → 监听 UnitDied 事件（召唤者）
     d. 被击杀 → 正常死亡流程
     e. 主动解散 → 召唤者操作
  2. 移除召唤物的战斗参与者状态
  3. 清理占用格子
  4. 触发死亡动画/消失特效
  5. 销毁召唤物 Entity（或隐藏，取决于表现需求）
  6. 释放召唤者的召唤槽位
  7. 发布 SummonExpired 事件
- **输出**：SummonExpired 事件（召唤者、召唤物、消失原因）
- **失败处理**：召唤物已消失时忽略（幂等）

### 5.3 召唤物控制

- **输入**：指令（攻击目标/移动到位置/使用能力/跟随模式/自主模式）
- **处理**：
  1. 检查指令是否在当前 AI 模式下允许
  2. 如果是玩家控制，检查召唤者当前是否有权下达指令（通常消耗附赠动作）
  3. 召唤物执行指令
  4. 如果无指令且 AI 模式为自主，SummonAI 自动决策
- **输出**：指令执行结果
- **失败处理**：指令不被 AI 模式允许/召唤者无控制权时忽略

## 6. 领域事件

| 事件名 | 触发时机 | 携带数据 | 订阅者 |
|--------|----------|----------|--------|
| SummonCreated | 召唤物被创建时 | caster_id, summon_entity_id, template_id, position, duration_type | Combat（注册参与者）、Tactical（更新占位）、UI（显示召唤物） |
| SummonExpired | 召唤物消失时 | caster_id, summon_entity_id, reason（expired/concentration_broken/caster_died/killed/dismissed） | Combat（移除参与者）、Tactical（释放格子）、UI（移除召唤物界面） |
| SummonCommand | 召唤物接受指令时 | caster_id, summon_entity_id, command_type（attack/move/ability/follow/auto） | UI（更新召唤物行为指示） |
| SummonSlotChanged | 召唤者的召唤占用变化时 | caster_id, summon_slots_used, summon_slots_max | Condition（检查召唤数量限制） |

### 事件订阅关系图

```
SummonCreated
    │
    ├──→ Combat：注册召唤物为战斗参与者（本轮最后行动）
    ├──→ Tactical：标记召唤物占用的格子
    ├──→ Effect：启动 Effect(Duration) 管理召唤持续时间
    ├──→ UI：显示召唤物图标/状态
    └──→ Cue：召唤特效

SummonExpired
    │
    ├──→ Combat：移除战斗参与者
    ├──→ Tactical：释放占用的格子
    ├──→ Effect：终止关联的 Effect(Duration)
    ├──→ UI：移除召唤物图标
    ├──→ Cue：消失特效
    └──→ 日志：记录召唤消失原因
```

## 7. 与已有架构的对齐校验

- ✅ 架构边界：Summon 域位于 `core/domains/summon/`，components.rs 定义 SummonBond/SummonDuration/SummonAI，systems/ 实现召唤创建/到期/控制/死亡联动系统，rules/ 定义专注/持续时间/占位规则
- ✅ 持续时间通过 Effect 管理：复用 Effect(Duration) 管理召唤物的存续时间，不走独立计时器
- ✅ 专注管理通过 Spell 领域：专注类召唤的维持依赖 Spell 领域的 Concentration 系统
- ✅ 召唤者死亡联动：监听 UnitDied 事件，级联移除所有相关召唤物
- ✅ 召唤物是独立 CombatParticipant：拥有独立先攻位置和行动权
