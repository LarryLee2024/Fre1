# 很多人误解DDD

很多人看到DDD就想到：

```text
Aggregate
Repository
Factory
Domain Service
Value Object
```

然后把游戏项目搞得像银行系统。

这对Bevy是灾难。

---

你真正应该学的是：

# DDD最核心的东西

只有一句话：

> **软件结构应该围绕业务领域组织，而不是围绕技术组织。**

这和你的 Feature First 几乎完全一致。

---

例如：

坏：

```text
components/
systems/
events/
resources/
```

这是技术视角。

---

好：

```text
battle/
character/
skill/
inventory/
equipment/
quest/
```

这是领域视角。

这其实已经是DDD思想。

---

# 你的项目最值得借鉴的是 Strategic DDD

不是 Tactical DDD。

---

# 一、Bounded Context（限界上下文）

这是DDD最有价值的东西。

---

你的项目天然存在：

```text
Character
Battle
Inventory
Quest
Map
Dialogue
```

几个大领域。

---

例如：

## Character Context

负责：

```text
职业
等级
属性
成长
```

---

## Battle Context

负责：

```text
伤害
行动顺序
技能释放
死亡
```

---

## Inventory Context

负责：

```text
背包
堆叠
重量
装备
```

---

## Quest Context

负责：

```text
任务
进度
奖励
```

---

每个Context内部可以自由演化。

不要互相穿透。

---

例如：

错误：

```rust
battle
    -> inventory.gold -= 100
```

---

正确：

```rust
BattleRewardGranted
```

```rust
Inventory
```

收到消息。

自己处理。

---

这其实就是：

```text
Bounded Context
```

---

# 四、领域规则集中

DDD有个原则：

> 规则属于领域

不要散落。

---

例如：

错误：

```rust
ui
battle
inventory
quest
```

都计算攻击力。

---

正确：

```text
character/stats/
```

统一计算。

---

这就是：

```text
Single Source of Truth
```

---

# 五、领域对象 vs ECS

这里要特别注意。

---

企业DDD：

```java
Character.attack(enemy)
```

---

Bevy：

```rust
Entity
Component
System
```

---

不要把DDD对象搬进ECS。

---

这是错误路线：

```rust
struct Character {
    fn attack(...)
}
```

---

应该：

```rust
AttackIntent
```

↓

```rust
DamageCalculation
```

↓

```rust
DamageApplied
```

↓

```rust
DeathDetected
```

---

DDD负责：

```text
领域边界
业务语言
业务规则
```

---

ECS负责：

```text
数据组织
执行模型
```

---

不要混。

---

# 最适合你项目的版本

如果专门针对：

* Bevy 0.18+
* DND风格SRPG
* 长期连载
* 大量内容更新

我会在你的铁律最前面增加一个总原则：

| 原则                 | 含义                                       |
| ------------------ | ---------------------------------------- |
| **领域优先于技术**        | 模块围绕 Battle、Character、Inventory、Quest 组织 |
| **统一领域语言**         | 一个概念只有一个名字                               |
| **限界上下文隔离**        | Battle 不直接操作 Inventory                   |
| **领域事件驱动协作**       | Context 间通过 Message/Observer 协作          |
| **规则集中管理**         | 一个规则只有一个真相来源                             |
| **ECS负责执行，领域负责建模** | 不把企业OOP搬进Bevy                            |

如果让我评价，你现在的整体方向其实已经不是普通的 Bevy 项目思维了，而是在向：

> **DDD（领域驱动） + ECS（数据驱动） + Feature First（业务驱动）**

三者融合的架构靠近。

对于大型 SRPG，这比单纯研究 Bevy API 本身重要得多，因为后期真正难维护的从来不是 ECS，而是越来越复杂的游戏规则。
